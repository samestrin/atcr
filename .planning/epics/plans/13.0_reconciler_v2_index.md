# Epic Index 13.0: Reconciler v2 Semantic Algorithms

- **Estimated time**: 4 weeks
- **Tasks**: 12
- **Components**: 4

## Context

Epic 8.0 extracts the existing v1 reconciler into a standalone module. While the v1 reconciler is robust (using basic text similarity and deduplication), it fails in edge cases where models use vastly different phrasing for the exact same underlying structural issue.

## Problem Statement

To maintain our architectural moat, the next iteration of the reconciler must "understand" code semantics deterministically.
1. Lexical limitations cause duplicate findings.
2. Line number drift causes fragmented findings.
3. Sub-optimal grouping leaves stragglers.
4. Flat vote counting fails to identify authoritative signals.

## Proposed Solution

Coordinate the rollout of the "v2" reconciler pipeline across 4 sprints:
- **[Sprint 13.0: Semantic Deduplication (NCD)](../active/13.0_semantic_ncd_deduplication.md)**
- **[Sprint 13.1: AST Plugin Architecture](../active/13.1_ast_plugin_architecture.md)**
- **[Sprint 13.2: Bipartite Matching & DBSCAN](../active/13.2_resolution_bipartite_dbscan.md)**
- **[Sprint 13.3: Deterministic PageRank](../active/13.3_signal_pagerank.md)**

## Acceptance Criteria

- [ ] AC1: All 4 sprints completed and integrated into `github.com/atcr/reconcile`.
- [ ] AC2: Benchmark suite proves 20% reduction in gray zone duplicates without increasing false merges.

## Success Criteria

### Business

## Technical Constraints

- Must be pure Go or dynamically loaded Wasm (no CGO dependencies).
- Must remain 100% deterministic and offline.
