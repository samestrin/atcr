# Tech Debt: Per-Call Client Telemetry (accurate API-call counting + latency histogram)

- **Estimated time**: 3–5 days

## Current State

`fanout.Result` carries `Turns` (tool-use round count) but no per-call attempt or timing data.
Two metrics gaps share this root cause:

1. **Undercounting `atcr_api_calls_total`** (`internal/fanout/metrics.go:37`): `recordAgentOutcome` assumes `Turns==0` means no HTTP request was made, so it zeroes `apiCalls` for any `context.DeadlineExceeded` or `context.Canceled` result. A single-shot agent whose context expires mid-flight (after one real HTTP round-trip) has `Turns==0` and counts 0 — undercounting by 1 exactly when a provider is degraded and the measurement matters most.

2. **Deferred latency histogram** (`internal/fanout/metrics.go:22–24`): `atcr_api_call_duration_seconds` was explicitly deferred because per-call start time lives inside `internal/llmclient` and is not surfaced on `Result`.

A `max(1, r.Turns)` patch for issue (1) was rejected: when `Turns==0` it cannot distinguish the single-shot-timed-out case (1 real call) from circuit-open fail-fast (0 calls, Epic 4.5 AC2) and cancel-before-send (0 calls), so it would overcount the AC2 path and phantom-inflate the metric.

## Pain Points

- `atcr_api_calls_total` undercounts by ≤1 per single-shot agent that times out mid-flight — bounded but misleading exactly during provider outages.
- No per-provider latency data; the histogram stub exists in `metrics.go` but is never populated.
- Both gaps block meaningful provider SLA and capacity dashboards.

## Proposed Solution

Surface a `CallsAttempted int` and `CallDurations []time.Duration` (or a compact `[]CallRecord`) from `internal/llmclient` onto `fanout.Result`, then wire them into `recordAgentOutcome`.

### Why together

Both gaps require the same structural change: the client must emit per-call data upward. Fixing them separately means two rounds of the same plumbing change.

### Migration Strategy

1. **Add `CallRecord` to `llmclient`** — `attempt()` stamps start/end time and whether the call reached the wire; `dispatch()` accumulates a `[]CallRecord` and returns it alongside `([]byte, error)`.
2. **Thread `CallRecord` onto `Result`** — `send()` / `Complete()` / `CompleteWithUsage()` return the slice; the fan-out engine attaches it to `Result`.
3. **Update `recordAgentOutcome`** — replace the `Turns`-based heuristic with `len(r.CallRecords)` for counting; feed durations into the histogram.
4. **Update tests** — `TestRecordAgentOutcomeContextCancelledBeforeRequest` becomes a per-record test (circuit-open and cancel-before-send have 0 records; single-shot timeout has 1 record with `ReachedWire=true`).

## Scope

### In Scope
- Add `CallRecord{ReachedWire bool, Duration time.Duration}` to `llmclient`
- Thread `[]CallRecord` from `dispatch` → `Complete`/`CompleteWithUsage` → `fanout.Result`
- Replace the `Turns`-based `apiCalls` heuristic in `recordAgentOutcome`
- Populate `atcr_api_call_duration_seconds` histogram
- Update affected tests

### Out of Scope
- Per-provider retry breakdown (separate observability epic)
- Surfacing `CallRecord` to callers outside the fan-out engine

## Clarifications

### Recorded 2026-06-20 (rubber duck — /execute-epic)

**Key Decisions:**
- `atcr_api_calls_total` adopts PER-ATTEMPT counting (Option A): one `CallRecord` per `c.attempt()` invocation, so internal HTTP retries each count. The counter total and the `atcr_api_call_duration_seconds` observation count are always equal, making them mutually validating.
- This is a deliberate semantic change from the prior `Turns`-based one-per-round-trip behavior (`defaultMaxRetries = 2` at `internal/llmclient/client.go:28` means a 3-attempt sequence previously counted 1; it now counts up to 3). A one-sentence comment in `internal/fanout/metrics.go` documents the change for dashboard owners.

**Scope Boundaries:**
- IN scope: `CallRecord{ReachedWire bool, Duration time.Duration}`; threading `[]CallRecord` through `dispatch` → `CompleteWithUsage` (4-value) and `Chat` (via `ChatResponse.CallRecords`) → `fanout.Result.CallRecords`; per-attempt counter fix; histogram population; affected test updates.
- OUT of scope: per-provider retry breakdown; surfacing `CallRecord` to callers outside the fan-out engine.

**Technical Approach:**
- `ReachedWire` is detected precisely via `net/http/httptrace` (`WroteRequest` hook) inside `attempt()` — the only reliable way to distinguish cancel-before-send (request never written → `ReachedWire=false`, counts 0, AC2) from a mid-flight timeout after the request was written (`ReachedWire=true`, counts 1, AC1).
- `Complete` (2-value) stays unchanged so `internal/doctor` is unaffected; only `CompleteWithUsage` and `Chat` surface records.
- `recordAgentOutcome`: when `Result.CallRecords != nil`, `apiCalls` = count of records with `ReachedWire==true` and each such record's `Duration` is observed into the histogram; when nil (test doubles / `Complete`-only path), it falls back to the existing `Turns`-based heuristic so all current tests stay green.

**Original Questions and Answers:**

   A: Option (A) per-attempt — matches the epic's "attempt" wording (migration step 3, AC3), is the cleaner design (counter and histogram are mutually validating), and avoids a separate "attempted-any" boolean. Add a one-sentence comment in `metrics.go` noting the semantic change from the prior `Turns`-based behavior so dashboard owners expect the inflation step-change.
