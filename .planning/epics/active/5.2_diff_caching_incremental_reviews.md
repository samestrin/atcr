# Epic Plan 5.2: Diff Caching & Incremental Reviews

**Estimated Durations**: 2 weeks  
**Status:** Draft (queued; depends on Epic 2.0 — complete)

## Objective

Implement diff caching to prevent re-reviewing identical file chunks across multiple test runs or minor commits, significantly reducing API costs and latency.

## Context

Users often run `atcr review` multiple times iteratively. Currently, a 1-line change triggers a full re-review of the entire diff, wasting tokens and time on unchanged files.

## Problem Statement

1. Running a review on a branch with 50 changed files costs the same whether 1 file or 50 files changed since the last run.
2. Fan-out review costs multiply this waste by the number of reviewers.

## Proposed Solution

- Hash individual file diffs or chunks.
- Store the findings for each hash in a local `.atcr/cache/` directory.
- On subsequent runs, skip API calls for chunks whose hash matches a cached entry, and simply inject the cached findings into the pool.
- Provide a `--no-cache` flag to bypass the cache.

## Acceptance Criteria

- [ ] Unchanged file chunks are served from local cache, avoiding LLM API calls.
- [ ] Cache keys incorporate the chunk diff hash and the reviewer model/persona.
- [ ] `--no-cache` flag forces a fresh review.
- [ ] Cache eviction or max-size policy implemented to prevent unbounded growth.

## Dependencies

- Epic 2.0 (Tool Loop).
