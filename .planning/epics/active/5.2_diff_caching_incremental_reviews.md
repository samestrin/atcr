# Epic Plan 5.2: Diff Caching & Incremental Reviews

**Estimated Durations**: 2 weeks  
**Status:** Draft (queued; depends on Epic 2.0 — complete)

## Objective

Implement diff caching to prevent re-reviewing identical file chunks across multiple test runs or minor commits, significantly reducing API costs and latency.

## Context

Users often run `atcr review` multiple times iteratively. Currently, a 1-line change triggers a full re-review of the entire diff, wasting tokens and time on unchanged files.

## Problem Statement

1. Running a review on a branch with 50 changed files costs the same whether 1 file or 50 files changed since the last run.
2. Fan-out review costs multiply this waste by the number of reviewers.

## Proposed Solution

- Hash individual file diffs or chunks.
- Store the findings for each hash in a local `.atcr/cache/` directory.
- On subsequent runs, skip API calls for chunks whose hash matches a cached entry, and simply inject the cached findings into the pool.
- Provide a `--no-cache` flag to bypass the cache.

## Acceptance Criteria

- [ ] Unchanged file chunks are served from local cache, avoiding LLM API calls.
- [ ] Cache keys incorporate the chunk diff hash and the reviewer model/persona.
- [ ] `--no-cache` flag forces a fresh review.
- [ ] Cache eviction or max-size policy implemented to prevent unbounded growth.

## Dependencies

- Epic 2.0 (Tool Loop).

## Clarifications

### Recorded 2026-06-20 (rubber duck — /execute-epic)

**Key Decisions:**
- Caching is PAYLOAD-LEVEL, not per-file. The fan-out engine concatenates all changed files into one text blob per payload mode (`buildPayloads`, internal/fanout/review.go:491) before any agent is called, so per-file granularity is impossible without re-architecting the engine. Cache a whole per-agent review result keyed on the full payload; replay it into the pool before reconciliation when the diff is unchanged.
- Cache key = `sha256(payload_bytes)` + `model_id` + `sha256(persona_text)`. Agent name is redundant once persona content + model are hashed. byte-budget/truncation is captured implicitly (different budget → different payload text → different hash).
- `min-severity` and `max-findings` are deterministic post-LLM filters (internal/fanout/postprocess.go:19), applied AFTER the API call, so they are NOT part of the key. The same cached LLM response is valid regardless of those filter thresholds.
- `--no-cache` bypasses cache READ but still WRITES fresh results (refreshes stale/suspect entries; every subsequent run benefits).
- Eviction: total-size cap with LRU eviction. Default 50 MB, overridable via `.atcr/config.yaml` as `cache_max_bytes *int64` (same optional-pointer pattern as payload_byte_budget/timeout_secs/max_parallel in internal/registry/project.go:32). No entry-count or TTL policy.
- Caching is scoped to the review fan-out stage ONLY. The verification/skeptic stage is excluded — it operates on reconciled findings, invokes live tool-loop code reads, and injects a non-deterministic sentinel (internal/verify/skeptic.go:24, rand.Uint32()) that makes content-hash keying unsafe by design.

**Scope Boundaries:**
- IN scope: cache package, key derivation, integration into the fan-out read/write path, `--no-cache` flag, size-based LRU eviction with config override.
- OUT of scope: re-architecting fan-out into per-file API calls (true incremental/partial cache hits — a separate epic); caching the verification/skeptic stage.

**Technical Approach:**
- Cache unit = per-agent raw review output (fanout `Result.Content`), replayed into the pool before reconciliation so reconcile/verify run normally on cached content.
- Reuse existing primitives: `atomicfs.WriteJSON` for storage, the `"sha256:"` hash format (reconcile/ambiguous.go:156), `.atcr/` root resolution. Cache lives under `.atcr/cache/` (gitignored like `.atcr/reviews`).
- Cache hit is content-addressed only; mtime is used solely for LRU eviction ordering, never for hit correctness.

**Original Questions and Answers:**
1. Q: Granularity — payload-level (A) or true per-file incremental (B)?
   A: Option A — payload-level caching. Per-file granularity requires re-architecting the engine and belongs in a separate epic.
2. Q: Cache-key composition — full key or narrower?
   A: Narrower — sha256(payload_bytes) + model_id + sha256(persona_text). Exclude min-severity/max-findings (post-LLM filters); byte-budget captured implicitly via payload text.
3. Q: --no-cache semantics — bypass read only (a) or full disable (b)?
   A: Option (a) — bypass read, still write fresh results to refresh stale entries.
4. Q: Eviction policy — 50 MB LRU + config override?
   A: Yes — 50 MB LRU default, overridable via .atcr/config.yaml as cache_max_bytes *int64. No entry-count or TTL policy.
5. Q: Cache scope — review fan-out only, excluding verification/skeptic?
   A: Yes — review fan-out only. Verification/skeptic excluded (non-deterministic sentinel, reconciled-findings input, live tool reads).
