# Feature Request: AXI Contextual Disclosure — Next-Step Suggestions

- **Estimated time**: 2-3 days
- **Tasks**: 6
- **Components**: `cmd/atcr`

- **Status**: DEFERRED — intentionally punted until the `--axi` output/command surface (Epics 31.0/31.1) has stabilized under real usage; do not start before then.

## Context

axi.md's Principle 9 ("Contextual Disclosure") asks output to append `help[]` lines suggesting concrete next-step command templates, carrying forward fixed disambiguating flags while leaving runtime values as placeholders (e.g. `<id>`). Epic 31.0 (AXI Compliance) and Epic 31.1 (Content-First Home View) deliberately scoped this principle out: unlike Principle 8 (a single, cheap, well-contained change), Principle 9 touches the output path of every command that produces a result an agent might act on next — `review`, `report`, `reconcile`, `verify`, `debate`, `status`, at minimum — and its content (which suggestions, for which commands) is a genuine design task, not just plumbing.

There is also a real overlap-risk to resolve before building this: `skill/SKILL.md` already ships as an installable Agent Skill that encodes "what to run next" as an explicit orchestration script for any agent using it (Principle 7's secondary recommendation, already satisfied — see Epic 31.1's Notes). Principle 9 mainly adds value for an agent driving the CLI *without* that Skill loaded; building it before the `--axi` surface and command set are stable risks a suggestion registry that goes stale the same way `internal/mcp`'s format-enum drifted before Sprint 25.0's TD-003 fix.

## Problem Statement

An agent invoking `atcr` commands directly (without the installed Skill) has no built-in signal for what to run next after a given command completes — it must already know the CLI's command graph, or spend a discovery turn inferring it, which is exactly the kind of avoidable overhead the AXI epics exist to eliminate.

## Proposed Solution

Build a per-command "next step" suggestion registry — a small, explicit map from each AXI-relevant command to its likely next commands, expressed as templates with fixed/disambiguating flags carried forward and runtime values left as placeholders (`<id>`). Append the resulting `help[]` block after each command's output (respecting `--axi` vs. human-mode rendering, reusing Epic 31.0/31.1's format plumbing rather than a third mode). Cover, at minimum: `review` → `report <id>`/`verify <id>`; `report`/`reconcile` → `debt list`/`verify <id>`; `verify`/`debate` → `report <id>`; `status` → `report <id>`/`resume <id>` (if pending).

## Tasks

- **T1**: Design the suggestion registry's data shape (command → candidate next-command templates, fixed flags vs. `<placeholder>` runtime values) and where it lives (e.g. `cmd/atcr/suggestions.go`).
- **T2**: Emit the `help[]` block after `atcr review`/`atcr resume` output, in both `--axi` and human modes, reusing Epic 31.0's context-propagation mechanism.
- **T3**: Emit the `help[]` block after `atcr report`/`atcr reconcile` output.
- **T4**: Emit the `help[]` block after `atcr verify`/`atcr debate` output.
- **T5**: Emit the `help[]` block after `atcr status` output (and Epic 31.1's home view, if still relevant at execution time).
- **T6**: Add a drift-guard test (mirroring the `internal/mcp` enum-drift precedent from Sprint 25.0/TD-003) so a future new/removed command cannot silently leave the suggestion registry stale.

## Acceptance Criteria

- [ ] AC1: Every command listed in the Proposed Solution emits a `help[]` block with concrete next-step command templates after successful output, in both `--axi` and human modes.
- [ ] AC2: Suggested templates carry forward fixed/disambiguating flags actually used in the current invocation and leave runtime values as `<placeholder>` tokens — never a guessed concrete id/value.
- [ ] AC3: A registry drift-guard test fails CI if a covered command is removed/renamed without updating its suggestion entries.
- [ ] AC4: No regression to any covered command's existing output when `help[]` emission is added — pre-existing content is unchanged, only appended to.

## Out of Scope

- Session-hook / plugin auto-install (Principle 7's primary recommendation) — separate from this epic; atcr's existing `skill/SKILL.md` already covers Principle 7's secondary bar.
- Any change to `skill/SKILL.md`'s own orchestration script — this epic is additive for CLI-only agents, not a replacement for the Skill's routing table.
- Reworking Epic 31.0/31.1's `--axi` payload shape itself.

## Dependencies

- Epic 31.0 (AXI Compliance) and Epic 31.1 (Content-First Home View) — reuses their format-dispatch/context-propagation plumbing, and this epic should not start until both have shipped and the `--axi` surface has seen real usage.
