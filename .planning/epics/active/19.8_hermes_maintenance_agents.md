# Epic Plan 19.8: Hermes Maintenance Agents — Mechanical, Judgment & Drafting

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Configure scheduled **hermes agents** to consume Epic 19.7's `atcr models check` primitives and offload the recurring persona-model maintenance across three clearly-separated roles — **mechanical** (deterministic catalog-diff → slug-bump PR, no LLM), **judgment** (classify drift; route minor vs. major/deprecation), and **drafting** (an LLM reads the vendor's updated prompting guide and drafts persona `.md` edits). Every change lands as a **pull request**, is **fixture-gated in CI**, and prompt changes require **human approval** — the agent is a contributor, never a committer to `main`.

## Context

**The real deliverable is the battle-tested prompt.** Model-tuned reviewer prompts are refined *in production*: they hallucinate, over-report, flag non-bugs, or miss real ones — and the maintainer tunes them over time until they earn trust. That human refinement loop is unavoidable and is 19.6's core value. Hermes automation does not replace it; it converts "write the re-tune from scratch" into "review a drafted re-tune," and it eliminates the purely mechanical slug/deprecation toil entirely.

**Split the labor by determinism.**

- **Mechanical work needs no LLM.** Detecting a newer family member, a deprecating `expiration_date`, or a missing slug, then bumping a lock, is pure string/timestamp comparison. An LLM here only *adds* hallucination risk (invented slugs) to a deterministic task. This layer is a scheduled wrapper over 19.7's `atcr models check --json` + a PR opener.
- **Judgment work is classification.** Minor bump → hand to mechanical. Major bump or deprecation → open a re-tune work item tagged with the vendor guide URL.
- **Drafting work is where an LLM helps, with a caveat.** Asking a flat-rate open model (Kimi K2.7 / GLM 5.2 / Qwen 3.7) to re-tune a prompt meant to embody *Anthropic's/OpenAI's/Google's own* prompting guidance is cross-vendor — the open model can mechanically transcribe "the guide now says X → change Y" but cannot judge fit for the target model. So its output is a **draft the maintainer reviews**, never an auto-merge.

**The guardrails already exist.** An auto-generated prompt is exactly 19.6's C3 threat model — untrusted input becoming a system prompt. 19.6 already ships the fixture gate + length cap + template allowlist that contain it. The hermes agent is just another untrusted contributor the existing machinery already handles.

## Proposed Solution

1. **Mechanical agent (no LLM).** Scheduled run of `atcr models check --json`; on a detected newer member / expiration / missing slug, open a **slug-bump PR** that advances the lock and lets CI's fixture gate verify. Auto-merge-on-green is an opt-in policy the maintainer sets.
2. **Judgment agent.** Classify each drift item: minor → mechanical PR; major or deprecation → open a **re-tune task** carrying the persona, the old/new model, and the vendor prompting-guide URL.
3. **Drafting agent (LLM).** On a re-tune task, fetch the vendor's current prompting guide, draft edits to the persona `.md`, and open a **separate** PR (never mixed with a mechanical slug bump). Backed by a configurable flat-rate reasoning model (Kimi/GLM/Qwen); output is explicitly a draft.
4. **Hard guardrails.**
   - PRs only — **never a direct commit to `main`** (rides the existing squash-to-main + CI releasability model).
   - **Separate PRs** for mechanical slug bumps vs. prompt re-tunes — a routine bump merges on green while a prompt change gets scrutiny, and a prompt hallucination never blocks a slug fix.
   - **Fixture gate CI-blocking** on both; **human approval required** on every prompt PR.
   - Reuse 19.6's fixture + length-cap + template-allowlist guardrails on any agent-authored prompt.
5. **Configuration surface.** How the maintainer assigns each hermes agent to a role and picks the drafting model — documented, with a default off/opt-in posture.

## Acceptance Criteria

- [ ] AC1: A scheduled mechanical agent runs `atcr models check --json` and opens a slug-bump PR (no LLM) when drift/expiration is detected; the PR passes the CI fixture gate before it can merge.
- [ ] AC2: The judgment step classifies drift as minor (→ mechanical) vs. major/deprecation (→ re-tune task with the vendor guide URL).
- [ ] AC3: The drafting agent produces prompt-edit PRs from the vendor prompting guide, backed by a configurable flat-rate model, opened **separately** from mechanical PRs.
- [ ] AC4: No agent ever commits to `main`; every agent change is a PR; prompt PRs require explicit human approval; mechanical PRs may auto-merge on green per maintainer policy.
- [ ] AC5: Agent-authored prompts pass the reused 19.6 C3 guardrails (fixture gate + length cap + template allowlist) as a hard CI gate.
- [ ] AC6: The role→agent and drafting-model configuration is documented with a safe default (opt-in), and a dry-run/preview mode exists before any PR is opened.

## Out of Scope

- **The resolution/lockfile/`models check` primitives themselves** — Epic 19.7 (this epic consumes them).
- **Community prompt submissions / crowd-sourced battle-tested prompts** — a later epic (under discussion); this epic is maintainer-side automation only.
- **Building the hermes agent framework itself** if it is an external system — this epic configures existing hermes agents against atcr's primitives.
- **Auto-merging prompt changes** — always human-reviewed.

## Dependencies

- **Epic 19.7 (Live Model Resolution)** — provides `atcr models check`, the lock, and the drift/expiration/major-bump signals this epic's agents act on.
- **Epic 19.6 (Community Registry Hub)** — provides the C3 guardrails reused to contain agent-authored prompts.
