# Epic Plan 1.1: Schema Reservations for Agentic Stages

**Estimated Durations**: 1-2 days

## Objective

Reserve schema room in the v1 public contracts — registry, findings JSON, manifest, status — for the agentic stages (Epics 2.0–5.0) **before** the v1 formats are published, so no later stage forces a format break. Documentation-and-schema only; no behavior changes.

## Context

Epic 1.0 publishes `atcr-findings/v1` and the registry schema as documented public contracts with an additive-only evolution policy. Epics 2.0 (tool-using reviewers) and 3.0 (adversarial verification) are committed for immediate execution and need new fields. Reserving them now costs near zero; retrofitting after publication costs a format version.

## Problem Statement

If v1 ships without reserved fields, Epic 2.0/3.0 must either bump the format version immediately after launch (bad optics, breaks early adopters) or overload existing fields (corrupts the contract).

## Proposed Solution

Additive, documented-as-reserved fields:

1. **Registry agent schema** — reserved fields, parsed and validated but inert in 1.x:
   - `tools: bool` (default false) — enables the tool loop (activated in 2.0)
   - `max_turns: int` (default 10) — agent-loop turn cap (2.0)
   - `tool_budget_bytes: int` — cumulative tool-result budget per agent (2.0)
   - `role: reviewer | skeptic | judge` (default reviewer) — stage assignment (3.0/4.0)
2. **findings.json record** — optional `verification` object (absent in 1.x): `{verdict: confirmed|refuted|unverifiable, skeptic: <agent>, notes: <string>}` (3.0). Renderers must tolerate its absence and presence.
3. **manifest.json** — optional `stages` array recording which stages ran (`["review"]` in 1.x; later `["review","verify","debate"]`).
4. **status.json per agent** — optional `turns`, `tool_calls`, `tool_bytes` counters (zero/absent in 1.x).
5. **docs/findings-format.md and docs/registry.md** — document every reserved field as "reserved for future stages; parsed, not yet acted on."

## Success Criteria

- [ ] Registry loader parses and validates the reserved fields (type errors fail at load) but no v1 code path acts on them.
- [ ] findings.json schema documents the optional `verification` block; `atcr report` renders findings identically with or without it.
- [ ] manifest.json carries `stages: ["review"]`.
- [ ] Format docs list all reserved fields with their owning future epic.
- [ ] No new dependencies; all tests green.

## Task Breakdown

1. Registry schema: add reserved fields + load-time type validation + tests.
2. findings.json / manifest.json / status.json: add optional fields to the shared types and writers; tolerant readers; tests.
3. Documentation updates (findings-format.md, registry.md).

## Out of Scope

Any behavior behind the reserved fields — that is Epics 2.0/3.0.

## Dependencies

Epic 1.0 tasks 2 (config), 7 (fan-out artifacts), 8 (stream package), 13 (docs) complete.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Reserved fields guess wrong and 2.0/3.0 need different shapes | Low | Low | Fields are optional and additive; renaming before activation is cheap since nothing acts on them |
