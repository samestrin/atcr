# Epic Plan 32.3: Sandbox Writable Overlay (Polyglot Auto-Fix Support)

- **Estimated time**: 1-2 days
- **Tasks/Components**: 3 tasks / 1 component
- **Execution**: init-plan

## Objective

Fix the `EROFS` (Read-Only File System) limitation in the ATCR Docker sandbox by implementing an ephemeral copy strategy, unlocking `--auto-fix` validation for non-Go ecosystems (Node, Rust, Python) that require write access to the project directory during testing.

## Context

Epic 32.0 containerized the `--auto-fix` validation step. To protect the host machine, the user's project directory is mounted read-only (`-v $PWD:/work:ro`). This works perfectly for Go (which writes its caches to a redirected `/scratch` mount), but fails closed for many other ecosystems. 
When `npm run build` writes to `dist/`, or `cargo build` writes to `target/`, or Python generates `__pycache__` next to source files, the read-only mount blocks the write. The validator crashes, the fix is assumed invalid, and the PR is silently discarded.

## Proposed Solution

1. **Mount to `/src` (Read-Only):** Modify `dockerRunArgs` in `internal/sandbox/docker.go` to mount the `SnapshotDir` to `/src:ro` instead of `/work:ro`.
2. **Ephemeral Copy Injection:** Modify `renderCommand` to inject a setup command (`cp -a /src/. /work/`) before the user's validation script executes.
3. **Execution in Writable Layer:** Run the user's test script inside `/work`, which is now part of the container's ephemeral, writable filesystem. This allows tools like `npm` and `cargo` to write their temporary build artifacts safely without mutating the host machine.

## Tasks

- **T1**: Update `dockerRunArgs` in `internal/sandbox/docker.go` to change the `-v` mount target from `/work:ro` to `/src:ro`.
- **T2**: Modify `renderCommand` in `internal/sandbox/docker.go` to automatically prepend `cp -a /src/. /work/ && cd /work` before the user's provided test script.
- **T3**: Add unit tests in `internal/sandbox/docker_test.go` that simulate a mock validation script attempting to create a file in the working directory, verifying it succeeds under the new ephemeral copy strategy.

## Acceptance Criteria

- [ ] The sandbox mounts the host code securely as read-only.
- [ ] Test scripts run inside the sandbox can successfully write new files into their working directory without hitting `EROFS` errors.
- [ ] The changes do not permanently mutate the user's local filesystem (the ephemeral copies die with the container).

### Items needing user confirmation (5)

- ⏸️ **Task 2 targets the wrong function — it cannot inject anything into the actual execution.** `renderCommand` (`internal/sandbox/docker.go:151-156`) only builds a human-readable log string used for evidence/logging (`res.Command`); it has zero effect on what the container actually executes. The real execution content is computed independently: `dockerRunArgs` appends `spec.Command` directly to argv, or `Run` sets `cmd.Stdin = strings.NewReader(spec.Script)` — neither reads `renderCommand`'s return value. A prior Epic 11.0 code review already scrutinized this exact function's fidelity to the real command (`.planning/epics/code-reviews/11.0_executing_reviewers/multi-agent/raw/bruce/review.md:19-29`, "Issue 1: `renderCommand` doesn't reflect the actual command") and concluded it is display-only — independent confirmation this is not a new observation. Suggested action: relocate the setup-injection logic into `dockerRunArgs`/`Run`, where the real argv/stdin is actually constructed.
- ⏸️ **Even relocated, Task 2's mechanism doesn't reach `--auto-fix`'s actual code path.** `RunSandboxedValidation` (`internal/verify/sandboxvalidate.go:43`) — the function that serves `--auto-fix` validation, this epic's whole objective — exclusively constructs `RunSpec{Command: argv}`, never `Script`, and this is a pinned invariant (`internal/verify/sandboxvalidate_test.go:71`: *"Script must never be populated on the argv path (RunSpec exactly-one invariant)"*). In `Command` mode, `dockerRunArgs` execs the image's argv directly (`args = append(args, cfg.Image); args = append(args, spec.Command...)`) with **no shell at all** — there is nothing to interpret a prepended `cp -a /src/. /work/ && cd /work` shell snippet even if it reached the container. As literally described, Task 2 has zero effect on `--auto-fix`'s validation runs — the epic's own AC2 would remain false for the code path it exists to fix. Suggested action: wrap the `Command`-mode exec in a shell invocation (e.g. `/bin/sh -c 'cp -a /src/. /work/ && cd /work && exec "$@"' -- <command...>`), mirroring whatever setup gets added for `Script` mode.
- ⏸️ **`/work` has no writable backing under the container's `--read-only` flag.** `dockerRunArgs` sets a global `"--read-only"` flag on the entire container rootfs; the only currently-writable location is the explicit `"--tmpfs", "/scratch:rw,exec,size=" + cfg.ScratchSize` mount. If Task 1 is implemented literally (rename the `-v` target from `/work:ro` to `/src:ro` and nothing else), `/work` becomes an ordinary path on the read-only rootfs with no volume/tmpfs backing it — `cp -a /src/. /work/` would still fail with `EROFS` (or the container may fail to start if the runtime can't even create the `--workdir` path on a read-only root). Suggested action: Task 1 needs an additional writable mount for `/work` (e.g. a new `--tmpfs /work:rw,exec,size=<budget>`, mirroring the existing `/scratch` pattern) sized for a full source-tree copy rather than `/scratch`'s 64MB build-cache-sized default — likely a new `DockerConfig` field, not a one-line mount-target rename.
- ⏸️ **Blast radius is wider than stated — this changes shared sandbox infrastructure, not just `--auto-fix`.** `dockerRunArgs`, `Run`, and `renderCommand` are shared by `--exec` (Epic 11.0, via `internal/tools/exec_tools.go:178,215`) as well as `--auto-fix` (Epic 32.0). The epic's Objective/Context frame this purely as an `--auto-fix` fix and Components Touched lists only `internal/sandbox/docker.go`/`docker_test.go`, but the change alters execution semantics for `--exec`'s Script-mode invocations too, and touches a documented security guarantee: `docs/execution.md:56-57,86,89` states *"/work (the snapshot) is read-only; the run cannot mutate your working tree"* as one of the sandbox's stated guarantees for `--exec`, and the package doc (`internal/sandbox/sandbox.go:9`) states *"a read-only view of the snapshot (the run cannot mutate the work tree)"* as a hard MUST for every `Backend.Run`. Suggested action: either explicitly scope this as a shared-infrastructure change (add `docs/execution.md` + `internal/tools/exec_tools_test.go` coverage to scope), or make the ephemeral-copy behavior conditional/opt-in so `--exec`'s existing guarantee is preserved for callers that don't need it.
- ⏸️ **An existing, currently-passing test pins the mount contract Task 1 would change.** `TestDockerRunArgs_HardeningFlagsPresent` (`internal/sandbox/sandbox_test.go:55`) asserts `assert.Contains(t, joined, "/tmp/snap:/work:ro", "snapshot must be a read-only mount")`. Task 1 as described would break this test; it isn't mentioned anywhere in Task 3's scope (T3 only describes *adding* new tests, not updating this existing pinned assertion). Suggested action: add updating this test to Task 1 or Task 3's scope explicitly.
