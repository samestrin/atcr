# Epic Plan 1.4: Parallel-Lane Concurrency Cap (max_parallel)

**Estimated Durations**: 1 day

## Objective

Bound the fan-out engine's parallel lane with a buffered semaphore sized from a new `max_parallel` setting resolved through the existing `CLI > project > registry > embedded` precedence chain — so a large roster cannot spawn an unbounded goroutine-per-slot burst against providers.

## Context

Epic 1.0 shipped the fan-out engine with one goroutine per non-serial slot (`internal/fanout/engine.go` `Run`), relying on real rosters being ≤~10 agents — comfortably within AC 01-04's "10 concurrent agent calls" target. The sprint's adversarial review evaluated a concurrency cap and recorded it as accepted risk TD-017: "a concurrency cap adds config surface not requested in v1; fix in v2 with a buffered semaphore sized from a new max_parallel setting." The binding Phase 3 clarification rejected `golang.org/x/time/rate` on the same keep-config-surface-small grounds. This epic is that recorded v2 work.

## Problem Statement

Every non-serial slot spawns its own goroutine with no semaphore or worker-pool bound. A future large roster (or a misconfigured one) fires all provider calls at once: provider-side rate limiting, local resource pressure, and burst cost are all uncontrolled, and there is no knob to tune it per project or per machine.

## Proposed Solution

1. **New setting `max_parallel`**: integer, resolved through the existing settings precedence chain (`CLI flag > .atcr/config.yaml > ~/.config/atcr/registry.yaml > embedded default`). Embedded default chosen to preserve current behavior for v1 rosters (≥10, matching the AC's concurrent-calls target; `0` = unbounded is a candidate escape hatch — decide at /init-plan).
2. **Buffered semaphore in the engine**: `Engine.Run` acquires a semaphore slot before each parallel-lane invocation; the serial lane is unaffected (already sequential). The WaitGroup drain guarantee (no leaked goroutines on cancellation) must be preserved.
3. **Validation**: negative values are a usage error (exit 2), consistent with the existing settings validation tier.
4. **Docs**: registry.md settings table and `atcr init` template comment gain the new key.

## Success Criteria

- [ ] `max_parallel` resolves through CLI > project > registry > embedded with the chosen default; invalid (negative) values fail with a usage error.
- [ ] With `max_parallel: N`, an engine test with a roster larger than N proves peak observed concurrency never exceeds N (the fake completer already tracks peak).
- [ ] Serial-lane behavior and the WaitGroup drain guarantee under cancellation are unchanged (existing engine tests stay green).
- [ ] registry.md documents the setting; `atcr init` template carries it as a comment.

## Task Breakdown

1. Settings: add `max_parallel` to registry/project schemas, `ResolveSettings`, and validation + tests.
2. CLI: `--max-parallel` flag wired into `registry.CLIOverrides` + tests.
3. Engine: buffered-semaphore bound on the parallel lane in `Engine.Run`, sized from the resolved setting + peak-concurrency tests.
4. Docs: registry.md settings table, init template comment.

## Out of Scope

- Rate limiting (requests/second) — the serial lane plus fallback chains remain the rate-limit mechanism.
- Per-provider or per-agent concurrency bounds (single global knob only).

## Dependencies

- Epic 1.0 core engine and settings precedence (complete, in code review).
