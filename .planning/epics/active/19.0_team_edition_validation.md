# Epic Plan 16.0: Team Edition Validation

**Estimated Durations**: 3 weeks
**Tasks/Components:** 8/1
**Execution:** execute-epic --strong
**Status:** Draft (queued; depends on Epic 15.0 website launch + Epic 10.0 leaderboard + 3.2 + 3.3)

## Objective

Validate demand, feature priorities, and price sensitivity for a Team Edition before committing to a 6-10 week build. Recruit volunteer engineering teams through atcr.dev, run a lightweight structured pilot, and produce a go/no-go memo with a ranked feature list.

## Context

Team Edition has the highest direct-revenue ceiling in the product concepts, but it also has the longest build, longest sales cycle, and least architectural moat. Building it without validated demand risks a large sunk cost for features nobody will pay for.

Unlike engineering epics, this epic does not ship production code. It ships evidence: a clear signal about which Team Edition features matter, what teams will pay, and whether the timing is right.

## Problem Statement

1. There is no confirmed enterprise demand for Team Edition features.
2. The relative priority of shared registry, finding history, audit trail, and trend dashboard is unknown.
3. There is no price-sensitivity data to size the revenue opportunity.
4. Without a recruitment channel, finding pilot teams requires personal network access that does not currently exist.

## Proposed Solution

### Phase 1 — atcr.dev landing page pilot integration & signup (1 week)

Integrate the signup flow into the atcr.dev site launched in Epic 15.0 with:
- A "Team Edition Pilot Program" signup form collecting:
  - Team size
  - Primary language/stack
  - Current code-review pain points
  - Willingness to pay (anchored range, optional)
  - Contact email

The site does not need a CMS or complex build. A static site generator (e.g., Hugo, Jekyll, or plain HTML) hosted on GitHub Pages or Cloudflare Pages is sufficient.

### Phase 2 — Lightweight prototypes (1-1.5 weeks)

Build throwaway prototypes for the three Team Edition pillars. These are deliberately thin — just enough to test the value, not enough to ship.

**Shared registry prototype:**
- A config-repo approach: a `registry.yaml` in a dedicated git repo that the team references via `ATCR_REGISTRY_URL`.
- Provider keys remain env-var only; no secrets manager integration yet.

**Finding history prototype:**
- `atcr history --since 30d --package internal/auth` reads from a repo-local `findings-history.jsonl` written by each `atcr review` run.
- Aggregation outputs a markdown table, not a dashboard.

**Audit trail prototype:**
- An append-only `audit.log.jsonl` in the review output directory with run timestamp, base/head SHAs, PR number, and findings summary.
- A one-page compliance report generator: `atcr audit-report --pr 1234`.

### Phase 3 — Pilot execution (1-1.5 weeks)

Recruit 3-5 volunteer teams from the atcr.dev signup list. Each pilot runs for 2 weeks with:
- Week 0: onboarding call, install ATCR + prototypes, configure shared registry.
- Week 1: mid-pilot check-in (async or 15-min call) — blockers and first impressions.
- Week 2: exit interview (30 min) structured around the feature priorities.

Exit interview questions:
- Which prototype would you miss most if it were removed?
- How do you currently solve this problem?
- What would make you pay for this?
- At what price per month would this be a no-brainer? Too expensive?
- What is the minimum feature set for a paid launch?

### Phase 4 — Validation memo (2-3 days)

Produce a one-page memo with:
- Number of signups, completions, and drop-offs.
- Feature priority ranking (shared registry, history, audit trail, dashboard).
- Price sensitivity summary.
- Go/no-go recommendation for Team Edition.
- If go: recommended MVP scope and pricing anchors.
- If no-go: what signal would change the decision.

## Acceptance Criteria

- [ ] 3-5 volunteer teams are recruited and onboarded.
- [ ] Shared registry prototype is validated by at least 2 teams.
- [ ] Finding history prototype is validated by at least 2 teams.
- [ ] Audit trail prototype is validated by at least 2 teams.
- [ ] Exit interviews are recorded or transcribed and stored in `.planning/product/validation/team-edition/`.

## Task Breakdown

1. **atcr.dev integration**: Integrate/embed pilot program signup form on the live site.
2. **Landing copy update**: Integrate pilot program CTA on the landing page.
3. **Shared registry prototype**: config-repo registry loading, env-var key sharing, team onboarding guide.
4. **Finding history prototype**: JSONL writer in `atcr review`, `atcr history` query CLI, markdown aggregation.
5. **Audit trail prototype**: immutable run log, `atcr audit-report --pr` generator.
6. **Pilot recruitment**: outreach to signup list, screening for team size and fit.
7. **Pilot onboarding**: install guide, weekly check-ins, exit interview script.
8. **Validation memo**: synthesis, go/no-go, MVP scope, pricing anchors.

## Out of Scope

- Full Team Edition build (dashboard, web UI, API layer, billing).
- Production secrets-manager integration.
- Long-term support or SLAs for pilot teams.
- Payment processing or paid pilot pricing.
- Real-time collaborative features.

## Dependencies

- ATCR OSS stable enough for pilot use.
- Disagreement Radar and local leaderboard shipped (credibility for atcr.dev).
- atcr.dev website launched (Epic 15.0).
- A way to collect signup data (Google Forms, Formspree, or self-hosted form).

## Revenue Model

This epic does not generate revenue. It de-risks the Team Edition revenue model by validating:
- Which features teams will pay for.
- What price points are plausible.
- Whether self-hosted positioning is a real differentiator.

Downstream revenue if validation is positive:
- Per-user Team Edition license.
- Enterprise support contracts.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| No teams sign up | Medium | High | Seed the pilot through GitHub README, leaderboard, and relevant communities (Go, Rust, security-focused forums) |
| Teams drop out before exit interview | Medium | High | Keep prototype setup under 30 minutes; offer async check-ins; compensate with a small gift card or swag |
| Feedback is too diverse to prioritize | Medium | Medium | Force-rank features in the exit interview; look for clusters, not unanimity |
| Volunteers expect free Team Edition forever | Medium | Medium | Set clear expectations in signup copy: "2-week pilot, no obligation, free during pilot only" |
| Pilot teams are too small to represent enterprise buyers | High | Medium | Ask about team size during signup; weight feedback by size; flag this limitation in the memo |

## Success Criteria

- [ ] 3+ teams complete the full 2-week pilot and exit interview.
- [ ] At least 2 teams express willingness to pay for a subset of Team Edition features.
- [ ] Memo contains a clear priority ranking with evidence quotes.
- [ ] Go/no-go decision is documented with conditions that would change it.
