# Epic Plan 37.0: Enterprise Accountability Gates (Digital Signatures)

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Create an "Accountability Gate" integration that explicitly requires a traceable human digital signature (or a verifiable `/approve` command) for pull requests flagged as HIGH/CRITICAL severity or touching protected paths. 

## Context

In high-compliance environments (e.g., SOC2, ISO 27001, FedRAMP), "looks good to me" by a bot isn't enough. When a change involves significant risk, auditors need proof of human authorization. Currently, AI tools speed up the development process, but often blur the lines of authorship and approval. ATCR needs a mechanism to enforce clear accountability so teams can stand behind what they merge.

## Proposed Solution

1. **VCS Integration (GitHub/GitLab):**
   - ATCR integrates via webhooks or actions into the Pull Request workflow.
2. **Dynamic Review Requirements:**
   - For routine PRs (LOW/MEDIUM severity findings, no protected paths), ATCR outputs findings and allows standard merge conditions.
   - For high-risk PRs (HIGH/CRITICAL severity findings, or paths matching Epic 36.0), ATCR locks the PR via a required status check.
3. **Signature Verification:**
   - The status check can only be cleared if a designated Code Owner leaves an explicit, traceable comment (e.g., `/atcr sign`) or uses a formal digital signature mechanism provided by the VCS.

## Acceptance Criteria

- [ ] ATCR can enforce a locking status check on GitHub/GitLab PRs based on finding severity.
- [ ] The lock can be cleared by an authorized human interaction (e.g., command, or digital signature).
- [ ] The accountability event is logged (tying into Epic 34.0 Enterprise Audit Logging).

## Components Touched

- `internal/integrations/vcs/` (Handling status checks and webhooks)
- `internal/reconciler/` (Triggering the accountability lock)
