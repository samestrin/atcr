# Feature Request: AXI Content-First Home View

- **Estimated time**: 1-2 days
- **Tasks**: 4
- **Components**: `cmd/atcr`

## Context

Epic 31.0 (AXI Compliance) implements `--axi` token-dense output, pagination, exit-code reconciliation, and stderr isolation for `atcr review`/`atcr report`, but explicitly scoped out axi.md's Principle 8 ("Content First"). This epic delivers Principle 8 as a small, self-contained follow-on — sequenced as `31.1` (not `31.0`'s own scope, and not folded into the later `48.0` epic) specifically so it lands **before** the `33.0`→`33.1`→`33.2` public-launch/atcr.dev sequence, since it is cheap, low-risk, and directly strengthens an eventual AXI-catalog listing.

## Problem Statement

Bare `atcr` (no arguments, no flags) currently calls `cmd.Help()` (`cmd/atcr/main.go`), showing static usage text. axi.md's Principle 8 argues a no-args invocation should show live, actionable data instead: an orchestrator or agent often runs a bare command just to sanity-check state before deciding what to do next, and every such invocation currently costs a wasted discovery turn returning nothing but help text it likely already has cached.

## Proposed Solution

Add a home-view branch to the root command's `RunE` for the true no-args case only. Cobra's `-h`/`--help` flags and `--version` already short-circuit before `RunE` runs, and every subcommand's own `Args`/`RunE` is untouched — this only changes what happens when `atcr` is invoked with literally nothing.

The home view shows:
1. The executable path, with the home-directory prefix rendered as `~` (per axi.md's example).
2. atcr's existing one-line description (already defined on the root command: "Agent Team Code Review — a review panel, not a reviewer").
3. Live state: resolve the current review directory via `cmd/atcr/anchor.go`'s `anchorDir("")` (the actual `.atcr/latest` resolution point — not `ReadReviewStatus` itself), then read its id/status via the same `fanout.ReadReviewStatus` reader `cmd/atcr/status.go` already uses. When `anchorDir("")` errors (no `.atcr/latest` pointer exists), catch it and render an explicit "no reviews yet, run `atcr review`" message with exit 0 — deliberately NOT `status.go`'s existing pattern of wrapping that same condition in `usageError()` (exit 2), since the home view must never error on a first-run/no-reviews state.

The view respects `--axi` (bare `atcr --axi`) by reusing Epic 31.0's `FormatAXI`/pagination/context-propagation plumbing — this is not a second, parallel output mode.

## Tasks

- **T1**: Add a no-args home-view branch to the root command's `RunE` in `cmd/atcr/main.go`, guarded so it only fires when zero args and zero flags are passed; `-h`/`--help`/`--version` and every subcommand remain on their existing paths.
- **T2**: Implement the home-view renderer: `~`-relativized executable path (`os.Executable()` + home-dir substitution), the existing one-line description, and live state by calling `anchorDir("")` (`cmd/atcr/anchor.go:18`) then `fanout.ReadReviewStatus` on the resolved directory — catching `anchorDir("")`'s error as the explicit no-reviews-yet state (exit 0), not as a usage error.
- **T3**: Wire `--axi` (bare) through Epic 31.0's format-dispatch/context-propagation mechanism so the home view's `--axi` output is one more consumer of that plumbing, not a new mode switch.
- **T4**: Add a golden/snapshot test pinning the non-`--axi` home-view output, plus regression tests confirming `--help`/`-h`/`--version` and all existing subcommands are unaffected — including explicitly updating `cmd/atcr/main_test.go`'s `TestRootCmd_BareInvocationShowsHelp`, which currently pins the pre-T1 bare-invocation-shows-help behavior and will fail once T1 lands.

### Advisory observations (4)

- ℹ️ **Existing path-relativization pattern to follow:** `relDisplay(base, path string) string` (`internal/tools/dispatch.go:369-377`) and `relativizePaths` (`internal/log/redact.go:155`) both relativize a path via `filepath.Rel` with a fallback to the absolute/original path on failure. Neither is a direct `~`-substitution drop-in (both relativize against an arbitrary base, not specifically the home directory), but T2's new helper should follow this same `filepath.Rel`-plus-fallback idiom rather than a bespoke string-prefix implementation. Separately, `expandHome` (`cmd/atcr/quickstart.go:279`) already does the *inverse* operation (`~` → full path) in the same package, and a known TD item flags duplicated home-expansion logic there — avoid adding a third divergent implementation.

- ℹ️ **`FormatAXI` confirmed not yet implemented — consistent with stated Dependency:** A codebase-wide search confirms `FormatAXI` exists only in Epic 31.0's planning documents (ACs, sprint-design, codebase-discovery.json), not anywhere in `cmd/` or `internal/`. This matches this epic's own Dependencies section, which already correctly frames Epic 31.0 as a not-yet-landed prerequisite. No action needed.

- ℹ️ **Minor task-description imprecision on args vs. flags:** The root command's `Args: usageArgs(cobra.NoArgs)` (`cmd/atcr/main.go:221`) already guarantees zero positional arguments whenever `RunE` fires without a subcommand. T1's phrasing "guarded so it only fires when zero args and zero flags are passed" is slightly imprecise: a persistent flag passed with no subcommand (e.g. `atcr --log-format json`) still reaches the same `RunE`. Worth deciding explicitly during implementation whether the home view should also render in that case (most likely yes) rather than leaving it undefined.

- ℹ️ **`ReadReviewStatus` lives outside declared COMPONENTS_TOUCHED, but as read-only reuse:** `fanout.ReadReviewStatus` (`internal/fanout/status.go:144`) is outside the plan's declared `cmd/atcr` component, but this mirrors how `cmd/atcr/status.go` already calls into `internal/fanout` today without listing it as a touched component — this is read-only dependency usage, not a modification, so it does not affect COMPONENT_COUNT or the scope-guard check.
