## Objective

Let the review pipeline ingest a standalone **diff file**, not just a `repo/base/head` git
triple. Add a diff-file → entries primitive in `internal/payload` and an exported
`internal/fanout` entry that builds a `PreparedReview` from it — so a loose unified diff can
be reviewed end to end on the exact same `modePayload` path production uses.

## Context

Every payload builder today is git-ref based: `payload.BuildEntries(ctx, mode, repo, base,
head)` (`internal/payload/builder.go:85`), `payload.BuildDiff` (`:60`), and the unexported
`buildPayloads(ctx, cfg, repo, base, head)` (`internal/fanout/review.go:516`) that calls
into them and feeds `ExecuteReview` (`internal/fanout/review.go:417`). There is no way to
hand the pipeline a diff that did not come from a git range.

Epic 10.2 (`atcr benchmark run`) needs exactly this: benchmark cases are versioned diff
files (`Case.Diff`), not git ranges. The 10.0 Q1b clarification names "a new diff-file→payload
ingestion path" as the precondition before benchmark cases can be reviewed. Building it as a
standalone epic keeps the seam clean (10.2 only *calls* the exported entry, it does not
modify `internal/fanout` or `internal/payload`) and delivers independent value: reviewing a
patch file directly, without a checkout.

## Problem Statement

1. The pipeline cannot consume a diff that is not the output of `git diff base..head`.
2. The diff→entries logic is internal to `internal/payload` and parameterized only by git
   refs; there is no entry that accepts pre-existing diff text or a diff file.
3. Without this, Epic 10.2 would have to parse diffs inline in `cmd/atcr`, bypassing the real
   payload path and risking scoring an artifact the production pipeline never produces.

## Proposed Solution

### Phase 1 — Diff-file → entries primitive (`internal/payload`) (1.5 days)

- Add a builder that takes diff text (or a diff-file path) and produces the same
  `[]FileEntry` shape `BuildEntries` returns, reusing the existing diff-parsing/budget
  internals so the output is byte-identical to a git-sourced payload for the same diff.
- Honor the existing safety guards: bound the read by `benchmark.MaxDiffBytes`-style cap and
  reject path traversal (mirror `isSafeRelPath`) when given a file path.
- No change to the git-ref builders — this is an additive sibling entry point.

### Phase 2 — Exported `PreparedReview`-from-diff entry (`internal/fanout`) (1 day)

- Add an exported function that assembles a `PreparedReview` from ingested diff entries +
  the configured roster, parallel to `PrepareReview` (`internal/fanout/review.go:190`) but
  sourced from the Phase 1 primitive instead of `buildPayloads`.
- The result must be accepted unchanged by `ExecuteReview` (`internal/fanout/review.go:417`)
  — same `Slots`/`modePayload` wiring, so downstream execution is identical to a git-range
  review.

### Phase 3 — Ingestion tests on the existing fixture (0.5 day)

- Drive the new path with the diffs already in `internal/benchmark/testdata/suite-valid/`
  (`case-01.diff`, `case-02.diff`) to assert a diff file produces the same entries a git
  range would, and that `ExecuteReview` runs against it with a stub `Completer` (no network).

## Acceptance Criteria

- [ ] A `internal/payload` entry turns diff text / a diff-file path into the same
      `[]FileEntry` shape as `BuildEntries`, byte-identical for the same diff content.
- [ ] The new primitive enforces a size cap and rejects path traversal when given a file path.
- [ ] An exported `internal/fanout` function builds a `PreparedReview` from an ingested diff
      that `ExecuteReview` accepts unchanged (same `modePayload`/`Slots` wiring).
- [ ] A test feeds `internal/benchmark/testdata/suite-valid/case-01.diff` through the path and
      asserts entry parity with a git-sourced equivalent.
- [ ] An end-to-end test runs `ExecuteReview` over an ingested diff with a stub `Completer`
      (no network); the existing git-ref builders are unmodified and still pass.

## Task Breakdown

1. **Payload diff-file primitive**: additive diff-text/diff-file → `[]FileEntry` builder in
   `internal/payload` reusing existing parse/budget internals; size cap + traversal guard.
2. **Fanout ingestion entry**: exported `PreparedReview`-from-diff function in
   `internal/fanout`, parallel to `PrepareReview`, accepted unchanged by `ExecuteReview`.
3. **Ingestion tests**: entry-parity + end-to-end `ExecuteReview` (stub `Completer`) using the
   existing `testdata/suite-valid` diffs.

## Out of Scope

- `atcr benchmark run` command, suite execution loop, scoring, `RunResult` assembly — Epic 10.2.
- Curated `standard-v1` suite content — external `atcr/benchmark-suite` (Epic 10.0 T3).
- Public leaderboard site / methodology — Epic 15.3.
- Any change to the git-ref builders or to `ExecuteReview`'s execution semantics.

## Dependencies

- **Epic 10.0** — the suite contract and `MaxDiffBytes`/`isSafeRelPath` guards to mirror. Shipped.
- **`internal/payload`** (`builder.go`) and **`internal/fanout`** (`review.go`) — the two
  components this epic modifies.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Ingested-diff entries diverge from git-sourced entries, so 10.2 scores an artifact the real pipeline never sees | Medium | High | Reuse the exact `internal/payload` parse/budget internals; assert byte-identical entry parity in tests |
| Oversized/hostile externally-sourced diff exhausts memory | Low | High | Size cap on read, mirroring `ReproHash`'s `MaxDiffBytes` guard |
| Path traversal via a diff-file path argument | Low | Medium | Reject non-safe relative paths, mirroring `isSafeRelPath` |
