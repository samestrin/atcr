# Feature Request: Local Feedback Loops (Ignore Patterns)

- **Estimated time**: 1 week
- **Tasks**: 5
- **Components**: `cmd/atcr`, `internal/reconcile`

## Context

A common pain point with AI code reviewers is the repetition of false positives. If a team has a specific macro, custom structural pattern, or internal convention that models consistently flag as an error, developers currently have to manually dismiss it on every PR. Competitors like Greptile use a centralized feedback loop (thumbs down) to tune their models globally. Since `atcr` is local and deterministic, we need a local equivalent.

## Problem Statement

When an `atcr` panel flags a false positive (e.g., an internal anti-pattern that is actually intentional in this codebase), it will likely flag the same issue again the next time that code is touched. There is no automated way to say "stop reporting this specific structural issue."

## Proposed Solution

Implement a local feedback loop via a `.atcr/ignore.yaml` file (or similar mechanism). When a developer sees a false positive, they can run `atcr ignore <finding-id>`.
1. The `ignore` command extracts the structural signature (AST pattern, regex, or specific file/line combination) of the finding.
2. It appends this rule to `.atcr/ignore.yaml`.
3. During the `atcr reconcile` phase, any findings that match an active ignore rule are automatically dropped (or downgraded to a "suppressed" state) before the final report is generated.

## User Personas

### Primary: Senior Engineer
- Does not want to review the same false positive from the AI twice.
- Wants a deterministic way to configure the AI to respect team conventions without having to write complex prompt engineering overrides for every persona.

## User Journeys

### Ignoring a False Positive
1. User runs `atcr review && atcr reconcile`.
2. The report surfaces a HIGH confidence finding: "Avoid using `unsafe.Pointer` here" (but this package specifically requires it).
3. User runs `atcr ignore 12345` (where 12345 is the finding ID).
4. `atcr` extracts the finding's context and adds a rule to `.atcr/ignore.yaml`.
5. On the next review of that file, the agents might still flag it, but `atcr reconcile` sees the match and filters it out of the final report.

## Tasks

- **T1**: Implement `atcr ignore <finding-id>` command in `cmd/atcr/ignore.go` that reads finding details and appends rules to `.atcr/ignore.yaml`.
- **T2**: Implement YAML configuration parser/writer in `internal/reconcile/ignore.go` (or similar) to handle `.atcr/ignore.yaml` structure.
- **T3**: Integrate ignore-checking into the reconciliation flow in [internal/reconcile/gate.go](file:///Users/samestrin/Documents/GitHub/atcr/internal/reconcile/gate.go).
- **T4**: Add robust matching for ignore rules to support scoping by file/package/pattern and match line numbers with tolerance for drift.
- **T5**: Update reconcile outputs to record ignored findings in raw `status.json` but remove them from `findings.json` and report markdown.

## Acceptance Criteria

- [ ] AC1: `atcr ignore <id>` successfully creates a targeted ignore rule based on an existing reconciled finding.
- [ ] AC2: `atcr reconcile` correctly parses `.atcr/ignore.yaml` and drops matching findings.
- [ ] AC3: Ignore rules can be scoped to specific files, packages, or specific string/regex patterns in the finding body.
- [ ] AC4: Suppressed findings are still logged in the raw `status.json` but omitted from `findings.json` and the final Markdown/GitHub check run.

## Technical Constraints

- The ignore mechanism must happen in the `reconcile` phase (deterministic) rather than the `review` phase (prompts). We don't want to burn tokens telling the LLM "don't flag X" unless X is pervasive enough to warrant updating the system prompt.
- Matching needs to be robust against minor line-number drift (similar to how the current AST grouping works).

## Out of Scope

- Centralized telemetry of ignored findings back to a global server (this violates the local-first philosophy).
- Fine-tuning models directly based on the ignore file.
