## Current State

The telemetry usage-ping endpoint (`defaultTelemetryEndpoint`, cmd/atcr/telemetry.go:19) and
the `--sync-cloud` push destination are compiled-in placeholders, both empty / explicitly
"not operational." With an empty endpoint every `telemetry.Client.Send` is an unconditional
no-op and no cloud-sync push targets a real server. Two hardening items surfaced during the
28.0 code review are correctly deferred until that backend exists, because they cannot be
specified or validated without it.

## Deferred Items (from Epic 28.0 code review, via /resolve-td clarifications 2026-07-16)

### 1. HMAC-harden HashPersonaID (TD-007)

**Source TD:** `internal/scorecard/telemetry.go:12` (HIGH; reviewers dax, kai, mira, pace)

**Problem:** `HashPersonaID` pseudonymizes a raw Persona ID with an UNSALTED SHA-256. Persona
IDs are a small, enumerable, often publicly-known set, so the digest is dictionary/rainbow
reversible — the "anonymized aggregation" privacy claim on the Persona Leaderboard is
therefore misleading.

**Intended fix:** Replace the unsalted digest with a keyed HMAC-SHA256 using a provisioned,
rotatable application secret (a server-side keyed hash is acceptable per the KB entry); add a
`TD-007` tracking label to the function.

**Why deferred (clarification Q1, 90% confidence):** Keep deferred until the real telemetry
endpoint decision. There is no live backend yet to define secret ownership, rotation, or
distribution for an HMAC pepper; provisioning now would only churn the AC-pinned
unsalted-SHA-256 digest values without a corresponding consumer.

### 2. Drain the telemetry client before process exit

**Source TD:** `cmd/atcr/main.go:52` (MEDIUM; two independent reviewers)

**Problem:** `telemetry.Client.Send` dispatches a detached goroutine (`WithoutCancel` + 3s
timeout) but nothing drains it before `main()` calls `os.Exit`. Production never calls
`client.Wait()` (the `Wait()`/`wg` seam exists but is test-only). Once a real endpoint is
wired the 3s request will almost never complete, dropping the ping on essentially every run.

**Intended fix:** Wire a bounded `telemetryClient.Wait()` (short cap ~1-2s) into `main()`
before `os.Exit` (or the shutdown path) so an in-flight ping can complete without blocking
indefinitely; the existing `Wait()`/`wg` seam was built for exactly this.

**Why deferred (clarification Q8, 85% confidence):** No code change needed now.
`defaultTelemetryEndpoint` is `""`, making every `Send` an unconditional no-op, so there is
nothing in flight to drain in production today. Revisit once a real endpoint is wired and
pings are actually in flight. This also needs a design decision on exposing the client to
`main()` without introducing a package-level singleton (the client is deliberately created
inside `newRootCmd()` and not a global).

## Objective

When a live telemetry / cloud-sync backend is provisioned, harden the persona-hash privacy
contract and ensure background pings actually complete before process exit.

## Acceptance Criteria

- [ ] AC1: `HashPersonaID` uses a keyed HMAC-SHA256 with a provisioned, rotatable secret; the
  secret's ownership, rotation, and distribution are documented. The `TD-007` label is on the
  function and the AC-pinned digest fixtures are updated to the new keyed values.
- [ ] AC2: A bounded `telemetryClient.Wait()` (~1-2s cap) runs before `os.Exit` in `main()`,
  exposed without a package-level singleton, so an in-flight ping can complete; a hung endpoint
  never blocks shutdown past the cap.
- [ ] AC3: With a real (test) endpoint configured, a review/reconcile run's usage ping is
  observed to be delivered (no longer dropped by immediate exit).

## Risks

| Risk | Mitigation |
|------|------------|
| Changing HashPersonaID digests breaks pinned fixtures | Update fixtures in the same change; the digest is intentionally not a stable public contract pre-activation |
| Exposing the telemetry client to main() reintroduces a global | Return it from newRootCmd or thread via a small accessor, not a package var |
| Bounded Wait still hangs shutdown | Cap the drain with a context/timer, never an unbounded Wait |
