# Sprint Plan 7.0.1: Executor Model Configuration

- **Estimated time**: 3 days
- **Tasks**: 4
- **Components**: 1

## Context

Sprint 7.0 introduced the **Executor Model** for generating inline code fixes. While the baseline functionality is powerful, it currently relies on a static, hardcoded prompt and default LLM API parameters (like temperature). 

## Problem Statement

A major pain point in automated remediation is style mismatch. If a project uses tabs but the executor model defaults to spaces, the generated fix will fail linting. If the team enforces specific architectural rules (e.g., "never use `lodash`"), the model might introduce a violation. Furthermore, using default temperatures (e.g., 0.7) can lead to unpredictable or non-deterministic code fixes.

## Proposed Solution

Extend the `executor` configuration schema (introduced in 7.0) to allow developers to tightly control the fix generation behavior:

1. **Temperature Control:** Allow configuring the API temperature (defaulting to `0.0` or a very low value to ensure deterministic, highly predictable code fixes).
2. **System Prompt Override:** Allow users to define a custom system prompt for the fixer persona.
3. **Coding Rules / Style Guidelines:** Allow passing an array of strings (e.g., `rules: ["Use tabs for indentation", "Avoid panic() in Go"]`) that are concatenated into the executor's context.

## User Personas

### Primary: Lead Architect (Alistair)
- Wants to ensure that auto-generated fixes adhere strictly to the project's established coding standards and pass CI linters without manual intervention.

## User Journeys

### Configuring the Fixer
1. Alistair opens `atcr.yaml` (or `registry.yaml`).
2. Adds a `rules` array under the `executor` block specifying project formatting guidelines.
3. Sets `temperature: 0.0`.
4. Runs `atcr review`. The generated fixes now perfectly match the surrounding codebase syntax.

## Acceptance Criteria

- [ ] AC1: The `Executor` schema supports `temperature` (float), `system_prompt` (string), and `rules` (array of strings).
- [ ] AC2: If `temperature` is specified, it is passed correctly to the underlying LLM provider client.
- [ ] AC3: If `rules` are provided, they are appended to the executor's context prompt prior to fix generation.
- [ ] AC4: Unit tests verify that the configuration values correctly hydrate the request payloads.

## Success Criteria

### Functional
- [ ] Users can dictate formatting rules without modifying ATCR source code.
- [ ] Temperature defaults to a deterministic low value if omitted.

### Business
- [ ] Reduces the number of "Lint Failed" CI runs resulting from ATCR-generated auto-fixes.

- Must map temperature controls abstractly so they work across Anthropic, OpenAI, and Google provider implementations.

## Out of Scope
