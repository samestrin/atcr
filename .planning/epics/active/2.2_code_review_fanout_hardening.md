# Epic Plan 2.2: Code Review Fan-out Hardening

- **Estimated time**: 2-3 days

## Context

The code review fan-out skill uses `~/.config/llm-tools/agents/registry.yaml` to orchestrate multi-agent code reviews. Each agent has a specialized role (correctness, performance, style, etc.) defined in its `system_prompt`. The fan-out sends the same diff to all agents, collects their `td-stream.txt` outputs, and merges them into a unified findings list.

Recent reconciles revealed two critical failures in this pipeline:

1. **Reviewer name hallucination**: The fan-out trusts the model's self-reported reviewer name (from the REVIEWER column in `td-stream.txt`) instead of stamping the registry agent key. When `bob2-backup` (NVIDIA-Nemotron-3-Super-120B-A12B) ran, it wrote "ace" as its reviewer name instead of "bob2-backup", making it impossible to trace findings back to the actual model.

2. **Role hallucination / runaway output**: `bob2-backup` was configured as a performance reviewer but produced 521 findings (498 blank-line style items), flooding the merged stream with noise. All other reviewers produced 0-4 findings each. The 498 style findings from one weak model drowned out the 3 HIGH and 8 MEDIUM findings from stronger models.

## Problem Statement

The fan-out pipeline has no guardrails against:
- **Unconstrained output volume**: A single agent can produce 500+ findings, overwhelming the merge/dedupe step and the TD README.
- **Role drift**: Models ignore their `system_prompt` role specialization and review outside their scope.
- **Identity drift**: Models invent reviewer names instead of using their assigned identity.

These failures waste tokens (generating findings that get discarded), clutter the TD README (486 blank-line items), and make it impossible to attribute findings to the correct model for quality tracking.

## Proposed Solution

Three-layer defense:

### Layer 1: Fan-out post-processing (enforcement)
- **Stamp reviewer name**: After collecting each agent's `td-stream.txt`, overwrite the REVIEWER column with the agent key from `registry.yaml`. Do not trust the model's self-reported name.
- **Enforce output caps**: Read `max_findings` from registry (if set) and truncate the agent's output to that many rows. Log a warning if truncated.
- **Enforce severity floor**: Read `min_severity` from registry (if set) and drop findings below that threshold. Log dropped count.

### Layer 2: Registry configuration (per-agent constraints)
Add optional fields to `registry.yaml`:

```yaml
agents:
  bob2-backup:
    provider: litellm
    model: NVIDIA-Nemotron-3-Super-120B-A12B
    temperature: 0.3
    lane: parallel
    scope: ["performance", "efficiency"]  # Restrict review focus
    min_severity: "MEDIUM"                # Drop LOW findings
    max_findings: 20                      # Cap output volume
    system_prompt: |
      You are Bob2, a code review agent specializing in performance.
      ...
```

| Field | Type | Purpose | Default |
|-------|------|---------|---------|
| `scope` | `string[]` | Categories this agent should focus on. Injected into prompt. | All categories |
| `min_severity` | `enum` | Minimum severity to report. Findings below this are dropped in post-processing. | `LOW` |
| `max_findings` | `int` | Maximum findings this agent can emit. Output truncated if exceeded. | Unlimited |

### Layer 3: Model selection (avoid weak models for open-ended review)
- Document which models are suitable for open-ended code review vs. narrow scopes.
- `bob2-backup` (Nemotron) demonstrated it cannot follow role instructions â€” remove from fan-out or constrain to `min_severity: HIGH` + `max_findings: 5`.
- Strong models (qwen-3.7-plus, deepseek-4, kimi-k2.6) can handle broader scopes.

## Requirements

### Functional
- Fan-out stamps reviewer name from `registry.yaml` agent key onto each `td-stream.txt` row after collection.
- Fan-out reads `scope`, `min_severity`, `max_findings` from registry (if present) and enforces them in post-processing.
- Fan-out logs warnings when output is truncated or findings are dropped.
- Reconcile skill correctly attributes findings to the registry agent name, not the model's self-reported name.

### Non-Functional
- Post-processing adds < 1 second per agent (deterministic filtering, no LLM calls).
- No breaking changes to existing `td-stream.txt` format â€” new columns are optional.
- Registry.yaml remains backward-compatible â€” new fields are optional with sensible defaults.

## Technical Approach

### Option A: Post-processing in fan-out skill (recommended)
- After the fan-out collects each agent's `td-stream.txt`, run a post-processing step:
  1. Parse each row.
  2. Overwrite REVIEWER column with agent key.
  3. Filter by `min_severity` (if set).
  4. Truncate to `max_findings` (if set).
  5. Write back to `td-stream.txt`.
- Pros: No changes to agent prompts or model behavior. Deterministic. Works with any model.
- Cons: Wastes tokens generating findings that get discarded.

### Option B: Prompt injection (scope enforcement)
- Read `scope` from registry and inject into the agent's system prompt before sending to the model.
- Example: "You are Bob2, specializing in performance and efficiency. Only report findings in these categories. Do not report style, naming, or formatting issues."
- Pros: Reduces wasted tokens (model doesn't generate out-of-scope findings).
- Cons: Model-dependent â€” weak models may still ignore the instruction.

### Option C: Both A + B (defense in depth)
- Inject scope into prompt (Option B) to reduce wasted tokens.
- Post-process to enforce caps and stamp reviewer name (Option A) as a safety net.
- Pros: Best of both worlds. Prompt guides the model; post-processing enforces hard limits.
- Cons: Slightly more complex fan-out logic.

**Recommended: Option C**

## Implementation Components

1. **Fan-out post-processing module**
   - Function: `postprocess_td_stream(agent_key, td_stream_path, registry_config)`
   - Steps:
     1. Parse `td-stream.txt` into rows.
     2. For each row: overwrite REVIEWER column with `agent_key`.
     3. Filter rows by `min_severity` (if set in registry).
     4. Truncate to `max_findings` (if set).
     5. Write back to `td-stream.txt`.
   - Log: "Agent {agent_key}: {original_count} findings â†’ {filtered_count} after post-processing (dropped {dropped_count}, truncated {truncated_count})"

2. **Registry schema extension**
   - Add optional fields: `scope`, `min_severity`, `max_findings`.
   - Update registry.yaml parser to read these fields.
   - Defaults: `scope = []` (all categories), `min_severity = "LOW"`, `max_findings = null` (unlimited).

3. **Prompt injection hook**
   - Before sending agent's system prompt to the model, check if `scope` is set.
   - If set, append: "Focus exclusively on these categories: {scope}. Do not report findings outside these categories."
   - This is a soft constraint â€” post-processing enforces the hard limit.

4. **Reconcile skill update**
   - No changes needed â€” the reconcile skill already reads the REVIEWER column from `td-stream.txt`. Since the fan-out now stamps the correct reviewer name, the reconcile skill will attribute findings correctly.

5. **Registry.yaml updates**
   - Add `scope`, `min_severity`, `max_findings` to `bob2-backup`:
     ```yaml
     bob2-backup:
       scope: ["performance", "efficiency"]
       min_severity: "MEDIUM"
       max_findings: 20
     ```
   - Optionally add constraints to other weak models (gemma-4-31b, nemotron-3-nano).

## Constraints

- Must remain backward-compatible with existing `registry.yaml` files that don't have the new fields.
- Post-processing must be deterministic (no LLM calls) â€” it's a simple filter/truncate operation.
- The fan-out skill must not break if `scope`, `min_severity`, or `max_findings` are missing.

## Out of Scope

- Changes to the reconcile skill's deduplication logic (it already works correctly once reviewer names are stamped).
- Changes to the TD README format (it already supports arbitrary reviewer names).
- Model-specific prompt tuning (that's a separate effort â€” this epic adds the infrastructure to enforce constraints, not the prompts themselves).
- Removing weak models from the registry (that's a policy decision â€” this epic adds the guardrails to constrain them).

## Risks

| Risk | Mitigation |
|------|------------|
| Post-processing breaks existing fan-out workflows | New fields are optional with sensible defaults; post-processing is a no-op if fields are missing. |
| Prompt injection weakens model output quality | Scope is a soft constraint; post-processing enforces hard limits. If prompt injection causes issues, disable it and rely on post-processing only. |
| Weak models still produce low-quality output even with constraints | Document model suitability in registry comments; recommend removing chronically underperforming models from the fan-out. |

## Rollback Plan

Post-processing is additive. If issues occur:
1. Remove `scope`, `min_severity`, `max_findings` from registry.yaml (fan-out falls back to defaults: no constraints).
2. Disable reviewer name stamping (fan-out uses model-reported names as before).
3. Existing fan-out workflows continue unchanged.
