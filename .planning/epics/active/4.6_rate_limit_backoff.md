# Epic Plan 4.6: Robust Rate-Limit & Backoff Handling

**Estimated Durations**: 1 week  
**Status:** Draft (renumbered 4.10 → 4.6 on 2026-06-19 so it executes immediately after Epic 4.5; depends on 4.5)

## Objective

Implement a unified, robust 429/Rate-Limit handling and backoff strategy across all LLM providers (Anthropic, OpenAI, Google).

## Context

As the number of models in the reviewer panel grows, and the number of chunks being processed in parallel increases, ATCR easily hits rate limits (429s). Currently, failure handling is inconsistent and lacks a coordinated backoff strategy, resulting in dropped chunks or failed reviews.

## Problem Statement

1. Hitting a 429 often causes an entire review chunk to fail or be silently dropped.
2. There is no unified retry-after logic across different provider SDKs.

## Proposed Solution

- Implement a central `RetryMiddleware` for all LLM client calls.
- Intercept 429 responses and read `Retry-After` headers if available.
- Use Exponential Backoff with Jitter for retries up to a configurable `max_retries` (default: 5).
- Expose circuit-breaking if backoff fails continuously to fail-fast the chunk gracefully — reusing Epic 4.5's breaker, not a second mechanism.

### Boundary with Epic 4.5 (Circuit Breaker)

Both epics live on the `internal/llmclient` error path and must not double-count 429s:

- A 429 is handled **here** (backoff + `Retry-After`) and, per 4.5 **AC10**, does **not** trip 4.5's breaker while retries remain.
- Only once `max_retries` is exhausted does the chunk surface a terminal failure; whether that terminal failure counts toward 4.5's breaker is the explicit hand-off point (default: it does not, since a sustained 429 is rate-limiting, not an outage — confirm during design).
- 5xx/timeout failures bypass this epic's 429-specific path and flow to 4.5's breaker as normal.

## Acceptance Criteria

- [ ] All LLM calls route through the retry middleware.
- [ ] 429 errors trigger exponential backoff with jitter.
- [ ] The system respects `Retry-After` headers from providers.
- [ ] Max retries and base delays are configurable.
- [ ] A chunk is only marked as failed after `max_retries` is exhausted.

## Dependencies

- **Epic 4.5 (Circuit Breaker)** — co-located on the `internal/llmclient` error path; this epic (4.6) executes immediately after 4.5. The 429/4xx boundary is defined by 4.5 AC10 (see "Boundary with Epic 4.5" above).
- New tunables (`max_retries`, base delays) extend the config-validation surface from **Epic 4.2** (config validation, complete) — update its schema/validation in lockstep.

## Clarifications

### Recorded 2026-06-19 (rubber duck — /execute-epic)

**Key Decisions:**
- The central retry engine already exists — `dispatch()` at `internal/llmclient/client.go:408` implements exponential backoff, jitter, 429 handling (`retryableStatus`), and verbatim `Retry-After` honoring (`parseRetryAfter`). Epic 4.6 is **expose the existing engine via config + reconcile the default**, NOT build a new/separate `RetryMiddleware` type (a second mechanism would duplicate working code and contradict the epic's own boundary note).
- Retry tunables live at **both** tiers, per-agent overriding global, mirroring the existing `TimeoutSecs` pattern: `Registry.MaxRetries *int` / `Registry.InitialBackoffMs *int` (global), `AgentConfig.MaxRetries *int` / `AgentConfig.InitialBackoffMs *int` (per-agent), `Settings.MaxRetries int` / `Settings.InitialBackoffMs int` (resolved), seeded in `ResolveSettings`, with `EffectiveMaxRetries(s)` / `EffectiveInitialBackoffMs(s)` override methods.
- Expose **only** `max_retries` and `initial_backoff_ms`. `backoff_factor` (1.5) and the `maxBackoff` cap stay as implementation constants — the AC does not require tuning them.
- Default `max_retries = 5` is applied at the **embedded/Settings tier in `ResolveSettings`**, NOT by changing the `defaultMaxRetries = 2` constant in `client.go`. The constant stays the naked-`New()` fallback (doctor self-test, bare/test clients); production review/resume paths get 5 by threading resolved Settings into `WithRetry()` at construction. Default `initial_backoff_ms = 500` likewise lives at the Settings tier.
- The terminal-failure → breaker boundary is **unchanged**: an exhausted 429 returns `Transient` wrapping `HTTPStatusError{Status: 429}`; `isBreakerFailure` returns false (429 < 500) so `send()` calls `RecordSuccess()` (`client.go:354`). Sustained rate-limiting never trips Epic 4.5's breaker — matches the epic default. No code change.

**Scope Boundaries:**
- IN scope: `max_retries` + `initial_backoff_ms` config fields on Registry + AgentConfig (YAML + validation mirroring `TimeoutSecs`); Settings resolution with embedded defaults (5 / 500ms) + `Effective*` override methods; wiring resolved retry settings into the production `llmclient.New(WithRetry(...))` construction at the review/resume paths.
- OUT of scope: a separate `RetryMiddleware` type; exposing `backoff_factor` / `max_backoff_ms`; changing provider SDKs (there is one shared OpenAI-compatible client, no per-provider SDKs to unify); any change to the circuit-breaker boundary; changing the `defaultMaxRetries = 2` constant.

**Technical Approach:**
- Mirror the `TimeoutSecs` shape exactly across `internal/registry/config.go` (structs + `validate()` + `applyDefaults`) and `internal/registry/precedence.go` (constants + `Settings` + `ResolveSettings` + `Effective*`).
- `WithRetry(maxRetries int, initialBackoff time.Duration, factor float64)` already exists; pass `EffectiveInitialBackoffMs` as `time.Duration(ms) * time.Millisecond` and keep `factor` = the existing `defaultBackoffFactor` constant.
- Thread resolved `Settings` into the production review/resume client construction; leave the doctor/serve bare-`New()` fallback on the built-in `2` default unless those paths already resolve Settings.
