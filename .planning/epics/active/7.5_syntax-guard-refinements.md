## Current State

The Epic 7.1 local syntax guard (`internal/verify/syntaxguard.go`) parses a generated fix with `go/parser` and flags it only when the content is plausibly Go code yet fails to parse. The guard is deliberately calibrated for **conservative recall**: it prefers a false negative (a malformed fix passes unflagged) over a false positive (a legitimate fix is flagged), because a false "invalid" flag degrades trust — the exact failure mode the guard exists to prevent.

`looksLikeGoCode` keys on line-structural signals (declaration keyword, trailing open brace, leading close brace). Fenced non-Go content is already suppressed via the `nonGoFenceLangs` map. The residual gap is **unfenced** non-Go content with block braces.

## Pain Points

### Unfenced JSON/config false positive (residual)
An unfenced, multi-line JSON or config snippet with block braces (`{` at line end, `}` at line start) satisfies `looksLikeGoCode`, fails to parse as Go, and is flagged `invalid_syntax` — a spurious flag on non-Go content. This is the residual false-positive surface after the 7.1 heuristic hardening. The common LLM output shape (a fenced ```` ```json ```` block) is already handled by `nonGoFenceLangs`; only the unfenced case remains, which is narrow.

## Proposed Solution

Detect obviously non-Go brace content before treating block braces as a Go code signal — e.g. recognize JSON-object / `key: value` line shapes (quoted keys, trailing commas, no Go declaration keywords) and exclude them from `looksLikeGoCode`. The detection must stay conservative: it should only suppress flagging (reduce false positives), never add new flagging, so it cannot regress the 7.1 trust guarantee.

## Scope

### In Scope
- Non-Go brace-content detection (JSON object / key:value lines) ahead of the block-brace Go signal in `looksLikeGoCode`
- Characterization tests for unfenced JSON/config fixtures asserting NoError
- Confirmation that all existing 7.1 syntax-guard tests still pass

### Out of Scope
- Multi-fence validation (validating fences beyond the first) — separately recorded as an accepted limitation (WONTFIX) in `extractFencedCode`'s doc comment
- Tightening `parseGoFix`'s three OR-strategies — recorded WONTFIX (conservative-recall by design)
- Any change that increases flagging / false positives
