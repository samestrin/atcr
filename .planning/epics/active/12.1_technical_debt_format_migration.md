# 12.1 Technical Debt Format Migration

- **Estimated time**: 2 days
- **Tasks/Components**: 3/1

## Context

Our team tracks technical debt in a massive Markdown table within `.planning/technical-debt/README.md`. As the project grows, this single file has become unwieldy, leading to developers skipping crucial details when documenting issues.

## Problem Statement

The current Markdown table format for tracking technical debt is hostile to long-form text. It encourages brevity, which frequently results in "won't fix, description too vague" closures. It also increases the likelihood of nasty merge conflicts when multiple developers add or modify items simultaneously.

## Proposed Solution

Migrate the single `.planning/technical-debt/README.md` file to a more robust, structured format. We will transition to a directory of YAML files (or a single JSONL file) that natively supports multi-line text descriptions, allowing developers to fully document the context, problem, and potential solution of each technical debt item without formatting restrictions.

## Requirements

### Functional
- Replace the Markdown table with a scalable storage format (e.g., individual YAML files per issue).
- Provide a simple script or ATCR command to generate a human-readable summary if needed.
- Preserve all existing technical debt items from the current table.

### Non-Functional
- Zero data loss during the migration.
- The new format must be easily readable and writable by humans.
- The new format must allow unconstrained multi-line strings for descriptions.

## Technical Approach

### Option A: Directory of YAML/Markdown Files (Recommended for 12.1)
- Create `.planning/technical-debt/items/`.
- Each technical debt item becomes an individual file (e.g., `TD-001-update-package.yaml` or `.md` with YAML frontmatter).
- Pros: Eliminates merge conflicts, supports full Markdown descriptions.

### Option B: Single JSONL File
- Pros: Easy for ATCR to parse and generate natively.
- Cons: Slightly harder for humans to edit multi-line strings manually compared to YAML.

*Decision for 12.1: Implement the Directory of YAML/Markdown files as it solves the immediate developer experience pain point without requiring major tooling changes.*

## Implementation Components

1. **Migration Script**
   - A one-off script to parse the existing Markdown table and output individual files.

2. **Schema Definition**
   - Define the required fields (ID, Status, Priority, Component, Description, etc.) for the new format.

3. **Format Update**
   - Update documentation to reflect the new process for logging technical debt.

## Constraints

- Existing tooling or review loops must not break during the migration.
- All columns from the current table must be accurately mapped to the new schema.

## Out of Scope

- Automated resolution of technical debt items (deferred).
- Complex CLI tooling (`atcr debt generate`, `atcr debt dashboard`) — deferred to Epic 18.0.

## Acceptance Criteria

- [ ] AC1: A script is created that successfully parses the current Markdown table.
- [ ] AC2: All existing technical debt items are converted to the new format with no data loss.
- [ ] AC3: The new format supports multi-line, detailed descriptions.
- [ ] AC4: The old `README.md` is replaced or updated to point to the new directory/format.

## Success Criteria

- [ ] Developers can log technical debt with full context without breaking table formatting.
- [ ] Merge conflicts on the technical debt log are eliminated.

## Rollback Plan

- Maintain a backup of the original `README.md` table on a separate branch.
- If the new format proves problematic, revert the migration commit.

## Security Considerations

- None. This is a documentation/planning format change.

## Timeline Milestones

1. Schema design and team agreement.
2. Migration script creation.
3. Execution and repository update.

## Clarifications

### Recorded 2026-06-26 (rubber duck — /execute-epic)

**Key Decisions:**
- ADDITIVE coexistence model. Build `.planning/technical-debt/items/*.md` as a point-in-time snapshot ALONGSIDE the existing `README.md` table. The README table stays the canonical machine-read source for all 6 Go MCP TD binaries (`group_td`, `td_stats`, `td_filter`, `td_dedupe`, `td_clean`, `td_validate`) and the 9 skills that read/write it — none of that tooling is touched in 12.1. The real cutover (making `items/` canonical + updating binaries/skills) is deferred to the follow-on epic (18.0, or renumbered 12.3).
- File format: `.md` with YAML frontmatter. Structured metadata (id, severity, status, category, file, est_minutes, source, sprint, date, group) lives in frontmatter; multi-line Problem/Fix/Notes live in the Markdown body.
- Migration tool: testable Go in a new `internal/tdmigrate/` package + a thin one-off `cmd/td-migrate` entry point (removable after the cutover; named outside Epic 18.0's planned `atcr debt` namespace). Not a shell script — the repo's TDD + 80%-coverage mandate requires testable Go.
- File naming: `TD-0001-<slug>.md`, sequential, zero-padded to 4 digits, slug derived from the problem text.
- Summary generator = the REVERSE of the migration (`items/*` → README table). This proves round-trip fidelity / zero data loss (AC2) and satisfies the "simple script/command to generate a human-readable summary" requirement without adding a separate digest. Round-trip fidelity is scoped to the DATA sections (dated `### [date] From Sprint:` item tables + Stats), not the static instructional preamble.

**Scope Boundaries:**
- IN scope: TD item schema (Go types + parse/serialize for `.md`+frontmatter and for the README table), `cmd/td-migrate` with `migrate` (README→items) and `generate` (items→README table) modes, actual generation of `items/` from the live README, a pointer/doc note in README describing the new format and process.
- OUT of scope: modifying any of the 6 Go MCP TD binaries or the 9 skills; making `items/` the canonical source; dual-write; the `atcr debt` CLI namespace (Epic 18.0/12.3); automated TD resolution.

**Technical Approach:**
- Snapshot is point-in-time: `items/` may drift from README once the next `/execute-*` run appends a row. The short coexistence window (closed by the immediate follow-on epic) is an accepted tradeoff; no dual-write in 12.1.
- Frontmatter values are simple scalars only (multi-line text stays in the body), so parsing is robust even for `file` values containing colons (e.g. `internal/stream/parser.go:42`).
