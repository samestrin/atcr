# Feature Request: AST Plugin Architecture

- **Estimated time**: 1.5 weeks
- **Tasks**: 4
- **Components**: 2
- **Execution**: execute-epic

## Context

The reconciler attempts to group findings based on line numbers. However, models are imprecise. A model might flag line 42, while another flags line 45, even though both refer to the exact same logic block.

## Problem Statement

-   **Line number drift**: Models frequently hallucinate or skew line numbers by ±3 lines. Current proximity grouping is brittle.
-   **Binary bloat**: Compiling Tree-sitter parsers for every language natively into the Go binary creates massive bloat and CGO headaches, preventing statically linked cross-compilation.

## Proposed Solution

1.  **Wasm Plugin Architecture**: Compile Tree-sitter parsers to WebAssembly (`.wasm`). The reconciler uses a lightweight Wasm runtime (e.g., Wazero) to dynamically load the `.wasm` parser only for the language currently being reviewed.
2.  **AST Isomorphism**: Map the raw line numbers flagged by models to their smallest covering Abstract Syntax Tree (AST) node.
3.  **Merkle Hashing**: Compare structural hashes of the AST nodes. If two models flag different lines but map to the same logical block, group them.

## Acceptance Criteria

- [ ] AC1: Wasm host runtime (wazero) integrated into the reconciler with zero CGO dependencies.
- [ ] AC2: Mechanism implemented to fetch and cache language-specific `.wasm` parser plugins dynamically.
- [ ] AC3: AST mapping correctly groups findings that are offset by whitespace or minor line-number drift.

## Success Criteria

### Functional
- [ ] Cross-compilation to all target architectures still works out-of-the-box (no CGO).
- [ ] AST mapping resolves 95% of off-by-one line discrepancies in the benchmark suite.

### Business
- [ ] Allows open-ended language support by just dropping in new `.wasm` files.

## Technical Constraints

- Must use Wazero or similar zero-CGO Wasm runtime for Go.

## Non-Functional Requirements

### Performance
- Wasm module instantiation must be cached so repeated parsing of the same file language is fast (<10ms overhead).

## Out of Scope
- Building Wasm parsers for every language on earth (we will start with Go and Python as proofs of concept).

## Clarifications

### Recorded 2026-06-27 (rubber duck — /execute-epic)

**Key Decisions:**
- **Parser acquisition (AC2):** Vendor the two PoC parsers (Go + Python) as `.wasm` in-repo under a `parsers/` directory and serve them via `go:embed`, mirroring the existing bundle pattern (`internal/personas/bundles.go:14`, `skill/skill.go`, `personas/personas.go`). "Fetch" = load from the embedded FS; "cache" = the wazero compiled-module instance cache (satisfies the <10ms NFR). A runtime override path (check a configurable directory first, fall back to embedded) satisfies the "drop in new .wasm files" business criterion as a non-blocking extension, not a PoC requirement. No network seam, no build-time Wasm toolchain in CI.
- **Feasibility spike-first (Q2):** Time-box ≤1 day to validate the *combined* engine+grammar `.wasm` path (built via `tree-sitter build --wasm`, which bundles the Emscripten-compiled runtime alongside the grammar) running in wazero with manually-satisfied `env` host imports — NOT a bare grammar `.wasm` (a non-starter by definition). If the Emscripten-ABI host-shim route is blocked, keep the plugin abstraction interface UNCHANGED and back each language with a pure-Go implementation (Go: stdlib `go/ast`/`go/parser`; Python: a pure-Go parser). The abstraction is the durable design asset, not the Wasm backend.
- **Grouping signal (Q3):** AST structural identity REPLACES the ±3-line proximity gate (`reconcile/cluster.go:5-7`, `lineProximity = 3`) as the PRIMARY grouping pre-filter. The ±3 window is retained only as a fallback when no `.wasm`/AST parser is available for a finding's language. Augmenting AST inside the ±3 window cannot satisfy AC3, because whitespace-induced offsets can exceed ±3 and cross-window grouping would remain structurally impossible.
- **Benchmark (Q4a):** No AST-grouping benchmark exists. Build a labeled fixture set during the epic, following 13.0's `ncd_corpus.json` pattern (~33 labeled pairs). The existing `reconcile/cluster_test.go` cases are unit-correctness tests for `Cluster()`, not an accuracy benchmark.
- **Adopt-or-revert policy (Q4b):** This is NOT 13.0's full revert. AC1+AC2 (wazero runtime + plugin caching) and the AST-mapping infrastructure have standalone downstream value — planned 13.2 (Bipartite+DBSCAN) sources edge weights from 13.1's AST isomorphism. Therefore: the wazero/plugin/AST-mapping plumbing SHIPS regardless of the benchmark outcome. ONLY the AC3 production default-flip (making AST identity the primary grouping signal in place of ±3) is gated on the benchmark validating it. If AC3 is falsified, the ±3 proximity gate remains the production default and the AST path ships as available-but-not-default infrastructure (reversible adoption posture); the plumbing is not reverted.
- **Go PoC parser (Q5):** Use tree-sitter-go.wasm through the plugin path for the Go PoC, subject to the Q2 spike gate. Both Go and Python implement the same Wasm-backed path; stdlib `go/ast` is the fallback only if the spike fails, not the default. The existing `go/ast` use in `internal/verify/syntaxguard.go:4-6` (LLM-fix syntax validation) is a separate concern and must not be conflated with reconciler AST isomorphism.

**Scope Boundaries:**
- IN scope: Go + Python PoC parsers; wazero host runtime with compiled-module cache; `go:embed`-vendored `.wasm`; AST line→smallest-covering-node mapping; Merkle structural hashing; AST-identity grouping replacing ±3 as primary signal; a labeled benchmark fixture set for AC3.
- OUT of scope: parsers for any language beyond Go + Python; a live network download mechanism for `.wasm` (embedded-only for the PoC; runtime-override dir is an optional extension); a CI Wasm build toolchain.

**Technical Approach:**
- wazero is the runtime (hard constraint); zero-CGO verified by a cross-compile check.
- Reconciler reads source from disk at `{Options.Root}/{finding.File}` during the dedupe phase to parse the AST (source is not currently passed in; it is available on disk).
- Plugin interface is the stable seam; Wasm vs pure-Go is an implementation detail behind it.
- Spike outcome (Wasm-backed vs pure-Go fallback per language) will be recorded in the final report and PR body.
