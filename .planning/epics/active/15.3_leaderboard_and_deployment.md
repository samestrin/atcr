# Sprint Plan 15.3: Leaderboard & Production Deployment

- **Estimated time**: 5 days
- **Tasks**: 6
- **Components**: 3
- **Execution**: execute-epic --strong

## Context

The final piece of the `atcr.dev` launch is establishing credibility (via the public model-evaluation leaderboard) and deploying the code so pilot signups can begin. This sprint integrates the evaluation data, sets up lead generation, and configures the automated deployment pipeline to static hosting.

## Problem Statement

We need to:
1. Port and format the evaluation table on the leaderboard page (`leaderboard.html`), displaying real findings and scores generated in Epic 10.0.
2. Establish the signup form on the Team Edition page (`team-edition.html`) to capture volunteer teams for the Epic 15.0 pilot program.
3. Configure static deployment pipelines (e.g., GitHub Pages or Cloudflare Pages) linked to the `atcr.dev` custom domain.
4. Finalize SEO metadata, favicons, and error pages (`404.html`).

## Proposed Solution

1. Port mockup structures for [leaderboard.html](file:///Users/samestrin/Documents/GitHub/atcr.dev/the-artifact/mockups/leaderboard.html) and [team-edition.html](file:///Users/samestrin/Documents/GitHub/atcr.dev/the-artifact/mockups/team-edition.html).
2. Populate the leaderboard ledger table using evaluation numbers derived from Epic 10.0.
3. Hook up the pilot signup form fields to a secure static form provider (such as Formspree, Netlify Forms, or Cloudflare Workers endpoint).
4. Configure DNS records for the custom `atcr.dev` domain and create a GitHub Actions deployment workflow.

## User Personas

### Primary: Software Engineer (Devin)
- Wants to compare model performance on the conformance suite to verify reviewer accuracy.
- Wishes to sign up their team for the Team Edition Pilot.

## User Journeys

### Leaderboard Review & Pilot Signup
1. Accesses `atcr.dev/leaderboard.html`.
2. Inspects model rankings, consensus scores, and benchmark counts.
3. Clicks the "Team Edition" navigation link to read the validation program criteria.
4. Fills out the pilot application form (team size, stack, pain points, email) and hits submit.
5. Receives a friendly confirmation screen.

## Acceptance Criteria

- [ ] AC1: The leaderboard page correctly displays ranked models with agreement metrics, structured columns, and no styling layout bugs.
- [ ] AC2: The Team Edition signup form handles field validations (e.g., email syntax, mandatory fields) and submits payload securely via HTTPS.
- [ ] AC3: Submitting the pilot form successfully records lead information and presents a clean thank-you confirmation state.
- [ ] AC4: The custom domain `atcr.dev` resolves properly with active HTTPS certificates.
- [ ] AC5: Merge events to the `main` branch of the `atcr.dev` repo trigger a GitHub Action that deploys the static files within 2 minutes.

## Success Criteria

### Functional
- [ ] Form submit round-trip takes less than 2 seconds.
- [ ] Static pages load correctly over IPv4 and IPv6 networks.

### Business
- [ ] Form inputs are structured cleanly so the pilot team coordinates can be exported easily in CSV/JSON format for validation tracking.

## Technical Constraints

- The deployment pipeline must run as an automated GitHub Action.
- All secrets (such as Formspree/API target IDs) must be injected securely or kept client-safe via obfuscation or proxy configurations.

## Out of Scope

- Building a private dashboard for signup management (spreadsheet/email lists are sufficient for validation).
- Live performance updating (updates will be batched manually or scheduled weekly).

## Dependencies

- Conformance benchmarking data generated in Epic 10.0.
- Custom domain registration (`atcr.dev`) ownership and DNS provider access.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Lead capture form gets targeted by spam bots | Medium | Medium | Integrate simple client-side honeypot fields or basic form provider bot protection filters |
| DNS propagation delay | Low | Low | Set short TTL values on the DNS records prior to switching the nameservers or custom domains |

- ⏸️ **Missing target repository / cross-repo execution mismatch (most severe finding):** This epic file lives in the `atcr` repo's `.planning/epics/active/`, so `/execute-epic` will resolve REPO_ROOT to `atcr` (the Go engine codebase) and will branch/commit/PR *within that repo*. But every deliverable here — `leaderboard.html`, `team-edition.html`, the GitHub Actions deploy workflow, the DNS records — targets the separate `~/Documents/GitHub/atcr.dev` repository (per parent Epic Index 15.0 AC1: "integrated into the `atcr.dev` repository"). Verified that `atcr.dev` currently has **no git remote configured at all** (`git remote -v` returns empty) and **no `.github/workflows/` directory** — so AC5 ("Merge events to the `main` branch of the `atcr.dev` repo trigger a GitHub Action") requires infrastructure that doesn't exist yet: pushing `atcr.dev` to GitHub, wiring Pages/Cloudflare Pages hosting, and DNS registrar access for the custom domain. `/execute-epic`'s single-repo branch → TDD → PR → merge flow cannot create a GitHub remote, configure DNS, or provision HTTPS certificates — none of that is a code change a RED→GREEN test can verify. This is the same root cause already flagged in Sprint 15.1's `/refine-epic` run (2026-07-02) and is more acute here since 15.3's own AC4/AC5 are infra/DNS/hosting acceptance criteria, not just file output. Suggested action: state explicitly where in `atcr.dev` the deploy workflow and DNS config should live, confirm `atcr.dev` has been pushed to a GitHub remote first, and decide whether this epic should instead run as a session scoped to the `atcr.dev` repo itself, or route through `/init-plan` if cross-repo output can't be avoided.
- ⏸️ **Hosting/form-provider contradiction:** The plan offers "GitHub Pages or Cloudflare Pages" for static hosting (Proposed Solution step 4, Technical Constraints) and, independently, "Formspree, Netlify Forms, or Cloudflare Workers endpoint" for the signup form (Proposed Solution step 3) as if freely combinable. Netlify Forms only processes submissions on sites actually built/deployed through Netlify's own pipeline — it does nothing on a GitHub Pages or Cloudflare Pages deployment. Suggested action: drop "Netlify Forms" from the option list unless Netlify is also chosen as the static host, or state explicitly that the hosting and form-provider choices are coupled.
- ⏸️ **Task count mismatch:** Header claims "Tasks: 6," but Problem Statement / Proposed Solution enumerate exactly 4 parallel top-level items (leaderboard port+populate, team-edition port+form, deployment pipeline, SEO/favicon/404) with no stated sub-task breakdown. Suggested action: either expand Proposed Solution to explicitly enumerate all 6 tasks, or correct the header to "Tasks: 4" (or 5–7, depending on how the leaderboard-port/populate and DNS/GitHub-Actions pairs are split).
- ⏸️ **team-edition.html mockup has no form to "hook up":** Verified `the-artifact/mockups/team-edition.html` directly — it contains bare `<input>` elements (email, team size) with no `<form>` wrapper, no `name=` attributes, no submit button, and no honeypot field. Proposed Solution step 3 ("Hook up the pilot signup form fields to a secure static form provider") implies an existing form just needs an `action` attribute wired up, but the actual mockup requires building the form structure (tag, field names, submit handler, confirmation state, honeypot per the Risks table) from scratch. Suggested action: reword step 3 to "build and wire" rather than "hook up," and add the honeypot field explicitly as a task deliverable.
- ⏸️ **No 404.html design source exists:** Problem Statement item 4 calls for finalizing "error pages (404.html)," but `the-artifact/mockups/` (the plan's cited design source, verified via `llm_support_multiexists`) contains no `404.html` — unlike the leaderboard and team-edition pages, this task has no mockup to port from. Suggested action: either add a task to design the 404 page from scratch (reusing `styles.css`/nav from the ported pages), or clarify it's out of scope for this sprint.
- ⏸️ **Existing `atcr leaderboard --export` command not referenced:** `cmd/atcr/leaderboard.go` (verified via grep) already implements a `leaderboard --export` subcommand that writes the "public v1 submission JSON" — exactly the Epic-10.0-derived data payload Proposed Solution step 2 needs ("Populate the leaderboard ledger table using evaluation numbers derived from Epic 10.0"). The plan doesn't mention this command or any data hand-off mechanism between the `atcr` engine repo and the `atcr.dev` site repo. Suggested action: name `atcr leaderboard --export` explicitly as the data source and describe how its JSON output gets transcribed (manually or via a build script) into the static HTML table.
