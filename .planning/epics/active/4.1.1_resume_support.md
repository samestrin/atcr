# Epic 4.1.1: Resume Support

**Status:** Draft  
**Created:** 2026-06-17  
**Priority:** Medium-High (efficiency, developer experience, cost reduction)

## Problem

When a code review panel is aborted (via Ctrl-C/SIGINT), fails due to transient network issues, or gets cancelled because of token budget/context timeouts, the user is left with a partially populated review folder. If they want to complete the review panel, they must rerun the review from scratch. This causes three main issues:

1. **Wasted Cost and Tokens:** LLM API tokens for previously completed agent personas are consumed again on the rerun.
2. **Delayed Feedback Loops:** Rerunning the entire panel (especially in multi-agent environments) takes significantly longer than only running the failed or interrupted agents.
3. **Loss of Consistent Context:** A completely new run might get different responses from non-deterministic models even on unchanged code, altering the original partially generated panel consensus.

## Root Cause

The `atcr review` CLI subcommand does not check for existing partial results or allow targeting an existing review ID directory to resume work. It always generates a new unique review ID, creates a new directory, and fans out requests to the entire roster of agents configured in `.atcr/config.yaml`.

## Solution

Introduce a `--resume <id-or-path>` flag on the `atcr review` command. This flag allows `atcr` to point to an existing review folder, parse the results of already-completed agents, filter the fanout queue to only run the remaining/pending/failed agents, and then write the new findings directly into the existing review directory structure before running reconciliation.

### Core Design

1. **CLI Flag Resolution:**
   ```bash
   atcr review --resume latest           # Resolves path from .atcr/latest
   atcr review --resume <id>             # Resolves .atcr/reviews/<id>
   atcr review --resume ./custom/path    # Resolves from explicit directory path
   ```

2. **Roster Filtering Logic:**
   - Load the `status.json` or `manifest.json` in the target directory to verify the review state.
   - If the file range (base/head git shas or payload content hashes) does not match the current working tree, abort the resume and print a warning.
   - Scan the `sources/` subdirectories. If `sources/<agent_name>/findings.txt` exists and is non-empty, mark that agent as **Completed**.
   - Filter the pool of agents to run: `ActiveAgentRoster = ConfiguredRoster - CompletedRoster`.

3. **Incremental Saving:**
   - Execute only the remaining agents.
   - Save the results into `sources/<agent_name>/findings.txt` inside the target directory.
   - Update `status.json` status to `"completed"` once the remaining queue successfully finishes.
   - Automatically trigger a fresh reconciliation (`atcr reconcile <dir>`) upon successful completion.

## Scope

### In Scope

- Adding the `--resume <id-or-path>` flag to `atcr review`.
- Review state detection (parsing `status.json` / `manifest.json` and scanning existing `sources/` directory).
- Roster filtering to skip already completed agents.
- Incremental writing of newly completed agent findings to the existing folder.
- Validation checks to ensure the git range (`base` and `head`) and files have not changed since the interrupted run.
- Integration tests simulating a partial run, interruption, and resume pipeline.

### Out of Scope

- Modifying the agent panel configuration mid-resume (the configuration must match the original panel).
- Interactive selection of which agents to resume (it resolves automatically based on missing or empty outputs).
- Support for resuming when the source code or git diff payload has changed (requires a new review run).

## Acceptance Criteria

- [ ] **AC1:** Running `atcr review --resume latest` resolves the latest review directory path from `.atcr/latest`.
- [ ] **AC2:** If all agents in the target review directory are already completed, the command exits cleanly with: `All configured agents already completed. Re-running reconciliation...` and triggers `atcr reconcile`.
- [ ] **AC3:** The tool compares the target git base/head range of the interrupted review. If the current workspace git range has changed, the resume operation aborts with a clear error and exit code **2**.
- [ ] **AC4:** The fanout engine skips any agent that has a non-empty `findings.txt` in its corresponding source directory, only fanning out to the pending/failed agents.
- [ ] **AC5:** As resumed agents complete, their findings are written cleanly to `sources/<agent>/findings.txt` and `status.json` is updated incrementally.
- [ ] **AC6:** When all resumed agents finish successfully, the review status is updated to `"completed"`.
- [ ] **AC7:** If the resumed run is interrupted again, the new partial findings are preserved, and `status: "interrupted"` is maintained.
- [ ] **AC8:** Unit tests cover roster filtering based on pre-existing findings files.
- [ ] **AC9:** Integration test: runs a mock review, interrupts it after 2 agents, runs `--resume`, and verifies all 4 agents have results and the final status is `"completed"`.

## Success Criteria

- [ ] Users can resume a failed or interrupted review without spending tokens on already completed agent queries.
- [ ] Review state is preserved atomically even if resumed runs are interrupted multiple times.
- [ ] CLI warns users when trying to resume a review where the underlying codebase diff has changed.

## Implementation Plan

### Phase 1: State Inspection & CLI Flags (1.5 days)
1. Add the `--resume` flag to the CLI (`cmd/atcr/review.go`).
2. Implement paths resolution for `latest`, IDs, or direct file directories.
3. Write functions to inspect the target folder, parsing `status.json` and analyzing the `sources/` directory structure.
4. Implement git range integrity verification (comparing original `manifest.json` git base/head hashes with current workspace).

### Phase 2: Roster Filtering & Resumed Fanout (1.5 days)
1. Modify the fanout queue generator to filter out agents whose output directories are already successfully populated.
2. Ensure the fanout engine executes with the restricted roster.
3. Update status reporting to show skipped/resumed agents.

### Phase 3: Verification & Reconciliation Integration (1 day)
1. Ensure the `status.json` and `manifest.json` are cleanly merged and updated.
2. Invoke the reconciler automatically after all resumed agents complete.
3. Write comprehensive unit and integration tests.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Code base shifts mid-resume, causing inconsistent reviews | Medium | High | Rigorous validation of git base and head SHAs; fail early if mismatch is detected. |
| Incomplete findings files are treated as complete | Low | Medium | Ensure validation checking parses the findings format and ensures files aren't just empty/corrupted before skipping. |
| Configuration changes (e.g. model swaps) break resume state | Low | Low | Compare active agent config with saved configuration in `manifest.json` and error if changes are incompatible. |

## Open Questions

1. **How should we handle partial findings that are empty (e.g. the agent completed but found no issues)?**
   - *Recommendation:* Write an explicit sentinel file or metadata field (e.g. `findings.json` with an empty array or `status.json` completion state) so we can differentiate between "unrun" and "run but found nothing".
   
2. **Should we allow resuming a run with a modified agent registry?**
   - *Recommendation:* No, it is safer to lock the panel configuration. If the registry or configured personas change, they should initiate a clean `atcr review`.
