# 18.0 Technical Debt Tooling

- **Estimated time**: 2 weeks
- **Tasks/Components**: 6/3

## Context

In Epic 12.1, we migrated our technical debt tracking from a cumbersome Markdown table to a structured file format. However, interacting with these files is still largely manual. As we transition to a Team Edition, we need automated tooling to manage, visualize, and prioritize this debt across the organization.

## Problem Statement

While the underlying format of our technical debt is now robust and scalable, developers and engineering managers lack unified tooling to quickly query, aggregate, and report on the overall health of the codebase. Manual inspection of individual YAML/JSONL files is inefficient for portfolio-level planning.

## Proposed Solution

Develop a suite of CLI commands and integrations within the ATCR engine specifically dedicated to technical debt management. This includes commands like `atcr debt list`, `atcr debt generate`, and automated GitHub/GitLab issue syncing.

## Requirements

### Functional
- Add an `atcr debt` CLI namespace.
- Provide filtering, sorting, and aggregation (e.g., by component, priority, or age).
- Automatically generate read-only Markdown dashboards from the underlying structured data.

### Non-Functional
- Commands must execute rapidly.
- The tooling must cleanly parse the schema established in Epic 12.1.

## Technical Approach

- Build a new `debt` package in the ATCR CLI.
- Implement a parser that reads the structured `.planning/technical-debt/items/` directory.
- Use a templating engine (like Go `text/template` or equivalent) to render aggregated dashboard views on demand.

## Implementation Components

1. **CLI Commands**
   - `atcr debt list`
   - `atcr debt add`
   - `atcr debt dashboard`

2. **Dashboard Generator**
   - A CI/CD friendly script that auto-generates a summary `README.md` whenever a debt item is added or resolved.

## Constraints

- Must work entirely locally without requiring external database dependencies.
- Must cleanly integrate into existing ATCR binaries.

## Out of Scope

- Auto-remediation of the actual code issues.

## Acceptance Criteria

- [ ] AC1: `atcr debt list` outputs a cleanly formatted table to the terminal.
- [ ] AC2: `atcr debt add` provides an interactive prompt to scaffold a new debt item.
- [ ] AC3: A dashboard generator can be hooked into a git pre-commit hook or CI pipeline.

## Success Criteria

- [ ] Team leads can instantly view the highest priority technical debt items.
- [ ] The overhead of managing and prioritizing debt is drastically reduced.

## Rollback Plan

- Disable the new CLI commands if they exhibit instability. The underlying data format remains untouched.

## Security Considerations

- Ensure the dashboard generator strips any potentially sensitive information if the dashboard is published publicly.

## Timeline Milestones

1. CLI Command Scaffolding.
2. Parser and Aggregation Logic.
3. Dashboard Rendering Template.
4. CI/CD Integration Examples.

## Clarifications

### Recorded 2026-07-03 (rubber duck — /execute-epic)

**Key Decisions:**
- `atcr debt list` / `dashboard` read the YAML shards under `.planning/technical-debt/items/` via the existing `tdmigrate.LoadShards()`. Epic 18.0 is the designated cutover that makes the shards a read source. Because migrate only flows README→shards and the shards are currently stale, the read path resyncs shards from the authoritative README (via `tdmigrate` migrate) before loading.
- `atcr debt add` writes to the authoritative README table (master) and regenerates the shards in the same step. Never writes a shard-only item (it would be destroyed by the next migrate). README-append format mirrors the existing `### [date] From …` section/row convention.
- `atcr debt add` is flag-driven as the primary, testable contract; an interactive TTY wizard is layered on top and only engages when required flags are absent AND stdin is a TTY. Streams are injectable (pattern from `cmd/atcr/quickstart.go`) so the flow is unit-testable without a TTY.
- `atcr debt dashboard` is a SEPARATE artifact from `td-migrate generate`. It renders an aggregation (counts by component/severity/age + top-priority list) to `.planning/technical-debt/DASHBOARD.md`. It NEVER writes `README.md`.
- Security scrub reuses `log.NewRedactor("")` (`Redact(string) string`, masks bearer/`sk-`-shaped tokens) applied unconditionally to free-text problem/fix/notes on the dashboard. File paths are NOT stripped (they are required core TD data and already public in the OSS tree). No `--public` flag added (YAGNI).
- Dashboard generator ships a `--check` drift mode (exit non-zero when the on-disk dashboard differs from freshly-generated output) and documented pre-commit + CI example snippets. The tracked `.githooks/` are NOT modified.

**Scope Boundaries:**
- IN scope: `atcr debt list` (filter/sort), `atcr debt add` (flag + interactive), `atcr debt dashboard` (aggregation + `--check` + scrub), a new `internal/debt` package built atop `internal/tdmigrate`, CI/CD example docs.
- OUT of scope: auto-remediation of code issues; changing the Epic-12.1 storage format; modifying live `.githooks/`; a `--public` scrub mode.

**Technical Approach:**
- Build `atcr debt` as a thin cobra layer (`newDebtCmd()` registered in `newRootCmd()`) over a new `internal/debt` package that reuses `tdmigrate` types (`Item`/`Shard`) and functions (`LoadShards`, `ParseREADME`, `GenerateTable`, migrate/regenerate).
- Aggregation (`internal/debt`) computes counts by component (path prefix), severity, and age (from shard `date`), plus a top-priority ordering (severity, then age).
- Reuse `log.NewRedactor("")` for the dashboard token scrub; pass `reviewRoot=""` so path relativization stays off.

**Original Questions and Answers:**

   A: Shards via `LoadShards()`; 18.0 is the cutover. Resync shards from the authoritative README (migrate) before reading, since shards are currently stale.
2. Q: Where does `atcr debt add` persist a new item?
   A: Write to the README table (master), then regenerate shards in the same step. Never shard-only.
3. Q: May `atcr debt add` be flag-driven primary with an optional interactive layer?
   A: Yes. Flags are the primary/testable contract; interactive wizard engages only when required flags absent AND stdin is a TTY; injectable streams for testability.
4. Q: Is `dashboard` a separate artifact from `td-migrate generate`, and what output path?
   A: Yes, separate. Aggregation rollup to `.planning/technical-debt/DASHBOARD.md`. Never write README.md.
5. Q: Security scrub — what is sensitive, on-by-default or `--public`, strip file paths?
   A: Scrub secret-shaped tokens (bearer/`sk-`) always via `log.NewRedactor("")`. Do not strip file paths. No `--public` flag.
6. Q: Hook integration — capability + docs, or modify live `.githooks/`?
   A: Capability (`--check` drift mode) + documented example snippets. Do not touch tracked `.githooks/`.
