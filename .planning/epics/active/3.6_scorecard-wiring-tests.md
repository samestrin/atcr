- **Estimated time**: ~1 hour

## Current State

Epic 3.4 made the scorecard diagnostics sink injectable (`ReadOpts.Writer`, `EmitOpts.Diag`) and added unit tests proving the routing **at the scorecard package layer**:
- `TestEmit_OrphanVerdictDiagnosticRoutesToDiagWriter` (`internal/scorecard/scorecard_test.go:295`)
- the buffer-injection read-path test (`internal/scorecard/scorecard_test.go:322`)

These lock **AC1** (diagnostics route to the injected writer) and **AC2** (the default-to-`os.Stderr` resolution) at the package boundary.

## Pain Points

The **wiring** that connects the real call sites to those injectable fields has **no dedicated test**:

- **AC3 — CLI entry points** pass `cmd.ErrOrStderr()` into the scorecard read/emit paths (the three scorecard-touching commands). Nothing asserts this; a future refactor could swap `cmd.ErrOrStderr()` back to a default writer and no test would fail.
- **AC4 — MCP handler** (`internal/mcp/handlers.go:220`) passes `EmitOpts{Diag: os.Stderr}` on the agentic path. Nothing asserts the handler passes a non-default writer.

This is silent-regression risk: the plumbing can break without any failing test.

## Proposed Solution

Add targeted wiring tests that prove the real call sites pass a non-default writer through to the scorecard layer.

### In Scope

- **AC3 (CLI):** Drive a scorecard command against a deliberately-malformed store and assert the read diagnostic reaches the command's `cmd.ErrOrStderr()` buffer (set via `cmd.SetErr(&buf)`). Cover the three scorecard-touching CLI entry points.
- **AC4 (MCP):** Assert the MCP `handleReconcile` path constructs `EmitOpts` with a non-default `Diag` writer. If/when MCP diagnostics are routed through `e.log` (see the deferred `internal/mcp/handlers.go:220` structured-logging TD), assert against a captured logger buffer instead.

### Out of Scope

- Changing the diagnostics convention itself (the `_, _ = fmt.Fprintf` discard is the intentional, codebase-wide best-effort pattern — confirmed wontfix during the 3.4 TD review).
- Adapting `e.log` into an `io.Writer` shim (tracked separately under the `internal/mcp/handlers.go:220` deferred item).
