# Epic 4.8: Improve MCP Server Test Coverage

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Medium (quality, regression safety)

## Problem

`internal/mcp` is the integration surface that exposes ATCR to IDE and agent clients over the MCP stdio protocol, yet it is the least-tested package in the codebase. It sits at **76.7%** statement coverage — below the project's 80% baseline (`testing.coverage_baseline: 80` in `.planning/.config/config.yaml`) — and the gap is concentrated in error paths and option parsing, exactly the branches a client hits when something goes wrong.

### Concrete Failure

A regression in MCP handler error handling ships silently. `handleVerify` (63% covered) builds the verify registry, parses an optional severity, and resolves the review dir; several of those error branches (`loadVerifyRegistry` 75%, `parseOptionalSeverity` 75%, `rangeError` 66.7%) have no test asserting the structured error a client receives. A future change that swallows or mis-maps one of these errors would pass CI, and an IDE client would get a wrong or empty error with no test catching it.

## Root Cause

Tests focus on happy paths. The existing suite (`handlers_test.go`, `handlers_verify_test.go`, `tools_test.go`, `server_test.go`) covers the success cases and a few rejections, but:

- Error/edge branches inside the handlers are under-covered (`handleVerify` 63%, `handleReport` 76.9%, `handleReview`/`registerTool` 80%, `handleStatus` 81.8%).
- Small parsing/helper functions are partially covered (`parseOptionalSeverity` 75%, `loadVerifyRegistry` 75%, `rangeError` 66.7%, `reportInputSchema` 83.3%).
- Server lifecycle code is effectively untested (`Serve` 0%, `drain` 0%) because it spans the stdio transport and process lifecycle, which the unit suite never drives.

## Solution

Raise `internal/mcp` to **≥ 80%** by adding targeted, table-driven tests for the under-covered handler error branches and helper functions, plus a lightweight lifecycle test (or an explicit, documented exclusion) for the server transport code. No production behavior changes — this is test-only remediation, minimizing or avoiding changes to non-test code.

### Test Approach

- **Table-driven handler tests**: for each handler, drive the documented error scenarios (missing/corrupt artifacts, invalid args, in-progress fan-out, unknown review id) and assert the structured error, not just `err != nil`.
- **Helper unit tests**: cover `parseOptionalSeverity` (valid, blank, invalid), `rangeError`, and `loadVerifyRegistry` failure directly.
- **Lifecycle**: either a minimal `Serve`/`drain` test over an in-memory pipe transport, OR an explicit decision to exclude the transport lifecycle from the coverage target with a documented rationale (see Open Questions).

## Scope

### In Scope

- New/expanded tests in `internal/mcp/*_test.go` for the under-covered handler error branches and helpers listed above.
- A lightweight server-lifecycle test (or a documented exclusion) for `Serve`/`drain`.
- Raising `internal/mcp` statement coverage to ≥ 80%.
- Minimal, test-only refactors if a function is genuinely untestable as written (e.g., extracting a pure helper) — kept to the absolute minimum and called out in review.

### Out of Scope

- Changing MCP handler behavior or the tool/JSON-schema contract.
- Adding new MCP tools or handlers.
- Cross-package coverage (only `internal/mcp` is in scope; other packages already meet the bar).
- A repo-wide CI coverage gate (tracked separately; see Relationship to Other Epics).
- The cmd/mcp radar-helper de-duplication TD item from Epic 3.2 (separate refactor).

## Implementation Plan

### Phase 1: Helper and parser coverage (0.5 days)

1. Add unit tests for `parseOptionalSeverity` (valid/blank/invalid).
2. Add tests for `rangeError` and `loadVerifyRegistry` failure paths.

### Phase 2: Handler error branches (1 day)

1. Extend `handlers_verify_test.go` to cover `handleVerify` registry/severity/range failures.
2. Extend `handlers_test.go` for `handleReport` radar + error branches and `handleStatus` corrupt-manifest.
3. Cover the remaining `handleReview`/`registerTool` branches.

### Phase 3: Lifecycle decision (0.5 days)

1. Attempt a minimal `Serve`/`drain` test over an in-memory transport.
2. If not feasible cleanly, record an explicit exclusion rationale and adjust the target accordingly.

### Phase 4: Verify and lock (0.25 days)

1. Confirm `internal/mcp` ≥ 80% and the full suite/gates are green.
2. (Optional) Note remaining intentionally-uncovered lines in a short comment.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `Serve`/`drain` are untestable without invasive refactor | Medium | Medium | Prefer an in-memory transport test; otherwise exclude explicitly rather than contort production code |
| Tests pin incidental error wording, becoming brittle | Medium | Low | Assert error type/category and key substrings, not full messages |
| Chasing the last few percent forces unnatural tests | Low | Low | Target ≥ 80% (the baseline), not 100%; document deliberate exclusions |
| A "test-only" change leaks into production behavior | Low | Medium | Keep non-test edits to extracted pure helpers only; call them out in review |

## Open Questions

1. **Should `Serve`/`drain` count toward the coverage target, or be excluded as lifecycle/transport code?**
   - Option A: write a minimal in-memory-transport lifecycle test.
   - Option B: exclude them explicitly and target ≥ 80% on the remaining statements.
   - Recommendation: attempt Option A; fall back to Option B with a documented rationale.

2. **How strictly should error messages be asserted?**
   - Recommendation: assert error type/category and a stable substring, not the full string, to avoid brittleness.
