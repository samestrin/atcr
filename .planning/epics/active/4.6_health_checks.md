# Epic 4.6: Health Checks for MCP Server

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Low (operational convenience, orchestration support)

## Problem

The ATCR MCP server (`atcr serve`) has no health check endpoint. Orchestration systems (Kubernetes, Docker Compose) cannot determine if the server is healthy without making a real MCP request. This makes liveness probes, readiness probes, and load balancer health checks impossible.

### Concrete Failure

A user deploys ATCR MCP server in Kubernetes. The server process is running but hung (deadlocked, out of memory, etc.). Kubernetes does not detect the unhealthy state because there is no health check endpoint. The pod continues to receive traffic, causing user requests to hang.

## Root Cause

The MCP server (`cmd/atcr/serve.go`, `internal/mcp/server.go`) implements the MCP protocol but does not expose a simple HTTP health check endpoint. The MCP protocol is JSON-RPC over stdio or HTTP, but health checks typically expect a simple HTTP GET returning 200 OK.

## Solution

Add a `/health` HTTP endpoint to the MCP server that returns 200 OK with a simple JSON body. The endpoint does not require MCP protocol negotiation.

### Core Design

```go
// internal/mcp/server.go
func (s *Server) registerHealthEndpoint() {
    http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        w.WriteHeader(http.StatusOK)
        json.NewEncoder(w).Encode(map[string]interface{}{
            "status":  "ok",
            "version": s.version,
            "uptime":  time.Since(s.startTime).Seconds(),
        })
    })
}
```

### Response Format

```json
{
  "status": "ok",
  "version": "1.2.3",
  "uptime": 3600.5
}
```

### Readiness vs Liveness

- **Liveness** (`/health`): Is the process running and responsive? Returns 200 if yes, 503 if no.
- **Readiness** (`/ready`): Is the server ready to accept MCP requests? Returns 200 if yes, 503 if no (e.g., still initializing).

For the initial scope, only `/health` is implemented. `/ready` can be added later if needed.

## Scope

### In Scope

- `/health` HTTP endpoint in MCP server.
- Returns 200 OK with JSON body (`status`, `version`, `uptime`).
- Endpoint does not require MCP protocol negotiation.
- Unit test: HTTP GET `/health` returns 200.
- Integration test: start MCP server, curl `/health`, verify response.

### Out of Scope

- `/ready` endpoint (readiness probe).
- Detailed health information (memory usage, goroutine count, etc.).
- Authentication for `/health` endpoint (health checks should be unauthenticated).
- HTTPS/TLS for `/health` (typically handled by load balancer or ingress).

## Open Questions

1. **Should `/health` check database connectivity or other dependencies?**
   - Option A: no — `/health` only checks if the process is running (simple).
   - Option B: yes — `/health` checks database, API keys, etc. (comprehensive).
   - Recommendation: Option A; ATCR has no database, and API key checks are done at request time.

2. **Should we implement `/ready` as well?**
   - Option A: no — `/health` is sufficient for now.
   - Option B: yes — `/ready` distinguishes liveness from readiness.
   - Recommendation: Option A; add `/ready` if users request it.
