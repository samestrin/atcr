# Epic Plan 1.3: Project-Level Registry Overlay

**Estimated Durations**: 1-2 weeks

## Objective

Let a repository define its own providers and agents in-repo (`.atcr/`), overlaying the user-level registry with project entries taking precedence by name — so a team can commit a complete, working atcr setup that every contributor gets without editing `~/.config/atcr/registry.yaml`.

## Context

Epic 1.0 shipped a two-file model: `~/.config/atcr/registry.yaml` (user level: provider and agent **definitions** plus user defaults) and `.atcr/config.yaml` (project level: roster **selection** plus shared-settings overrides). The precedence chain `CLI > project > registry > embedded` already covers settings, and `.atcr/personas/` already shadows user personas. But definitions are user-only: every `.atcr/config.yaml` roster entry must name an agent that exists in the user registry (`internal/registry/config.go` validation), so a cloned repo cannot be self-contained.

## Problem Statement

A team cannot commit its review roster's agent/model/provider definitions to the repo. Onboarding requires each contributor to mirror agent definitions into their home registry by hand, and drift between contributors' registries silently changes what a "review" means per machine. The existing precedence philosophy (project shadows user, per item, first hit wins) stops one tier short of the data that matters most.

## Proposed Solution

1. **Project registry overlay**: support `providers:` and `agents:` sections at the project level (file shape — dedicated `.atcr/registry.yaml` vs new sections in `.atcr/config.yaml` — decided at /init-plan; see Clarifications). Strict parsing (`KnownFields`) like every other config file.
2. **Name-level merge, project wins**: effective registry = user registry merged with project entries; a project agent or provider with the same name shadows the user one entirely (no field-level deep merge — whole-entry replacement keeps reasoning simple). New names are added.
3. **Validation over the merged view**: roster references, fallback chains (dangling/cycle checks), persona refs, and payload/timeout/temperature validation all run against the merged registry, so a project agent may fall back to a user agent and vice versa, and errors name the source file.
4. **Key handling unchanged**: project providers still reference keys only via `api_key_env`; secrets never enter the repo. This preserves the existing redaction/trust discipline.
5. **Provenance surfaced**: `atcr doctor` (Epic 1.2) and any verbose/config-dump output label each effective entry's source tier (project / user / embedded-default), so shadowing is visible rather than silent.
6. **Docs**: registry.md gains an overlay section with the updated precedence diagram:
   `CLI flag > .atcr/* (project) > ~/.config/atcr/* (user) > embedded`, now uniform across settings, personas, AND definitions.

## Security Considerations (first-class, not an afterthought)

A repo-defined provider means a cloned repository can direct requests to an arbitrary `base_url` while naming any `api_key_env` — a malicious repo could exfiltrate a real key from the contributor's environment to an attacker endpoint. Mitigation options to be settled at /init-plan (one or a combination):

- Require explicit one-time trust for project-defined **providers** (e.g. `atcr trust` writing a hash-pinned allow entry under the user dir), while project **agents** referencing user-defined providers need no prompt.
- Restrict project providers to `api_key_env` names matching an allowlist declared in the user registry.
- Print a loud first-use banner naming each project provider's base_url and key env.

## Success Criteria

- [ ] A fresh clone containing project-level provider+agent definitions runs `atcr review` end-to-end with no edits to the user registry (given the key env vars are set).
- [ ] A project agent with the same name as a user agent fully shadows it; removal of the project entry restores the user definition.
- [ ] Fallback chains spanning tiers validate at load (dangling refs and cycles fail fast, error names the defining file).
- [ ] Unknown keys in the project registry file are load errors (strict parsing parity).
- [ ] Project providers cannot read secrets from the repo — only `api_key_env` indirection, enforced by schema.
- [ ] The chosen trust mitigation is implemented and tested (a repo-defined provider cannot silently receive a key without it).
- [ ] Config provenance (project/user/embedded) is visible in at least one inspection surface (doctor output or a config dump).
- [ ] docs/registry.md documents the overlay, merge semantics (whole-entry shadowing), and the trust model; all tests green.

## Task Breakdown

1. Loader: locate + strictly parse the project registry source; merge into the user registry (whole-entry, project wins); thread source-file attribution into entries + tests.
2. Validation: run roster/fallback/persona/range checks over the merged view with tier-aware error messages + tests (incl. cross-tier fallback cycles).
3. Trust model: implement the chosen mitigation for project-defined providers + tests.
4. Provenance: expose entry source tier to doctor/config-dump surfaces + tests.
5. Docs: registry.md overlay section, precedence diagram update, README note; `atcr init` template comment updates.

## Out of Scope

- Field-level deep merge of individual agent entries (whole-entry shadowing only, v1 of the overlay).
- Remote/shared registries (URLs, git refs) — local files only.
- Storing API keys anywhere in any config file (unchanged invariant).
- Renaming `~/.config/atcr/` or relocating existing files.

## Dependencies

- Epic 1.0 core registry/loader (complete, in code review).
- Epic 1.1 schema reservations should land first so the overlay file reserves the same future-stage fields (`tools`, `role`, etc.).
- Epic 1.2 `atcr doctor` is the natural provenance display surface (soft dependency — a config dump works too).

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Malicious repo exfiltrates an API key via project provider base_url | Low | High | Trust gate for project providers (see Security Considerations); key env indirection only |
| Merge semantics confusion (which definition won?) | Medium | Medium | Whole-entry shadowing only; provenance labeling in doctor/config dump |
| Validation error messages lose clarity across tiers | Medium | Low | Every load error names the defining file and tier |

### Recorded 2026-06-11 (rubber duck — /execute-epic)

**Key Decisions:**
- File shape: project definitions live in a **dedicated `.atcr/registry.yaml`** (mirrors the user-level filename; keeps `.atcr/config.yaml` as a pure selection+settings file). The new file reuses the existing `Provider`/`AgentConfig` structs and `decodeStrictYAML` (KnownFields parity, incl. Epic 1.1 reserved fields).
- Merge: load the project registry, merge whole-entry into the user `Registry` (project wins by name, new names added), then run `ValidateFallbacks` + validation + `ValidateAgainst` over the merged view. Each effective entry carries a source tier (`project` | `user` | `embedded`).
- Trust model: **option (a) + (c)** — `atcr trust` (new cobra subcommand → `internal/registry/trust.go`) writes a hash-pinned allow file at `~/.config/atcr/trusted_providers.yaml`; the hash is SHA-256 over `base_url + "\x00" + api_key_env` (reuse `HashBytes` from `internal/reconcile/ambiguous.go`). A project-defined provider that is not trusted produces a hard `usageError` that points the user at `atcr trust` (the gate). A loud first-use banner naming each active project provider's base_url + key env is printed after the gate passes (UX confirmation, not the control). Project **agents** that reference an existing user-defined provider need no trust prompt.
- Provenance: extend **`atcr doctor`** — add a `SOURCE` column to the table (`internal/doctor/render.go`) and a `source` field (omitempty) to `AgentResult` (`internal/doctor/run.go`). No separate `atcr config dump` command in this epic.

**Scope Boundaries:**
- IN scope: `.atcr/registry.yaml` loader, whole-entry name-keyed merge (project wins), merged-view validation with tier-aware error attribution, `atcr trust` gate + hash-pinned allow file + first-use banner, `atcr doctor` SOURCE provenance, docs/registry.md overlay section + precedence diagram + `atcr init` template comment.
- OUT of scope: field-level deep merge of entries, remote/shared registries, relocating `~/.config/atcr/`, storing API keys in any config file, a standalone `atcr config dump` command.

**Technical Approach:**
- Project registry discovery follows the existing convention: located relative to `root="."` (no walk-up; matches how `cmd/atcr` resolves `.atcr/config.yaml` today).
- Trust check inserts at the overlay-merge step in the load path: after merging project providers, each project-tier provider is checked against the allow file before any invocation reaches `llmclient.Complete`. Missing entry → `usageError`.
- Source tagging is threaded from the load layer (registry merge) through to the render layer (doctor); embedded defaults tag as `embedded`.

**Original Questions and Answers:**
1. Q: File shape — dedicated `.atcr/registry.yaml` or new `providers:`/`agents:` sections inside `.atcr/config.yaml`?
   A: Dedicated `.atcr/registry.yaml`. config.yaml stays a pure selection+settings file; a separate file mirrors the user-level filename and lets `ValidateAgainst()` merge the two registries before checking roster references without touching `ProjectConfig`.
2. Q: Trust mitigation for project-defined providers — (a) explicit `atcr trust` gate, (b) api_key_env allowlist, (c) loud banner only, or (d) a combination?
   A: Option (d) = (a) + (c). `atcr trust` is the hard gate (missing trust entry is a `usageError` that blocks execution); the loud first-use banner is a UX aid printed after the gate passes, not the security control. Reuses the existing `HashBytes`/sha256 pattern; pins (base_url, api_key_env). Option (b) rejected — it does not pin base_url, so a "trusted" env name could still be sent to an attacker host.
3. Q: Provenance surface — extend `atcr doctor` or add a separate `atcr config dump`?
   A: Extend `atcr doctor`. Add a `SOURCE` column to the table and a `source` (`project|user|embedded`, omitempty) field to `AgentResult`. A separate `config dump` command is out-of-scope scope expansion the success criterion does not require.
