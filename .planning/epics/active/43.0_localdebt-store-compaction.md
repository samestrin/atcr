# Epic 43.0: Local Debt Store Compaction

- **Estimated time**: ~4 hours (240 min)

- **Source**: Promoted from technical debt (Epic 24.0 code review, reviewer claude)

## Current State

The local technical-debt store (`internal/localdebt`) is an append-only JSONL ledger,
sharded by month. Every state change appends a **full record copy**:

- A reconcile run appends an open record per finding.
- `atcr debt resolve --status resolved|wontfix` appends another full record copy carrying the terminal status.
- The record id is line-sensitive (`record.go` `StampID` hashes file + line + problem), so a
  finding that drifts by a line gets a **fresh id** ‚Äî a fresh open record plus, on dismissal, a
  fresh wontfix record.

There is **no compaction, GC, or cap** on the store. Every command that reads the store
(`ReadAll`, `selectOpenDebt`, `persistLocalDebt`) re-reads and folds the entire history on each
invocation.

## Pain Points

### Unbounded Growth
- A churny false positive (one that drifts line-to-line across runs) accumulates a new
  open+wontfix record pair on every line drift.
- Shard files grow without bound; superseded closed records are never reclaimed.

### O(total-records) On Every Command
- `ReadAll`/`selectOpenDebt`/`persistLocalDebt` fold the **full history** each call, so per-command
  cost tracks total history, not live findings.
- The read path already bounds a single JSONL line at `maxLineBytes = 1 MiB`
  (`internal/localdebt/store.go:18`) and **silently drops** any line over that limit ‚Äî a growth
  path can therefore lose records silently.

## Proposed Solution

Add a compaction/GC path so on-disk size tracks **live** findings, not total history.

### Approach
1. **Fold-and-rewrite compaction** ‚Äî read the full store, fold by id to the effective record
   (latest open record, or the highest-precedence terminal record for closed ids), drop
   superseded records, and rewrite shards atomically.
2. **New CLI command** ‚Äî `atcr debt compact` (name TBD) to trigger compaction on demand;
   optionally a periodic/threshold-triggered fold.
3. **Concurrency safety** ‚Äî compaction rewrites must not race the store's existing per-write
   atomic-append guarantee (`store.go` Append). Define a safe rewrite protocol (temp file +
   atomic rename, or an advisory lock consistent with the codebase's mkdir-lock convention).
4. **Back-compat / migration** ‚Äî a compacted shard must remain readable by the current
   `ReadRecords` stream parser and schema-version guard.

## Scope

### In Scope
- Compaction fold logic in `internal/localdebt`.
- A CLI command in `cmd/atcr` to invoke it.
- Concurrency-safe atomic shard rewrite.
- Tests: simulate many resolve/wontfix cycles, assert bounded on-disk size after compaction and
  no loss of the effective (live) findings.

### Out of Scope
- Changing the line-sensitive id scheme (`StampID`) ‚Äî separate concern.
- Resolved-regression re-opening semantics (`persistLocalDebt` seeding + `selectOpenDebt` fold) ‚Äî
  explicitly deferred at the Epic 24.0 review; a separate follow-up if wanted.
- The append-only write path's atomicity guarantee ‚Äî preserved, not replaced.

## Acceptance Criteria

- [ ] AC1: A compaction path folds the append-only store by id and drops superseded records, so
  a store with N resolve/wontfix cycles of the same finding compacts to O(live findings).
- [ ] AC2: After compaction, on-disk shard size is bounded by live-finding count, not total
  history (verified by simulating many resolve cycles and asserting bounded size).
- [ ] AC3: Compaction is concurrency-safe against the store's atomic-append writes ‚Äî no record is
  torn or lost when a compaction overlaps a concurrent append.
- [ ] AC4: Compacted shards remain readable by the current `ReadRecords` parser and schema-version
  guard (no back-compat break).
- [ ] AC5: The effective open backlog (`selectOpenDebt` output) is identical before and after
  compaction ‚Äî compaction changes storage, never which findings are open.

## Risks

| Risk | Mitigation |
|------|------------|
| Compaction races a concurrent append and tears/loses a record | Temp-file + atomic rename or advisory lock; preserve the per-write append guarantee |
| Fold picks the wrong effective terminal status | Reuse the precedence rule already added at `debt_resolve.go` (`higherClosedStatus`: wontfix outranks resolved) |
| Silent record loss via the 1 MiB line cap | Ensure compacted records stay well under `maxLineBytes`; surface any drop |
