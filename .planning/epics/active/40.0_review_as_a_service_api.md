# Epic Plan 40.0: Review-as-a-Service API

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

> **Note:** This epic is tracked here for product roadmap visibility, but the actual execution (building the API and worker) will occur in the `atcr.dev` infrastructure repository, not the core `atcr` repository.

## Objective

Build a hosted, scalable API backend that wraps the ATCR core engine, allowing teams to `POST` a Git diff and receive a reconciled multi-model code review without installing the binary or managing API keys locally.

## Context

While the local CLI is excellent for privacy and BYO-Keys, many mid-market and enterprise teams prefer the simplicity of a SaaS endpoint. The Review-as-a-Service (RaaS) API will serve as the primary scalable revenue driver (Phase 3 of the Monetization Roadmap), providing a frictionless on-ramp for teams that want review orchestration fully managed in the cloud.

## Proposed Solution

1. **API Server:** 
   - Build an HTTP REST/gRPC API using the public `atcr` Reconciler as a library.
   - Endpoints: `POST /v1/reviews` (Submit diff), `GET /v1/reviews/{id}` (Poll status).
2. **Asynchronous Processing:** 
   - Integrate a task queue (e.g., Redis/Celery or Go equivalent) to handle long-running model fanouts asynchronously.
3. **Managed Keys & Billing:** 
   - Integrate Stripe for subscription billing or per-review micro-transactions.
   - The API will utilize our own enterprise API keys (OpenRouter/Anthropic), abstracting token management from the user.

## Acceptance Criteria

- [ ] A developer can `POST` a unified diff and receive a polling URL.
- [ ] The API successfully orchestrates the multi-model Reconciler and returns a structured JSON report matching the CLI output.
- [ ] The API is secured via Bearer tokens tied to billing accounts.
- [ ] The telemetry/cloud-sync privacy contract is finalized: the backend provisions the HMAC pepper secret so `HashPersonaID` can move from unsalted SHA-256 to keyed HMAC-SHA256, and the payload `Model` field is validated against a backend-owned non-sensitive enum at the boundary (see Deferred Technical Debt below).
- [ ] The client telemetry ping is delivered reliably (bounded drain before process exit) and the review-path send/gate ordering is covered by an end-to-end test, so the SaaS ingest never loses or misreports run outcomes.

## Deferred Technical Debt (from Sprint 28.0 — telemetry_expansion_cloud_sync)

The following sprint-28.0 findings were deferred to this epic because they are gated on backend decisions this epic owns (the `atcr.dev` endpoint, the HMAC pepper secret, and the payload contract), or are the client-side reliability/coverage half of that backend work. The backend design here must account for them:

- **TD-006 — telemetry delivery is fire-and-forget** (`cmd/atcr/main.go:46`): `Client.Send` POSTs on a detached goroutine, then `main()` returns / `os.Exit`s immediately, so a configured endpoint may never receive the ping. Needs a bounded drain (`telemetryClient.Wait()` under a short ~1s timeout) before exit. Once a real ingest endpoint exists (this epic), unreliable delivery becomes a data-loss bug.
- **TD-007 — `HashPersonaID` is unsalted SHA-256** (`internal/scorecard/telemetry.go:18`): pseudonymous but dictionary-reversible over the small enumerable set of persona names. Hardening to keyed HMAC-SHA256 needs a backend-provisioned application pepper secret and a revision to the AC 03-01/03-04 plain-SHA-256 digest pins. Blocked on this epic's key/secret management.
- **TD-008 — payload `Model` copied unhashed on an asserted (not enforced) non-PII assumption** (`internal/scorecard/telemetry.go:30`; same pattern on the outbound cloud-sync path): a custom/local/fine-tuned model name could carry sensitive data and would transmit verbatim. Needs boundary enforcement (validate `Model` against a backend-owned non-sensitive enum, or scrub it) rather than a comment-only assertion.
- **TD-011 — no review-path end-to-end telemetry test** (`cmd/atcr/review.go:397`): only the reconcile call site has a counting-send e2e test. Needs a review-path equivalent (a `runReviewGated` helper over the existing mock-backend fixture, mirroring `runReconcileGated`), so the SaaS ingest contract is verified on both paths. This test is the prerequisite that gates the two items below.
- **review.go:409 — review telemetry `status` predates the gate**: the `review_run` event is Sent with a status derived from the fan-out `err` BEFORE the in-process reconcile/verify/debate + `--fail-on` gate runs, so a review whose agents succeed but whose gate later fails (exit 1) still reports `status="success"` — telemetry disagrees with the exit code. The review-path analog of TD-009 (already fixed for reconcile). Fix is to defer the Send so it reads the finalized `err` (mirroring the deferred sync-cloud push and `reconcile.go`), verified by the TD-011 e2e test above.
- **cmd/atcr/telemetry.go:60 — telemetryGate re-reads project config on the review path**: `telemetryGate()` calls `registry.LoadTelemetrySetting(".")` a second time after `fanout.LoadReviewConfig` already loaded the same config, resolving the opt-out through a different parse path than the strict roster load (they could disagree after a schema change, with no test binding them). Thread the already-loaded config's telemetry field into the gate on the review path. Entangled with the review.go:409 Send region and its TD-011 test, so scoped here with them.
