# Epic Plan 19.5: Response-Truncation Failover for Reviewer & Fixer

- **Estimated time**: 2-3 days
- **Tasks/Components**: 5/2
- **Execution**: execute-epic --strong

## Objective

Stop a truncated, runaway model response from being silently recorded as a clean "no issues found" review (or an empty fix). Detect a response that hit the token ceiling with zero usable output and route it into the existing fallback chain instead.

## Context

A reasoning-capable local model (e.g. Ornith-1.0-35B thinking-on, evaluated 2026-07-04) can burn its entire completion budget inside its `<think>` block and stop on `finish_reason=length` before emitting any pipe-delimited findings. Measured rate in a 15-call reviewer bench: **3/15 calls** (the adversarial and advocate personas on large diffs) produced non-empty rambling that parsed to **zero findings**.

`internal/llmclient/chat.go:190-196` already guards the *empty*-content truncation case (non-standard `finish_reason` + empty content + no tool_calls → error → fallback) and sets `resp.Truncated = true` on `finish_reason == "length"`. The gap: a truncated response with **non-empty** content that parses to **zero findings** falls through the guard. Fanout parses it, gets no findings, and records the slot as `StatusOK` — a **silent false all-clear**. This is more dangerous than a crash, because a crash already descends the fallback chain (`internal/fanout/engine.go`, Primary → Fallbacks until `StatusOK`) and a false-clean does not.

The fixer path (`internal/verify/executor.go`) is partly protected — a runaway fix fails `validateGoFixSyntax` and the `--auto-fix` validation gate, so garbage never lands — but a truncated fixer still burns a turn and emits no patch silently instead of failing over.

## Task Breakdown

1. Thread the response-truncation signal into slot classification: consult `ChatResponse.Truncated` (and `FinishReason == "length"`) where fanout builds a slot `Result` so the status logic can see it (it is set on the response today but not consulted by the reviewer status path).
2. In fanout, classify a reviewer slot as `StatusFailed` when the response is truncated **and** yields **zero** parsed findings, so the existing fallback chain engages. Preserve a truncated response that still parsed **≥1** finding as `StatusOK`, keeping the partial findings and attaching a `truncated` warning marker (do not discard partial signal).
3. Apply the same truncated-and-empty → fail-and-fallback rule to the executor/fixer path in `internal/verify/executor.go`, so a runaway fix retries via its fallback rather than returning no patch silently.
4. Emit a status/telemetry marker (`response_truncated`, `truncated_zero_findings`) attributable per agent so runaway rates are observable across runs.
5. Tests: (a) truncated + 0 findings → `StatusFailed` + `FallbackUsed`; (b) truncated + N findings → `StatusOK` + warning marker + partial findings preserved; (c) executor truncated + no patch → fail/fallback, no false success.

## Acceptance Criteria

- [ ] A reviewer response with `finish_reason=length` and zero parsed findings is recorded as `StatusFailed`, and the slot's fallback agent runs (`FallbackUsed=true`).
- [ ] A reviewer response with `finish_reason=length` and ≥1 parsed finding stays `StatusOK`, keeps its findings, and carries a `truncated` warning in the slot record / status.json.
- [ ] An executor/fixer response that truncates with no usable patch fails over instead of returning a silent no-op success.
- [ ] A per-agent `truncated_zero_findings` count is visible in status.json (or the run's telemetry) for observability.
- [ ] `go test ./...` passes, including new cases in `internal/fanout/*_test.go` and `internal/verify/*_test.go` covering the three scenarios above.
- [ ] No behavior change for the normal `finish_reason=stop` path (regression-guarded by existing fanout tests).

## Components Touched

- `internal/fanout` (slot-result classification in engine/loop, truncation marker in postprocess/status)
- `internal/verify` (executor truncation → fallback)

Signal source only (no logic change): `internal/llmclient` already exposes `ChatResponse.Truncated` / `FinishReason`.

## Rollout

This work de-risks pointing the `llm-large` litellm alias (on orchestrator.lan) at Ornith-1.0-35B (thinking-on) as the reviewer/fixer backend. The switch is a served-model change **behind the existing alias only** — reviewer agents and the executor keep referencing `llm-large`; no registry `model_name` churn. Land this sprint (failover) **before** flipping the alias, so a runaway degrades to a fallback instead of a silent clean review or a silent no-op fix.

## Out of Scope

- **Preventing** runaways rather than making them safe (this sprint makes them *safe*, not *rare*): `max_tokens` sizing, per-agent timeouts, diff chunking, and tightening the adversarial/advocate persona prompts. Note the runaway-prone personas observed in benchmarking — **brad** and **ronin** — live in the operator's `~/.config/llm-tools/agents/registry.yaml` (user config), **not** the repo's `personas/*.md` (whose tested persona, `bruce`, did not run away), so their tightening is an operator change execute-epic cannot make. The `truncated_zero_findings` telemetry from Task 4 identifies which agents are repeat offenders, so any later tightening is data-driven, not speculative.
- Registry `fallback:` pairing for specific model-backed agents: lives in the user's `~/.config/llm-tools/agents/registry.yaml`, not repo code; this sprint only guarantees a truncated slot *reaches* the fallback chain.
- vLLM guided/structured decoding to force the pipe-delimited (reviewer) or fenced-code (fixer) output shape: heavier, separate effort; hold in reserve if failover alone proves insufficient.
- Changing `defaultMaxTokens` (`internal/fanout/review.go:943`).

## Dependencies

- Existing fallback chain and `Result` status machinery in `internal/fanout/engine.go` (Primary → Fallbacks, `StatusOK`/`StatusFailed`/`StatusTimeout`).
- `ChatResponse.Truncated` / `FinishReason` already set in `internal/llmclient/chat.go`.
- Executor validation gate in `internal/verify/executor.go` (`validateGoFixSyntax` + `--auto-fix` local validation).
