# Feature Request: Brace-Language AST Parsers (TS/JS, PHP, Rust, Bash)

- **Estimated time**: 3 days
- **Tasks**: 5
- **Components**: 1
- **Execution**: execute-epic

## Context

Epic 13.1 shipped the AST plugin architecture: a wazero host that loads language parsers compiled to `.wasm`, maps a finding's line to its smallest covering AST block, and groups findings by a structural Merkle hash so duplicate findings group together even when their reported line numbers drift. Go and Python ship as the proofs of concept. 13.1 established the business criterion that new languages are added by "dropping in a new `.wasm` file" plus a one-line registry entry.

This epic exercises that criterion by adding the languages the maintainer actually writes — TypeScript/JavaScript, PHP, Rust, and Bash — so AST-isomorphism grouping covers real review surface beyond Go and Python.

## Problem Statement

- **Coverage gap**: Today only `.go` and `.py` findings benefit from AST grouping. Findings in `.ts`/`.tsx`/`.js`/`.jsx`/`.php`/`.rs`/`.sh`/`.bash` fall back to the ±3 line-proximity gate, which 13.1's benchmark showed both under-groups (misses line drift > 3) and over-groups (false-merges adjacent-but-different blocks).
- **Per-language cost would not scale if done naively**: Writing a separate parser per language repeats most of the work. The four target languages are all brace-delimited (`{ }` blocks), so a single shared brace-block parser parameterized by a small per-language keyword/naming table covers all of them — the cost is one parser plus four small config tables, not four parsers.

## Proposed Solution

1. **Shared brace-block parser**: A single Go-compiled wasip1 `.wasm` reactor (same plugin interface, memory protocol, and JSON contract as 13.1's `goparser`/`pyparser`) that recovers block structure by tracking `{ }`/`( )` depth and naming each block from the nearest preceding declaration keyword. It is parameterized by a per-language table of block-introducing keywords (e.g. `function`/`class`/`=>` for TS, `function`/`class`/`trait` for PHP, `fn`/`impl`/`trait`/`mod` for Rust, `function`/`name()` for Bash).
2. **Per-language configs**: One keyword/naming table each for TypeScript/JavaScript, PHP, Rust, and Bash, plus the file-extension → language mapping (`.ts`,`.tsx`,`.cts`,`.mts`,`.js`,`.jsx`,`.mjs` → `ts`; `.php` → `php`; `.rs` → `rust`; `.sh`,`.bash` → `bash`).
3. **Registry + benchmark**: Embed the new `.wasm` via `go:embed`, add the `builtinParsers` and `LanguageForExt` entries, and extend `testdata/corpus.json` with labeled drift/no-drift pairs per language so the existing accuracy benchmark guards each one.

The parser is a heuristic block-structure parser (strategy B, like the 13.1 Python plugin), used ONLY for grouping — never for semantic analysis. A misparse degrades to line-proximity grouping for that finding; it cannot break a reconcile.

## Acceptance Criteria

- [ ] AC1: A single shared brace-block parser plugin parses all four languages, parameterized by a per-language keyword/naming table (no separate parser binary per language).
- [ ] AC2: `.ts`/`.tsx`/`.cts`/`.mts`/`.js`/`.jsx`/`.mjs`, `.php`, `.rs`, and `.sh`/`.bash` findings resolve to a parser via `LanguageForExt`, and findings in the same block group across line drift > 3.
- [ ] AC3: The accuracy benchmark (extended corpus) holds AST recall ≥ 0.95 and precision ≥ proximity on the new languages, the same bar 13.1 set.

## Success Criteria

### Functional
- [ ] Cross-compilation to all target architectures still works out-of-the-box (no CGO; new `.wasm` are embedded bytes, not host code).
- [ ] Repeat-parse overhead per language stays within the 13.1 cache NFR (<10ms after first compile).
- [ ] Disabling AST grouping (`ATCR_DISABLE_AST_GROUPING`) still reverts cleanly to line proximity for the new languages.

### Business
- [ ] AST-isomorphism grouping covers the maintainer's full primary writing stack (Go, Python already done; TS/JS, PHP, Rust, Bash added here).
- [ ] Adding the next brace language is a config-table + corpus change, not new parser code.

## Technical Constraints

- Reuse 13.1's `internal/astgroup` plugin interface, wazero host, memory protocol, and `Node` JSON contract unchanged — this epic adds parsers and registry entries only.
- The `reconcile` library MUST stay zero-dependency (`go mod tidy ./reconcile` yields an empty require block); all new code lives in the root module under `internal/astgroup`.
- Parsers are Go-compiled wasip1 `.wasm` reactors built via the existing `internal/astgroup/parsers/build.sh`; no new toolchain (no emscripten/tree-sitter) is introduced.

## Non-Functional Requirements

### Performance
- Module instantiation cached per language so repeated parsing of the same language is fast (<10ms overhead), matching 13.1.
- Keep the shared parser `.wasm` size in line with the existing plugins (~3–4 MB); a single shared parser avoids multiplying vendored binary size across four languages.

## Out of Scope
- Ruby and PowerShell. Ruby uses `end`-keyword/indentation block structure (a different parser family closer to the Python plugin, not the brace parser); PowerShell is low-demand. Capture both as technical debt and revisit when those review surfaces become relevant.
- Real grammar-accurate parsing (tree-sitter/emscripten). The heuristic brace parser is sufficient for grouping; a misparse only falls back to proximity. Grammar-accurate parsing is a separate future investment if the AST layer is ever used for more than grouping.
- Semantic/type-aware analysis of any language.

## Dependencies
- Epic 13.1 (AST Plugin Architecture) — **complete and merged** (PR #109, merge commit `45efed18`). This epic depends only on the architecture 13.1 shipped.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Heuristic parser misreads braces/keywords inside strings, comments, regex, or Bash heredocs, producing spurious blocks | Medium | Low | Track string/comment/heredoc state in the scanner (as the 13.1 Python plugin does for triple-quoted strings); on misparse the finding falls back to line proximity — grouping degrades, never breaks. Corpus cases cover string/heredoc edge cases per language. |
| Per-language naming nuance (TS arrow functions/decorators, Rust `impl` blocks, PHP namespaces, Bash `function` vs `name()` forms) reduces grouping precision | Medium | Medium | Keep the keyword/naming table per language explicit and corpus-guarded; precision is measured by the benchmark before adoption. |
| Shared parser becomes a tangle of per-language special cases | Low | Medium | Keep language differences in data (the keyword/naming table), not control flow; the scanner stays language-agnostic over braces + the table. |

## Clarifications

### Recorded 2026-06-28 (rubber duck — /execute-epic)

**Key Decisions:**
- Build FOUR separate `.wasm` binaries (`ts`/`php`/`rust`/`bash`) from ONE parameterized Go source (`braceparser`), baking each language's keyword/naming table in at compile time (Go build tags). The host's existing language string (`LanguageForExt` → `Parser(lang)` → `builtinParsers[lang]`) is the discriminator — exactly how `go`/`python` work today. NO in-guest content detection.
- The host (`host.go`) and `grouper.go` remain UNCHANGED. The `parse(ptr,n)` ABI, memory protocol, and `Node` JSON contract are reused verbatim. `WithArgs(lang)` (host.go:193) is only a documented fallback if a single distributable `.wasm` ever becomes a hard shipping requirement — it is NOT used here.
- "Single shared binary" / "~3–4 MB" appears only in the Proposed Solution / NFR sections, not the binding Technical Constraints; it is consciously relaxed. Four vendored `.wasm` (~3–4 MB each) is accepted as the cost of the deterministic, host-driven language discriminator.
- Bash scope = function-level grouping ONLY (brace `{ }` function bodies). Bash end-keyword control flow (`if/fi`, `for/done`, `case/esac`) is OUT of scope (same end-keyword family as the explicitly-excluded Ruby); those findings degrade to ±3 line proximity.
- The guest ABI (`alloc`/`free`/`parse`/`emit`/`pins`) is duplicated a third time in `braceparser` rather than extracted into a shared guest package now (extraction needs cross-module `replace` coordination in build.sh, out of this epic's "add parsers only" scope); captured as a LOW technical-debt item.

**Scope Boundaries:**
- IN scope: `braceparser` Go source (one shared `{ }`/`( )` depth scanner + four compile-time language tables), build.sh entries producing `ts`/`php`/`rust`/`bash` `.wasm` + refreshed `SHA256SUMS`, `go:embed` + `builtinParsers` + `LanguageForExt` entries (`.ts`/`.tsx`/`.cts`/`.mts`/`.js`/`.jsx`/`.mjs`→`ts`; `.php`→`php`; `.rs`→`rust`; `.sh`/`.bash`→`bash`), extended `testdata/corpus.json` + benchmark coverage.
- OUT of scope: Ruby, PowerShell, grammar-accurate parsing, semantic/type analysis, ANY `host.go`/`grouper.go` change, in-guest language detection, Bash control-flow grouping.

**Technical Approach:**
- One `braceparser` module; a language-agnostic `{ }`/`( )` depth scanner with per-language string/comment/heredoc state + a block-introducing keyword table, selected at compile time via build tag.
- Emit only host-recognized `blockKinds` (`file`/`func`/`class`/`if`/`for`/`while`/`switch`/`block`); anonymous/object-literal braces are depth-tracked but left unnamed so sibling blocks keep distinct structural addresses (AST false-positives stay 0).
- New corpus pairs authored to keep `TestBenchmark_ASTGroupingAccuracy` green: recall ≥ 0.95, `astFP == 0`, AST resolves every >3-line-drift positive proximity misses.

**Original Questions and Answers:**
1. Q: Language-selection mechanism for the single shared binary — in-guest auto-detect, host `WithArgs(lang)` tweak, or four separate `.wasm` binaries?
   A: Four separate `.wasm` from one parameterized source, table baked in at compile time. The host already holds the lang string (grouper.go:158); the binary is the discriminator, same as go/python. Reject in-guest detection (fragile, discards known lang). `WithArgs(lang)` is the fallback only if a single distributable `.wasm` ever becomes mandatory; it leaves `parse(ptr,n)` unchanged. (Confidence 92%)
2. Q: Is function-level Bash grouping (brace blocks only) sufficient, or is control-flow-level Bash grouping (if/fi, for/done, case/esac) also expected?
   A: Function-level is the correct, complete scope. The Bash table names only `function`/`name()` (brace-delimited). Control-flow delimiters are end-keyword structures like the excluded Ruby blocks; adding them would violate AC1 or require a second parser family. Out-of-function findings fall back to ±3 proximity. (Confidence 94%)
