# Epic Plan 3.4: Per-Run Scorecard

**Estimated Durations**: 1 week

## Objective

Emit a normalized per-reviewer eval record alongside each reconcile run, accumulate those records into a local store, and expose them via `atcr scorecard` (single run) and `atcr leaderboard` (aggregated view across runs). This is the monitoring foundation for the quality pipeline and the data prerequisite for the public Model-Eval Leaderboard (Epic 6.0).

## Context

Every `atcr reconcile` run already computes everything needed for a per-reviewer quality signal: which reviewer raised which finding, and which findings were corroborated by 2+ independent reviewers. After Epic 3.0, findings also carry a `survived_skeptic` signal (verified/refuted/unverifiable). The byproduct of doing the work IS an evaluation dataset — but currently that dataset is discarded after each run.

This epic is purely a persistence + projection step over existing reconcile (and optionally verify) output. No new analysis. The local `atcr leaderboard` CLI is independently valuable for "for my codebase, which model finds the most real bugs at what cost?", and the same record schema becomes the submission format for the public leaderboard (Epic 6.0).

## Problem Statement

1. Every review run produces quality signal (corroboration rates, costs, latencies) that is discarded — there is no way to answer "is our review quality improving over time?"
2. Model and persona selection is currently intuition-driven; no per-run data exists to compare performance across reviewer configurations.
3. Epic 6.0 (Model-Eval Leaderboard) needs a stable, versioned per-run record schema — this epic defines and produces it.

## Proposed Solution

### Per-run scorecard record (emitted per reviewer per run)

```json
{
  "schema_version": 1,
  "run_id": "2026-06-14T10:00:00Z-abc123",
  "reviewer": "bruce",
  "model": "claude-sonnet-4-6",
  "role": "reviewer",
  "findings_raised": 12,
  "findings_corroborated": 7,
  "findings_solo": 5,
  "corroboration_rate": 0.58,
  "findings_verified": 4,
  "findings_refuted": 1,
  "survived_skeptic_rate": 0.57,
  "cost_usd": 0.04,
  "tokens_in": 14200,
  "tokens_out": 4000,
  "latency_ms": 9100
}
```

Fields `findings_verified`, `findings_refuted`, and `survived_skeptic_rate` are omitted when `verification.json` is absent (i.e., `atcr verify` has not run). The schema is versioned (`schema_version`) so Epic 6.0's public submission format can evolve independently.

### Storage

- `~/.config/atcr/scorecard/` — append-only JSONL file per calendar month (`2026-06.jsonl`). Local, never committed.
- Each run appends one record per reviewer + one aggregate record for the run.
- Records are small (~500 bytes each); even 1000 runs/month is ~500KB.

### CLI

- `atcr scorecard [id-or-path]` — display the scorecard for a single run (table: reviewer, model, findings raised/corroborated, cost, latency).
- `atcr leaderboard [--since 30d] [--model X] [--persona X]` — aggregate across stored records; output ranked table by corroboration rate and cost-per-corroborated-finding. Defaults to last 30 days.
- `atcr leaderboard --export` — emit the versioned JSON submission format for Epic 6.0's public leaderboard (anonymized, no provider keys or repo content).

### When scorecard is emitted

Automatically at the end of `atcr reconcile` (and optionally `atcr verify`). No new flag required; the record is written silently alongside `reconciled/summary.json`. Suppressed with `--no-scorecard`.

## Acceptance Criteria

- [ ] `atcr reconcile` writes a per-reviewer scorecard record to `~/.config/atcr/scorecard/YYYY-MM.jsonl` after each run.
- [ ] Record fields: schema_version, run_id, reviewer, model, role, findings_raised, findings_corroborated, findings_solo, corroboration_rate, cost_usd, tokens_in, tokens_out, latency_ms.
- [ ] When `reconciled/verification.json` is present, record also includes findings_verified, findings_refuted, survived_skeptic_rate.
- [ ] `atcr scorecard [id-or-path]` displays a formatted table for a single run.
- [ ] `atcr leaderboard` aggregates across stored records and ranks by corroboration rate; `--since` filters by date range.
- [ ] `atcr leaderboard --export` emits the versioned public submission JSON (no PII, no repo content, no provider keys).
- [ ] `--no-scorecard` suppresses writing to the local store.
- [ ] Schema is versioned; a future schema change increments `schema_version` and old records remain readable.
- [ ] Docs: scorecard.md (schema, storage location, CLI usage, privacy model).

## Task Breakdown

1. **Scorecard emitter**: compute per-reviewer fields from reconcile output; write to monthly JSONL.
2. **Verification integration**: when verification.json present, fold in survived_skeptic_rate and friends.
3. **`atcr scorecard` command**: read and display a single run's scorecard.
4. **`atcr leaderboard` command**: aggregate + rank across stored records; `--since` and `--model`/`--persona` filters.
5. **`--export` format**: versioned public submission JSON (schema v1); anonymization pass.
6. **`--no-scorecard` flag** on `atcr reconcile`.
7. **Docs + tests**: scorecard.md, unit tests for the emitter and aggregator.

## Out of Scope

- Public leaderboard site or submission infrastructure (Epic 6.0).
- Team-shared scorecard storage (Epic 7.x Team Edition).
- Real-time dashboard or web UI.
- Persona quality scoring beyond the standard reviewer metrics.

## Dependencies

- Epic 1.0 reconcile output (findings.txt, summary.json) — complete.
- Epic 3.0 verification.json — optional; scorecard works without it and gains the skeptic-rate columns when 3.0 data is present.
- **Client `usage` parsing (hard prerequisite).** `cost_usd`, `tokens_in`, and `tokens_out` come from the provider's `usage` block, which `internal/llmclient` does **not** currently decode — `chatResponse` / `chatToolResponse` drop it entirely. Without this, those columns are *always* empty, not just on providers that omit usage. The client must decode `usage`, thread tokens up to the scorecard emitter, and compute `cost_usd` (per-model rate) before this epic's cost/token ACs can pass. Tracked in the TD README (2026-06-14 "From Review: llmclient"); fold into this epic's first task if not yet resolved.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Scorecard JSONL grows unbounded | Low | Low | Monthly rotation; document `~/.config/atcr/scorecard/` location; user can delete old months |
| Export format becomes a public API surface locked too early | Medium | Medium | Version from day one; `schema_version` field; document that v1 is experimental until Epic 6.0 stabilizes it |
| Cost/token fields unavailable from some providers | Medium | Low | Fields are omitempty; missing data degrades gracefully (shown as `—` in the table). NOTE: graceful degradation assumes the client *parses* `usage` at all — it currently does not (see Dependencies). That is a hard prerequisite, not a degradation case. |
