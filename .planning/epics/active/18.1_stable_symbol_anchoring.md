# Feature Request: Stable Symbol Anchoring for Technical Debt

- **Estimated time**: 3 days
- **Tasks/Components**: 3/2
- **Execution**: execute-epic

## Context

During a `/resolve-td` session, code modifications introduce line-number drift. Subsequent technical debt items in the same file immediately point to invalid/stale line numbers. The `resolve-td` skill implements a relocation phase that searches for a `RELOCATE_KEY` to find shifted code, but deriving this key on-the-fly via the resolver LLM is fragile for generic or vague findings.

## Problem Statement

Line-number drift renders downstream TD links in `.planning/technical-debt/README.md` stale. Resolving drift dynamically during execution relies on the LLM generating a correct `RELOCATE_KEY` from the problem text, which often fails for generic items, leading to skipped fixes or incorrect applications.

## Proposed Solution

1.  **Reconciler Symbol Generation**: Update the `atcr` reconciler/reporting engine to resolve findings to their enclosing AST block name (using the existing `internal/astgroup` AST parsers).
2.  **ToC Formatting**: Prepend the resolved block/symbol name in parentheses to the `Problem` cell in `.planning/technical-debt/README.md` (e.g. `(classifyHeader) Problem description...`). This maintains 100% schema backward-compatibility for the legacy scripts.
3.  **`resolve-td` Skill Integration**: Update the `resolve-td` instructions in `/Users/samestrin/Documents/GitHub/claude-prompts/.claude/skills/resolve-td/instructions.md` to parse the `(symbolName)` prefix from the `Problem` cell and immediately use it as the `RELOCATE_KEY` for dynamic relocation.

## Acceptance Criteria

- [ ] AC1: The reconciler/report generator extracts the enclosing AST node name and prepends it as a parenthesized prefix in the `Problem` column of `README.md`.
- [ ] AC2: If no AST parser is available for the language or the finding is not in a named block, omit the prefix (graceful degradation).
- [ ] AC3: The `resolve-td` skill extracts this parenthesized symbol (if present) and successfully uses it to relocate shifted code blocks during execution.

## Technical Constraints

- Do not alter the column structure of the technical debt Markdown table (to avoid breaking legacy scripts).
- Reconciler changes must remain in the root Go module (no dependencies added to the `reconcile` library).

## Non-Functional Requirements

- Symbol extraction overhead must be negligible during the reconcile phase.

## Out of Scope

- Symbol resolution for unsupported/non-brace-based languages (these will fall back to standard keyword grep relocation).

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Duplicate symbol names in the same file | Low | Medium | If multiple methods have the same name (e.g. overloaded methods), rely on surrounding context or order; fallback to proximity if ambiguous. |

## Clarifications

### Recorded 2026-07-04 (rubber duck — /execute-epic)

**Key Decisions:**
- Symbol resolution walks the AST covering chain from the deepest covering block up to the root and returns the first node with a non-empty `Name` (a func/method/class/type). It does NOT use the literal deepest covering block, which is frequently an unnamed `if`/`for`/`block`. `astgroup.CoveringBlock`/`coveringChain` (`internal/astgroup/cover.go`) already produce the full descent chain.
- The `(symbolName)` prefix is added to `JSONFinding.Problem` in the atcr reconciler's emit layer (`internal/reconcile`), stamped at the derived-field point in `RunReconcile` (`internal/reconcile/gate.go`) BEFORE `Emit` runs. The reconciler never writes `README.md`; the prefix propagates into `.planning/technical-debt/README.md` downstream via the external `llm_support_group_td` flush.
- The `(symbolName)` prefix is display-only ToC sugar and is NOT a grouping key (grouping already uses the Merkle/address key), so duplicate symbol names are harmless for AC1/AC2.

**Scope Boundaries:**
- IN scope (this atcr PR, rides CI): T1 symbol resolution (`internal/astgroup`), T2 prefix stamping (`internal/reconcile`), T3a a new `docs/technical-debt-format.md` documenting the Problem-cell contract (mirrors the `docs/code-review-backend.md` external-consumer pattern).
- OUT of scope for this PR: T3b — editing `claude-prompts`'s `resolve-td/instructions.md` to parse the prefix (satisfies AC3). Done out-of-band in that separate repo; no CI. NOT vendored into atcr (Epics 20.0/20.1 scope out touching the private claude-prompts skills from atcr's tree).
- The zero-dependency `./reconcile` library is NOT touched; all changes stay in the root Go module.

**Technical Approach:**
- Add an exported `astgroup` helper (nearest-enclosing-named-block over the existing chain) plus a `Grouper` method that resolves a `Finding`'s file+line to a symbol, reusing the existing tree cache / canonical-path / Host machinery.
- Prefix format is literal `(symbolName) ` — parens, bare identifier, single trailing space, then the original problem text — anchored at position 0. Emit NOTHING (byte-identical Problem cell) when no named block resolves (AC2 graceful degradation: unsupported language, absent/unparseable file, or no named ancestor). The identifier is a bare AST name with nothing to escape.
