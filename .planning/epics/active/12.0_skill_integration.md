## Objective

Replace the `llm-support` CLI backend in the existing `execute-code-review` skill's
multi-agent mode with `atcr`, so the skill's multi-agent fan-out uses atcr's
heterogeneous reviewer panel and deterministic reconciler instead of the legacy
`review_direct` / `multi_review` commands. Update `/reconcile-code-review` to accept
atcr's output structure as a source.

## Context

The existing `execute-code-review` skill (in
`~/Documents/GitHub/claude-prompts/.claude/skills/execute-code-review/`) has two modes:

1. **Default mode** (phases 1-8): Sprint verification pipeline — work-item review, test
   execution, coverage gates, adversarial risk analysis. Sprint-coupled; stays as-is.
2. **Multi-agent mode** (`--multi-agent`): Thin orchestration over `llm-support
   review_direct` / `multi_review`. The CLI handles range resolution, fan-out, and merge.

ATCR replaces the reviewer panel — heterogeneous personas, deterministic reconcile with
clustering/dedupe/confidence, payload modes, and a clean findings format. The end state:
the skill's multi-agent mode calls `atcr review --output-dir ${CR_OUT_DIR}` + `atcr
reconcile`, reads atcr's actual output format, and `/reconcile-code-review` accepts
atcr's output structure.

The skill lives in
`~/Documents/GitHub/claude-prompts/.claude/skills/execute-code-review/multi-agent.md`.
The reconcile skill lives in
`~/Documents/GitHub/claude-prompts/.claude/skills/reconcile-code-review/`.

## Problem Statement

1. `multi-agent.md` Phase 2 Step 6 invokes `llm-support review_direct` / `multi_review`
   — single-backend (openclaw or direct API), no cross-reviewer agreement scoring.
2. ATCR provides the differentiated capability (heterogeneous panel, deterministic
   reconcile, confidence scoring) but the existing skill cannot consume it.
3. The two systems produce different output structures: the skill expects `td-stream.txt`
   + `multi-review-summary.json`; atcr writes `sources/pool/findings.txt` +
   `sources/pool/summary.json` + `reconciled/` artifacts.
4. `/reconcile-code-review` auto-discovers sources by `td-stream.txt` filename; it must
   also discover atcr's `reconciled/findings.txt`.

## Proposed Solution

### Changes to `multi-agent.md`

**Phase 2, Step 6** — replace the `llm-support` invocation:

Before:
```
llm-support review_direct \
  --config "${CONFIG_PATH}" \
  --repo "${REPO_ROOT}" \
  ${RANGE_FLAGS} \
  ${SCOPE_FLAG} \
  --output-dir "${CR_OUT_DIR}"
```

After:
```
atcr review \
  --output-dir "${CR_OUT_DIR}" \
  ${RANGE_FLAGS}
```

Followed by:
```
atcr reconcile "${CR_OUT_DIR}"
```

Notes:
- `--config` and `--repo` are not needed — atcr reads config from `.atcr/config.yaml` and
  `~/.config/atcr/registry.yaml`, and the repo is the CWD.
- `${SCOPE_FLAG}` (`--sprint-plan`) is dropped — atcr has no sprint-plan concept.
  Reviewers fall back to diff-touched-line scope.
- `${RANGE_FLAGS}` maps directly: `--base`, `--head`, `--merge-commit` are the same.

**Phase 3, Step 8** — update output verification:

Before:
```
paths: ["${CR_OUT_DIR}td-stream.txt", "${CR_OUT_DIR}multi-review-summary.json"]
```

After:
```
paths: [
  "${CR_OUT_DIR}sources/pool/findings.txt",
  "${CR_OUT_DIR}sources/pool/summary.json",
  "${CR_OUT_DIR}reconciled/findings.txt",
  "${CR_OUT_DIR}reconciled/summary.json"
]
```

**Phase 3, Step 9** — update summary reporting: read `reconciled/summary.json` for
`sources` (count), `total_findings`, `partial`. Display format unchanged; only the data
source changes.

**Phase 1, Step 3** — add an `atcr --version` pre-flight check (backend selection). If the
binary is not found, display an actionable error and HARD STOP.

### Changes to `/reconcile-code-review`

1. Add `reconciled/findings.txt` as an additional source pattern (alongside
   `td-stream.txt`).
2. When a source is `reconciled/findings.txt`, read it as 9-column format
   (SEVERITY|FILE:LINE|PROBLEM|FIX|CATEGORY|EST_MINUTES|EVIDENCE|REVIEWERS|CONFIDENCE)
   instead of 8-column.
3. Sources under `.atcr/reviews/` are NOT auto-discovered — only sources under the
   sprint's `code-review/` directory (where `--output-dir` points).

## Refinements (2026-06-26) — Follow-up: corrections & decisions

This block corrects one finding from the morning run and records the resolution decisions. The original section above is preserved for history.

### Correction to "External repo target"

The morning finding overstated the barrier. It is true that `/execute-epic`'s **automation** (branch → PR → merge → archive) is hardwired to the cwd repo and cannot drive an external repo. It is **not** true that the external repo is inaccessible: `claude-prompts` is a sibling git repo with a GitHub origin (`samestrin/claude-prompts`) that can be edited, branched, committed, and PR'd directly from a session in `atcr`. The conflation of "the `/execute-epic` skill's git flow is single-repo" with "the external repo can't be touched" was wrong.

### Decisions (resolved)

- **D1 — Execution path:** Implement directly in `claude-prompts` on a feature branch + PR (squash-merge, delete branch). `/execute-epic` is not used. Independent of Epic 12.1 (TD format migration), which is a separate atcr-local epic with no dependency on 12.0.
- **D2 — Pre-flight scope (T4):** In addition to `atcr --version`, the Phase 1 pre-flight checks that `.atcr/config.yaml` exists in the reviewed repo and HARD STOPs with an actionable message if absent. This is **fail-fast UX**, not a fix for a silent gap — `atcr review` already errors clearly (`internal/registry/project.go`: `no roster found: .atcr/config.yaml not found ... run 'atcr init'`) and exits non-zero. The skill check mirrors that contract so messages do not diverge. The global `~/.config/atcr/registry.yaml` is one-time user setup atcr's own error covers, so it is not gated on.
- **D3 — Task granularity:** Original T1/T2/T3/T7 all edit the same file (`multi-agent.md`) and are merged into one cohesive "edit multi-agent.md" unit. This drops the effective task count below `/execute-epic`'s ≤6 guard, though D1 makes that guard moot. T5 (reconcile skill) and T6 (validation) stand.
- **D4 — Reconcile source is atcr's 8-col pool file, NOT the 9-col reconciled file (supersedes the original Proposed Solution + Success Criteria):** The original plan had `/reconcile-code-review` discover and parse atcr's `reconciled/findings.txt` in 9-column format. Verification of the live skill (`reconcile-code-review/instructions.md:213`) showed it does **not** parse columns in-model — it delegates to the `llm_support_td_dedupe` binary, which normalizes **5/6/8/10**-column widths (not 9), and that binary lives in a third repo (llm-tools) out of this epic's scope. Meanwhile atcr's per-source `sources/pool/findings.txt` is **8-column** (`SEVERITY|FILE:LINE|PROBLEM|FIX|CATEGORY|EST_MINUTES|EVIDENCE|REVIEWER`, `internal/stream/parser.go:27`) — byte-identical to the skill's existing 8-col source contract. The skill now discovers that 8-col pool file, which `td_dedupe` ingests unchanged and which lets the skill recompute REVIEWERS-union + CONFIDENCE across atcr's individual reviewers alongside the other sources (better cross-source confidence than ingesting atcr's pre-collapsed 9-col blob). **Success Criteria bullets referencing "reconciled/findings.txt (9-column)" are therefore superseded by "sources/pool/findings.txt (8-column)".** No 9-column parser is needed anywhere.
