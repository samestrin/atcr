# Epic Plan 26.0: .atcrignore Token Waste Protection

- **Estimated time**: TBD
- **Tasks/Components**: 3 tasks / 2 components
- **Execution**: execute-epic

## Objective

Prevent ATCR from wasting LLM tokens (and money) on auto-generated code, vendored dependencies, or massive lockfiles by respecting `.gitignore` and introducing a native `.atcrignore` filtering layer.

## Context

If a developer runs a codegen tool or bumps a massive `vendor/` dependency, the resulting git diff can be hundreds of thousands of lines. While Epic 19.5 handles truncation and dropping files to fit the token budget, it does so based on size/priority. It is far safer to explicitly never send these files to the payload builder in the first place.

## Proposed Solution

1. **File Filtering Layer**: Update the diff ingestion logic in `internal/payload` (`diff.go`'s `changedFiles`/`changedFilesMemo`, ahead of `budget.go`'s `ApplyByteBudget` truncation pass) to completely exclude files that match standard ignore patterns.
2. **.gitignore Support**: By default, parse and respect the repository's `.gitignore`.
3. **.atcrignore Support**: Support a dedicated `.atcrignore` for files that should be committed to git but ignored by the AI (e.g., `go.sum`, `package-lock.json`, `docs/`). Repo-root-only, one gitignore-syntax pattern per line, purely additive to `.gitignore`'s exclusions (no `!` negation support).

## Acceptance Criteria

- [ ] Files matching `.gitignore` are excluded from the review payload.
- [ ] Files matching `.atcrignore` are excluded from the review payload.
- [ ] `atcr review` logs a trace/debug message noting which files were skipped due to ignore rules.

## Components Touched

- `internal/payload` (file-list filtering — `diff.go`/`builder.go`, ahead of the existing byte-budget pass)
- `cmd/atcr` (CLI interface)

## Out of Scope

- Complex globbing beyond standard `.gitignore` syntax.

## Dependencies

- None.

### Items needing user confirmation (2)

⏸️ **Components mismatch — `internal/gitrange` is architecturally the wrong "or" alternative.** Plan lists `internal/payload/builder.go` or `internal/gitrange` as the filtering-layer location. Verified: `internal/gitrange` (`resolver.go`) only resolves the base/head SHAs via git plumbing (merge-base, default-branch detection, shallow-clone checks) — it never enumerates files or computes diffs, so it has no file-level data to filter. The actual file list comes from `git diff --name-status` in `internal/payload/diff.go`'s `changedFiles`/`changedFilesMemo`, called by `builder.go`'s `BuildEntries`/`BuildDiff`/`BuildBlocks`/`BuildFiles`, ahead of the existing `ApplyByteBudget` truncation pass (`internal/payload/budget.go`). Suggested action: drop `internal/gitrange` from Components Touched and Proposed Solution item 1 — the filter belongs entirely in `internal/payload`. Proposed COMPONENTS_TOUCHED: `internal/payload`, `cmd/atcr` (count unchanged at 2 — still fits the scope guard).

⏸️ **`.atcrignore` location and precedence are undefined.** The plan doesn't say whether `.atcrignore` is repo-root-only or per-directory like `.gitignore`, nor its precedence relative to `.gitignore` (can an `.atcrignore` entry re-include a file `.gitignore` excludes? does it support `!` negation?). Given the plan's own Out-of-Scope bound on "complex globbing," suggested addition to Proposed Solution item 3: "`.atcrignore` is repo-root-only, one gitignore-syntax pattern per line, purely additive to `.gitignore`'s exclusions (no negation support)." This is the smallest testable surface and mirrors the stated simplicity boundary.

### Advisory observations (3)

ℹ️ **No existing gitignore-pattern-matching capability.** Verified via grep across `internal/` + `go.mod`/`go.sum` — no vendored `.gitignore`-glob library and no hand-rolled matcher exists today (the only `.gitignore` references found are test fixtures, `atcr init`'s own `.atcr/.gitignore` scaffold, and an unrelated dotfile-extension guard in `internal/verify/select.go`). Recommend a vetted third-party library (e.g. `github.com/sabhiram/go-gitignore`) over hand-rolling `.gitignore` syntax (negation, `**`, anchoring, and directory-only patterns are easy to get subtly wrong) — consistent with the plan's own "no complex globbing" scope bound.

ℹ️ **Suggested architecture: pre-filter stage ahead of the byte-budget pass.** `internal/payload/budget.go`'s `ApplyByteBudget` already sheds oversized entries by size (it does not classify by filename pattern today — "generated files/lockfiles" in its doc comment describes an emergent effect of size-based shedding, not a name-based classifier). The ignore-filter reads naturally as an earlier stage over the same file list — right after `changedFilesMemo` in `diff.go`, before `ApplyByteBudget` runs — so it composes with existing truncation instead of duplicating file-list plumbing.

ℹ️ **Logging convention already exists for AC #3.** `internal/payload/diff.go`'s `gitRunner` already exposes a `log() *slog.Logger`. Recommend AC #3 point at this concretely: "skipped files are logged at `slog` debug level via the existing `gitRunner` logger" — makes the AC testable rather than a vague "trace/debug message."
