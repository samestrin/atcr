# Tech Debt: Thread an injectable diagnostics writer through the scorecard read+emit API

- **Estimated time**: ~1 day
- **Plan #**: 3.4
- **Slug**: scorecard-diagnostics-writer
- **Source**: Deferred from `/resolve-td --group=8` (Sprint 3.3_per_run_scorecard code review)

## Current State

The scorecard store and emitter write operational diagnostics directly to the
process-global `os.Stderr` rather than to a writer threaded from the caller:

- `internal/scorecard/store.go` — `ReadRecords` ("skipping malformed record", "skipping over-long line"), `FindByRunID` ("run spans adjacent month files")
- `internal/scorecard/scorecard.go` — `Emit` / `verdictTallies` ("write failed", "verification read/parse failed", "verification finding … has no matching raised finding")

Because the sink is the global `os.Stderr`, these warnings cannot be captured,
redirected, or asserted in tests, and they bypass any writer the CLI already
threads for the rest of its output.

## Pain Points

- **Not testable** — orphan-verdict and malformed-record warnings can only be
  verified by capturing the global `os.Stderr`, so the diagnostics added in
  Sprint 3.3 are guarded only behaviorally (count assertions), not by their text.
- **Not redirectable** — the CLI threads `cmd.ErrOrStderr()` for its own output,
  but scorecard diagnostics escape that, so `--quiet`/redirection cannot reach them.
- **Inconsistent** — the rest of the command tree sources its error writer from
  cobra; the scorecard package is the outlier writing to a package-global.

## Proposed Solution

Thread a single injectable `io.Writer` (or a small logger) through the scorecard
read + emit entry points and their callers as one designed package-API change:

- Add the writer to `Append`, `ReadRecords`, `FindByRunID`, `ReadAll`, `Emit`, and `EmitForReconcile` (or a package-level option struct that carries it).
- CLI caller (`cmd/atcr/reconcile.go`) passes `cmd.ErrOrStderr()`.
- Default to `os.Stderr` when no writer is supplied so existing behavior is preserved.

## Scope

### In Scope
- The scorecard public read+emit API signatures (or an options-struct carrier)
- CLI caller wiring (`cmd/atcr`)
- MCP caller wiring (`internal/mcp/handlers.go`)
- Converting the existing `fmt.Fprintf(os.Stderr, …)` diagnostics to the threaded writer
- Tests asserting diagnostic text via an injected buffer

### Out of Scope
- The diagnostics' content/wording (unchanged)
- The JSONL record schema and aggregation logic

## Key Design Gap (why this is not an inline fix)

The MCP entry point `internal/mcp/handlers.go:218` calls `EmitForReconcile` with
**no cobra `cmd`** to source a writer from, so the CLI's `cmd.ErrOrStderr()`
precedent does not transfer. The package change must define how the MCP path
supplies (or defaults) its writer — a deliberate API decision, not a mechanical
substitution. This is precisely why it crosses the group-8 boundary and was
deferred from `/resolve-td` rather than resolved inline.

## Acceptance Criteria

- [ ] AC1: A test injects a buffer and asserts on a scorecard diagnostic's text (e.g. the orphan-verdict warning), proving the writer is honored.
- [ ] AC2: No `fmt.Fprintf(os.Stderr, …)` remains in `internal/scorecard/` for operational diagnostics (verified by grep); each goes through the threaded writer.
- [ ] AC3: CLI reconcile diagnostics route through `cmd.ErrOrStderr()`.
- [ ] AC4: The MCP `handleReconcile` path supplies a defined writer (or documented default) with no regression to emission.
- [ ] AC5: Default behavior (no writer supplied) still writes to `os.Stderr`; all existing scorecard tests pass.

## Risks

| Risk | Mitigation |
|------|------------|
| Signature churn across two entry-point packages | Prefer an options struct carrying the writer over positional params |
| MCP path has no cmd writer | Decide a default (os.Stderr) or thread a handler-level writer; document the choice |
| Silent behavior change | Default to os.Stderr when unset; cover with an existing-behavior test |
