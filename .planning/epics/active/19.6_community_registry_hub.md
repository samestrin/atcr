## Objective

Make the community persona channel the **canonical** source of reviewer personas — fetched from the product repo (`samestrin/atcr`) rather than compiled into the binary — and ship a **model-indexed, human-named persona library** that a user can discover *by the model they already have* ("I have DeepSeek → find the DeepSeek persona → install it"). Pair this with onboarding documentation that leads first-time setup with the monetizing flat-rate path (`atcr quickstart` → Synthetic) and treats frontier-provider personas as opt-in, out of the default funnel.

## Context

**What already exists (verified this session):**

- **The community distribution mechanism** (Epic 9.0): `atcr personas install/search/list/upgrade/remove`, fetching persona YAML + an `index.json` from a base URL. Default is hardcoded at `internal/personas/client.go:24` → `https://raw.githubusercontent.com/atcr/personas/main`, overridable via `ATCR_PERSONAS_URL`. Bundles install (`bundle/<name>`) already work (`bundle/django`, `bundle/go-production` ship today).
- **The monetizing onboarding already ships**: `atcr quickstart` (an interactive wizard) writes a `synthetic` provider registry, sets up `LLM_SYNTHETIC_API_KEY`, shows the `https://synthetic.new/` signup link (`--open` launches it), scaffolds `.github/workflows/atcr.yml`, and binds one synthetic agent per persona so the roster resolves. It is documented in `README.md:59` and the commands table. So "a new user gets a working panel with one command" is **already solved**, pointed at the flat-rate provider the maintainer earns referral credit from.
- **Persona binding today is generic, not tuned**: `internal/quickstart/manifest.go:183` binds models to personas **round-robin** (`m.Models[i%len(m.Models)]`) — not by matching a model's strength to a review lens. The maintainer's hand-tuned `~/.config/atcr/registry.yaml` (11 lenses, per-lens model choices) is the real prior art; nothing packages that as installable content.
- **Search is name/description only**: `PersonaIndexEntry` is `{Name, Version, Description, Path}` and `Search` (`internal/personas/search.go`) matches `Name`/`Description` substrings. The persona YAML carries `provider`/`model`, but that metadata never reaches the index, so discovery *by model* is impossible today without hacking the model name into free-text.

**Single-repo decision (2026-07-07):** persona content lives in **`samestrin/atcr` itself** — the product repo — not a separate `atcr/personas` org repo. One repo, one maintenance surface. The default fetch URL is repointed from `atcr/personas` → `samestrin/atcr` (+ an in-repo community-persona path). Because the content is in-repo, the normal single-repo pipeline **can** author it (the earlier "external repo, can't auto-execute" constraint no longer applies). Two accepted consequences: (1) anonymous `raw.githubusercontent` fetch only works once `samestrin/atcr` is **public** (after product launch), so the live install flow is exercised against mock/local servers until then — the existing tests already do this; (2) persona edits land on `main`, so the community-persona directory is treated as always-releasable (the squash-to-main + CI workflow already enforces this).

## Problem Statement

Three gaps, none of which the retired docs-only scope addressed:

1. **Model-specific content is stale-by-design if embedded.** Models change too fast to bake any model-named persona into a binary release; the canonical source must be fetched, not compiled in.
2. **No discovery by model.** A user who has a specific model (DeepSeek, Claude, Qwen, …) cannot ask atcr for the persona tuned to it — the index carries no model metadata and search cannot match it.
3. **No curated, tuned, installable library.** `quickstart` produces a generic round-robin panel; there is no human-named, per-model-tuned, task-scoped persona library a user can pull from — and the existing built-in stragglers (`sentinel`/`tracer`/`idiomatic`) break the all-human-names convention.

## Proposed Solution

Four capability areas (detailed decomposition is `/init-plan`'s job; these are the requirements):

1. **Community-canonical distribution from the product repo.** Repoint the default fetch URL from `atcr/personas` → `samestrin/atcr` (+ in-repo community-persona path). Treat the fetched community layer as canonical; no model-specific persona content is the binary's source of truth. `init`/`quickstart` obtain personas by **fetch-and-pin** (pin a version for reproducibility; `atcr personas upgrade` to advance). Define offline behavior (the tool needs network to review anyway; a `--offline` stub is acceptable). Preserve backward compatibility for users with existing on-disk personas.
2. **Structured model metadata + model-aware discovery.** Add `provider`/`model` (and `tasks`/`tags` as warranted) to the `index.json` entry schema and its generation, and make `atcr personas search` match them (and/or add `--model`/`--provider` filters) — so "I have model X → find its persona" works from **structured data**, never from a model name stuffed into a filename or description. Make the convention enforceable: the authoring contract requires the bound model to appear in structured metadata, asserted by the fixture test.
3. **A model-indexed, human-named persona library (in-repo content).** Author personas each bound to a specific `provider`+`model`, **human-named** (all human, no role-based names), prompt-phrased per that model's own vendor/official prompting guidance, and scoped to the review task that model is best suited for. Cover both frontier providers (Claude / GPT / Gemini) **and** strong flat-rate open models (DeepSeek / Qwen / Kimi / GLM / …). Each persona ships a passing fixture per `docs/personas-authoring.md`, and appears in `index.json` discoverable by model. **Migrate the built-in stragglers into this layout as they are renamed** (`sentinel`→`sasha`, `tracer`→`penny`, `idiomatic`→`ingrid`, with `ingrid` generalized beyond Go) — folding in Epic 23.0 so there is never a mixed-naming state.
4. **Onboarding-hierarchy documentation.** Lead first-run (`README.md` Quickstart + persona docs) with `atcr quickstart` (Synthetic — the monetizing default). Document **DashScope/Alibaba** as the secondary flat-rate option (docs-only this epic — no `quickstart` wiring). Reference **Chutes → Featherless** (in that order) as "more models, but slower / tighter context / concurrency limits — explore, not default." Add an **Advanced** note on **LiteLLM** as an OpenAI-compatible proxy to aggregate providers behind one endpoint. Position frontier/majors personas as opt-in "bring your own frontier key," kept out of the default funnel. Document the discover-and-install-by-model flow.

## Rescope & Pivot (2026-07-07)

This section records why the epic changed from its original docs-only framing. It is additive; prior history is preserved below.

**Original framing (retired):** "Author 3 majors personas (Claude/GPT/Gemini) in an external `atcr/personas` repo and add a docs recommendation to install them" — a 2/12 docs-only sprint with the persona content tracked as an un-executable external dependency.

**What discovery surfaced:**

1. **The core goal was already met.** `atcr quickstart` already delivers a one-command working panel, pointed at Synthetic — the maintainer's *monetizing* (referral-credit) path. The retired scope would have added a *majors* pack recommendation to the same first-run surface, steering new users toward providers that pay the maintainer nothing and cost users the most (flagship API × a multi-agent panel). Promoting the majors at the highest-intent onboarding surface actively diluted the funnel.
2. **The differentiated feature is majors-only by nature** ("tuned per each provider's official prompting guide"), which is structurally the part that can't monetize — so the value had to be reframed, not shipped as-is.
3. **The real gaps were architectural, not documentation:** models change too fast to embed, so the canonical source must be fetched; and there was no way to discover a persona by model. These are code features, not a markdown edit.
4. **Single-repo simplification.** Distributing from `samestrin/atcr` (not a separate `atcr/personas` org repo) removes the cross-repo constraint, collapses the previously-split "external content epic" back into this one, and lets the normal pipeline author the content.
5. **Naming consistency is a hard rule.** All personas use human first names — no mixed state — so the 23.0 straggler rename folds into the migration.

**Net:** the epic became a real in-repo feature (fetch-canonical distribution + model-aware discovery + a tuned, human-named library + onboarding-hierarchy docs), superseding the docs-only version. Complexity and plan type change accordingly; `Execution: init-plan` (the full pipeline) is retained and the maintainer runs it.

### Clarifications (2026-07-06)

This section records answers to open questions raised during `/init-plan`, additive to the content above.

#### Q1: Does this epic duplicate the existing model-tuned persona panel already in production?

**Answer:** No — they solve different problems and neither replaces the other.

The user's global `~/.config/atcr/registry.yaml` (ported from a legacy tool, `llm-tools`) already tunes *which model* plays each of 11 review lenses (`bruce`, `greta`, `kai`, `mira`, `dax`, `pace`, `vera`, `brad`, `archer`, `ronin`, `otto`) — e.g. `archer` → `glm-5.2` "for instruction precision", `kai` → `kimi-k2.7-code` ("thinking mode fixes temperature at 1.0"). This panel is in active production use and is prior art to preserve, not replace by this epic.

However, every model in that panel is MiniMax, Qwen, Kimi, Deepseek, GLM, Gemma, Nemotron, poolside, or OpenAI's own open-weight `gpt-oss-20b` (used by `vera`, which uses OpenAI's "Harmony" response format — a thin adjacent touchpoint, though `vera`'s own prompt does not use Harmony-specific conventions). None of the existing personas' prompts are phrased per Anthropic Claude's, OpenAI GPT's, or Google Gemini's own official prompting guide — that dimension of tuning does not exist anywhere yet, confirmed by inspecting every persona file in both the repo's `personas/` and the user's `~/.config/atcr/personas/`.

This epic's net-new scope is specifically that gap, requiring new research (reading each provider's official prompting guide) rather than reusing the legacy-ported panel.

#### Q2: How many new personas, and how are they bound to models?

**Answer:** 3 new personas — one for each of the top-3 frontier providers (Anthropic, OpenAI, Google), not "1-2" as originally scoped. Each:
- Has prompt phrasing written specifically per that provider's own official prompting guide (not a generic template reused across providers).
- Defaults to that provider's flagship model as primary (e.g. Claude Opus), mirroring the existing `registry.yaml` convention of a same-family fallback (e.g. `bruce`/`bruce-backup`) — each new persona should define a primary (flagship) + fallback (a lighter/cheaper model in the same provider family, e.g. Claude Sonnet) so the prompt phrasing remains valid across that provider's tiers rather than being brittle to one specific checkpoint.

This corrects the epic's original Objective/Proposed Solution/Acceptance Criteria wording ("1-2 new persona YAMLs... e.g. a Claude-tuned variant and a GPT-4-tuned variant") to 3: Anthropic, OpenAI, and Google, each with a flagship+fallback pair.

> **Note (2026-07-07 rescope):** Q2's "3 majors personas" is now a *subset* of the broader model-indexed library (AC3) — the library also covers strong flat-rate open models (DeepSeek/Qwen/Kimi/GLM), and the majors are positioned as opt-in "bring your own key," not a first-run recommendation. The flagship+fallback pairing convention still applies.
