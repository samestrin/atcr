# Epic Plan 1.5: Review Status Lifecycle (artifact contract, stale detection)

**Estimated Durations**: 1-2 weeks

## Objective

Make the review status lifecycle coherent end-to-end: one designed write-ordering contract for the status-bearing artifacts (manifest.json, summary.json), atomic-feeling status reads, and a distinct terminal state for dead reviews — without adding a competing state-bearing artifact or breaking the pinned `ReviewStatus` consumer shape.

## Context

Epic 1.0 shipped status as a derivation over two artifacts: manifest.json (roster, provenance) and sources/pool/summary.json (completion signal — absence means in_progress). AC 04-04 pins the `ReviewStatus` shape `{review_id, status, agent_count, agents_done, agents_pending}`; the MCP `StatusResult` is a type alias of it and the CLI `atcr status` consumes the same struct. In-sprint triage deferred the two-step torn-read concern as TD-023. The clarification adjudication (2026-06-11) settled the design direction: summary.json already IS the completion sentinel; a third snapshot artifact would recreate the dual-source-of-truth problem; the two-step read, dead-review terminal state, and partial-flag ordering are one coupled design on one writer/reader pair and must land together. (The partial-flag write reorder — finalized manifest before summary.json — was adjudicated as an immediate re-attempt fix; this epic owns the remaining lifecycle work and must not contradict it.)

## Problem Statement

1. **Torn reads (TD-023)**: `ReadReviewStatus` reads manifest.json then summary.json in two separate steps while the background fan-out is writing both; each file is atomic (temp+rename) but the pair is not.
2. **No terminal state for dead reviews**: absent summary.json is reported as in_progress unconditionally. A fan-out that dies after scaffolding (post-fan-out persistence failure, killed MCP server) leaves `atcr status` reporting in_progress forever, the Skill polling loop cannot distinguish running from dead, and `.atcr/latest` keeps pointing at the orphan.

## Proposed Solution

1. **Keep summary.json as the sole completion sentinel** — no new status.json snapshot, no new sentinel file. Design the read path so the manifest+summary pair is interpreted consistently with the (re-attempted) manifest-before-summary write ordering.
2. **Persist the effective timeout in the manifest**: add an optional `timeout_secs` (omitempty) field to `payload.Manifest` — additive to the plan-owned schema, degrades gracefully for old manifests (zero value = unknown).
3. **Stale detection in `ReadReviewStatus`**: when summary.json is absent AND `StartedAt + timeout_secs` (plus a grace margin) has elapsed, report status `"stale"` — a new additive value beside in_progress/completed/failed (AC 04-04 enum extension recorded as a contract amendment). The state is honestly labeled as inferred, not observed.
4. **Post-fan-out persistence failure maps to `failed`**, not a second new state: on a WritePool/WriteManifest failure, ExecuteReview writes a best-effort failure record the reader understands through the existing summary-derived path where possible.
5. **Preserve the `ReviewStatus` shape**: `{review_id, status, agent_count, agents_done, agents_pending, partial}` stays intact for the CLI and MCP alias; only the `status` value set grows.
6. **Docs/AC**: record the enum extension and the stale semantics in the AC 04-04 contract and skill polling guidance (poll loop treats `stale` as terminal).

## Success Criteria

- [ ] `payload.Manifest` carries the effective timeout (`timeout_secs`, omitempty); old manifests without it still load and report status without stale inference.
- [ ] A scaffolded review with no summary.json and `StartedAt + timeout` elapsed reports status `stale` (test with an injected stale manifest); within the window it still reports `in_progress`.
- [ ] An injected WritePool failure yields a state the reader reports as `failed`, not eternal `in_progress` (test).
- [ ] `ReviewStatus` JSON shape is unchanged apart from the new `status` value; MCP `StatusResult` alias and `atcr status` output stay compatible (existing tests green).
- [ ] Read-path interpretation is consistent with the manifest-before-summary write ordering (no torn-pair misreport under mid-write reads; concurrency test or documented invariant).
- [ ] AC 04-04 contract amendment and skill polling guidance recorded.

## Task Breakdown

1. Manifest: additive `timeout_secs` field + ExecuteReview/PrepareReview stamping + backward-compat tests.
2. Reader: stale inference in `ReadReviewStatus` (absent summary + elapsed timeout + grace) + `RunStale` constant + tests.
3. Failure marker: post-fan-out persistence-failure path writes a reader-understood failure record + tests.
4. Torn-read coherence: verify/encode the manifest-before-summary ordering invariant on the read side + concurrency test.
5. Surfaces: CLI `atcr status` and MCP handler pass-through of the new state (alias untouched) + tests.
6. Docs: AC 04-04 amendment, skill polling guidance, registry.md/docs notes as applicable.

## Out of Scope

- A consolidated status.json snapshot artifact or any new sentinel file (rejected by adjudication — would recreate dual source of truth).
- Per-agent incremental progress mid-run (agents_done granularity — separate open TD item on status.go:65).
- Changing the `ReviewStatus` field set or the MCP tool schema shape.

## Dependencies

- The adjudicated re-attempt fix reordering ExecuteReview (finalized manifest before summary.json) should land first — this epic's read-path design assumes that ordering.
- Epic 1.0 fan-out and status reader (complete, in code review).

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Stale inference fires on a legitimately slow run | Medium | Medium | Grace margin atop the persisted effective timeout; status string honestly signals inference |
| Enum extension breaks a consumer hard-coding the three states | Low | Medium | Contract amendment recorded in AC 04-04; grep consumers (CLI, MCP, skill) in-task |
| Clock skew between writer and reader hosts | Low | Low | Same-host tool in practice; document the assumption |
