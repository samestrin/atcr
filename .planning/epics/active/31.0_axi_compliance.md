## Objective

Implement an `--axi` (Agent eXperience Interface) output mode to make `atcr` a foundational tool for *other* autonomous agents. By stripping human-ergonomic formatting (Markdown, ANSI colors, ASCII tables) in favor of token-dense TOON/JSON output, predictable pagination, and strict exit codes, `atcr` becomes safely composable in agentic workflows (e.g., an autonomous sweeper invoking `atcr` to review its own generated code).

## Context

The AXI (Agent eXperience Interface) design principles argue that CLIs designed for humans (with rich formatting, colored output, and verbose logs) are actively hostile to LLM agents. They consume excessive context window tokens and are brittle to parse. Currently, ATCR outputs beautifully formatted Markdown and terminal tables.

By offering a first-class `--axi` flag, ATCR bridges the gap between human engineering teams and autonomous swarms. An agent writing code can run `atcr review --axi`, consume the findings at a fraction of the token cost, and fix its own code programmatically based on the predictably structured output.

## Proposed Solution

1. **Token-Dense Output Mode (`--axi`)**: Intercept the standard CLI output writers (`internal/cli/formatters`) and replace them with a TOON (Token Optimized Object Notation) or compact JSON formatter when the flag is passed. This must strip all ANSI color codes, table padding, and visual dividers.
2. **Predictable Exit Codes**: Ensure `atcr review` and other subcommands exit with distinct codes:
   - `0`: Success (No findings or all findings below threshold).
   - `1`: Actionable Findings (Skeptic-verified findings were identified).
   - `2`: Internal/Syntax Error (Invalid YAML, model API failure, etc.).
3. **Pagination & Truncation Guidelines**: When emitting diffs or massive lists of findings, enforce a strict line cap (default 500 lines) and emit a "truncated" flag. Allow this to be overridden via an `ATCR_AXI_MAX_LINES` environment variable. The calling agent must be able to cleanly handle massive PRs without blowing its context window.
4. **Stderr Isolation**: Strictly separate diagnostic/progress logs (stderr) from the actual review payload (stdout). If an agent runs `atcr review > findings.json`, `findings.json` must be perfectly clean.

## Acceptance Criteria

- [ ] `atcr review --axi` outputs a clean, machine-readable (TOON/JSON) payload free of ANSI codes or markdown tables.
- [ ] Stderr is strictly used for progress bars/logs; stdout contains only the final payload.
- [ ] Exit codes deterministically reflect the review outcome (`0` = clean, `1` = findings, `2` = error).
- [ ] The `atcr` docs include a section on "Agentic Consumption" explaining how to orchestrate `atcr` in larger swarms.
