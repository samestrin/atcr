# Epic 7.4: Executor Agent Mode

**Status:** Draft
**Created:** 2026-06-21
**Priority:** Medium (quality enhancement, opt-in)
**Estimated time:** 1 week
**Tasks/Components:** 3/1

## Problem

Epic 7.0's executor generates fixes from a pre-loaded code snippet (±N lines around `file:line` from the review snapshot). For simple findings this is sufficient. For complex ones — cross-file issues, problems involving types or interfaces defined elsewhere, cascading dependencies — the snippet may not capture enough context for a precise fix. A developer fixing that finding by hand would read imports, check callers, look at the type definition. The executor currently cannot.

## Solution

An optional `agent_mode` flag on the executor registry configuration. When enabled, the executor receives the same read-only dispatcher the skeptics use (already open during `runVerify`) and runs a tool loop — calling `read_file`, `search_files`, etc. — to explore the codebase before proposing a fix. When disabled (default), the executor uses the Epic 7.0 snippet-based path unchanged.

```yaml
# registry.yaml
executor:
  name: opus
  provider: anthropic
  model: claude-opus-4-8
  persona: fixer
  role: executor
  agent_mode: true      # opt-in; default false
  max_tool_calls: 10    # budget cap; default 10
```

**How agent mode works:**
1. `runVerify` already has the dispatcher open when skeptics run
2. For each eligible finding, instead of a single-turn `callExecutor`, dispatch `invokeExecutor` — a tool loop analogous to `invokeSkeptic`
3. The executor explores the code (reads files, searches), then returns `{"fix": "..."}` JSON
4. If the loop times out, trips the budget, or errors, the finding gets a `FixWarning` — graceful degradation, never a failed run

## Key Design Decisions

### 1. Off by default
- Default `agent_mode: false` — Epic 7.0 snippet path, no change to existing behavior
- Users opt in by setting `agent_mode: true` on the executor block
- Rationale: multiple tool-call round-trips add latency and token cost that not every user wants to pay

### 2. Reuse the existing dispatcher — no second snapshot
- The snapshot (`snapshotRoot`) and dispatcher are already built lazily in `runVerify` when at least one skeptic runs
- The executor borrows the same dispatcher; no second `buildDispatcher` call, no second git checkout
- If no skeptics run (no eligible findings or harness unavailable), agent mode degrades gracefully to the snippet path

### 3. `invokeExecutor` mirrors `invokeSkeptic` structurally
- Same tool loop pattern (read-only dispatcher), different goal (generate a fix, not issue a verdict)
- Executor prompt is constructive: "Read the code, then propose the minimal fix." — no adversarial framing
- Response schema: `{"fix": "...", "explanation": "..."}` — simpler than the verdict schema
- `max_tool_calls` caps total tool calls per finding (default 10, configurable)

### 4. Graceful degradation
- Tool loop timeout or error → `FixWarning = "agent_mode failed: ..."`, finding emitted without fix
- No `agent_mode` capability but `agent_mode: true` set AND dispatcher unavailable → fall back to snippet path, log warning
- Never fails the run or drops a finding

### 5. Configurable budget cap
- `max_tool_calls` on the executor block (default 10) bounds cost per finding
- Executor cannot read more files than the cap allows; beyond the cap it must propose the fix from what it has
- Mirrors `DefaultLimits()` used for skeptics

### `invokeExecutor` (new function in `internal/verify/executor.go`)

```go
// invokeExecutor runs the executor in agent mode: it drives a tool loop against
// disp (the same dispatcher skeptics use), then parses the JSON fix response.
// It never returns an error — failures produce a FixWarning string instead.
func invokeExecutor(ctx context.Context, exec ExecutorConfig, finding reconcile.JSONFinding,
    cc fanout.ChatCompleter, disp Dispatcher) (fix string, warn string) {

    limits := tools.DefaultLimits()
    if exec.MaxToolCalls > 0 {
        limits.MaxCalls = exec.MaxToolCalls
    }

    prompt := buildExecutorAgentPrompt(finding) // no pre-loaded entries — reads via tools
    // tool loop identical to invokeSkeptic's loop, different response schema
    resp, tripped, err := runToolLoop(ctx, exec.toAgent(), prompt, cc, disp, limits)
    if err != nil || len(tripped) > 0 {
        return "", fmt.Sprintf("agent_mode failed: %v (tripped: %v)", err, tripped)
    }
    fix, err = parseExecutorResponse(resp)
    if err != nil {
        return "", fmt.Sprintf("agent_mode parse error: %v", err)
    }
    return fix, ""
}
```

### Call site in `runVerify` (extends Epic 7.0 fix-generation block)

```go
// After verdict application, before computeFindingsBytes:
if reg.Executor != nil {
    for i, f := range findings {
        if !eligibleForFix(f, reg) {
            continue
        }
        var fix, warn string
        if reg.Executor.AgentMode && disp != nil {
            fix, warn = invokeExecutor(ctx, *reg.Executor, f, cc, disp)
        } else {
            entries := readSnippetFromSnapshot(snapshotRoot, f.File, f.Line, 30)
            fix, warn = callExecutor(ctx, *reg.Executor, f, entries, cc)
        }
        if warn != "" {
            findings[i].FixWarning = warn
        } else {
            findings[i].Fix = fix
            findings[i].Executor = reg.Executor.Name
        }
    }
}
```

### Executor Agent Prompt

```
You are a code fix generator. Your job is to produce the minimal, correct fix for the
following finding. You have access to tools to read files and search the codebase.

1. Read the file at the cited location first.
2. Follow imports or callers if needed to understand the full context.
3. Propose the minimal change that fixes the problem without breaking existing behavior.
4. Preserve existing style and conventions.

<finding>
**Problem:** ...
**Severity:** ...
**Location:** file:line
**ReviewerFix:** (existing reviewer suggestion, if any)
**Evidence:** ...
</finding>

Return a JSON object and nothing else:
{"fix": "one-sentence or inline description of exact change", "explanation": "why this fixes it"}
```

## Acceptance Criteria

- [ ] AC1: `agent_mode: false` (default) behaves identically to Epic 7.0 — no regression, no new snapshot I/O
- [ ] AC2: `agent_mode: true` enables the tool loop for fix generation; executor reads at minimum `file:line` before proposing a fix
- [ ] AC3: `max_tool_calls` is respected; executor must emit a fix from available context when cap is reached
- [ ] AC4: Tool loop timeout or error produces `FixWarning` on the finding; the run continues and all other findings are emitted normally
- [ ] AC5: Agent mode reuses the already-open dispatcher; no second `buildDispatcher` call, no second git checkout
- [ ] AC6: If dispatcher is unavailable (no skeptics ran, harness failed) and `agent_mode: true`, executor falls back to snippet path with a logged warning
- [ ] AC7: Unit tests cover: agent_mode=false (snippet path), agent_mode=true success, agent_mode=true tool-loop failure (warning, no crash), max_tool_calls cap
- [ ] AC8: Registry validation rejects `agent_mode: true` when executor block is absent

## Success Criteria

- [ ] Executor in agent mode reads the actual code at `file:line` before every fix
- [ ] Fix quality measurably improves over snippet-only for cross-file findings (manual review of 5 sample findings)
- [ ] No second snapshot checkout; wall-clock overhead is tool-call latency only
- [ ] `agent_mode: false` users see zero change in behavior or performance

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Executor runs many tool calls per finding, high cost | Medium | Medium | `max_tool_calls` cap; default 10 is conservative |
| Agent loop times out on slow providers | Low | Low | Timeout → FixWarning, run continues |
| Executor hallucinates code it didn't actually read | Medium | Medium | Prompt instructs it to read first; fix is advisory only |
| Dispatcher unavailable when `agent_mode: true` (no skeptics) | Low | Low | Falls back to snippet path with warning |

## Open Questions

1. **Should `invokeExecutor` share a common tool-loop function with `invokeSkeptic`, or duplicate the loop?**
   - Option A: Extract `runToolLoop(ctx, agent, prompt, cc, disp, limits)` helper shared by both
   - Option B: Duplicate the loop (simpler, no abstraction)
   - Recommendation: Option A — the loop logic is identical; deduplication reduces drift risk

2. **Should `max_tool_calls` default to the same `DefaultLimits().MaxCalls` as skeptics, or a separate constant?**
   - Recommendation: separate constant `defaultExecutorMaxToolCalls = 10` — executor's task is narrower than a skeptic's and should be cheaper
