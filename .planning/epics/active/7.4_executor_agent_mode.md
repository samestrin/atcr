# Epic 7.4: Executor Agent Mode

**Status:** Draft
**Created:** 2026-06-21
**Priority:** Medium (quality enhancement, opt-in)
**Estimated time:** 1 week
**Tasks/Components:** 3/1

## Problem

Epic 7.0's executor generates fixes from a pre-loaded code snippet (Â±N lines around `file:line` from the review snapshot). For simple findings this is sufficient. For complex ones â€” cross-file issues, problems involving types or interfaces defined elsewhere, cascading dependencies â€” the snippet may not capture enough context for a precise fix. A developer fixing that finding by hand would read imports, check callers, look at the type definition. The executor currently cannot.

## Solution

An optional `agent_mode` flag on the executor registry configuration. When enabled, the executor receives the same read-only dispatcher the skeptics use (already open during `runVerify`) and runs a tool loop â€” calling `read_file`, `search_files`, etc. â€” to explore the codebase before proposing a fix. When disabled (default), the executor uses the Epic 7.0 snippet-based path unchanged.

```yaml
# registry.yaml
executor:
  name: opus
  provider: anthropic
  model: claude-opus-4-8
  persona: fixer
  role: executor
  agent_mode: true      # opt-in; default false
  max_tool_calls: 10    # budget cap; default 10
```

**How agent mode works:**
1. `runVerify` already has the dispatcher open when skeptics run
2. For each eligible finding, instead of a single-turn `callExecutor`, dispatch `invokeExecutor` â€” a tool loop analogous to `invokeSkeptic`
3. The executor explores the code (reads files, searches), then returns `{"fix": "..."}` JSON
4. If the loop times out, trips the budget, or errors, the finding gets a `FixWarning` â€” graceful degradation, never a failed run

## Key Design Decisions

### 1. Off by default
- Default `agent_mode: false` â€” Epic 7.0 snippet path, no change to existing behavior
- Users opt in by setting `agent_mode: true` on the executor block
- Rationale: multiple tool-call round-trips add latency and token cost that not every user wants to pay

### 2. Reuse the existing dispatcher â€” no second snapshot
- The snapshot (`snapshotRoot`) and dispatcher are already built lazily in `runVerify` when at least one skeptic runs
- The executor borrows the same dispatcher; no second `buildDispatcher` call, no second git checkout
- If no skeptics run (no eligible findings or harness unavailable), agent mode degrades gracefully to the snippet path

### 3. `invokeExecutor` mirrors `invokeSkeptic` structurally
- Same tool loop pattern (read-only dispatcher), different goal (generate a fix, not issue a verdict)
- Executor prompt is constructive: "Read the code, then propose the minimal fix." â€” no adversarial framing
- Response schema: `{"fix": "...", "explanation": "..."}` â€” simpler than the verdict schema
- `max_tool_calls` caps total tool calls per finding (default 10, configurable)

### 4. Graceful degradation
- Tool loop timeout or error â†’ `FixWarning = "agent_mode failed: ..."`, finding emitted without fix
- No `agent_mode` capability but `agent_mode: true` set AND dispatcher unavailable â†’ fall back to snippet path, log warning
- Never fails the run or drops a finding

### 5. Configurable budget cap
- `max_tool_calls` on the executor block (default 10) bounds cost per finding
- Executor cannot read more files than the cap allows; beyond the cap it must propose the fix from what it has
- Mirrors `DefaultLimits()` used for skeptics

### `invokeExecutor` (new function in `internal/verify/executor.go`)

```go
// invokeExecutor runs the executor in agent mode: it drives a tool loop against
// disp (the same dispatcher skeptics use), then parses the JSON fix response.
// It never returns an error â€” failures produce a FixWarning string instead.
func invokeExecutor(ctx context.Context, exec ExecutorConfig, finding reconcile.JSONFinding,
    cc fanout.ChatCompleter, disp Dispatcher) (fix string, warn string) {

    limits := tools.DefaultLimits()
    if exec.MaxToolCalls > 0 {
        limits.MaxCalls = exec.MaxToolCalls
    }

    prompt := buildExecutorAgentPrompt(finding) // no pre-loaded entries â€” reads via tools
    // tool loop identical to invokeSkeptic's loop, different response schema
    resp, tripped, err := runToolLoop(ctx, exec.toAgent(), prompt, cc, disp, limits)
    if err != nil || len(tripped) > 0 {
        return "", fmt.Sprintf("agent_mode failed: %v (tripped: %v)", err, tripped)
    }
    fix, err = parseExecutorResponse(resp)
    if err != nil {
        return "", fmt.Sprintf("agent_mode parse error: %v", err)
    }
    return fix, ""
}
```

### Call site in `runVerify` (extends Epic 7.0 fix-generation block)

```go
// After verdict application, before computeFindingsBytes:
if reg.Executor != nil {
    for i, f := range findings {
        if !eligibleForFix(f, reg) {
            continue
        }
        var fix, warn string
        if reg.Executor.AgentMode && disp != nil {
            fix, warn = invokeExecutor(ctx, *reg.Executor, f, cc, disp)
        } else {
            entries := readSnippetFromSnapshot(snapshotRoot, f.File, f.Line, 30)
            fix, warn = callExecutor(ctx, *reg.Executor, f, entries, cc)
        }
        if warn != "" {
            findings[i].FixWarning = warn
        } else {
            findings[i].Fix = fix
            findings[i].Executor = reg.Executor.Name
        }
    }
}
```

## Acceptance Criteria

- [ ] AC2: `agent_mode: true` enables the tool loop for fix generation; executor reads at minimum `file:line` before proposing a fix

**Key Decisions:**
- The plan's pseudocode references symbols that do not exist in the Epic 7.0 codebase
  (`runToolLoop`, `exec.toAgent()`, `invokeSkeptic` in skeptic.go, `tools.DefaultLimits().MaxCalls`).
  Implementation follows the actual code: the tool loop is `fanout.Engine`, and
  `invokeExecutor` mirrors `invokeSkeptic` (internal/verify/invoke.go) â€” building a
  tool-enabled `fanout.Agent` and running a throwaway `fanout.NewEngine(cc, WithDispatcher(disp))`.
- `max_tool_calls` maps to the fanout Agent's `MaxTurns` budget (the real per-finding
  tool-call cap; engine default is already 10). `tools.Limits` caps result SIZES, is baked
  into the shared dispatcher at construction, and AC5 forbids rebuilding it â€” so it is not
  the right knob.
- AC8 is satisfied-by-construction: `agent_mode` is a field on `ExecutorConfig`, and
  `Registry.Executor` is `*ExecutorConfig`. An absent `executor:` block leaves the pointer
  nil and `validateExecutor()` returns immediately (`if e == nil { return nil }`,
  internal/registry/config.go:486). The rejected state is inexpressible in YAML. The
  meaningful replacement: a unit test asserting an absent-executor config is valid, plus
  validation rejecting an explicitly-set `max_tool_calls` outside bounds.
- `max_tool_calls` is validated within `1..MaxAgentTurns` (1..1000) via a new named
  constant `MaxExecutorToolCalls = MaxAgentTurns`, consistent with the codebase's
  bounded-count-field pattern (max_turns 1..MaxAgentTurns, max_findings 1..MaxFindingsCap).
  Unset or â‰¤0-after-parse defaults to 10 at call time.
- The executor block gets NO new function-calling flag. When `agent_mode: true`, the
  executor's fanout.Agent is built with `SupportsFC: true` (opting into agent mode asserts
  a tool-capable model); a non-FC model degrades to single-shot gracefully in the engine.
- `parseExecutorResponse` reuses the existing `extractJSONObject` helper (verdict.go).

**Scope Boundaries:**
- IN scope: `AgentMode bool` + `MaxToolCalls int` on ExecutorConfig; `validateExecutor`
  bound check + `MaxExecutorToolCalls` constant; `invokeExecutor`, `buildExecutorAgent`,
  `buildExecutorAgentPrompt`, `parseExecutorResponse` in internal/verify/executor.go;
  agent-mode branch wired into `generateFixes` with `cc fanout.ChatCompleter` threaded
  through; fallback to snippet path (+ logged warning) when dispatcher/cc unavailable;
  unit + integration tests; update examples/registry-with-executor.yaml and docs/registry.md.
- OUT of scope: any change to the Epic 7.0 snippet path's behavior (agent_mode=false is

  top-level config surface.

**Technical Approach:**
- `generateFixes` gains a `cc fanout.ChatCompleter` parameter; runVerify already has `cc`

- Failure isolation matches Epic 7.0: a tool-loop timeout, tripped budget, provider error,
  or parse failure sets `FixWarning` on the finding and never fails the run or drops a finding.
