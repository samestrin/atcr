# Sprint Plan 14.1: Documentation & Reviewer Personas

- **Estimated time**: 4 days
- **Tasks**: 5
- **Components**: 3

## Context

A command-line developer utility is only as good as its documentation. Developers need to know:
1. How to run reviews and reconcile findings.
2. How the config schemas are defined.
3. Who the individual reviewer personas are (their specializations, focal points, and limitations).
This sprint addresses porting and refining the `docs.html`, `faq.html`, and `personas.html` layouts.

## Problem Statement

We need to provide:
1. Fully populated and structured CLI instruction documents covering standard commands (`atcr review`, `atcr reconcile`, `atcr scorecard`).
2. An interactive FAQ page handling frequently asked questions regarding offline capabilities, costs, and token consumption.
3. A beautiful "Reviewer Directory" showcasing the panel members (dossier cards for Greta, Bruce, Kai, Mira, Otto, etc.) with consistent visual design cues (top keylines reflecting persona accent colors and code sample outputs).

## Proposed Solution

1. Port mockup structures for [docs.html](file:///Users/samestrin/Documents/GitHub/atcr.dev/the-artifact/mockups/docs.html) and [faq.html](file:///Users/samestrin/Documents/GitHub/atcr.dev/the-artifact/mockups/faq.html) to the production source.
2. Port and format [personas.html](file:///Users/samestrin/Documents/GitHub/atcr.dev/the-artifact/mockups/personas.html) structure, ensuring the reviewer dossiers are complete.
3. Ensure absolute styling consistency with the shared CSS layout tokens (IBM Plex fonts and HSL colors).

## User Personas

### Primary: Lead Architect (Alistair)
- Needs to review configuration structures and custom rules.
- Wants to understand which reviewer personas (e.g. Mira for security, Otto for architecture) should be included in their team review panels.

## User Journeys

### Reviewing Personas & Reading Docs
1. Navigates to the "Personas" page.
2. Reads the dossiers for each model persona.
3. Navigates to the "Docs" page to see how to enable specific personas in their `atcr.yaml` configuration file.
4. Copies example configuration values from the documentation block.

## Acceptance Criteria

- [ ] AC1: `docs.html` contains detailed references for command execution, configuration parameters, and environment keys.
- [ ] AC2: `faq.html` provides a clean accordion-style layout or clear text block segments matching the design theme.
- [ ] AC3: `personas.html` renders all reviewers as distinct cards with top color-keylines corresponding to their signature colors (`--src-greta`, `--src-mira`, etc.).
- [ ] AC4: Canonical findings on the persona cards are rendered using the dark code artifact block styling.

## Success Criteria

### Functional
- [ ] Tab and anchor links navigate correctly to page segments without offset errors.
- [ ] High-contrast readability is preserved for long-form prose.

## Technical Constraints

- HTML files must use standard relative links (`index.html`, `docs.html`, etc.) for portability.
- Avoid external CSS libraries; styling details must reside purely in `styles.css`.

## Out of Scope

- Integrating live leaderboard data (covered in Sprint 14.2).
- Deploying the site to production host environments (covered in Sprint 14.2).
