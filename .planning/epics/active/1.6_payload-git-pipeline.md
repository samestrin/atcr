# Epic Plan 1.6: Payload Git Pipeline Batching

**Estimated Durations**: 1 week

## Objective

Batch the payload builders' per-file git classification into one `--numstat`/`--name-status` pass plus a single split whole-range diff, so building blocks/files payloads costs a constant number of git processes instead of 4-5 per changed file.

## Context

Epic 1.0 shipped the payload engine with a per-file fan-out: `BuildEntries` lists changed files once (`git diff --name-status -M`), then `fileBody` issues up to 4-5 additional git processes per file — `--numstat` (binary check), `--function-context`, `--unified=10` (fallback), `show` (head content), and `--unified=0` (changed ranges) in `internal/payload/builder.go` and `internal/payload/diff.go`. This was a recorded sprint deferral (sprint-plan.md: "builder.go per-file fan-out | N×4-5 git processes | Deferred → TD-011"). The rename-pairing fix for `internal/payload/diff.go:78` (clarification Q3, 2026-06-11) introduces the split-whole-range-diff mechanism this epic generalizes: parse one `-M` diff of the full range and split it per file on column-0 `diff --git` boundaries keyed against the `changedFiles` list.

## Problem Statement

For a review range touching N files, blocks/files payload builds spawn O(N) git processes (2-3 extra per file beyond the classification pass). On large ranges this dominates payload build time, and the duplicated per-file invocations have already produced contract drift (trimmed bodies, rename pathspec loss) because each helper re-derives state the whole-range diff already contains. The modes are functionally correct today — this is performance and structural debt, not a correctness bug.

## Proposed Solution

1. **Single classification pass**: derive change kind, rename pairing, and binary status for all files from one `--name-status -M` plus one `--numstat` invocation over the whole range (replacing the per-file `isBinary` calls).
2. **Single split diff per mode**: run the whole-range diff once per needed variant (`--function-context`, `--unified=10`, `--unified=0`) and split per file on `diff --git` boundaries keyed against the changed-file list, instead of one git process per file per variant.
3. **Preserve contracts**: verbatim bodies (raw bytes, trailing-newline fidelity), the blocks-mode fallback with its slog record, binary/deleted markers, and the entries-join-to-flat-payload bridge all remain byte-identical where unaffected; existing builder tests pin this.
4. **Rename pairing for free**: the split diff carries `-M` rename pairing, subsuming the per-file pathspec problem fixed for diff.go:78.

## Success Criteria

- [ ] Building a blocks or files payload for a range touching N files issues a constant number of git processes (independent of N, excluding `show` for files-mode head content).
- [ ] All existing `internal/payload` tests pass unchanged, including verbatim-body, sentinel, fallback-record, rename, binary, and entries-bridge tests.
- [ ] A benchmark or test demonstrates the process-count reduction on a multi-file range.
- [ ] The blocks-mode function-context fallback and its slog record survive the restructure.

## Out of Scope

- Changing payload output formats or sentinel/marker contracts.
- The byte-budget drop policy (tracked separately at `internal/payload/budget.go:34`).
- Error-classification changes in git helpers beyond what the restructure requires (tracked at `internal/payload/diff.go:122`).

## Dependencies

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Diff splitter mis-attributes hunks to files | Low | High | Split on column-0 `diff --git` boundaries keyed against the same `-M` changed-file list; pin with multi-file + rename + binary tests |
| Behavior drift in fallback/marker paths | Medium | Medium | Existing tests pin verbatim bodies, fallback record, and markers; keep them green throughout |
| Memory growth holding whole-range diffs in memory | Low | Low | Payloads already concatenate all bodies in memory; the byte budget bounds final size |
