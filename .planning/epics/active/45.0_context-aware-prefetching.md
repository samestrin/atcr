# Feature Request: Context-Aware Pre-Fetching

- **Estimated time**: 2 weeks
- **Tasks**:
  - T1: Extract key symbols (function names, types, imports) from changed blocks in the payload generation phase.
  - T2: Query local index (e.g., `qdrant_fts.db` or similar) for symbols and retrieve the 2-3 most relevant snippets.
  - T3: Implement context token/byte capping and fallback logic.
  - T4: Inject "Context Definitions" into the review payload generation flow.
  - T5: Add unit/integration tests to verify the pre-fetching latency and accuracy.
- **Components**:
  - `internal/payload`
  - `internal/fanout`

## Context

Competitor products like Greptile build a full semantic graph of a repository to provide deep, whole-codebase context to their AI reviewers. While `atcr` agents can use tools (`tools: true` for `read_file`, `grep`) to explore a repo, each tool call consumes tokens, extends review time, and requires a highly capable (and expensive) reasoning model. We need a way to provide relevant global context upfront without the latency of multi-turn tool usage.

## Problem Statement

Reviewer agents receive the `blocks`, `diff`, or `files` payload, which focuses only on the changed regions. If a changed function calls an obscure utility function in another file, the LLM won't know what that utility does unless it explicitly uses a tool to look it up, leading to potential hallucinations or false positives about missing logic.

## Proposed Solution

Introduce a lightweight, automated semantic pre-fetching mechanism into the `atcr review` payload generation phase:
1. When generating the payload, extract the key symbols (function names, types, imports) from the changed blocks.
2. Query a local semantic or structural index (e.g., leveraging the existing `qdrant_fts.db` or `llm-semantic-mcp`) to find the 2-3 most relevant file snippets or definitions.
3. Inject these snippets into the payload as a "Context Definitions" section *before* passing the payload to the LLMs.

## User Personas

### Primary: Software Engineer
- Wants the AI to understand project-specific utility functions and types without having to explain them in the PR description.
- Wants fast reviews without the latency overhead of 5+ multi-turn tool calls per agent.

## User Journeys

### Code Review with Pre-fetched Context
1. User runs `atcr review` on a branch that heavily uses a custom internal `DatabaseClient` class.
2. The payload generator detects the usage of `DatabaseClient`, queries the local index, and retrieves the class definition.
3. The LLMs receive the diff *and* the `DatabaseClient` definition.
4. The LLMs provide accurate feedback based on the actual method signatures of `DatabaseClient` rather than hallucinating them.

## Acceptance Criteria

- [ ] AC1: Payload generation phase includes an optional step to query a local index for related symbols.
- [ ] AC2: Pre-fetched context is capped at a configurable byte/token limit to avoid blowing out the context window.
- [ ] AC3: Pre-fetched context is deterministically identical for all agents on the same run (unless mode-specific overrides are applied).
- [ ] AC4: Performance overhead of context pre-fetching is < 2 seconds per review.

## Technical Constraints

- Must run locally without requiring a massive cloud-hosted index.
- Should leverage existing tools in the repository (e.g., `qdrant_fts.db`) or standard local indexing solutions.
- Must fall back gracefully (empty context) if the index is unavailable or unpopulated.

## Out of Scope

- Full repository graph generation (we are not building a language server from scratch).
- Multi-repository context sharing.
