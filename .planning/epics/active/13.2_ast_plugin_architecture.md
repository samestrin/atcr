# Sprint Plan 13.2: AST Plugin Architecture

**Estimated Duration**: 1.5 weeks
**Status**: Draft

## Objective

Introduce AST-aware structural hashing to map flagged lines to their covering AST nodes. To avoid bloat and CGO dependencies, implement a dynamic plugin system (e.g., using WebAssembly) to load language-specific parsers on demand.

## Problem Statement

-   **Line number drift**: Models frequently hallucinate or skew line numbers by ±3 lines. Current proximity grouping is brittle.
-   **Binary bloat**: Compiling Tree-sitter parsers for every language natively into the Go binary creates massive bloat and CGO headaches.

## Proposed Solution

1.  **Wasm Plugin Architecture**: Compile Tree-sitter parsers to WebAssembly (`.wasm`). The reconciler dynamically loads the `.wasm` parser only for the language currently being reviewed (e.g., loading `go.wasm` or `python.wasm`).
2.  **AST Isomorphism**: Map the raw line numbers flagged by models to their smallest covering Abstract Syntax Tree (AST) node.
3.  **Merkle Hashing**: Compare structural hashes of the AST nodes. If two models flag different lines but map to the same logical `for` loop or `if` block, group them deterministically.

## Acceptance Criteria

- [ ] AC1: Wasm host runtime (e.g., wazero) integrated into the reconciler with zero CGO dependencies.
- [ ] AC2: Mechanism to fetch and cache language-specific `.wasm` parser plugins dynamically.
- [ ] AC3: AST mapping correctly groups findings that are offset by whitespace or minor line-number drift.
