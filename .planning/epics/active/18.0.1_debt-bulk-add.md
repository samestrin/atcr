## Objective

Add a batched write path to the technical-debt writer so adding N items costs one
README rewrite, one shard sync, and one stats refresh instead of N of each. The
single-item interactive path (`atcr debt add`) stays exactly as-is.

## Current State

`debt.AppendItem` (`internal/debt/add.go:167`) is the single authoritative TD
writer. Per call it:

1. Acquires the shared README lock (`withReadmeLock`),
2. Reads + re-parses the README (`os.ReadFile` â†’ `insertRow` â†’ `ParseREADME`),
3. Writes the README,
4. Regenerates the entire shard store via a full `SyncShards` migrate,
5. Refreshes the `## Stats` block via `RefreshStats` (another read + parse + write).

At N=1 this is correct and cheap. Adding N items in a loop, however, is O(N) full
README rewrites and ~2N shard-directory rewrites â€” the cost the original review
finding flagged.

## Why this is deferred (not built now)

There is **exactly one caller** of `AppendItem` today â€” `cmd/atcr/debt_add.go:167`,
the single-item `atcr debt add`. No code path bulk-adds. Building `AppendItems`
now would ship an exported API with zero callers (speculative, against epic 18.0's
YAGNI stance), and inventing `atcr debt add --from <file>` to justify it would
re-open the already-completed 18.0 contract (whose AC scoped `add` as single-item).

## Revisit Trigger

Build this the moment a **real bulk-add caller** is introduced â€” e.g. an
`atcr debt add --from <file>` import command, a migration/backfill that files many
rows, or any loop over `AppendItem` in production code. Until then this stays
deferred.

## Proposed Solution (when triggered)

Expose `debt.AppendItems(readmePath, itemsDir string, entries []<section+item>, stderr io.Writer) error` that:

- Acquires `withReadmeLock` **once** for the whole batch.
- Reads + parses the README once, inserts every row into the in-memory content,
  validates the mutated table parses cleanly once, writes the README once.
- Runs `SyncShards` once after all rows are inserted.
- Runs `RefreshStats` once.
- Preserves `AppendItem`'s best-effort rollback semantics (restore original bytes
  if shard sync or stats refresh fails) at batch granularity.

Keep `AppendItem` as the single-item entry point (it may delegate to
`AppendItems` with a one-element slice, or stay independent â€” implementer's call).

## Scope

### In Scope

## Acceptance Criteria

- [ ] AC1: `debt.AppendItems` files a batch of N items with exactly one README write, one `SyncShards`, and one `RefreshStats` (assert via call counts or instrumentation).
- [ ] AC2: A benchmark shows the 100-item batch performs materially fewer full-store rewrites than 100 single adds (target: 1 README write + 1 shard sync vs 100 + 100).
- [ ] AC3: Batch write is atomic â€” a failure mid-batch leaves the README and shards at their pre-call state (rollback verified by test).
- [ ] AC4: The single-item `AppendItem` path and its existing tests remain unchanged and green.
- [ ] AC5: Concurrent batch + single-item writers still serialize correctly through the shared `td-readme.lock` (extend the existing concurrency test).

## Risks

| Risk | Mitigation |
|------|------------|
| Batch rollback leaves partial state | Snapshot original README bytes before the batch; restore on any post-write failure, mirroring `AppendItem`. |
| Batch interleaves with a skill session mid-write | Single lock acquisition for the whole batch already prevents this â€” do NOT release between rows. |
| API drift from single-item path | Have `AppendItem` delegate to `AppendItems` with len==1 so both share one code path. |
