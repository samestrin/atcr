# Epic Plan 27.0: Local Model / Ollama Persona Integration

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Deliver a "Zero Data Egress" out-of-the-box local persona (using Ollama or llama.cpp) in the community registry to completely resolve enterprise privacy objections during onboarding.

## Context

While LiteLLM and proxy configurations are supported, privacy-conscious enterprise teams often refuse to send proprietary code to external APIs like OpenAI or Anthropic. Having a first-class, officially supported local persona (e.g., pointing to `localhost:11434` with a highly optimized prompt for `Ornith 1.0` or `Qwen 3.6`) makes the "privacy first" adoption path friction-free.

## Proposed Solution

1. **Local Persona Definition**: Author a persona using the human naming convention (e.g., `Ollie (Local)`) in `registry.yaml` configured to hit a standard local Ollama/llama.cpp endpoint.
2. **Targeted Persona Pack (The 3-Tier Strategy)**: Instead of one generic "local" persona, ship a focused pack of personas tuned for specific hardware tiers:
   - **Tier 1: 32GB Dense** (e.g. `Gemma 4`, `Qwen 3.6`): High reasoning, standard context (32k). Best for file-by-file review.
   - **Tier 2: 32GB Long-Context** (e.g. `Ornith 1.0`): Optimized for massive context (256k on 32GB VRAM). Best for full-repo architectural reviews.
   - **Tier 3: 64GB+ Heavyweight** (e.g. `Llama 4 70B` or `DeepSeek V4 Pro`): For Mac Mini M4 Pro or dual-GPU setups. Fits 70B+ quants with massive context, rivaling frontier cloud APIs. 
   Creating a new persona just requires a new YAML file and prompt tuning (not new Go code), so expanding this library is low-effort/high-reward.
3. **Documentation**: Add a "Local / Privacy-First Quickstart" to the docs explaining how to `brew install ollama`, pull the model, and run `atcr` entirely offline.

## Acceptance Criteria

- [ ] `registry.yaml` includes a ready-to-use local persona.
- [ ] A local model can successfully execute a review pipeline using this persona without external network calls.
- [ ] Documentation clearly outlines the zero-egress setup.

### Items needing user confirmation (7)

1. ⏸️ **Mechanism mismatch: `registry.yaml` is not the community hub.** No `registry.yaml` exists as shared/community content anywhere in this repo. `~/.config/atcr/registry.yaml` (and the optional `.atcr/registry.yaml` project overlay) is the user-local **provider/agent config** file (`docs/registry.md:5-18`) — not a persona registry. The actual community persona distribution channel — built by **Epic 19.6**, this epic's own listed dependency — is `personas/community/*.yaml` + `personas/community/index.json` (confirmed on disk: `milo.yaml`, `anthony.yaml`, `gia.yaml`, etc.). Epic 19.6 explicitly ruled this out: *"Any `registry.yaml` task→model mapping schema — the existing named-agent + roster model stands"* (Out of Scope) and *"this epic does not touch `ATCR_REGISTRY_URL`/`registry.yaml` hosting."* All three concrete citations of `registry.yaml` in this plan (Proposed Solution item 1, Acceptance Criteria, Components Touched) point at the wrong file/mechanism.

2. ⏸️ **A community persona is a 3-file unit, not a config edit.** Every existing community persona ships as a triple: `<slug>.yaml` (metadata: name/provider/model/persona/role), `<slug>.md` (the actual prompt template, e.g. `personas/community/milo.md`), and `community/testdata/<slug>_fixture.patch` (a fixture case). All three are embedded into the binary via `//go:embed` globs (`personas/community.go:17,24,32`) — new files under `personas/community/` are auto-discovered with **no Go code edit needed** — but every persona must pass `internal/personas/community_fixture_test.go:19` (`TestTemplateFixtureRunner_CommunityPersonasPass`), which asserts exactly one passing fixture case per persona. The plan's Task 1 ("Author a persona ... in registry.yaml") does not account for the `.md` prompt or the fixture-test requirement.

3. ⏸️ **Naming convention mismatch.** Commit `73afa1f0` updated this plan's example from `local-reviewer` to `Ollie (Local)` to "match the human naming convention," but every existing built-in and community persona uses a single bare human first name with **no** parenthetical qualifier or model name (anthony, sonny, gia, milo, flint, delia, quinn, celeste, glenna, gene; built-ins bruce/dax/greta/ingrid/kai/mira/otto/penny/sasha). Model identity lives only in the structured `provider`/`model` fields, never in the display name. `Ollie (Local)`, `Gemma 4`, `Qwen 3.6`, `Ornith 1.0`, `Llama 4 70B`, `DeepSeek V4 Pro` all diverge from this convention.

4. ⏸️ **Self-contradiction between item 1 and item 2 of Proposed Solution.** Item 1 says "Author a persona ... (e.g., `Ollie (Local)`)" — singular, one persona. Item 2 opens with "Instead of one generic 'local' persona, ship a focused pack of personas tuned for specific hardware tiers" — directly negating item 1's own example as the discouraged "one generic persona." The two items read as alternatives, not a sequence.

5. ⏸️ **The provider-level building block already exists.** `docs/providers.md:20-22` already documents, under a "Direct (always works, zero infrastructure)" section:
   ```yaml
   local:
     base_url: http://localhost:11434/v1   # ollama, llama.cpp, vllm, ...
     api_key_env: OLLAMA_API_KEY
   ```
   and `internal/registry/config_test.go` already has a passing test for this exact provider shape (a `local-llm` provider at `http://localhost:11434/v1`). So "pointing to a standard local Ollama/llama.cpp endpoint" is **already shipped and documented** — the real net-new work is narrower than the Objective states: a tuned persona (prompt + fixture) plus a connecting walkthrough, not a provider-config feature.

6. ⏸️ **`api_key_env` is required-non-empty; no keyless-local path is addressed.** `internal/registry/config.go:686-687` makes `api_key_env` mandatory for every provider, and `internal/llmclient/client.go:325-327` errors ("API key env var %s is not set") if that env var isn't set to a non-empty value at invoke time — even for a keyless local Ollama endpoint. Neither this plan nor the existing `docs/providers.md` example addresses this: the user will still need to `export` some placeholder value for `OLLAMA_API_KEY`. Minor, but the "Zero Data Egress... out-of-the-box" framing and AC2 ("execute a review pipeline... without external network calls") imply more friction-free than this actually is today.

7. ⏸️ **Documentation target under-specified.** `docs/personas-install.md:256` already has a `## Provider tiers beyond Synthetic` section covering DashScope, Chutes→Featherless, and LiteLLM as sibling tier write-ups. Task 3 ("Add a Local/Privacy-First Quickstart") doesn't cite this file/section, risking a disconnected or duplicate doc page instead of a natural extension of the existing structure.
