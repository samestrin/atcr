# Tech Debt: Distinguish free vs paid-but-uncorroborated `cost_per_corroborated_finding_usd`

- **Estimated time**: 1 week
- **Tasks/Components**: 4/2

## Current State

`scorecard.PublicRecord.CostPerCorroboratedFindingUSD` is a plain non-pointer `float64` with no `omitempty` tag (internal/scorecard/export.go:40). Both the production leaderboard `--export` path and the benchmark scorer therefore emit `0.0` in two very different situations:

1. The reviewer genuinely cost $0 (free/stub model).
2. The reviewer cost real money but matched zero planted/defect categories.

Because the field has no absent/N/A state, these cases are byte-identical on the public leaderboard, making a paid-but-ineffective reviewer appear as best-in-class efficient.

## Pain Points

### Leaderboard Misreading
- `cost_per_corroborated_finding_usd == 0.0` is ambiguous: it could mean "free" or "expensive but useless".
- Readers who look only at cost efficiency cannot distinguish the two without cross-referencing `corroboration_rate` and `findings_raised_avg`.

### Schema Freeze Conflict
- Epic 10.2 froze `scorecard.PublicRecord` for byte-compatibility with production.
- Any change to the field type (pointer, omitempty, or companion flag) is a schema change that must be applied to both production and benchmark paths together, not patched locally into the benchmark scorer.

## Evidence

### TD item driving this plan:
- internal/benchmark/score.go:109 — "scoreOne sets CostPerCorroboratedFindingUSD only when matchedFindings>0. A reviewer with real CostUSD>0 but zero matched planted categories reports 0 — byte-identical to a genuinely free reviewer."

### Existing parity precedent:
- `internal/scorecard/export.go:272-274` — `costPer(totalCost, corroborated)` returns `0` when `corroborated <= 0`.
- `internal/benchmark/score.go:118-120` — the benchmark scorer uses `if matchedFindings > 0 { ... }`, producing the same `0.0`.
- `internal/scorecard/export_test.go:121` already asserts "no corroborated findings => 0, never Inf/NaN", confirming `0.0` is the intended sentinel today.

## Proposed Solution

Introduce an explicit "N/A" or absent state for `cost_per_corroborated_finding_usd` in the public schema, and render it consistently from both production export and benchmark run. Options to evaluate:

1. Change `CostPerCorroboratedFindingUSD` to `*float64` with `omitempty` (breaking change; requires leaderboard renderer to handle absent).
2. Add a companion enum/boolean field (e.g., `cost_per_corroborated_finding_usd_available`) to disambiguate.
3. Keep `0.0` in the raw JSON but update the leaderboard UI/rendering layer to display "N/A" when `corroboration_rate == 0 && findings_raised_avg > 0`.

The preferred option should be chosen after weighing backward compatibility against clarity.

## Scope

- Update benchmark scorer (`internal/benchmark/score.go`) to match production exactly.

### Out of Scope
