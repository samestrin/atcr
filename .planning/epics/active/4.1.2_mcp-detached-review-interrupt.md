# Epic 4.1.2: Mark Serve-Mode Detached Reviews Interrupted on Shutdown

**Status:** Draft
**Created:** 2026-06-18
**Priority:** Medium (CLI/MCP parity, observability)

## Problem

In MCP serve mode, a background fan-out is detached from the request so the
review survives the handler returning. `reviewContext` strips cancellation via
`context.WithoutCancel(ctx)` (`internal/mcp/handlers.go:88`), so a SIGINT to the
MCP server never cancels — nor marks interrupted — an in-flight detached review.

CLI mode promises an `interrupted` status on shutdown
(`internal/fanout/review.go:361` keys the interrupted marker off
`errors.Is(ctx.Err(), context.Canceled)`), but for a detached review that
predicate is always false. The review is allowed to finish (the intended MCP
design), yet it never receives the interrupted marker the CLI guarantees — so a
review cut off by a server shutdown is left `in_progress` on disk with no
interrupted signal.

## Root Cause

`context.WithoutCancel(ctx)` intentionally severs the cancellation signal so a
handler return does not abort the background run. That same severance also hides
a genuine server-shutdown signal from the fan-out, leaving no path for the
interrupted marker to be set on a detached review.

## Solution

Thread an interrupt-aware signal into the detached fan-out **without**
reintroducing the request-return cancellation that `WithoutCancel` exists to
prevent. Two candidate designs to evaluate:

1. **Decoupled interrupt context** — derive the detached context from
   `context.WithoutCancel(ctx)` but layer a separate server-lifecycle
   cancellation (cancelled only on server shutdown, never on handler return), so
   the existing `ctx.Err()`-based interrupted marker fires for the right reason.
2. **Post-hoc marker** — on server shutdown, walk in-flight detached reviews and
   stamp the interrupted marker out-of-band, leaving the fan-out context
   untouched.

Either must coexist with the existing `5s` drain and must not regress the
handler-return detachment that serve mode depends on.

## Scope

### In Scope
- A server-shutdown signal reaching detached reviews.
- Detached reviews receiving the `interrupted` marker on server shutdown,
  matching CLI semantics.

### Out of Scope
- Cancelling/aborting detached reviews on handler return (must stay detached).
- Changes to CLI-mode interrupt handling (already correct).

## Acceptance Criteria

- [ ] AC1: A detached MCP review in flight when the server begins shutdown is
  marked `interrupted` on disk (not left `in_progress`), matching CLI behavior.
- [ ] AC2: A handler returning normally does NOT mark its detached review
  interrupted — detachment is preserved.
- [ ] AC3: The existing serve-mode drain behavior is unchanged.
- [ ] AC4: No false `completed` status is ever produced for an interrupted run.

## Risks

| Risk | Mitigation |
|------|------------|
| Reintroducing handler-return cancellation | Derive the interrupt signal from server lifecycle only, not the request context |
| Racing the drain | Sequence the marker after/within the existing drain window |
