## Objective

Implement `atcr benchmark run` — the subcommand that executes each case in a benchmark
suite through the real review pipeline, scores the findings against each case's planted
defect categories, and emits a suite-tagged `RunResult` JSON. This closes the loop between
the 10.0 suite contract (`Load`/`Validate`) and the 10.0 `export` envelope (which already
reads a `run` result), so a reproducible public-leaderboard score can be produced end to end.

## Context

Epic 10.0 built everything around `benchmark run` but not `run` itself: the suite manifest
loads and validates, the reproducibility hash is computed, and `export` already consumes a
`RunResult` it cannot yet produce. The reason `run` was held back (per the 10.0 Q1b
clarification) is three concrete blockers:

1. **Ingestion gap.** The review pipeline ingests **git refs** via
   `internal/fanout/review.go:516` `buildPayloads(ctx, cfg, repo, base, head)` — not loose
   diff files. Benchmark cases are versioned diff files referenced by `Case.Diff`, so a new
   **diff-file → payload** ingestion path is required before a case can be reviewed.
2. **Execution is network-bound.** `internal/fanout/review.go:417`
   `ExecuteReview(ctx, completer, *PreparedReview)` drives real model calls
   (`llmclient.New()`), so `run` lives at the CLI/integration layer and is exercised with a
   stub `Completer` in tests — not a unit test against a fixed transcript.
3. **Rubric is external.** The planted-defect scoring rubric (`Case.ExpectedCategories`)
   ships as curated content in the external `atcr/benchmark-suite` repo (T3), not in this
   tree. 10.1 builds the scorer against the contract and a **local fixture suite**; the
   curated `standard-v1` content lands separately.

## Problem Statement

1. `atcr benchmark export` is wired to a run-result file that nothing in the tree can
   currently generate — the public-submission path dead-ends without `run`.
2. The review pipeline cannot consume a standalone diff file, only a `repo/base/head`
   triple, so suite cases cannot be fed to it.
3. There is no scorer that turns a review's findings into a comparable per-reviewer score
   (precision/recall against the case's expected categories) for the leaderboard.

## Proposed Solution

### Phase 1 — Diff-file ingestion path (2 days)

- Add a diff-file → `modePayload` path alongside `buildPayloads`, so a `Case.Diff` (a
  versioned unified-diff file resolved relative to the suite root via the existing
  `isSafeRelPath` guard) can be turned into a `PreparedReview` without a git checkout.
- Reuse the same payload shape `ExecuteReview` already expects; no change to the executor.
- Enforce the existing `MaxDiffBytes` cap on read (a hostile externally-sourced suite must
  not trigger unbounded allocation — same guarantee `ReproHash` already makes).

### Phase 2 — `atcr benchmark run` subcommand (2 days)

- `atcr benchmark run --suite-path <dir> [--out <run-result.json>]` (mirrors the
  `verify`/`export` flag style in `cmd/atcr/benchmark.go`).
- Flow: `Load` + `Validate` the manifest → for each `Case`, ingest its diff (Phase 1),
  build a `PreparedReview`, and `ExecuteReview` against the configured roster → collect
  findings per reviewer.
- Aggregate into the existing `benchmark.RunResult{Suite, SuiteVersion, GeneratedAt,
  Reviewers []scorecard.PublicRecord}` and write it as the JSON that `benchmark export`
  already reads via `--in`. `GeneratedAt` is injected (not `time.Now`) to stay reproducible,
  matching `BuildSubmission`'s `submittedAt` convention.

### Phase 3 — Scoring against expected categories (1 day)

- For each case, compare the reviewer's finding categories against `Case.ExpectedCategories`
  (the planted defects) to compute per-reviewer precision/recall/corroboration, folded into
  the `scorecard.PublicRecord` fields the public envelope already carries.
- Categories are matched on the normalized set already validated by 10.0 (`Validate` rejects
  empty/duplicate categories), so the scorer can trust the manifest's contract.
- Re-scrub on the way out (defense-in-depth): `Reviewers` may carry reviewer identifiers, so
  apply the same `ScrubPublicRecord` pass `BuildSubmission` applies before the record is
  written.

### Phase 4 — Local fixture suite + docs (1 day)

- A minimal in-repo **fixture** suite (one or two tiny diffs with known expected categories)
  to drive integration tests deterministically with a stub `Completer` — independent of the
  external `atcr/benchmark-suite` content.
- Document the `run → export` flow in the benchmark README; note that the public board only
  accepts `source == "benchmark-suite"` submissions (`SourceBenchmarkSuite`), and that the
  curated `standard-v1` content is sourced externally.

## Acceptance Criteria

- [ ] `atcr benchmark run --suite-path <dir>` loads + validates the suite, executes each
      case's diff through the review pipeline, and writes a `benchmark.RunResult` JSON.
- [ ] A diff-file ingestion path exists that turns a `Case.Diff` into a `PreparedReview`
      without a git checkout, honoring `isSafeRelPath` and `MaxDiffBytes`.
- [ ] The run-result written by `run` is consumed unchanged by `atcr benchmark export --in`
      (round-trip: `run` → `export` produces a valid suite-tagged `Submission`).
- [ ] Findings are scored against `Case.ExpectedCategories` into per-reviewer
      `scorecard.PublicRecord` fields; reviewer PII is re-scrubbed before write.
- [ ] `GeneratedAt` is injectable so two runs over the same suite + transcript are
      byte-identical (reproducibility parity with `ReproHash` / `BuildSubmission`).
- [ ] A local fixture suite drives an end-to-end `run` integration test with a stub
      `Completer` (no network); benchmark README documents the `run → export` flow.

## Task Breakdown

1. **Diff-file ingestion**: diff-file → `modePayload`/`PreparedReview` path beside
   `buildPayloads`, with `isSafeRelPath` + `MaxDiffBytes` guards (`internal/fanout`).
2. **`atcr benchmark run`**: new subcommand in `cmd/atcr/benchmark.go`; wire
   `Load`/`Validate` → per-case `ExecuteReview` → `RunResult` writer.
3. **Scorer**: finding-category precision/recall vs `Case.ExpectedCategories` →
   `scorecard.PublicRecord`; re-scrub via `ScrubPublicRecord`.
4. **Fixture suite + docs**: in-repo fixture suite, end-to-end integration test with stub
   `Completer`, benchmark README `run → export` section.

## Out of Scope

- Curated `standard-v1` suite content — external `atcr/benchmark-suite` (Epic 10.0 T3).
- Public static leaderboard site + methodology page — Epic 15.3.
- `demonstrated_rate` column — depends on Epic 11.0 (no code yet); dropped in 10.0.
- CI integration / auto-running the suite on every commit.
- Changes to the `verify`/`export` subcommands or the `Submission` envelope (10.0, frozen).

## Dependencies

- **Epic 10.0** — the suite contract (`Load`/`Validate`/`ReproHash`, `RunResult`,
  `Submission`, `BuildSubmission`) and the `benchmark export` consumer. Hard prerequisite,
  shipped.
- **`internal/fanout`** — `ExecuteReview` (`review.go:417`) and the payload shape
  `buildPayloads` (`review.go:516`) produces.
- **External `atcr/benchmark-suite` (T3)** — needed for a *real* `standard-v1` score; 10.1
  itself ships and tests against a local fixture suite, so it does not block on T3.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `run` is network-bound and hard to test deterministically | High | Medium | Drive end-to-end tests with a stub `Completer` + local fixture suite; keep real-model runs out of unit scope |
| Category scoring is a coarse proxy for review quality | Medium | Medium | Score on the validated category set only; label precision/recall explicitly as a proxy, consistent with 10.0's corroboration caveat |
| Diff-file ingestion diverges from git-ref payloads, scoring an artifact the real pipeline never sees | Medium | High | Reuse the exact `modePayload` shape `ExecuteReview` consumes; assert parity in tests |
| External suite content (T3) absent, blocking a real score | High | Low | Ship against a local fixture suite; `standard-v1` is additive once `atcr/benchmark-suite` lands |
| Hostile/oversized externally-sourced diff exhausts memory | Low | High | Enforce `MaxDiffBytes` on ingestion, same guard `ReproHash` already applies |
