# Feature Request: Technical Debt Metadata Pipeline Enhancement

- **Estimated time**: 1 week
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Context

In Epic 12.1, we migrated our technical debt storage to a sharded YAML format (`.planning/technical-debt/items/`). However, the context captured for each finding is often too thin for the `resolve-td` engine to autonomously understand and fix the bug. While the initial Markdown table summarizes the problem, the detailed narrative, full justifications, and code references exist in the original code review reports (e.g., `code-review/claude/review.md`).

## Problem Statement

When the automated resolving engine reads a technical debt item, it lacks the rich context necessary to accurately generate a fix. The link between the summarized finding in the TD store and the detailed review report is lost after reconciliation.

## Proposed Solution

Enhance the reconciliation pipeline to preserve and link detailed context for each technical debt finding:
1. **Populate the YAML `notes` field:** Modify the reconciler to extract the detailed justification/explanation from the source review log (e.g., `review.md`) and inject it into the unconstrained `notes` field of the sharded YAML item.
2. **Back-references:** Automatically include a hyperlink back to the specific line or section of the original code review report in the finding's description or metadata.
3. **Integration with Epic 18.1:** Ensure these metadata enhancements work seamlessly alongside the stable AST symbol anchoring introduced in Epic 18.1.

## Acceptance Criteria

- [ ] AC1: The reconciler successfully parses the detailed justification from the source `review.md` for each finding.
- [ ] AC2: The parsed justification is written to the `notes` field in the generated `.yaml` shard under `items/`.
- [ ] AC3: A direct back-reference (e.g., file path or URL) to the original code review report is appended to the item's metadata or `Problem` description.
- [ ] AC4: The `resolve-td` skill is updated to read the `notes` field and back-reference when formulating a fix strategy.

## Out of Scope

- Retroactively updating older, already-resolved technical debt items.

### Items needing user confirmation (5)

1. ⏸️ **Ambiguous target: "the reconciler" is not one system.** "Modify the reconciler to extract the detailed justification/explanation from the source review log (e.g., `review.md`) and inject it into the unconstrained `notes` field of the sharded YAML item." Verified in the live codebase: `internal/tdmigrate.ParseREADME` (`internal/tdmigrate/parse.go:24`) — the only path that writes `items/*.yaml` — parses **exclusively** the README's 9/11-column table rows and has zero awareness of `review.md` or `td-stream.txt`; the table itself carries no free-text notes column, so every shard's `notes` field (`internal/tdmigrate/item.go:53`) is empty by construction today (`grep -rn '\.Notes ='` across the repo: 0 matches). `internal/reconcile` (atcr's in-run multi-agent finding-dedup engine) is likewise unrelated to the TD store or `review.md`. The system that *does* read `review.md`/`td-stream.txt` narratives and write into `.planning/technical-debt/README.md` is the `/reconcile-code-review` skill, which lives in the **separate** `claude-prompts` repository and calls the externally-installed `llm-support-mcp` `group_td`/`td_dedupe` binaries (a third codebase). That is the most likely target for AC1, but the plan never names it, and none of these three systems currently touches both `review.md` *and* a `.yaml` shard in the same step. **Suggested action:** name the specific component (skill / binary / Go package) "the reconciler" refers to before this plan reaches an execution flow — this single word is the crux of AC1/AC2 and is unresolvable from the plan text alone.

2. ⏸️ **`items/` is a dormant, one-time migration snapshot — AC2 targets a directory nothing keeps live.** All 33 files under `.planning/technical-debt/items/` share the identical mtime `2026-06-26 21:56:51` (the one-time `td-migrate migrate` run from Epic 12.1's merge), while `.planning/technical-debt/README.md` has been edited as recently as `2026-07-02` — every TD item added in the past week has no corresponding shard. `td-migrate` (`cmd/td-migrate/main.go`) is not invoked from any skill, CI workflow, or git hook in either `atcr` or `claude-prompts` (verified by repo-wide grep — zero hits). Epic 12.1 explicitly deferred "making the shards canonical" to "the follow-on epic (18.0, or renumbered 12.3)"; Epic 18.0 (also active) scopes only `atcr debt` CLI/dashboard tooling and does not perform that cutover. **Suggested action:** either scope in wiring `td-migrate migrate` into a live trigger (pushing this plan past a small task budget), or explicitly document that AC2's shard write only takes effect the next time someone manually runs `td-migrate migrate`.

3. ⏸️ **`resolve-td` has no existing linkage from a README row back to its shard file — AC4 needs a new lookup path.** `resolve-td`'s item selection goes through `llm_support_td_filter`, which parses the flat 11-column README row (`Group | checkbox | Severity | File | Problem | Fix | Category | Est Minutes | Source | Reviewers | Confidence`) — verified via `claude-prompts/.claude/skills/resolve-td/instructions.md:441` and `:452`; there is no Notes column and no reference to `items/*.yaml` anywhere in that skill today. For AC4 to work, resolve-td needs a new way to map a selected table row to its owning shard entry — no existing ID/index does this (shard filenames key by `date_label`, not by finding). Compounding this, Epic 18.1's Technical Constraints explicitly forbid adding columns to the README table ("Do not alter the column structure of the technical debt Markdown table"), so a Notes column can't be bolted onto the table as a shortcut — the shard-lookup path is the only option, and it does not exist yet.

4. ⏸️ **Epic 18.1 (the stated dependency) has not merged.** The plan states "Integration with Epic 18.1: Ensure these metadata enhancements work seamlessly alongside the stable AST symbol anchoring introduced in Epic 18.1" as though 18.1 already shipped. Verified via `llm_support_multiexists`: `18.1_stable_symbol_anchoring.md` is still under `.planning/epics/active/`, not `.planning/epics/completed/` — it has not yet merged. **Suggested action:** sequence 18.3 to start after 18.1 merges, or soften the phrasing until 18.1 is confirmed complete.

5. ⏸️ **AC3 hedges between two back-reference targets with different technical costs.** "A direct back-reference (e.g., file path or URL) to the original code review report is appended to the item's metadata **or** Problem description." These are not interchangeable: "metadata" is a shard-only (`items/*.yaml`) field, cheap to add; "Problem description" is the README table's `Problem` cell, whose column structure Epic 18.1 explicitly forbids altering (see item 3 above). **Suggested rewrite:** "AC3: A direct back-reference (file path or URL) to the original code review report is appended to the shard item's metadata. The README table's `Problem` cell is unchanged, per 18.1's column-stability constraint." (was: "...appended to the item's metadata or `Problem` description.")
