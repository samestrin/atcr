# Feature Request: Technical Debt Metadata Pipeline Enhancement

- **Estimated time**: 3-4 days
- **Tasks/Components**: 4/1
- **Execution**: execute-epic

## Context

atcr's shared reconciliation engine (`internal/reconcile`) already produces two artifacts per review source: a terse `findings.txt` row and a full narrative `review.md` (written by every pool agent and the host reviewer — see `skill/SKILL.md`). `atcr reconcile` currently reads only `findings.txt`; the narrative context in `review.md` is discarded at reconciliation time and never reaches `reconciled/findings.json`. Every downstream consumer of that JSON — today the private `.planning/technical-debt/` pipeline (`claude-prompts:/reconcile-code-review` + `/resolve-td`), and in the future the standalone public TD-resolve capability tracked in Epic 20.1 — inherits this same thinness, because they all consume the same engine output.

## Problem Statement

Downstream TD-resolution consumers lack the rich context needed to autonomously understand and fix a finding, because the shared engine drops each finding's originating narrative before anything downstream ever sees it. Fixing this once, at the engine layer, benefits every consumer instead of requiring each one (private today, public tomorrow) to independently re-derive the mapping from raw `review.md` files.

## Proposed Solution

Enrich the shared reconciliation engine's output so the narrative context survives past reconciliation:

1. **Extend `internal/reconcile.JSONFinding`** (`internal/reconcile/emit.go`) with a `Notes`/`Justification` field.
2. **Populate it at reconcile time:** teach `atcr reconcile` to associate each source's `review.md` narrative with its findings (best-effort match by `file:line` / section) and fill the new field when a match is found. No match is not an error — the field stays empty.
3. **Back-reference:** add a `SourceReport` field (path + line/section within the source's `review.md`) to `JSONFinding` alongside the notes, so a consumer can navigate to full detail without re-deriving the mapping. This lands only in the reconciled JSON — never appended to the legacy `.planning/technical-debt/README.md` table's `Problem` cell, whose column structure Epic 18.1 explicitly freezes ("Do not alter the column structure of the technical debt Markdown table").
4. **Document the contract:** update `docs/findings-format.md` to describe the new fields as an additive, backward-compatible extension to the existing reconciled-findings contract.

## Acceptance Criteria

- [ ] AC1: `atcr reconcile` extracts the relevant narrative section from each source's `review.md` for a finding, when available.
- [ ] AC2: `reconciled/findings.json` carries the extracted narrative as a `notes`/`justification` field per finding (`internal/reconcile.JSONFinding`).
- [ ] AC3: A back-reference (source path + section/line within that source's `review.md`) is included in the reconciled JSON alongside the finding. It is never written into the README table's `Problem` cell.
- [ ] AC4: `docs/findings-format.md` documents the new fields as part of the reconciled-findings contract, so downstream consumers (private and future public) can adopt them without re-deriving the schema.

## Out of Scope

- Modifying the private `claude-prompts` skills (`execute-code-review`, `reconcile-code-review`, `resolve-td`) to consume the new fields — that repo-external work is not something this epic's execution can touch; it happens separately once these fields ship.
- Building a public-facing TD tracking/resolve capability that consumes these fields — tracked in Epic 20.1 (Public TD Resolve Skill).
- Wiring `.planning/technical-debt/items/*.yaml` shard regeneration into a live trigger. The dormant-pipeline gap identified during refinement (all shards share a single stale mtime; `td-migrate` isn't invoked by any skill/CI/hook) is real, but no longer load-bearing here once notes are available at reconcile time instead of being reconstructed later from a table that never had them. Candidate for its own TD item if still wanted.
- Retroactively updating older, already-resolved technical debt items.

## Dependencies / Related

- Epic 18.1 (Stable Symbol Anchoring) — the README table's frozen column structure (Technical Constraint) is why AC3's back-reference lands only in the JSON, never the `Problem` cell.
- Epic 20.1 (Public TD Resolve Skill) — the primary new consumer of the fields this epic adds; depends on this epic shipping first.
- `claude-prompts:/reconcile-code-review` and `/resolve-td` — the private pipeline's eventual consumer of these fields; updated separately, outside this repo.

## Refinements (2026-07-03) — Follow-up: corrections & decisions

All 5 user-confirm items above were discussed and resolved. The plan body (Context, Problem Statement, Proposed Solution, Acceptance Criteria, Out of Scope) was rewritten to reflect these decisions, and a new Dependencies / Related section was added.

### Decisions (resolved)

- **D1 — Retarget from "the reconciler" (ambiguous) to `internal/reconcile` (supersedes item 1):** Rather than the private `claude-prompts:/reconcile-code-review` skill, the fix lands in atcr's own shared engine. Both `review.md` (the narrative) and `findings.txt` (the terse row) already exist per-source in every `atcr review` run — the engine just never correlates them. Fixing this once in `internal/reconcile` benefits every consumer (private today, public via Epic 20.1 tomorrow) from a single change, and keeps this epic's execution entirely inside the `atcr` repo.
- **D2 — Shard-pipeline dormancy (item 2) is no longer load-bearing:** Populating notes at reconcile time (before a TD item is ever written) avoids the lossy "reconstruct from a table that never had notes" problem entirely. Wiring `td-migrate` into a live trigger is now an independent, optional cleanup, not a dependency of this epic.
- **D3 — Row↔shard linkage (item 3) no longer applies to this epic:** With notes attached at creation time rather than reconstructed afterward, there is no separate artifact to join. Whatever writes a TD item (private today, public in 20.1) reads directly from the enriched `reconciled/findings.json`.
- **D4 — Epic 18.1 sequencing (item 4) confirmed:** Still blocked behind 18.1 merging, per the frozen-column constraint referenced in the new Dependencies section — no change needed beyond noting it.
- **D5 — AC3 target (item 5) resolved as JSON-only, not the table's `Problem` cell:** Confirmed against 18.1's Technical Constraint ("Do not alter the column structure of the technical debt Markdown table") — the back-reference lands in `reconciled/findings.json` only.
- **D6 — AC4 rescoped:** The original AC4 ("the `resolve-td` skill is updated...") named a private, external-repo skill this epic's execution cannot touch. Replaced with a documentation AC (`docs/findings-format.md` describes the new contract) so any consumer — private or the new public Epic 20.1 — can adopt the fields without this epic reaching outside its own repo.
- **D7 — New Epic 20.1 (Public TD Resolve Skill) created** to own the public-facing consumer of these fields, previously undiscussed scope surfaced while resolving item 1.

**Net effect:** the epic is now scoped entirely within `atcr` (`internal/reconcile` + `docs/findings-format.md`), with no cross-repo or dormant-pipeline dependencies.
