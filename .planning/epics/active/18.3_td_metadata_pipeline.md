# Feature Request: Technical Debt Metadata Pipeline Enhancement

- **Estimated time**: 1 week
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Context

In Epic 12.1, we migrated our technical debt storage to a sharded YAML format (`.planning/technical-debt/items/`). However, the context captured for each finding is often too thin for the `resolve-td` engine to autonomously understand and fix the bug. While the initial Markdown table summarizes the problem, the detailed narrative, full justifications, and code references exist in the original code review reports (e.g., `code-review/claude/review.md`).

## Problem Statement

When the automated resolving engine reads a technical debt item, it lacks the rich context necessary to accurately generate a fix. The link between the summarized finding in the TD store and the detailed review report is lost after reconciliation.

## Proposed Solution

Enhance the reconciliation pipeline to preserve and link detailed context for each technical debt finding:
1. **Populate the YAML `notes` field:** Modify the reconciler to extract the detailed justification/explanation from the source review log (e.g., `review.md`) and inject it into the unconstrained `notes` field of the sharded YAML item.
2. **Back-references:** Automatically include a hyperlink back to the specific line or section of the original code review report in the finding's description or metadata.
3. **Integration with Epic 18.1:** Ensure these metadata enhancements work seamlessly alongside the stable AST symbol anchoring introduced in Epic 18.1.

## Acceptance Criteria

- [ ] AC1: The reconciler successfully parses the detailed justification from the source `review.md` for each finding.
- [ ] AC2: The parsed justification is written to the `notes` field in the generated `.yaml` shard under `items/`.
- [ ] AC3: A direct back-reference (e.g., file path or URL) to the original code review report is appended to the item's metadata or `Problem` description.
- [ ] AC4: The `resolve-td` skill is updated to read the `notes` field and back-reference when formulating a fix strategy.

## Out of Scope

- Retroactively updating older, already-resolved technical debt items.
