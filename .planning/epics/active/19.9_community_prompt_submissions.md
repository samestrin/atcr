# Epic Plan 19.9: Community Prompt Submissions & Battle-Tested Prompt Collection

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Establish a **GitHub-native** intake and curation flow so users contribute refined, battle-tested reviewer prompts back to the canonical library — and so the maintainer learns *which* prompts need tuning from **opt-in, content-free** signal — without building a marketplace, website, or hosted registry (explicitly ruled out in Epic 19.6). The battle-tested prompt is the real deliverable of the whole persona effort; this epic is the flywheel that lets a growing user base improve it.

## Context

Model-tuned reviewer prompts are refined *in production*: they hallucinate, over-report, flag non-bugs, or miss real ones, and get tuned over time until they earn trust. Two distinct things flow back from users, and they need different plumbing:

- **The prompt itself** is *code* — a file. The intake pipeline already exists: a PR to `samestrin/atcr`, gated by the 19.6 fixture gate + human review. The only real work is **friction reduction** (turn "I improved my local copy" into "I opened a PR" in one command).
- **The quality signal** ("this prompt over-reports") is *data*. Because atcr reviews **private code**, this signal must be collected as **opt-in, aggregate, content-free** telemetry — never a line of code or a finding body leaves the machine. Epic 24.0 (false-positive / wontfix dismissal) already *produces* the signal; Epic 28.0 (telemetry / cloud sync) is the transport.

This composes with the rest of the persona program into one living-library flywheel: **drift detected (19.7) → hermes drafts (19.8) → maintainer + users refine in production → dismissal signal (24.0/28.0) flags weak prompts → refined prompt submitted back (this epic) → graduates to official.**

## Proposed Solution

1. **Prompt intake — `atcr personas submit <name>`.** Takes the user's locally-tuned `.atcr/personas/<name>.md`, runs the fixture gate locally, and opens a fork+PR to `samestrin/atcr` via `gh`. Fixture-gated, human-reviewed — no new hosting.
2. **Two-tier curation.** A submitted prompt lands in a `community-contributed` namespace (fixture-passing but unvetted); the maintainer battle-tests it and **graduates** it to the official set. Attribution/credit in the persona carries provenance and is the incentive to submit.
3. **Quality signal (opt-in, content-free).** Aggregate counters per persona+model (e.g. `dismissed`/`confirmed` from the 24.0 dismissal signal) transported via 28.0. Hard privacy line: **zero code and zero finding content ever leaves the machine** — counters and identifiers only, opt-in, with a visible local preview of exactly what would be sent.
4. **Close the loop.** Surface the signal to the maintainer (which prompts over/under-report) so it drives which personas the hermes drafting agent (19.8) and community submissions target next.

## Acceptance Criteria

- [ ] AC1: `atcr personas submit <name>` runs the fixture gate locally and opens a fork+PR of the local persona to the canonical repo; a failing fixture blocks submission with a clear error.
- [ ] AC2: Submitted prompts land in a `community-contributed` tier distinct from the official set, carry attribution/provenance, and require maintainer graduation to become official.
- [ ] AC3: Quality signal is opt-in, aggregate, and content-free — only per-persona+model counters and identifiers are transmitted; a local preview shows exactly what would be sent; nothing is sent by default.
- [ ] AC4: The signal is surfaced to the maintainer in a form that identifies over/under-reporting prompts, closing the loop to 19.8's drafting and this epic's submissions.
- [ ] AC5: No marketplace/website/hosted-registry surface is introduced; the entire flow is GitHub-PR-native (honors the 19.6 out-of-scope constraint).
- [ ] AC6: `go test ./...` passes; docs document the submit flow, the two-tier curation model, and the exact content-free telemetry contract.

## Components Touched

- `cmd/atcr` (`personas submit`; any signal preview/report surface)
- `internal/personas` (community-contributed tier, provenance/attribution)
- Telemetry integration (rides Epic 28.0 transport; consumes Epic 24.0 dismissal signal)
- `docs/`

## Out of Scope

- **A marketplace / website / hosted persona registry / persona-ranking UI** — explicitly rejected in Epic 19.6; this epic is GitHub-PR-native only.
- **Transmitting any code or finding content** — the privacy line is absolute; only content-free aggregate counters + identifiers, opt-in.
- **The dismissal signal source (24.0) and telemetry transport (28.0) themselves** — this epic consumes them; it does not build them.
- **The hermes drafting agent (19.8)** — separate; this epic feeds it targets via the signal loop.

## Dependencies

- **Epic 19.6 (Community Registry Hub)** — the in-repo persona units, fixture gate, and human-review intake this epic lowers friction into.
- **Epic 24.0 (false-positive / wontfix dismissal)** — produces the quality signal.
- **Epic 28.0 (telemetry / cloud sync)** — the opt-in, content-free transport for the signal.
- **Epics 19.7 / 19.8** — complete the living-library flywheel this epic closes.
