# Epic Plan 19.9: Community Prompt Submissions (Intake & Curation)

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Establish a **GitHub-native** intake and curation flow so users contribute refined, battle-tested reviewer prompts back to the canonical library — turning "I improved my local copy" into "I opened a PR" in one command — **without** building a marketplace, website, or hosted registry (explicitly ruled out in Epic 19.6). The battle-tested prompt is the real deliverable of the whole persona effort; this epic is the **intake half** of the flywheel that lets a growing user base improve it. (The opt-in, content-free *quality signal* that tells the maintainer which prompts to tune is split into **Epic 30.0**, which depends on this epic plus 24.0/28.0.)

## Context

Model-tuned reviewer prompts are refined *in production*: they hallucinate, over-report, flag non-bugs, or miss real ones, and get tuned over time until they earn trust. The refined **prompt itself** is *code* — a file. The intake pipeline already exists: a PR to `samestrin/atcr`, gated by the 19.6 fixture gate + human review. The real work here is **friction reduction** (one command to fork+PR a locally-tuned persona) and **curation structure** (a place for unvetted submissions to land before a maintainer graduates them).

This is the intake half of the living-library flywheel: **drift detected (19.7) → hermes drafts (19.8) → maintainer + users refine in production → refined prompt submitted back (this epic) → graduates into the vetted community library.** The *signal* that closes the loop back to 19.8's drafting (which prompts over/under-report) is **Epic 30.0**.

## Proposed Solution

1. **Prompt intake — `atcr personas submit <name>`.** Takes the user's locally-tuned persona, runs the fixture gate locally (reusing `personas.TestPersona` / `TemplateFixtureRunner`, `internal/personas/test.go`), and opens a fork+PR to `samestrin/atcr` via `gh`. Fixture-gated, human-reviewed — no new hosting. Slots in as a new subcommand alongside the existing `install`/`list`/`search`/`remove`/`test`/`upgrade` (`cmd/atcr/personas.go`).
2. **Two-tier curation via a `submitted` status.** Provenance and vetting status are **orthogonal axes**. Provenance is the existing `Source` field (`built-in` | `community`, `internal/personas/list.go:22`) — every submission is and stays `community`. This epic adds a *status* axis: a submitted prompt lands as **`submitted`** (fixture-passing but unvetted); the maintainer battle-tests it and **graduates** it — promoting it into the shipped, vetted `personas/community/` library. Graduation flips the status, not the provenance. Attribution/credit carried in the persona is the incentive to submit.

## Acceptance Criteria

- [ ] AC1: `atcr personas submit <name>` runs the fixture gate locally and opens a fork+PR of the local persona to the canonical repo; a failing fixture blocks submission with a clear error.
- [ ] AC2: A submitted prompt lands with a `submitted` status (fixture-passing but unvetted) layered on the existing `community` source; it carries attribution/provenance and requires maintainer **graduation** (promotion into the vetted `personas/community/` library) to become vetted. The `submitted` status is distinct from `Source` and does not overload the existing "community-contributed" provenance meaning.
- [ ] AC3: No marketplace/website/hosted-registry surface is introduced; the entire flow is GitHub-PR-native (honors the 19.6 out-of-scope constraint).
- [ ] AC4: `go test ./...` passes; docs document the submit flow and the two-tier (`submitted` → graduated) curation model.
