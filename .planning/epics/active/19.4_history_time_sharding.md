# Feature Request: Time-Based Sharding for Findings History

- **Estimated time**: 1-2 days
- **Tasks/Components**: 3/2
- **Execution**: execute-epic

## Context

Epic 19.0 introduced the ability to persist findings from every `review` run into a local `findings-history.jsonl` file. Because ATCR is moving toward a Team Edition, this history needs to be committed to version control so the entire team can query trends and track code health over time. 

## Problem Statement

Committing a single, infinitely growing `findings-history.jsonl` file to Git will cause severe repository bloat. Every time a developer runs a review, new lines are appended, creating a new Git blob for the entire file. Over thousands of reviews, this monolithic file becomes a massive, constantly churning commit burden.

## Proposed Solution

Implement time-based sharding for the findings history:
1. **Write Path**: Instead of appending to a flat `findings-history.jsonl`, the review runner will append to a monthly sharded file (e.g., `.planning/history/2026-07.jsonl`). 
2. **Read Path**: The `atcr history` command and `internal/history` reader will be updated to glob/load all `.jsonl` files in the `.planning/history/` directory before applying the `--since` and `--package` filters across the combined data stream.
3. **Legacy Migration**: If a flat `findings-history.jsonl` exists from earlier 19.x sprints, it will be transparently loaded as read-only legacy data. No complex write-migrations will be performed.

## Acceptance Criteria

- [ ] AC1: `atcr review` appends new records into a file named with the current year and month (e.g., `YYYY-MM.jsonl`) under `.planning/history/`.
- [ ] AC2: `atcr history` seamlessly queries across multiple sharded files without requiring the user to specify which shard to read.
- [ ] AC3: Older monthly shards stop receiving writes (and thus stop creating new Git blobs) once the month rolls over.
- [ ] AC4: All existing `cmd/atcr/history_test.go` and `internal/history/*_test.go` test cases pass with the new storage layout.

## Technical Constraints

- Sharding must occur automatically without user configuration.
- The read path must remain fast enough that scanning 1-2 years of sharded JSONL files does not degrade the `atcr history` command's UX.
