# Epic Plan 24.0: False Positive / WontFix Dismissal Loop

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Provide a seamless mechanism for developers to permanently dismiss a specific technical debt finding (e.g., mark as False Positive or WontFix) so that ATCR's agents learn to ignore it in subsequent review loops.

## Context

ATCR successfully surfaces technical debt and automatically routes it to the `.planning/technical-debt/README.md` or `.atcr/reviews/` sharded backend. However, if an agent hallucinates or flags a legitimate but accepted pattern (a false positive), the developer currently has no "dismiss" button. The finding will repeatedly surface, causing alert fatigue. 

## Proposed Solution

1. **Dismiss Command**: Add an `atcr dismiss <finding-id> --reason <fp|wontfix>` subcommand to `cmd/atcr` that appends a `dismissed` status record to `internal/localdebt` (extending its existing terminal `resolved`/`deferred` status contract), so `selectOpenDebt` folds the id out of the open backlog.
2. **Reconciliation Ignore Logic**: Update the reconciler (`internal/reconcile/gate.go`) and prompt payload builder (`internal/payload`) so that dismissed findings whose `internal/localdebt` id (`history.FindingID(file, line, problem)`) has a `dismissed` status record are automatically filtered out of future review outputs.
3. **Audit Log**: Append a dismissal audit record (id, reason, timestamp) via `internal/audit` (capture.go/writer.go), OR carry the reason on the `internal/localdebt` `dismissed` status record — pick one so the audit surface is testable.

## Acceptance Criteria

- [ ] A finding can be explicitly marked as dismissed/WontFix.
- [ ] A dismissed finding does not reappear in the reconciled output on subsequent runs against the same code structure.
- [ ] The dismissal state is preserved across runs.

## Components Touched

- `cmd/atcr` (CLI interface — `atcr dismiss` subcommand)
- `internal/localdebt` (persistence — `.atcr/debt/` backlog; extends existing resolved/deferred status)
- `internal/reconcile` (filtering — extend gate.go)
- `internal/payload` (prompt payload builder — T2)
- `internal/audit` (audit log — T3)

## Out of Scope

- Team-wide synchronization of dismissed findings (handled by Epic 19.x remote registry fetching / shared history if needed, but the core engine logic is local).

## Dependencies

- None.

T2's dismiss-filter should extend `gate.go`, not introduce a parallel filter. If filtering lives at the gate, `internal/payload` may need no change (the payload builder simply never sees dismissed findings) — which would drop COMPONENT_COUNT from 5 to 4.

- ℹ️ **Two TD stores — disambiguate:** `internal/debt` writes the versioned `.planning/technical-debt/README.md` (private pipeline); `internal/localdebt` writes the uncommitted `.atcr/debt/YYYY-MM.jsonl` (public/standalone, Epic 20.1). The plan's Context references the README AND the `.atcr/reviews/` backend, and AC #2 says "does not reappear in the reconciled output" — but the reconciled output differs per store. Specify which store(s) hold dismiss state and which output the AC targets.

- ℹ️ **Cross-epic dependency / downstream contract:** Epic 30.0 (`30.0_community_prompt_quality_signal.md`) and completed 19.9 both consume "the 24.0 dismissal signal" — `dismissed`/`confirmed` events aggregated per persona+model.
