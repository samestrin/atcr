# Epic 4.4: Metrics and Observability

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Medium (operational visibility)

## Problem

ATCR has no metrics beyond structured logging (Epic 4.0). Operators cannot answer questions like:

- How many reviews fail per day?
- What is the p95 review latency?
- Which provider fails most often?
- How many API calls does a typical review make?
- What is the agent success rate by persona?

Without metrics, performance regressions and provider issues go undetected until users complain.

## Root Cause

There is no metrics collection infrastructure. The codebase has no counters, histograms, or gauges. Timing information is logged at debug level but not aggregated.

## Solution

Introduce a lightweight metrics package that collects counters and histograms in-process. Expose metrics via a `/metrics` endpoint (MCP server mode) or print a summary at the end of each review (CLI mode).

### Core Design

```go
// internal/metrics/metrics.go
package metrics

import (
    "sync"
    "sync/atomic"
    "time"
)

// Registry holds all metrics collectors.
type Registry struct {
    mu       sync.RWMutex
    counters map[string]*Counter
    histograms map[string]*Histogram
}

// Counter is a monotonically increasing counter.
type Counter struct {
    name  string
    value atomic.Int64
}

func (c *Counter) Inc()          { c.value.Add(1) }
func (c *Counter) Add(n int64)   { c.value.Add(n) }
func (c *Counter) Value() int64  { return c.value.Load() }

// Histogram tracks the distribution of a metric (e.g., latency).
type Histogram struct {
    name    string
    mu      sync.Mutex
    values  []float64
    sum     float64
    count   int64
}

func (h *Histogram) Observe(v float64)
func (h *Histogram) Percentile(p float64) float64
func (h *Histogram) Mean() float64

// NewRegistry creates a new metrics registry.
func NewRegistry() *Registry

// Global registry (singleton).
var DefaultRegistry = NewRegistry()

// Convenience functions.
func Counter(name string) *Counter
func Histogram(name string) *Histogram
```

### Integration Points

**CLI mode:**
- At the end of each review, print a summary:
  ```
  Review completed in 142.3s
  Agents: 8/10 succeeded, 1 failed, 1 timed out
  API calls: 12 (3 retries)
  Findings: 5 (2 HIGH, 3 MEDIUM)
  ```

**MCP server mode:**
- Expose `/metrics` endpoint in Prometheus text format.
- Metrics are cumulative since server start.

### Implementation

```go
// cmd/atcr/review.go
func runReview(cmd *cobra.Command, _ []string) error {
    metrics.Counter("atcr_reviews_total").Inc()
    start := time.Now()
    defer func() {
        metrics.Histogram("atcr_review_duration_seconds").Observe(time.Since(start).Seconds())
    }()

    // ... review logic

    // At end, print summary
    fmt.Fprintf(cmd.OutOrStdout(), "Review completed in %.1fs\n", time.Since(start).Seconds())
    fmt.Fprintf(cmd.OutOrStdout(), "Agents: %d/%d succeeded\n",
        result.Summary.Succeeded, result.Summary.Total)
}
```

```go
// internal/fanout/engine.go
func (e *Engine) invokeAgent(ctx context.Context, a Agent) Result {
    metrics.Counter("atcr_agents_total").Inc()
    start := time.Now()
    defer func() {
        metrics.Histogram("atcr_agent_duration_seconds").Observe(time.Since(start).Seconds())
    }()

    r := e.invokeSingleShot(ctx, a)
    switch r.Status {
    case StatusOK:
        metrics.Counter("atcr_agents_succeeded").Inc()
    case StatusFailed:
        metrics.Counter("atcr_agents_failed").Inc()
    case StatusTimeout:
        metrics.Counter("atcr_agents_timed_out").Inc()
    }
    return r
}
```

## Scope

### In Scope

- `internal/metrics` package with `Counter`, `Histogram`, and `Registry`.
- Metrics collection in fanout engine (agent invocations, API calls).
- Metrics collection in CLI (review duration, findings).
- Summary printed at end of CLI review.
- `/metrics` endpoint in MCP server mode (Prometheus text format).
- Unit tests for counter, histogram, and registry.
- Integration test: run a review, verify metrics collected.

### Out of Scope

- External metrics backends (Prometheus, Datadog, StatsD) — metrics are in-process only.
- Metrics export to files or remote endpoints (MCP `/metrics` is the only export).
- Distributed tracing (OpenTelemetry) — separate concern.
- Alerting on metrics thresholds — separate concern.
- Per-review metrics persistence (metrics are ephemeral, reset on process exit).

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Metrics overhead slows down reviews | Low | Low | Counters are atomic; histograms use mutex only for percentile calculation |
| Histogram memory grows unbounded | Medium | Medium | Cap histogram at 10,000 samples; discard oldest |
| `/metrics` endpoint is a security risk | Low | Low | MCP server is local-only; document that `/metrics` should not be exposed publicly |

## Relationship to Other Epics

- **Epic 4.0 (Structured Logging)**: metrics complement logging; logging is for debugging, metrics are for monitoring.
- **Epic 4.4 (Circuit Breaker / Provider Health)**: circuit breaker uses metrics to detect provider failures.

## Open Questions

1. **Should metrics persist across reviews?**
   - Option A: no — metrics reset on process exit (simple).
   - Option B: yes — metrics are written to a file and reloaded on startup (complex).
   - Recommendation: Option A; persistent metrics are a separate concern.

2. **Should we use a third-party metrics library?**
   - Option A: no — custom implementation is simple enough.
   - Option B: yes — use `prometheus/client_golang` for robustness.
   - Recommendation: Option A; the metrics needs are minimal.
