# Epic 4.2: Configuration Validation

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Medium (fail fast, prevent wasted API calls)

## Problem

ATCR loads configuration from YAML files (`.atcr/config.yaml`, persona registry, project overrides) but does not validate the configuration comprehensively at load time. Invalid or missing fields surface late — during execution — wasting API calls and confusing users.

### Concrete Failures

1. **Missing required fields.** A persona YAML omits `provider` or `model`. The error surfaces when the agent is invoked, after other agents have already started.

2. **Invalid enum values.** `payload_mode: "invalid"` in config. The error surfaces when the payload is built, after the review directory is created.

3. **Type mismatches.** `timeout: "300"` (string instead of int). The error surfaces when the timeout is applied, mid-review.

4. **Semantic errors.** `min_severity: "CRITICAL"` but `max_findings: 0`. The constraint is nonsensical but not caught until a finding is filtered.

5. **Provider misconfiguration.** `provider: "anthropic"` but `ANTHROPIC_API_KEY` is not set. The error surfaces at the first API call.

## Root Cause

Configuration loading (`internal/registry/config.go`) unmarshals YAML into structs but does not validate beyond what the YAML decoder catches (type mismatches). There is no schema validation, no required-field checking, and no semantic validation (e.g., "if X is set, Y must also be set").

## Solution

Add a validation layer that runs after config loading and before execution. Validate required fields, enum values, type constraints, and semantic rules. Fail fast with a clear error message.

### Core Design

```go
// internal/registry/validate.go
package registry

// ValidationError is a configuration error with a clear message.
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("config validation failed: %s: %s", e.Field, e.Message)
}

// Validate checks the resolved configuration for errors.
// Returns a list of validation errors (empty if valid).
func (c *Config) Validate() []error {
    var errs []error

    // Required fields
    if c.Registry.DefaultProvider == "" {
        errs = append(errs, &ValidationError{"registry.default_provider", "required"})
    }

    // Enum validation
    validPayloadModes := map[string]bool{"diff": true, "blocks": true, "files": true}
    if c.Registry.DefaultPayloadMode != "" && !validPayloadModes[c.Registry.DefaultPayloadMode] {
        errs = append(errs, &ValidationError{"registry.default_payload_mode",
            fmt.Sprintf("invalid value %q; must be one of: diff, blocks, files", c.Registry.DefaultPayloadMode)})
    }

    // Semantic validation
    for _, persona := range c.Registry.Personas {
        if persona.MinSeverity != "" && persona.MaxFindings != nil && *persona.MaxFindings == 0 {
            errs = append(errs, &ValidationError{
                fmt.Sprintf("persona[%s].max_findings", persona.Name),
                "max_findings=0 with min_severity set is nonsensical",
            })
        }
    }

    return errs
}
```

**Integration:**

```go
// cmd/atcr/review.go
func runReview(cmd *cobra.Command, _ []string) error {
    cfg, err := fanout.LoadReviewConfig(".", cliOverrides(cmd))
    if err != nil {
        return usageError(err)
    }

    // Validate configuration before execution
    if errs := cfg.Validate(); len(errs) > 0 {
        var msgs []string
        for _, e := range errs {
            msgs = append(msgs, e.Error())
        }
        return usageError(fmt.Errorf("configuration errors:\n  - %s", strings.Join(msgs, "\n  - ")))
    }

    // ... rest of review logic
}
```

### Validation Categories

1. **Required fields.** `default_provider`, `default_payload_mode`, persona `name`, persona `model`.

2. **Enum values.** `payload_mode` (diff, blocks, files), `min_severity` (LOW, MEDIUM, HIGH, CRITICAL), `provider` (openai, anthropic, openrouter).

3. **Type constraints.** `timeout` > 0, `byte_budget` >= 0, `max_findings` >= 0, `max_parallel` >= 0.

4. **Semantic rules.** If `min_severity` is set, `max_findings` must be > 0 (or nil). If `provider` is set, the corresponding API key env var must be set (or the persona is marked `optional: true`).

5. **Cross-field dependencies.** If `fallbacks` is set, the fallback personas must exist in the registry.

## Scope

### In Scope

- `internal/registry/validate.go` with `Validate()` method on `Config`.
- Validation of required fields, enum values, type constraints, and semantic rules.
- Clear error messages with field names and expected values.
- Integration in `cmd/atcr/review.go`, `cmd/atcr/reconcile.go`, `cmd/atcr/verify.go` (all commands that load config).
- Unit tests for each validation category.
- Integration test: invalid config → clear error message before execution.

### Out of Scope

- JSON Schema validation (overkill for this use case).
- Config file auto-generation or interactive setup.
- Validation of persona prompt content (out of scope; prompts are opaque strings).
- Validation of project-specific overrides (separate concern; this epic focuses on the base config).

## Acceptance Criteria

- [ ] AC1: Missing `default_provider` in config produces error: "config validation failed: registry.default_provider: required".
- [ ] AC2: Invalid `payload_mode: "invalid"` produces error: "config validation failed: registry.default_payload_mode: invalid value \"invalid\"; must be one of: diff, blocks, files".
- [ ] AC3: `timeout: -1` produces error: "config validation failed: registry.timeout: must be > 0".
- [ ] AC4: `min_severity: "CRITICAL"` with `max_findings: 0` produces error: "config validation failed: persona[...].max_findings: max_findings=0 with min_severity set is nonsensical".
- [ ] AC5: Validation errors are returned as `usageError` (exit code 2), not generic failure (exit code 1).
- [ ] AC6: All validation errors are reported at once (not one-at-a-time).
- [ ] AC7: Validation runs before any API calls or review directory creation.
- [ ] AC8: `go test ./internal/registry/...` includes tests for each validation category.
- [ ] AC9: Integration test: `atcr review` with invalid config fails fast with clear error.

## Success Criteria

- [ ] Invalid configurations are caught before execution.
- [ ] Error messages clearly identify the field and expected value.
- [ ] No API calls are made with invalid configuration.
- [ ] No review directories are created for invalid configurations.
- [ ] Users can fix config errors without inspecting logs or stack traces.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Validation is too strict, breaks existing configs | Medium | High | Start with minimal validation (required fields, enums); add semantic rules incrementally |
| Validation is too slow (blocks startup) | Low | Low | Validation is in-memory; should be <1ms |
| Error messages are unclear | Medium | Medium | Use field names from YAML, not Go struct names; test error messages in unit tests |
| Validation misses a real error | Medium | Medium | Add validation rules as bugs are reported; validation is not exhaustive |

## Open Questions

1. **Should validation be exhaustive or minimal?**
   - Option A: exhaustive — validate every field, every constraint.
   - Option B: minimal — validate only required fields and common mistakes.
   - Recommendation: Option B; exhaustive validation is a large effort and can be added incrementally.

2. **Should validation run in `atcr doctor`?**
   - Option A: yes — `atcr doctor` should validate config and report errors.
   - Option B: no — `atcr doctor` focuses on API key connectivity, not config validation.
   - Recommendation: Option A; `atcr doctor` is the natural place for a config check.

3. **Should we support a `--validate` flag to validate config without running a review?**
   - Option A: yes — `atcr validate` checks config and exits.
   - Option B: no — validation runs automatically before every command.
   - Recommendation: Option B; a separate command is overkill.
