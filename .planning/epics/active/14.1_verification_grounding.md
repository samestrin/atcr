# Epic 14.1: Verification & Grounding

- **Estimated time**: 1 week
- **Tasks/Components**: 3/2
- **Execution**: execute-epic

## Context

During a multi-review using `atcr`, the system can fan out to multiple LLM personas. When presented with large amounts of code, the LLMs frequently hallucinate "Technical Debt" (TD) items that do not exist, flagging unchanged code, completely fabricated snippets, or irrelevant issues.

## Problem Statement

A user recently received 523 TD items, of which 513 were complete hallucinations. This runaway generation destroys trust in the review process and makes the `atcr` tool effectively useless for large codebases. We must force the LLMs to ground their responses in the actual diff provided.

## Proposed Solution

Implement a robust grounding mechanism that prevents an LLM from successfully submitting a hallucinated TD item. 

1. **Require Exact Quotes/Snippets**: Update the persona prompts to mandate that every TD item cite the exact line numbers or the exact code snippet from the diff.
2. **Programmatic Validation**: Implement a new Go function that runs before passing the persona's findings to the reconciler. This function will check if the referenced snippet/lines actually exist in the git patch. If not, the TD item is silently dropped.
3. **Limit Scope to Patch**: Update the prompt constraints so the LLMs are explicitly forbidden from commenting on code outside the "added" or "modified" lines of the diff.

## User Personas

### Primary: Software Engineer
- Wants actionable, real feedback on their PRs.
- Has zero tolerance for reviewing 500+ hallucinated items.

## User Journeys

### Code Review execution
1. User runs `atcr review`.
2. The fanned-out personas generate TD items and cite exact line numbers.
3. The programmatic validator parses the response, cross-references it with the patch, and drops hallucinations.
4. Only grounded, real TD items are passed to the Reconciler.

## Acceptance Criteria

- [ ] AC1: Persona prompts are updated to strictly require `file` and `line` properties (or equivalent).
- [ ] AC2: A validation step drops any TD item whose `file:line` reference cannot be found in the diff.
- [ ] AC3: Prompts are updated to explicitly reject analyzing code that is not part of the `+` or `-` lines in the patch.

## Technical Constraints

- Must intercept the review items before they hit the reconciler pipeline.
- Must accurately map line numbers from the LLM response to the actual git patch line numbers.

## Non-Functional Requirements

### Performance
- The patch-validation step should add negligible overhead (<500ms) to the pipeline.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| LLMs provide "approximate" lines | High | High | Use fuzzy matching or snippet text matching if line numbers are slightly off, to avoid dropping real issues. |
