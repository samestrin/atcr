# Feature Request: Benchmark Run Resumability & Checkpointing

- **Estimated time**: 2 days
- **Tasks/Components**: 4/2
- **Execution**: execute-epic --strong
- **Status**: Draft (2026-06-25 — reconstructed from the 10.2 resolve-td clarification that deferred it)

> **Provenance.** This feature was carved out — but never written up as its own epic —
> during Epic 10.2 (Benchmark Run & Scoring). The deferral is recorded verbatim in
> `.planning/.knowledge/clarifications-10.2_benchmark_run_scoring-Q1.md`: the residual
> concern that "N-1 cases of paid LLM work [are] lost on one transient failure" is "a
> SEPARATE low-priority resumability/checkpointing feature, not a correctness bug." Epic
> 10.2 intentionally aborts the whole run on a total-roster case failure to avoid emitting
> a poisoned, non-reproducible `RunResult`; this epic adds the checkpointing that makes that
> abort recoverable instead of total. Numbered 10.x to preserve the benchmark lineage
> (10.0 leaderboard → 10.1 ingestion → 10.2 run+scoring → 10.3 resumability).

## Context

`atcr benchmark run` (`cmd/atcr/executeBenchmarkRun`, `cmd/atcr/benchmark_run.go`) executes
each suite case's diff through the production review path and accumulates one
`scorecard.PublicRecord` per reviewer across every case. A benchmark over a real suite is
many cases × many reviewers of **paid** LLM work, run serially in a single loop with a
function-local `accs` accumulator (`cmd/atcr/benchmark_run.go:57`).

## Problem Statement

Any per-case error inside the loop returns immediately
(`return nil, fmt.Errorf(...)` at `cmd/atcr/benchmark_run.go:63,82,86,91,95`), which
discards the entire `accs` map by design. A single transient failure — a total-roster
case failure (no reviewer saw the case), a network blip, a rate-limit, a read error — on
case N throws away the completed, already-paid-for work of cases 1..N-1. The operator's
only recovery is to re-run the whole suite from scratch and pay for all of it again.

This is deliberately *not* a correctness bug: aborting rather than scoring a transient
total failure as recall-0 is the right call (see 10.2 Q1 — scoring it recall-0 would
conflate infrastructure failure with a genuine missed defect and corrupt every reviewer's
recall metric). The gap is purely **cost recovery**: there is no checkpoint to resume from.

## Proposed Solution

Add opt-in checkpointing to `benchmark run` so a completed-case's scored results are
durably recorded before the next case starts, and a re-run resumes from the first
unscored case instead of restarting.

1. **Checkpoint after each scored case.** Persist the per-case scored outcome (the
   `benchmark.CaseScore` + usage-gated cost/latency per reviewer, keyed by case index)
   to a run-scoped checkpoint file as soon as a case completes successfully — before the
   loop advances to case i+1.
2. **Resume from checkpoint.** On a re-run of the same suite (matched by the existing
   `ReproHash` reproducibility key), skip cases already present in the checkpoint and
   replay their scored outcomes into `accs`, executing only the unscored remainder.
3. **Reuse the existing resume precedent.** The fan-out layer already has resume
   infrastructure (`internal/fanout/resume.go` `RebuildPool`, Epic 4.1.1) and the
   `ReproHash` contract (`internal/benchmark`); model the checkpoint format and the
   suite-identity guard on those rather than inventing a new mechanism.
4. **Final result is identical.** A resumed run must produce a byte-identical
   `RunResult` to an uninterrupted run over the same suite + transcript — the
   reproducibility contract `export` depends on (`generatedAt` injected, deterministic
   aggregation order) must hold across the resume boundary.

## Acceptance Criteria

- [ ] AC1: `benchmark run` writes a checkpoint entry for each case immediately after it is
      scored and before the next case begins; a process killed mid-suite leaves a
      checkpoint containing exactly the completed cases.
- [ ] AC2: Re-running the same suite resumes from the first unscored case — already-scored
      cases are not re-executed (verified by asserting the injected `Completer` receives
      zero calls for checkpointed cases).
- [ ] AC3: A resumed run produces a `RunResult` byte-identical to an uninterrupted run over
      the same suite + transcript (reproducibility contract preserved across resume).
- [ ] AC4: Resume is guarded by suite identity — a checkpoint from a different suite or a
      changed `ReproHash` is rejected (or ignored), never silently mixed into a new run.
- [ ] AC5: Checkpointing is opt-in via an explicit flag; default behavior is unchanged and
      the abort-on-total-roster-failure semantics from 10.2 remain intact.

## Success Criteria

- [ ] A transient failure on case N no longer forfeits the paid work of cases 1..N-1.
- [ ] Resume path is zero-cost under the stub-`Completer` test path (no network in tests).
- [ ] No regression to the 10.2 reproducibility / determinism contract.

## Technical Constraints

- Keep `internal/benchmark` a live-LLM-free leaf (suite contract + scorer only); the
  checkpoint orchestration lives in `cmd/atcr` (the composition root), consistent with the
  10.2 placement rationale (`cmd/atcr/benchmark_run.go:26-29`).
- Reuse `ReproHash` for suite identity and the `internal/fanout/resume.go` pattern for the
  on-disk format rather than introducing a parallel mechanism.
- Preserve deterministic aggregation order (`sort.Strings(order)`,
  `cmd/atcr/benchmark_run.go:119`) and injected `generatedAt` so resumed results stay
  reproducible.

## Out of Scope

- Changing the abort-on-total-roster-failure decision itself (10.2 Q1 — that is correct
  and stays).
- Parallelizing case execution (the loop stays serial; this epic only adds durability).
- Partial-case (mid-roster) checkpointing — a case is the atomic checkpoint unit.
- Automatic retry/backoff of the failed case (the operator re-runs; resume picks up where
  it stopped).

## Dependencies

- Epic 10.2 (Benchmark Run & Scoring) — completed; provides `executeBenchmarkRun`, the

- Epic 10.1 (Diff-File Ingestion) — completed; the `PrepareReviewFromDiff` entry each case

## Clarifications

### Recorded 2026-06-25 (rubber duck — /execute-epic)

**Key Decisions:**
- The opt-in checkpoint flag is path-valued: `--checkpoint <file>`. Absent = checkpointing
  off (default behavior unchanged, satisfying AC5 with no derivation logic). This matches
  every other filesystem-location flag on `benchmark run` (`--suite-path`, `--out`) and the
  fanout resume precedent (caller-supplied paths only).
- Suite-identity mismatch on resume (AC4) is FAIL-CLOSED: a checkpoint whose recorded
  `ReproHash` (or suite / suite_version) differs from the current suite aborts with a clear
  error instructing the operator to remove the stale checkpoint to start fresh. It is never
  silently discarded/overwritten — mirroring `internal/fanout/resume.go`'s `ErrRangeChanged`
  / `ErrRosterChanged` hard-abort contract, and required by AC4's "never silently mixed".

**Scope Boundaries:**
- IN scope: path-valued `--checkpoint` flag; per-case checkpoint write (atomic temp+rename)
  after each case is scored and before the next begins; ReproHash-guarded resume that replays
  checkpointed cases into the `accs` accumulator and executes only the unscored remainder;
  byte-identical `RunResult` across the resume boundary.
- OUT of scope: changing the 10.2 abort-on-total-roster-failure decision; parallelizing the
  case loop; mid-roster/partial-case checkpointing (a case stays the atomic unit);
  auto-retry/backoff of the failed case.

**Technical Approach:**
- Checkpoint type + read/write/resume logic lives in `cmd/atcr` (new file
  `benchmark_checkpoint.go`), keeping `internal/benchmark` a live-LLM-free leaf.
- Checkpoint format: a single JSON file headed by `{repro_hash, suite, suite_version}` plus a
  list of completed case entries (keyed by case index), each carrying per-reviewer
  `{agent, model, persona, expected, raised, usage_reported, cost_usd, latency_ms}` — exactly
  the fields the run loop folds into `accs`. Replayed in case-index order so float cost sums
  and latency medians reconstruct identically (AC3).
- `executeBenchmarkRun` takes a `checkpointPath string`; empty preserves the 10.2 path verbatim.

**Original Questions and Answers:**
1. Q: Flag shape for the opt-in checkpoint — `--checkpoint <file>` or boolean-with-derived-path?
   A: `--checkpoint <file>` (path-valued). Matches `--suite-path`/`--out` and the fanout
      caller-supplied-path precedent; absent = off, satisfying AC5 with no derivation logic.
2. Q: AC4 mismatch behavior — fail-closed error, or ignore-and-overwrite?
   A: Fail-closed. Mirrors `ErrRangeChanged`/`ErrRosterChanged`; AC4's "never silently mixed"
      makes ignore-and-overwrite non-compliant.
