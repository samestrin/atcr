# Epic Plan 19.10: Per-Model Payload Sizing for the Multi-Agent Reviewer

- **Estimated time**: 2–3 days
- **Tasks/Components**: 6/2

## Objective

Size each reviewer's payload to **its own model's token context window** (reserving the output-token budget), and degrade gracefully on overflow, so a large diff no longer silently guts the multi-agent review. Today a single global **byte** budget is shipped to a heterogeneous roster whose models span 32k→144k-token windows: the payload overflows the small-context models outright, leaves no output headroom on the large ones, and slow local backends time out — so a 6,400-line sprint returned **1 finding from 11 reviewers (5 ok, 3 timeout, 3 failed)**.

## Context

Confirmed from the `19.6_community_registry_hub` multi-agent review run (2026-07-08). Diff under review: **101 files / 6,429 insertions**. Artifacts: `.planning/sprints/active/19.6_community_registry_hub/code-review/multi-agent/`.

**Confirmed root cause — the payload sizer has no per-model token awareness.**

1. A single global **byte** budget is applied to every agent — `.atcr/config.yaml:21` `payload_byte_budget: 524288` (512 KB). `ApplyByteBudget` (`internal/payload/budget.go:46`) sheds files by **bytes** against that one value, applied identically at `internal/fanout/review.go:464` and `:726`. There is **no per-model context-window table anywhere** in the codebase, and no bytes↔tokens conversion.

2. Because the budget is global, **every agent got the same payload and dropped the same 102 files**; `blocks.txt` came out at 506 KB (just under the 512 KB cap).

3. 512 KB ≈ ~147k tokens. `otto`'s provider measured it exactly: `Input length (147295 tokens) exceeds the maximum allowed length (144941 tokens)`. That one payload was shipped unchanged to:
   - `otto` (144,941-token model) → over by ~2.4k tokens → **failed**.
   - `dax` (32,768-token model) → over by ~115k tokens → **failed**.

4. The 8,192 output request is **never reserved**. `internal/fanout/review.go:948-958` defines `const defaultMaxTokens = 8192`, applied via `maxTokensPtr()` to every call, model-agnostic. `dax`'s litellm error reports the boundary as `24577` input tokens = **32768 − 8192 + 1**, i.e. input+output blew the window by exactly one token past the (never-reserved) output budget.

**Secondary failure axes (same run):**

- **No graceful degradation.** litellm `context_window_fallbacks` is unset ("No fallback model group found"), so overflow **kills** the agent instead of falling back or truncating. `otto`'s error literally names the escape hatch: "enable `--allow-auto-truncate`".
- **Timeout too tight for slow backends.** `greta`/`vera`/`brad` hit `context deadline exceeded` against the local litellm endpoint; `brad` at exactly 599,998 ms — the `timeout_secs: 600` wall (`.atcr/config.yaml:20`, `DefaultTimeoutSecs = 600` at `internal/registry/config.go:25`), made worse by the oversized payload.

The irony: a code-review product cannot review its own large sprint. Fixing this early (19.10, ahead of epics 20.0–31.1) means every downstream epic's `/execute-code-review` runs against a working reviewer.

## Task Breakdown

1. Add a per-model **context-window resolver** — a static table keyed by model id with a conservative default for unknown models — exposing the token window for each roster model. Keep it deterministic (no hot-path network call); dynamic/live resolution is out of scope (see below).
2. **Reserve the output budget and size per model (root cause #1).** Derive an *effective per-agent* byte budget = `min(global payload_byte_budget, (contextWindow − defaultMaxTokens − promptOverhead) × bytesPerToken)` and apply it per agent (replacing the single global cap at `internal/fanout/review.go:464` and `:726`), so each model receives a payload sized to *its* window with the 8,192 output tokens reserved. This moves the payload build from one shared `blocks.txt` to per-agent slices.
3. **Graceful degradation on overflow.** Enable litellm `context_window_fallbacks` and/or pass the auto-truncate signal so a too-large payload downgrades to a fallback model or truncates rather than returning `status=failed` (the `dax`/`otto` path).
4. **Scale the request timeout to payload size** (or apply a higher ceiling for slow local backends) so a large-but-valid payload no longer hits the 600 s wall (the `greta`/`vera`/`brad` path).
5. **Diagnosability.** Record per-agent effective budget, resolved model window, and reserved output tokens in the pool `summary.json`, so a future run is inspectable without reverse-engineering provider error strings.
6. **Audit (tail-end).** Re-run the multi-agent review over a large diff (the 19.6 range, base `f9d5161…` → head `b6bcb67…`, 101 files) and assert: zero `ContextWindowExceededError` across the roster, zero context-deadline timeouts on the previously-failing models, and grounded findings from multiple agents (not 1). Capture before/after counts in this epic's evidence.

## Acceptance Criteria

- [ ] Each reviewer's payload is sized to its own model window: for a model with window `W` and output cap `O`, the built payload's estimated input tokens ≤ `W − O`. A unit test covers both a 32k-window model and a 144k-window model.
- [ ] A regression test asserts the input budget reserves the output cap, so the `24577 + 8192 > 32768` overflow (the `dax` arithmetic) can no longer occur.
- [ ] Re-running the exact 19.6 range (base `f9d5161…` → head `b6bcb67…`) produces **zero** `ContextWindowExceededError` across the roster, and the two previously-failed context-overflow agents (`dax`, `otto`) complete.
- [ ] On an intentionally oversize payload, an agent **degrades** (fallback model or truncation) instead of returning `status=failed` — verified by a test/fixture.
- [ ] The three previously-timed-out agents (`greta`, `vera`, `brad`) complete on a large-but-valid payload without hitting the wall (timeout scales with payload, or a raised ceiling is applied).
- [ ] Pool `summary.json` records per-agent effective budget, resolved model window, and reserved output tokens.

### Advisory observations (3)

- ℹ️ **Hold COMPONENT_COUNT ≤ 2 by keeping T2/T4 in `internal/payload` + `internal/fanout` — the resolvers they read live in `internal/registry`.** Timeout resolution (`EffectiveTimeoutSecs` at `internal/registry/config.go:912`, `DefaultTimeoutSecs` at `:25`) and byte-budget resolution (`PayloadByteBudget` precedence in `internal/registry/precedence.go` / `project.go`) live in `internal/registry` — a component the plan does NOT declare. The declared 2-component boundary holds only if T2's per-agent effective-budget computation and T4's timeout scaling READ the already-resolved registry values and apply them in `payload`/`fanout` (e.g. scale the resolved `TimeoutSecs` where the deadline is set, `review.go:516`), rather than modifying registry's resolvers or adding a config-schema field (already deferred in Out of Scope). If the implementer edits `internal/registry`, COMPONENT_COUNT becomes 3 and the epic exceeds `/execute-epic`'s ≤2 guard → route to `/init-plan`. This makes the plan's own Scope Note concrete.

- ℹ️ **T2's `bytesPerToken` should use the conservative measured ratio (~3.5 B/token), not the codebase's ~4 B/token comment.** `internal/registry/project.go:89` comments "512 KiB ≈ 128k tokens" (≈4.1 bytes/token), but the 19.6 run measured 506 KB ≈ 147k tokens (≈3.5 bytes/token, `otto`'s provider count). T2's effective-budget formula must use the lower bytes/token figure so it OVER-reserves (smaller payload, safer); 4 B/token would under-count tokens and re-introduce overflow on token-dense payloads. Consider deriving the constant from the worst-case observed ratio plus a safety margin.

- ℹ️ **No existing per-model token-window table to reuse — T1 is genuinely net-new; do not confuse it with `MaxContextLines`.** Codebase search confirmed the plan's claim: no per-model context-window table and no bytes↔tokens conversion constant exists anywhere. Note that `DefaultMaxContextLines`/`MaxContextLines` (`internal/registry/config.go:36`, `:346`) is a per-chunk diff **line-count** budget, NOT a token window — unrelated to T1. Name the new resolver distinctly (e.g. `ContextWindowTokens`) so the two concepts don't collide for future readers.
