# Epic Plan 12.2: Sprint Plan Scoping in ATCR

- **Estimated time**: 3 days
- **Tasks/Components**: 5/2
- **Status**: Draft

## Context

Our legacy `llm-support` engine (in `llm-tools`) supported a `--sprint-plan` flag. This flag allowed developers to pass a sprint or epic planning markdown file to scope the code review. The content was injected into the prompt as a `SCOPE CONSTRAINT`, instructing the AI reviewer models to only flag findings in files and areas directly related to the active tasks, while ignoring unrelated changes (like standard formatting updates or third-party dependency bumps) that happened to be in the diff.

In the current ATCR implementation, this capability was dropped (as documented in Epic 12.0: *"atcr has no sprint-plan concept. Reviewers fall back to diff-touched-line scope"*). Losing this context results in increased noise and off-topic findings in reviews.

## Problem Statement

Without a sprint-plan constraint, ATCR's reviewer personas analyze the entire git diff indiscriminately. If a pull request includes necessary but out-of-scope changes (e.g., package updates, mechanical refactorings, formatter applications), the model generates findings for them. This creates noise, increases review times, and leads to developers having to close generated technical debt cards as "Won't Fix (Out of Scope / Vague)".

To ensure high-quality, targeted feedback, ATCR must natively support sprint-plan scoping.

## Proposed Solution

Introduce sprint-plan scoping support to ATCR:
1. Add a `--sprint-plan` CLI flag to `atcr review` that accepts a path to a markdown plan.
2. Read and parse the sprint plan file, safely handling missing or empty files.
3. Construct a structured `SCOPE CONSTRAINT` prompt block.
4. Inject this block into the LLM prompt directly before the diff payload so reviewer personas are constrained by the active sprint goals.

## Requirements

### Functional
- CLI accepts a `--sprint-plan` path flag.
- Missing file at the specified path is ignored silently (defaulting to diff-wide review).
- Unreadable file at the specified path warns the user on `stderr` but does not crash the review execution.
- If a valid plan is provided, wrap it in a `SCOPE CONSTRAINT` template block and inject it into the reviewer prompts.
- Ensure the prompt constraint instructs the LLMs to suppress findings for files/changes unrelated to the work items defined in the plan.

### Non-Functional
- The injection must happen immediately before the payload/diff text so models do not miss it.
- Minimal overhead: reading the plan file must run in less than 5ms.

## Technical Approach

### 1. CLI and API Changes
- Add `SprintPlanPath` field to `ReviewRequest` struct in `internal/fanout/review.go`.
- Map the `--sprint-plan` flag in `cmd/atcr/review.go` and `cmd/atcr/flags.go` to `ReviewRequest.SprintPlanPath`.

### 2. Loading the Plan
- Implement a helper function `readSprintPlan(path string) (string, error)` (modeled after the legacy implementation in `llm-tools`) that handles existence checks and empty file logic.

### 3. Prompt Construction & Injection
- Expose the constraint block in the prompt rendering lifecycle.
- To ensure the scope constraint is placed directly before the diff without requiring edits to all existing persona files, we can construct the `SCOPE CONSTRAINT` string and prefix it directly to `PayloadContext.Payload` (or add a dedicated `ScopeConstraint` field to `PayloadContext` and update the default base persona template to render it).
- *Recommended Approach*: Add `ScopeConstraint` string field to `PayloadContext` in `internal/payload/template.go` and prefix it to the payload during prompt assembly in `internal/fanout/review.go` for seamless backwards compatibility.

## Implementation Components

1. **CLI Flag & Request Propagation**
   - Register `--sprint-plan` flag in `cmd/atcr/flags.go`.
   - Update `ReviewRequest` struct to carry the path.

2. **Sprint Plan Reader**
   - Add `internal/support/sprint_plan.go` to load, sanitize, and format the scope constraint block.

3. **Engine and Prompt Integration**
   - Read the sprint plan during `PrepareReview` / `PrepareReviewFromDiff`.
   - Feed the constraint block into `PayloadContext` when constructing reviewer slots in `internal/fanout/review.go`.

4. **Integration Testing**
   - Add unit tests verifying:
     - Missing file is ignored.
     - Empty file is ignored.
     - Valid file formats the block correctly.
     - Prompt contains the constraint block.

## Constraints

- Changing the prompt content must not break the `diffCacheKey` calculations (since `diffCacheKey` hashes the final rendered prompt, it will naturally invalidate and compute a new key if a sprint plan changes, which is correct).

## Out of Scope

- Automated task matching (e.g. checking git diff paths against described tasks programmatically in Go). Scoping is left to the LLM's comprehension of the injected prompt block.

## Acceptance Criteria

- [ ] AC1: `atcr review --sprint-plan <path>` is a valid CLI flag.
- [ ] AC2: A missing or empty sprint plan file allows the review to proceed without warnings.
- [ ] AC3: An unreadable sprint plan file (e.g. permission error) prints a warning to stderr but allows the review to proceed.
- [ ] AC4: A valid sprint plan file injects the `SCOPE CONSTRAINT` prompt block before the diff payload.
- [ ] AC5: Roster runs with cache enabled recalculate the cache key when the sprint plan changes.

## Success Criteria

- [ ] AI reviews of large PRs with unrelated changes generate 0 findings in the out-of-scope files when constrained by a sprint plan.

## Rollback Plan

- If scoping causes issues, running reviews without `--sprint-plan` operates identically to the baseline engine.

## Security Considerations

- The sprint plan file is trusted local operator input. No additional sandbox validation is required.

## Timeline Milestones

1. CLI flag and Request wiring.
2. Sprint plan reader helper implementation.
3. Prompt injection and test suite verification.
