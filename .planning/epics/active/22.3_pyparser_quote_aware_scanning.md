# Epic Plan 22.3: Quote/Escape-Aware Scanning in pyparser

- **Estimated time**: 0.5 day
- **Tasks/Components**: 2/1

## Objective

Make `pyparser`'s `scanTripleQuotes` and `stripComment` quote/escape-aware so a `#` inside a triple-quoted string, or a triple-quote appearing inside a `#` comment, no longer flips the multi-line-string state machine incorrectly and silently erases real code from structural hashing.

## Context

`internal/astgroup/parsers/src/pyparser/main.go:112` documents this heuristic limitation but does not bound it: on unusual source, the structural hash can diverge from the true AST because arbitrary spans of real code get misclassified as string content and dropped from `significantLines`. This was previously deferred as "accepted PoC scope," but has no epic plan backing that deferral.

## Task Breakdown

1. Restrict `scanTripleQuotes` to run only over the code portion of a line (skip scanning after an unquoted `#`).
2. Skip triple-quote detection when it occurs inside a single-line string literal.

## Acceptance Criteria

- [ ] A regression fixture with a `#` containing a triple-quote inside a comment is classified correctly (comment, not string).
- [ ] A regression fixture with a string literal containing `#` is classified correctly (string, not comment).
- [ ] Existing `pyparser` structural-hash tests pass unchanged for well-formed input.

## Components Touched

- `internal/astgroup/parsers/src/pyparser`

## Out of Scope

- The shared guest ABI extraction across parsers (tracked separately in epic 22.2).
- Full AST-accurate Python parsing — this remains a heuristic, proximity-grouping fallback by design.

## Dependencies
