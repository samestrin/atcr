# Epic Plan 7.1: Local Syntax/Compile Guard for Fixes

**Estimated Durations**: 1 week  
**Tasks/Components:** 3/1
**Status:** Draft (queued; depends on Epic 7.0)

## Objective

Provide a fast, local validation layer that ensures generated fixes are syntactically valid before they are presented to the user.

## Context

Epic 7.0 introduces an Executor Model for Fix Generation. However, LLMs can generate syntactically invalid code. Running a full test suite (Epic 11.0) is slow; a fast local syntax check can catch trivial errors immediately.

## Problem Statement

1. Generated fixes might introduce syntax errors.
2. Presenting syntax errors degrades trust.

## Proposed Solution

- Apply the generated fix to an in-memory copy of the file.
- Run a fast language-specific syntax check (e.g., `go build`, `ast.ParseFile` for Go, `python -m py_compile` for Python).
- If the syntax check fails, mark the fix as `invalid_syntax` or ask the executor to retry.

## Acceptance Criteria

- [ ] Fixes are applied to in-memory/temp files and parsed.
- [ ] Support built-in syntax checking for at least Go.
- [ ] Invalid fixes are either retried (up to N times) or flagged in the output.

## Dependencies

- Epic 7.0 (Executor Model).

## Clarifications

### Recorded 2026-06-22 (rubber duck — /execute-epic)

**Key Decisions:**
- The `Fix` field is definitionally free-form ("corrected code or a precise change instruction", `internal/verify/executor.go:236`), so the guard uses a code-aware, conservative strategy: a heuristic detects whether the Fix string plausibly contains Go source before attempting `go/parser`; clearly-prose change-instructions pass through untouched (no false "invalid" flag). Only fixes that look like code but fail to parse are marked invalid.
- Invalid fixes are flagged (not retried). The existing `FixWarning` field (`internal/reconcile/emit.go:133`, already written at `executor.go:128,134`) carries an `invalid_syntax: <parser error>` message — a third assignment site, zero new types/fields. The AC permits either branch; flag-only is chosen for minimum surface area.
- Go-only. Python is a parenthetical example in the prose, not an AC line; the project is a pure Go module with no language-detection infrastructure.

**Scope Boundaries:**
- IN scope: Go syntax validation (via stdlib `go/parser`) of generated fixes inside `generateFixes`, run immediately after the fix string is produced and before `f.Fix` is committed; marking syntactically-invalid code fixes via `FixWarning`.
- OUT of scope: applying fixes to the working tree; languages beyond Go; full type-check/compile (syntax/parse only); executor retry (no `fix_max_retries` config); changes to the Epic 7.0 executor output contract.

**Technical Approach:**
- Pure stdlib (`go/parser`, `go/token`) — no shelling to `go build`/`go vet`, no new runtime dependencies, no temp module.
- Code-vs-prose heuristic: attempt to parse as a full file when a `package` clause is present; otherwise attempt to parse the fix wrapped in a synthetic compilation unit; only when the content is plausibly Go code AND fails to parse is it flagged `invalid_syntax`.
- Status surfaced through the existing `FixWarning` field; the attempted fix remains visible alongside the warning.

**Original Questions and Answers:**
1. Q: The Fix is free-form, not a patch/full file — which validation strategy (code-aware conservative / full-file only / change Epic 7.0 first)?
   A: (a) Code-aware, conservative. Detect plausible Go source before running `go/parser`; never flag prose change-instructions as invalid. Honors the existing API contract without changing Epic 7.0's output shape.
2. Q: Invalid-fix handling — retry up to N times, or flag only?
   A: (a) Flag only. Reuse the existing `FixWarning` field with an `invalid_syntax: <parser error>` message; no new config or types. Retry deferred as materially larger surface.
3. Q: Go-only for this epic (Python out of scope)?
   A: Yes, Go-only. AC requires "at least Go"; Python is only a parenthetical example and the project has no language-detection infrastructure.
