# Epic Plan 2.1: Manifest Review-Stage Recording

**Estimated Durations**: TBD

## Objective

Record `snapshot_mode`, `head_sha`, and `snapshot_worktree_path` into the `stages.review` block of `manifest.json` when `SnapshotFor` is wired into the agent loop, satisfying AC 03-02 Scenario 5 and AC 03-03 Scenarios 4-5.

## Context

This work was deferred from Epic 2.0 (tool-using reviewers) per sprint-plan ┬¦2.5.A. The snapshot machinery (`internal/tools/snapshot.go`) is complete and tested; the gap is that `SnapshotFor` is not yet called from the agent loop, so no code path currently writes these fields into the manifest review stage.

TD item origin: `internal/tools/snapshot.go` ŌĆö `[ ]` LOW, from sprint 2.0_tool_using_reviewers.

## Problem Statement

`manifest.json` `stages.review` currently omits:
- `snapshot_mode` ŌĆö `"live"` or `"worktree"` depending on git state
- `head_sha` ŌĆö the commit SHA the snapshot was taken at
- `snapshot_worktree_path` ŌĆö the worktree path if mode is `"worktree"`, empty otherwise

AC 03-02 Scenario 5 requires these fields to be present in the manifest after a review run. AC 03-03 Scenarios 4-5 require test assertions that verify the fields for both live and worktree modes.

## Proposed Solution

When wiring `SnapshotFor` into the agent-loop call site:

1. Capture the `SnapshotResult` returned by `SnapshotFor`.
2. Pass `snapshot_mode`, `head_sha`, and `snapshot_worktree_path` into the `internal/payload/manifest.go` review-stage writer.
3. Add the manifest assertion tests from AC 03-02/03-03 covering both live and worktree paths.

## Acceptance Criteria

- [ ] `manifest.json` `stages.review` contains `snapshot_mode`, `head_sha`, and `snapshot_worktree_path` after a review run
- [ ] AC 03-02 Scenario 5 assertion test passes
- [ ] AC 03-03 Scenarios 4-5 assertion tests pass (live mode and worktree mode)
- [ ] Existing snapshot and manifest tests unaffected
