# Epic 4.5: Circuit Breaker and Provider Health Tracking

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Medium (resilience, cost savings)

## Problem

When an LLM provider (OpenAI, Anthropic) is experiencing an outage or degraded performance, ATCR continues to send requests to that provider for every agent. A review with 10 agents using the same provider will make 10 independent attempts, each timing out or failing, wasting time and API quota.

### Concrete Failure

OpenAI is down for 30 minutes. A user runs `atcr review` with 5 OpenAI agents. Each agent tries 3 times (retries from Epic 2.2), waiting 120 seconds per timeout. Total time wasted: 5 agents × 3 retries × 120s = 30 minutes. All 15 API calls fail. The review produces no results.

## Root Cause

Each agent invocation is independent. There is no shared state tracking provider health across agents. The retry logic in `internal/llmclient/client.go` retries individual requests, but there is no global "this provider is failing, stop trying" signal.

## Solution

Implement a circuit breaker pattern per provider. Track recent API call success/failure rates. When a provider's error rate exceeds a threshold, "open the circuit" and skip that provider for a cooldown period. Agents using that provider fail fast (or fall back to their fallback chain).

### Core Design

```go
// internal/circuitbreaker/breaker.go
package circuitbreaker

import (
    "sync"
    "time"
)

type State int

const (
    StateClosed   State = iota // normal operation
    StateOpen                  // circuit tripped, fail fast
    StateHalfOpen              // testing if provider recovered
)

type Breaker struct {
    mu              sync.Mutex
    provider        string
    state           State
    failureCount    int
    successCount    int
    lastFailure     time.Time
    threshold       int           // failures before opening
    cooldown        time.Duration // how long to wait before half-open
}

func New(provider string, threshold int, cooldown time.Duration) *Breaker

// Allow returns true if the request is allowed (circuit closed or half-open).
// Returns false if circuit is open (fail fast).
func (b *Breaker) Allow() bool

// RecordSuccess records a successful API call.
func (b *Breaker) RecordSuccess()

// RecordFailure records a failed API call.
func (b *Breaker) RecordFailure()

// State returns the current circuit state.
func (b *Breaker) State() State

// Registry holds breakers per provider.
type Registry struct {
    mu       sync.RWMutex
    breakers map[string]*Breaker
}

func NewRegistry() *Registry
func (r *Registry) Get(provider string) *Breaker

// Global registry.
var DefaultRegistry = NewRegistry()
```

### Integration with llmclient

```go
// internal/llmclient/client.go
func (c *Client) Complete(ctx context.Context, inv Invocation) (string, error) {
    breaker := circuitbreaker.DefaultRegistry.Get(inv.Provider)

    // Check circuit before attempting
    if !breaker.Allow() {
        return "", &CircuitOpenError{Provider: inv.Provider}
    }

    // Make the API call
    result, err := c.send(ctx, resolveEndpoint(inv.BaseURL), key, body)

    // Record outcome
    if err != nil {
        breaker.RecordFailure()
        return "", err
    }
    breaker.RecordSuccess()
    return result, nil
}
```

### Integration with fanout engine

When an agent's API call fails with `CircuitOpenError`, the engine treats it as a permanent failure (not retryable) and moves to the fallback chain immediately.

### Tuning Parameters

- **Threshold**: 3 consecutive failures → open circuit.
- **Cooldown**: 60 seconds → transition to half-open.
- **Half-open**: 1 successful call → close circuit; 1 failure → reopen.

These are hardcoded initially; can be made configurable later.

## Scope

### In Scope

- `internal/circuitbreaker` package with `Breaker` and `Registry`.
- Circuit breaker integration in `internal/llmclient/client.go`.
- `CircuitOpenError` type for fail-fast signaling.
- Fanout engine treats `CircuitOpenError` as permanent failure (triggers fallback).
- Metrics: `atcr_circuit_breaker_state` gauge (labels: provider, state).
- Unit tests for breaker state transitions.
- Integration test: simulate provider failure, verify circuit opens.

### Out of Scope

- Per-agent circuit breakers (breaker is per-provider, shared across agents).
- Configurable thresholds (hardcoded initially).
- Circuit breaker persistence (state resets on process exit).
- Manual circuit breaker control (`atcr circuit-breaker reset` command).

- [ ] AC1: After 3 consecutive API failures for a provider, the circuit opens.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Circuit opens too aggressively (transient blip) | Medium | Medium | Threshold of 3 consecutive failures is conservative; can increase if needed |
| Circuit stays open too long (provider recovered) | Low | Low | 60-second cooldown is short; half-open state tests recovery |
| Circuit breaker adds latency | Low | Low | Circuit check is in-memory (<1μs) |
| Race condition in concurrent breaker access | Low | High | Mutex protects state; test with concurrent access |

## Open Questions

1. **Should the threshold be per-agent or per-provider?**
   - Option A: per-provider (all agents using OpenAI share one breaker).
   - Option B: per-agent (each agent has its own breaker).
   - Recommendation: Option A; provider outages affect all agents using that provider.

2. **Should we distinguish between different error types?**
   - Option A: all errors count toward circuit (simple).
   - Option B: only 5xx and timeouts count; 4xx (auth errors) do not.
   - Recommendation: Option B; auth errors are permanent and should not trip the circuit.
