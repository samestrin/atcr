# Epic Plan 19.2: Shared Registry Remote Fetch

- **Estimated time**: 2-3 days
- **Tasks/Components**: 3/2
- **Execution**: execute-epic --strong

## Objective

Let a team share one registry by pointing atcr at a remote `registry.yaml` via `ATCR_REGISTRY_URL`, reusing the existing loader and env-var key contract.

## Context

`internal/registry` already loads `registry.yaml` and defers API-key reads to runtime env vars (`TestRegistryLoad_APIKeyNotReadAtLoad`, `docs/registry.md`, `internal/llmclient/client.go:resolveKey`). Sharing a team registry is a thin extension — fetch a remote file — not a new subsystem.

## Task Breakdown

1. Extend the `internal/registry` loader to fetch and parse a remote `registry.yaml` when `ATCR_REGISTRY_URL` is set, falling back to the local file when unset.
2. Validate/merge the remote registry against the existing schema; guarantee API keys are never read from the remote file — env-var-only, unchanged.
3. Document the config-repo pattern (a `registry.yaml` in a shared git repo referenced by URL) in `docs/registry.md`.

## Acceptance Criteria

- [ ] With `ATCR_REGISTRY_URL` pointing at a reachable `registry.yaml`, providers/personas load from the remote source.
- [ ] API keys still resolve only from env vars; a key present in the remote file is ignored (with a warning).
- [ ] An unreachable/invalid URL exits with a clear error and the documented local-fallback behavior.
- [ ] `go test ./...` passes; covered by `internal/registry/*_test.go`.

## Components Touched

- `internal/registry` (loader extension)
- `docs/registry.md` (onboarding note)

## Out of Scope

Secrets-manager integration; auth on the registry URL beyond what `http.Get` supports; registry write/publish tooling.

## Dependencies

- `internal/registry` loader (exists).
