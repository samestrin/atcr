# Epic 14.3: Context-Aware Diff Chunking

- **Estimated time**: 2 weeks
- **Tasks/Components**: 4/3
- **Execution**: execute-epic --strong

## Context

LLMs suffer from attention degradation when fed excessively large inputs. If `atcr` attempts to review a massive git range (e.g., thousands of lines) in a single prompt, the LLM often loses its place and enters a "runaway generation" loop, outputting hundreds of hallucinated TD items. However, naive per-file chunking introduces a massive API cost multiplier by duplicating the system prompt for every file.

## Problem Statement

We currently lack guardrails on the size of the context we provide to individual reviewer personas. Massive diffs directly correlate with hallucination rates, but fanning out requests indiscriminately drives up API costs unnecessarily for budget-conscious users.

## Proposed Solution

1. **Optional Chunking Strategies**: Introduce an opt-in chunking strategy. By default, `atcr` uses a `bulk` strategy (sending the entire diff in one prompt) to keep API costs strictly bounded. Users can opt into a `chunked` strategy for higher accuracy on large PRs.
2. **Persona Context Limits**: Add a `max_context_lines` (or `max_tokens`) field to `AgentConfig` in `internal/registry/config.go` (surfaced via `registry.yaml`). This allows different models to have tailored thresholds based on their capabilities and cost profiles.
3. **Context-Aware Bin Packing**: When the `chunked` strategy is enabled, the fan-out dispatcher will use bin packing. Instead of sending one file per request, it will group multiple files together into a single chunk until it hits the persona's specific context limit. This minimizes API request duplication while strictly respecting each model's attention window.

## Acceptance Criteria

- [ ] AC1: Introduce a configuration flag to toggle between `bulk` (default) and `chunked` review strategies.
- [ ] AC2: Add a `max_context_lines` field to `AgentConfig` in `internal/registry/config.go` (with a reasonable default if omitted).
- [ ] AC3: Implement a bin-packing algorithm in the dispatcher that groups files into chunks without exceeding a persona's `max_context_lines`.
- [ ] AC4: Ensure the Reconciler correctly merges findings from multiple chunks originating from the same persona.

## Technical Constraints

- The bin-packing algorithm must ensure that a single file is never split across chunks (if a single file exceeds the limit, it should be sent as its own chunk, possibly triggering a warning).
- The Reconciler must uniquely identify findings without treating chunks from the same persona as different reviewers.
- The bin-packing algorithm (`internal/fanout/chunker.go`) must implement its own per-file diff splitting from the raw diff text rather than calling into `internal/payload/` (which is package-private). Split on `diff --git a/` boundaries and count lines via `strings.Count`. This keeps the component count to 3 (`internal/registry`, `internal/fanout`, `internal/reconcile` + `reconcile/`) and avoids cross-package coupling for what is a ~20-line utility.
- Chunk result directories must be named `<agent-name>` (not `<agent-name>:chunk-N`) so the Reconciler attributes all chunks from the same persona to a single logical source. Epic 14.2 (active) implements a Two-Persona Consensus Rule that counts distinct persona sources — if chunk directories use per-chunk names, 14.2's filter will treat them as N voters instead of 1, producing false consensus. This epic (14.3) must be merged before Epic 14.2 executes.

## Non-Functional Requirements

### Scalability
- The system must comfortably handle fanning out multiple chunks across 3+ personas without hitting standard API rate limits (requires concurrency controls and backoff).

### Cost Efficiency
- Bin packing must demonstrably reduce the number of API requests compared to naive per-file chunking.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Loss of cross-file context | High | High | The Reconciler will still have access to the global context. If a bug requires seeing two files, the file-level reviewer might miss it. This is an accepted tradeoff of chunking. |
| Single file exceeding context | Medium | Low | If a single file exceeds the persona's `max_context_lines`, send it as an isolated chunk and optionally log a warning that the file exceeds the optimal context window. |

## Clarifications

### Recorded 2026-07-01 (rubber duck — /execute-epic)

**Key Decisions:**
- Chunk attribution uses **Option A (aggregate-at-source)**: in chunked mode the fan-out engine loops a persona's N bin-packed chunk calls internally and merges their raw outputs into the single `raw/agent/<persona>/` artifact directory with `Reviewer=<persona>` stamped once, before the artifact-write step. This requires zero change to the already-merged Epic 14.2 reconciler. Option B (per-chunk subdirs) is rejected — `internal/fanout/artifacts.go` (writePool) hard-errors on duplicate agent directories and `agentDirName()` only produces flat single-segment names.
- `max_context_lines` default = **1500** diff lines per chunk (≈22k–27k tokens; under the 128k `DefaultPayloadByteBudget` and the documented 40k-token conservative floor).
- The strategy toggle is a **run-wide** `review_strategy: bulk|chunked` on Settings/Registry (default `bulk`), resolved once per run like `PayloadMode`/`MaxParallel`/`PayloadByteBudget`. The global flag only decides *whether* to bin-pack; each bin's size is governed by the individual agent's own `max_context_lines`.
- Epic 14.2 is already merged; its ordering clause (line 33) is stale/inverted. Build 14.3 for compatibility with the live 14.2 reconciler and do NOT modify, relax, or touch 14.2's consensus-counting logic. `reconcile/consensus.go` counts distinct `Finding.Reviewer` values, so it is resilient to chunk fan-out as long as each finding's `Reviewer` is the plain persona name.

**Scope Boundaries:**
- IN scope: registry schema (`max_context_lines` on `AgentConfig`, `review_strategy` on Settings), the chunker + chunked dispatch path, same-persona chunk aggregation in the fan-out engine, `docs/registry.md` update.
- OUT of scope: touching `internal/payload/` (chunker implements its own `diff --git a/`-boundary split per line 32); changing bulk-mode behavior; modifying Epic 14.2 consensus logic; cross-file context recovery within a chunk (accepted tradeoff).

**Technical Approach:**
- `internal/fanout/chunker.go` splits the raw diff on `diff --git a/` boundaries, counts lines via `strings.Count`, and bin-packs into bins ≤ the persona's `max_context_lines` using **greedy next-fit in original file order** (files accumulate into the current bin until the next file would overflow it, then a new bin opens). Next-fit is chosen over first-fit-decreasing because it matches the epic prose ("group multiple files together into a single chunk until it hits the persona's specific context limit") and preserves the diff's presentation order; both satisfy AC3 and the Cost-Efficiency NFR. A single file exceeding the limit becomes its own oversized chunk with a warning.
- Retry/backoff/concurrency for the extra chunk calls is already provided by Epic 4.6 (`internal/llmclient`) — no new backoff code.
