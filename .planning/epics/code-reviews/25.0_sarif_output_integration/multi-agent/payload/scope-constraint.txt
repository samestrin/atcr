# Epic Plan 25.0: SARIF Output Integration

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Support exporting ATCR review findings in standard SARIF (Static Analysis Results Interchange Format) so they feed GitHub Advanced Security's Code Scanning ("Security" tab) and GitLab CI's native SAST report widget — the two centralized, cross-repo security surfaces ATCR's existing PR-check/inline-comment integration (`atcr github`) does not reach.

## Context

Currently, ATCR writes rich markdown files locally, and the maintained composite GitHub Action (`atcr github` + `action.yml`, see `docs/github-action.md`) already posts a PR check and inline "Files Changed" comments directly via the GitHub API — that part of the enterprise-integration story is solved and is explicitly out of scope here. What ATCR does not yet do is feed the *centralized* security surfaces enterprises also expect: GitHub Advanced Security's org-wide Code Scanning "Security" tab (aggregated across repos/branches, with its own dedup/suppression/trend workflow) and GitLab CI's native SAST report widget in the MR view. Both ingest SARIF, not markdown/JSON, so SARIF export is the missing piece for those two surfaces specifically.

## Proposed Solution

1. **SARIF Formatter**: Add a new render target in `internal/report` that maps ATCR's reconciled findings (file, line, message, severity) to the SARIF JSON schema, following the existing `Format*` pattern in `internal/report/render.go` (`FormatMarkdown`/`FormatJSON`/`FormatChecklist`).
2. **CLI Flag**: Add `--format=sarif` to `atcr report` (the existing render command — `atcr review`/`atcr reconcile` do not render output and should not gain a duplicate flag).
3. **CI Example**: Document a GitHub Actions workflow step demonstrating how to pipe `atcr review && atcr reconcile && atcr report --format=sarif` to GitHub's `codeql-action/upload-sarif` (and the GitLab CI equivalent for its SAST widget).

## Acceptance Criteria

- [ ] `atcr report --format=sarif` produces valid SARIF JSON.
- [ ] The SARIF output correctly maps ATCR severities (CRITICAL, HIGH, etc.) to SARIF levels.
- [ ] File paths and line numbers correctly anchor to the git diff.
- [ ] A documentation example exists showing the CI integration (GitHub Code Scanning upload, and the GitLab CI SAST-widget equivalent).

## Components Touched

- `internal/report`

- `docs/`

## Out of Scope

- `atcr github`'s existing direct-API PR check and inline-comment flow — already shipped, unaffected by this epic.
- Direct API calls to post SARIF results to GitHub/GitLab (`codeql-action/upload-sarif` and GitLab's native SARIF ingestion handle that; ATCR only needs to produce the file).

Only `cmd/atcr/report.go` wires `--format` (`cmd/atcr/report.go:25`) through `internal/report.Render()` (`internal/report/render.go:48`), which already switches on `FormatMarkdown`/`FormatJSON`/`FormatChecklist` (`internal/report/render.go:24-26`) — the exact extension point a `FormatSarif` case would slot into. Proposed Solution item 2 and AC1 (`atcr review --format=sarif produces valid SARIF JSON`) should instead target `atcr report --format=sarif`. The CI-example task/AC4 should follow suit: pipe `atcr report --format=sarif` (run after `atcr review && atcr reconcile`, consistent with the existing two-step pattern documented in `docs/ci-integration.md`), not `atcr review --format=sarif` directly.
Suggested action: reword Proposed Solution item 2, AC1, and item 3/AC4 to target `atcr report`.

Context/rationale overlaps with already-shipped `atcr github` PR-comment integration.** ATCR already ships a maintained composite GitHub Action (`atcr github` command, `action.yml`, documented in `docs/github-action.md`) that posts a PR check **and inline "Files Changed" comments** directly via the GitHub API — the exact "inline annotations on PRs" outcome this epic's Context cites as motivation for SARIF ("developers expect AI review findings to appear as inline annotations on their Pull Requests... just like SonarQube or CodeQL").
