+# Tech Debt: Distinguish free vs paid-but-uncorroborated `cost_per_corroborated_finding_usd`
+
+- **Estimated time**: 1 week
+- **Tasks/Components**: 4/2
+
+## Current State
+
+`scorecard.PublicRecord.CostPerCorroboratedFindingUSD` is a plain non-pointer `float64` with no `omitempty` tag (internal/scorecard/export.go:40). Both the production leaderboard `--export` path and the benchmark scorer therefore emit `0.0` in two very different situations:
+
+1. The reviewer genuinely cost $0 (free/stub model).
+2. The reviewer cost real money but matched zero planted/defect categories.
+
+Because the field has no absent/N/A state, these cases are byte-identical on the public leaderboard, making a paid-but-ineffective reviewer appear as best-in-class efficient.
+
+## Pain Points
+
+### Leaderboard Misreading
+- `cost_per_corroborated_finding_usd == 0.0` is ambiguous: it could mean "free" or "expensive but useless".
+- Readers who look only at cost efficiency cannot distinguish the two without cross-referencing `corroboration_rate` and `findings_raised_avg`.
+
+### Schema Freeze Conflict
+- Epic 10.2 froze `scorecard.PublicRecord` for byte-compatibility with production.
+- Any change to the field type (pointer, omitempty, or companion flag) is a schema change that must be applied to both production and benchmark paths together, not patched locally into the benchmark scorer.
+
+## Evidence
+
+### TD item driving this plan:
+- internal/benchmark/score.go:115-117 — "scoreOne sets CostPerCorroboratedFindingUSD only when matchedFindings>0. A reviewer with real CostUSD>0 but zero matched planted categories reports 0 — byte-identical to a genuinely free reviewer."
+
+### Existing parity precedent:
+- `internal/scorecard/export.go:272-274` — `costPer(totalCost, corroborated)` returns `0` when `corroborated <= 0`.
+- `internal/benchmark/score.go:115-117` — the benchmark scorer uses `if matchedFindings > 0 { ... }`, producing the same `0.0`.
+- `internal/scorecard/export_test.go:121` already asserts "no corroborated findings => 0, never Inf/NaN", confirming `0.0` is the intended sentinel today.
+- `internal/scorecard/export.go:39` — `SurvivedSkepticRate *float64` with `omitempty` already uses the identical pattern (absent key = no data, present key = real value including 0.0) for a different field on the same struct.
+
+## Proposed Solution
+
+**Decided:** change `CostPerCorroboratedFindingUSD` to `*float64` with `omitempty` on `scorecard.PublicRecord`. A nil pointer omits the JSON key (absent = "no corroborated findings, cost-per is undefined"); a non-nil pointer carries the real value, including a genuine `0.0` should that ever occur. This mirrors `SurvivedSkepticRate *float64` (`internal/scorecard/export.go:39`), which already uses this exact pattern on the same struct to distinguish "no verification ran" (key absent) from "verification ran, rate is 0" (key present, value 0.0) — the identical shape of ambiguity this epic addresses for cost-per-corroborated.
+
+Options considered:
+
+1. **Chosen.** Change `CostPerCorroboratedFindingUSD` to `*float64` with `omitempty`. Consistent with the `SurvivedSkepticRate` precedent already on this struct; the field is genuinely optional data, not always-present.
+2. Add a companion enum/boolean field (e.g., `cost_per_corroborated_finding_usd_available`) to disambiguate. Rejected: introduces a second convention for the same "value is optional" problem `SurvivedSkepticRate` already solves with a pointer — unnecessary duplication of an idiom the schema already has.
+3. Keep `0.0` in the raw JSON but update the leaderboard UI/rendering layer to display "N/A" when `corroboration_rate == 0 && findings_raised_avg > 0`. Rejected: no in-repo renderer consumes the exported `PublicRecord` JSON for human display (see Files Affected) — this option has no addressable implementation target in this codebase.
+
+## Scope
+
+### In Scope
+- Change `scorecard.PublicRecord.CostPerCorroboratedFindingUSD` to `*float64` with `omitempty` in `internal/scorecard/export.go` (Option 1, see Proposed Solution — decided, not open).
+- Update production export path (`internal/scorecard/export.go`) to emit the new representation.
+- Update benchmark scorer (`internal/benchmark/score.go`) to match production exactly.
+- Update field-reference documentation (`docs/scorecard.md:242`, `docs/benchmark.md:128`) to describe the new absent/N/A representation.
+- Update tests in both scorecard and benchmark packages.
+
+### Out of Scope
+- Changing benchmark scorer behavior before the schema decision is made.
+- Altering other PublicRecord fields.
+- New leaderboard features beyond the cost-per ambiguity fix.

-
-- **Estimated time**: 1 week
-- **Tasks/Components**: 4/2
-
-## Current State
-
-`scorecard.PublicRecord.CostPerCorroboratedFindingUSD` is a plain non-pointer `float64` with no `omitempty` tag (internal/scorecard/export.go:40). Both the production leaderboard `--export` path and the benchmark scorer therefore emit `0.0` in two very different situations:
-
-1. The reviewer genuinely cost $0 (free/stub model).
-2. The reviewer cost real money but matched zero planted/defect categories.
-
-Because the field has no absent/N/A state, these cases are byte-identical on the public leaderboard, making a paid-but-ineffective reviewer appear as best-in-class efficient.
-
-## Pain Points
-
-### Leaderboard Misreading
-- `cost_per_corroborated_finding_usd == 0.0` is ambiguous: it could mean "free" or "expensive but useless".
-- Readers who look only at cost efficiency cannot distinguish the two without cross-referencing `corroboration_rate` and `findings_raised_avg`.
-
-### Schema Freeze Conflict
-- Epic 10.2 froze `scorecard.PublicRecord` for byte-compatibility with production.
-- Any change to the field type (pointer, omitempty, or companion flag) is a schema change that must be applied to both production and benchmark paths together, not patched locally into the benchmark scorer.
-
-## Evidence
-
-### TD item driving this plan:
-- internal/benchmark/score.go:109 — "scoreOne sets CostPerCorroboratedFindingUSD only when matchedFindings>0. A reviewer with real CostUSD>0 but zero matched planted categories reports 0 — byte-identical to a genuinely free reviewer."

-- `internal/benchmark/score.go:118-120` — the benchmark scorer uses `if matchedFindings > 0 { ... }`, producing the same `0.0`.
-- `internal/scorecard/export_test.go:121` already asserts "no corroborated findings => 0, never Inf/NaN", confirming `0.0` is the intended sentinel today.
