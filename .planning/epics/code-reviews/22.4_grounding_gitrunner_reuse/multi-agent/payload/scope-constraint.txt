# Epic Plan 22.4: Reuse Payload Builder's Diff in Grounding Data

- **Estimated time**: 0.5 day
- **Tasks/Components**: 1/1

## Objective

Stop `computeGroundingData` from re-running a fresh `git` diff that the payload builder already computed for the same range, eliminating ~4 duplicate git subprocesses per review and per resume.

## Context

`internal/fanout/review.go:470` builds a fresh `gitRunner` and re-runs `validateRange` + `--name-status` + `--unified=0` for a `base..head` range the payload builder already diffed moments earlier in the same run. This is a real, measurable per-run cost (git subprocess spawns), not a by-design tradeoff, and had no citation of any kind in the TD README.

## Task Breakdown

1. Reuse the payload builder's memoized `rangeChunks`/`gitRunner` for the same `base..head` range in `computeGroundingData` instead of recomputing the zero-context diff from scratch.

## Acceptance Criteria

- [ ] `computeGroundingData` no longer spawns its own `git` subprocesses when a memoized diff for the same range is already available from the payload builder.
- [ ] Grounding behavior (which findings get dropped/exempted) is unchanged — this is a pure performance fix, verified by existing `internal/fanout` grounding tests passing unchanged.
- [ ] A benchmark or subprocess-count assertion confirms the reduction (e.g. asserting `git` is invoked once per range, not twice).
