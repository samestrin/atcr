# Epic 4.1: Graceful Shutdown and Signal Handling

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** High (data integrity, user experience)

## Problem

ATCR has no signal handling for SIGTERM or SIGINT. When a user presses Ctrl-C during a review, the process dies immediately. This creates three concrete problems:

1. **Lost work.** A review that has run for 5 minutes with 8/10 agents completed produces no output. The partial results are discarded.

2. **Inconsistent state.** Review artifacts (directories, `status.json`, partial agent outputs) may be left on disk in an incomplete state. Subsequent runs or `atcr status` may misinterpret the state.

3. **No cleanup.** Goroutines running agent invocations are abandoned mid-flight. HTTP connections to providers are not closed cleanly. File handles may not be flushed.

## Root Cause

There is no `signal.Notify` in the codebase. The CLI does not intercept SIGINT/SIGTERM, so the default Go behavior (immediate termination) applies. The fanout engine has no mechanism to cancel in-flight agent invocations cooperatively.

## Solution

Intercept SIGINT/SIGTERM in `cmd/atcr/main.go`, cancel the root context, and allow the fanout engine to shut down gracefully. Save partial results before exiting.

### Core Design

```go
// cmd/atcr/main.go
func main() {
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    // Intercept SIGINT/SIGTERM
    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
    go func() {
        <-sigCh
        fmt.Fprintln(os.Stderr, "\nReceived interrupt, shutting down gracefully...")
        cancel()
        // Wait a bit for graceful shutdown, then force exit
        <-time.After(10 * time.Second)
        fmt.Fprintln(os.Stderr, "Graceful shutdown timed out, forcing exit")
        os.Exit(1)
    }()

    root := newRootCmd()
    if err := root.ExecuteContext(ctx); err != nil {
        // ... existing exit code logic
    }
}
```

**Fanout engine cooperation:**
- `internal/fanout/engine.go` already checks `ctx.Err()` before each agent invocation.
- When the context is cancelled, in-flight agents continue to completion (or timeout), but no new agents start.
- Partial results are returned and written to disk.

**Partial result saving:**
- `cmd/atcr/review.go` checks if `result != nil` after `ExecuteReview` returns an error.
- If partial results exist, they are written to `status.json` with `status: "interrupted"`.
- A warning is printed: "Review interrupted. Partial results saved to <dir>. Resume with `atcr review --resume <id>`."

### Resume Support (Out of Scope, Future)

A future epic could add `--resume <id>` to continue an interrupted review from where it left off. This epic focuses on clean shutdown and partial result preservation, not resumption.

## Scope

### In Scope

- Signal interception in `cmd/atcr/main.go`.
- Root context cancellation on SIGINT/SIGTERM.
- Graceful shutdown message to stderr.
- Partial result preservation in `status.json` with `status: "interrupted"`.
- Fanout engine already cooperates (context checks in place); no changes needed.
- Unit tests for signal handling (mock signal, verify context cancellation).
- Integration test: start a review, send SIGINT, verify partial results saved.

### Out of Scope

- Resume support (`--resume <id>`) — future epic.
- Timeout tuning (10-second grace period is hardcoded; can be made configurable later).
- Cleanup of orphaned review directories (separate concern).
- Handling SIGHUP or other signals.

## Open Questions

1. **Should the grace period be configurable?**
   - Option A: hardcoded 10 seconds (simple).
   - Option B: `--shutdown-timeout` flag (flexible).
   - Recommendation: Option A for now; make configurable if users request it.

2. **Should SIGTERM behave differently from SIGINT?**
   - Option A: same behavior (both trigger graceful shutdown).
   - Option B: SIGTERM is "softer" (allow more time), SIGINT is "harder" (faster exit).
   - Recommendation: Option A; both mean "shut down" in practice.

3. **Should we print a summary of completed agents before exiting?**
   - Option A: yes ("3/10 agents completed, results saved to ...").
   - Option B: no (just the directory path).
   - Recommendation: Option A; users want to know what was saved.

## Clarifications

### Recorded 2026-06-17 (rubber duck — /execute-epic)

**Key Decisions:**
- The "interrupted" marker is recorded via **Option A**: add `Interrupted bool` to
  `payload.Manifest` and derive a new `RunInterrupted` ("interrupted") state in
  `fanout.ReadReviewStatus`, mirroring the existing `Partial bool` precedent. This keeps
  `atcr status` and the MCP `atcr_status` handler as the single source of truth.
- **Q1/Q3 tension resolved in favor of Option A (over the epic's "no fanout changes
  needed" line).** Rationale: the epic's own Problem #2 names `atcr status` misinterpreting
  interrupted state as a defect to fix. Status is *derived* at read-time, not stored, so a
  CLI-only sentinel (Option B) would leave `atcr status`/MCP reporting "completed (partial)"
  or "stale" — failing the stated requirement. The Solution's scope note was an inaccurate
  authoring assumption; the Problem statement (the actual requirement) governs.
- Interruption is detected via `errors.Is(ctx.Err(), context.Canceled)` on the **parent**
  context (the review's own timeout lives on a child `runCtx`), set inside `ExecuteReview`
  where the manifest is finalized, and re-checked in `cmd/atcr/review.go` for the warning.
- The interrupt warning prints "N/M agents completed; partial results saved to <dir>. Run
  'atcr status <id>' to inspect." — it does NOT reference a `--resume` flag (out of scope).
- SIGINT handling is applied **uniformly** to all subcommands including `atcr serve` (its
  5s drain fits inside the 10s force-exit window).
- 10-second grace then force-exit with **exit code 1** (hardcoded, no flag). SIGTERM ==
  SIGINT. On graceful interrupt the CLI also exits 1 (consistent with the force path) and
  skips downstream reconcile/verify. `result` is nil-guarded before printing counts,
  falling back to `prep.Dir`/`prep.ID`.

**Scope Boundaries:**
- IN scope: signal handling + root context cancel (`cmd/atcr/main.go`); `Interrupted`
  manifest field (`internal/payload/manifest.go`); set-on-cancel in `internal/fanout/review.go`
  (`ExecuteReview`); `RunInterrupted` derive in `internal/fanout/status.go`; interrupt warning
  + early-return in `cmd/atcr/review.go`; unit + integration tests.
- OUT of scope: `--resume`, configurable shutdown timeout, orphan-dir cleanup, SIGHUP.

**Technical Approach:**
- The fanout engine already cooperates (drains the WaitGroup, short-circuits unstarted
  slots to StatusTimeout) — no `engine.go` change. Completed agents' artifacts are already
  persisted by `WritePool`; the new work is the marker + warning, not result re-plumbing.
- `main.go` extracts a testable seam (`installSignalHandler` with injected signal channel,
  `forceExit`/timeout package vars) so AC8 (signal→ctx cancel) and AC7 (force-exit code 1)
  are unit-testable without `os.Exit` killing the test.
