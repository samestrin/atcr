# Epic 7.0: Executor Model for Fix Generation

**Status:** Draft
**Created:** 2026-06-14
**Priority:** High (differentiator, cost optimization)
**Estimated time:** 2 weeks
**Tasks/Components:** 4/1

## Problem

The Finding Remediation concept (auto-fix findings) is valuable but has a cost/quality problem:

**Current approach:**
- All N models in the panel suggest fixes
- N× cost for fix generation (if you have 3 models, you pay for 3 fix generation passes)
- Inconsistent fixes (different models suggest different solutions, different styles)
- Harder to validate (which fix is correct?)

**The lgtmaybe approach:**
- Single model generates inline comments with fixes
- Cheaper (1× cost)
- Consistent (one model, one style)
- But: single-model review (misses bugs, as lgtmaybe's creator admits)

**The opportunity:**
Combine ATCR's multi-model review (catches more bugs) with a single executor model for fix generation (cheaper, more consistent, can use a more capable model).

## Solution

Add an **executor model** to the registry that's used exclusively for fix generation:

```yaml
# registry.yaml
agents:
  - name: bruce
    provider: anthropic
    model: claude-sonnet-4-6
    persona: correctness
    role: reviewer  # This model reviews (part of the panel)
  
  - name: greta
    provider: openai
    model: gpt-4-turbo
    persona: design
    role: reviewer  # This model reviews (part of the panel)
  
  - name: kai
    provider: google
    model: gemini-pro
    persona: production
    role: reviewer  # This model reviews (part of the panel)

executor:
  name: opus
  provider: anthropic
  model: claude-opus-4-6
  persona: fixer
  role: executor  # This model generates fixes (not part of the review panel)
```

**How it works:**
1. **Review phase:** Panel of N reviewer models identifies findings (consensus, confidence, adversarial verification)
2. **Fix phase:** Single executor model generates fixes for verified findings (HIGH/CRITICAL only, or all findings above a confidence threshold)
3. **Output:** Findings include the fix generated by the executor model

**Example output:**
```
# atcr-findings/v1
HIGH|internal/auth/login.go:42|SQL injection vulnerability|Use parameterized query: `db.Query("SELECT * FROM users WHERE id = ?", userID)`|security|30|Found by bruce, greta, kai; confidence HIGH; verified by skeptic|fix: Replace line 42 with parameterized query...|executor:opus
```

## Key Design Decisions

### 1. Executor is optional
- If no executor is configured, ATCR works as before (no fix generation)
- Existing registries don't break
- Users can opt-in to fix generation

### 2. Executor only runs on verified findings
- Don't waste tokens generating fixes for LOW/MEDIUM findings
- Only generate fixes for findings that:
  - Have confidence HIGH (2+ models agreed)
  - Survived adversarial verification (post-Epic 3.0)
  - Are above a configurable severity threshold (e.g., MEDIUM+)

### 3. Executor can be a different (better) model
- Review panel: use cheaper models (Sonnet, GPT-4-turbo, Gemini)
- Executor: use more capable model (Opus, Fable) for fix generation
- Cost optimization: spend more on fix quality, less on review breadth

### 4. Executor has a different persona
- Review personas: correctness, design, production, security, etc.
- Executor persona: "fixer" — focused on generating minimal, correct fixes
- Prompt template: "Given this finding, generate the minimal code change that fixes it. Preserve existing style and conventions."

### 5. Fix generation is a separate phase
- Review phase: fan-out to N models, reconcile, verify
- Fix phase: single call to executor model with verified findings
- Clear separation of concerns
- Easier to test and debug

### Review Pipeline Changes

Current pipeline:
1. Parse payload
2. Fan-out to N reviewer models
3. Reconcile findings
4. Adversarial verification (post-3.0)
5. Emit findings

New pipeline:
1. Parse payload
2. Fan-out to N reviewer models
3. Reconcile findings
4. Adversarial verification (post-3.0)
5. **Generate fixes (if executor configured)** — NEW
6. Emit findings (with fixes)

### Fix Generation Logic

```go
func generateFixes(findings []Finding, executor Executor, payload Payload) []Finding {
    // Filter to verified findings above threshold
    var toFix []Finding
    for _, f := range findings {
        if f.Confidence == "HIGH" && f.Severity >= config.MinSeverityForFix {
            toFix = append(toFix, f)
        }
    }
    
    // Batch fix generation (one call per finding, or batch multiple findings)
    for i := range toFix {
        fix, err := callExecutor(executor, toFix[i], payload)
        if err != nil {
            toFix[i].FixWarning = "fix generation failed: " + err.Error()
        } else {
            toFix[i].Fix = fix
            toFix[i].Executor = executor.Name
        }
    }
    
    return findings  // Updated with fixes
}
```

### Output Format Changes

Ensure compatibility with the standard 9-column format defined in Epic 12.0:

```
SEVERITY|FILE:LINE|PROBLEM|FIX|CATEGORY|EST_MINUTES|EVIDENCE|REVIEWERS|CONFIDENCE
```

Where the fix is directly populated into the `FIX` column (column 4). The `EVIDENCE` column is extended to include the executor metadata:
- "Found by bruce, greta; confidence HIGH; verified by skeptic; fix by opus"

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Executor model generates bad fixes | Medium | High | Use a capable model (Opus/Fable); validate fixes compile/pass tests (post-Epic 11.0) |
| Fix generation is slow | Low | Medium | Batch fix generation (one call with multiple findings); use fast model |

## Open Questions

1. **Should the executor be a separate model, or can it be one of the reviewer models?**
   - Option A: Separate model (cleaner separation, can use different model)
   - Option B: One of the reviewers (cheaper, but less flexible)
   - Recommendation: Option A (separate model)

2. **Should fix generation be batched (one call with multiple findings) or per-finding?**
   - Option A: Per-finding (simpler, easier to debug)
   - Option B: Batched (fewer API calls, faster)
   - Recommendation: Option A first, then optimize to B if needed

3. **Should we validate fixes (compile, test) before emitting?**
   - Option A: No validation (fast, but risky)
   - Option B: Validate with executing reviewers (post-Epic 11.0)
   - Recommendation: Option A first, then B when 11.0 lands

4. **What's the default severity threshold for fix generation?**
   - Option A: HIGH only (conservative, fewest fixes)
   - Option B: MEDIUM+ (more fixes, more value)
   - Option C: All findings (most fixes, but may include noise)
   - Recommendation: Option B (MEDIUM+)

## Clarifications

### Recorded 2026-06-21 (rubber duck — /execute-epic)

**Key Decisions:**
- Fix generation folds into the existing shared `internal/verify` pipeline (`runVerify`) as a phase that runs after verdict application / confidence recompute and before the atomic artifact flush (pipeline.go:341-362), so fixes propagate into `findings.json`/`findings.txt` with no new commands, artifact paths, or entry points. It is exercised by `atcr verify`, `atcr review --verify`, and the `atcr_verify` MCP tool alike.
- The executor receives real source-code context via **pre-loaded snippet** (Epic 7.0 tier): after verdict application, read ±30 lines around `finding.File:finding.Line` from the already-open review snapshot and embed it in the executor prompt. One LLM call per finding, no tool loop, no second snapshot checkout. (Optional dispatcher-driven "agent mode" is deferred to Epic 7.4.)
- Fix targeting uses HIGH-or-better confidence: `reconcile.ConfidenceAtOrAbove(f.Confidence, reconcile.ConfHigh)` (so VERIFIED — the top tier promoted by the verify stage — is included), AND severity at or above `min_severity_for_fix` (default MEDIUM, epic Open-Q4 Option B), using the same AtOrAbove gate pattern as severity gates.
- Output stays the standard 9-column format: the generated fix populates column 4 (FIX); executor attribution ("fix by <name>") is appended to the EVIDENCE column. No 10th column is added — the authoritative format spec (epic lines 169-176) overrides the loose illustrative example on line 69.
- Executor calls are per-finding (Open-Q2 Option A) with no fix validation/compile/test (Open-Q3 Option A, deferred to Epic 11.0). A failed executor call sets a non-fatal fix warning on that finding and never fails the run (mirrors verify's failure-isolation contract).

**Scope Boundaries:**
- IN scope: optional top-level `executor:` registry section + validation (Role == "executor", Provider must reference a defined provider); fix-generation phase in the verify pipeline; FIX/EVIDENCE population; config options (`min_severity_for_fix`, `fix_timeout`, `batch_fixes`); docs + example registries (with and without executor).
- OUT of scope: PR inline-comment posting / GitHub Action (Epic 7.3); fix validation via compile/test (Epic 11.0); dispatcher-driven executor "agent mode" (Epic 7.4).

**Technical Approach:**
- Registry gets `Executor *ExecutorConfig` (Name, Provider, Model, Persona, Role); validation mirrors existing reviewer/skeptic role checks.
- Executor LLM calls go through the existing `llmclient` single-shot `Complete` path, injected via a package-level seam (not a new `runVerify` parameter) to avoid churning the ~29 existing `runVerify` test call sites.
- The snapshot the snippet is read from is the same one the verify stage opens for skeptics; snapshot need is extended to also fire when an executor is configured and at least one fixable finding exists. On snapshot unavailability, fix generation degrades gracefully (warning, finding still emitted) rather than failing the run.
- Tests use an injected fake completer (no network); a live end-to-end executor call is not exercised in CI.
