# Epic 5.0: File Path Validation and Correction

**Status:** Draft
**Created:** 2026-06-14
**Priority:** High (correctness issue)

## Problem

When reviewing code, models sometimes hallucinate file paths — they reference files that don't exist in the repository. The current parser (`internal/stream/parser.go`) extracts file paths from findings but **never validates that the files actually exist**.

This causes two problems:

1. **Lost findings**: A good TD item with a hallucinated path (e.g., `internal/auth/validator.go` when the actual file is `internal/auth/validate.go`) becomes useless because you can't navigate to the file to fix it.

2. **Silent failure**: The finding is parsed successfully, written to `findings.txt`, and appears in reports — but the file path is wrong. Users only discover this when they try to open the file and it doesn't exist.

This is especially problematic in the legacy code review system where TD items are reconciled from multiple sources. If a path is wrong, the TD item is effectively lost.

## Root Cause

The `splitFileLine` function in `internal/stream/parser.go` (lines 214-228) extracts the file path from the FILE:LINE column but performs no validation:

```go
func splitFileLine(s string) (string, int) {
    idx := strings.LastIndex(s, ":")
    if idx < 0 {
        return s, 0
    }
    suffix := s[idx+1:]
    if suffix == "" {
        return s[:idx], 0
    }
    n, err := strconv.Atoi(suffix)
    if err != nil {
        return s, 0
    }
    return s[:idx], n
}
```

The function splits on the last colon, parses the line number, and returns the file path — but never checks if the file exists.

## Solution

Add a **file path validation step** to the reconcile pipeline that:

1. **Validates file existence** after parsing findings
2. **Flags findings with non-existent paths** (adds a warning or metadata field)
3. **Optionally attempts path correction** (fuzzy matching, similar file names)
4. **Preserves the finding** even if the path is wrong (don't discard it)

### Validation Strategies

#### 1. File Existence Check
After parsing, check if the file exists:
```go
if _, err := os.Stat(finding.File); os.IsNotExist(err) {
    finding.PathValid = false
    finding.PathWarning = "file not found"
}
```

#### 2. Fuzzy Matching (Optional)
If the file doesn't exist, try to find a similar file:
- Same directory, similar name (e.g., `validator.go` → `validate.go`)
- Same name, different directory (e.g., `internal/auth/validator.go` → `pkg/auth/validator.go`)
- Use Levenshtein distance or similar algorithm

If a match is found with high confidence, correct the path and note the correction.

#### 3. Metadata Tracking
Add fields to the `Finding` struct:
```go
type Finding struct {
    // ... existing fields ...
    PathValid    bool   // true if file exists, false otherwise
    PathWarning  string // warning message if path is invalid
    PathCorrected string // original path if corrected, empty otherwise
}
```

### Integration Points

The validation should run:
1. **After parsing** (in `ParseSource` and `ParseModelOutput`)
2. **Before reconciliation** (so invalid paths are flagged early)
3. **In the report generator** (so users see warnings)

## Clarifications

### Recorded 2026-06-20 (pre-execution design resolution — /execute-epic --strong)

**Base directory for file-existence validation (resolves the plan's core architectural gap):**
- The plan proposed `os.Stat(finding.File)` inside the parser (`splitFileLine`). The parser is pure and has no repo root, so validation **cannot** live there. The Root Cause framing in this plan is therefore the *symptom site*, not the *fix site*.
- The target repo root is the **process CWD**. `atcr` runs from inside the repo under review — both CLI and MCP hardcode `Repo: "."` / `Root: "."` (`cmd/atcr/review.go:165`, `internal/mcp/handlers.go:205`). Finding file paths (e.g. `src/auth.go`) are CWD-relative.
- Precedent to mirror: the `verify` stage already threads a `repoRoot` (always `"."`) and stats files against it (`cmd/atcr/verify.go:61`, `internal/verify/pipeline.go:91`).

**Validation design:**
- Add `PathValid bool` and `PathWarning string` to `stream.Finding`.
- Add a `ValidatePaths(findings []Finding, root string)` helper in `internal/stream` that stats `filepath.Join(root, finding.File)`; `root` defaults to `"."` so unit tests inject a temp dir.
- Run validation in the **reconcile pipeline** (where CWD == repo root), not in the parser.
- Avoid the `PathValid` zero-value trap: an unvalidated finding must not read as "invalid." Validity is meaningful only after `ValidatePaths` runs (warning-only signal / explicit gate), so re-parsed reconciled rows are never falsely flagged.

**Report surfacing (AC3):**
- Findings reach reports as `reconcile.Merged` (rendered by `internal/reconcile/emit.go`) and `report.JSONFinding` (rendered by `internal/report/render.go`) — NOT `stream.Finding`. Path validity must thread through both so the warning appears in `report.md` and `atcr report`.

**Scope decisions:**
- IN scope: AC1–AC6 (existence validation; `PathValid`/`PathWarning` flagging; report warnings in both renderers; findings preserved, never discarded; unit tests for valid/invalid/typo/wrong-dir; integration test with a hallucinated path).
- OUT of scope (deferred to technical debt): AC7 fuzzy matching / path correction — explicitly optional; the plan's own Open Questions recommend the safe default (Option B, correction behind a flag). Line-number validation and content validation remain out of scope per the plan.
