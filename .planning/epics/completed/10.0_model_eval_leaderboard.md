## Objective

Treat ATCR as a model-evaluation engine and expose the accumulated signal as a leaderboard — first local (`atcr leaderboard`, already defined in Epic 3.3), then as an opt-in public submission format and a static leaderboard site. The asset sold is a trusted measurement that falls out of the architecture already built.

## Context

There is no trusted, reproducible answer to a question every team building with LLMs now asks: **which model is actually good at code review, and at what cost?** Generic leaderboards (lmarena, MMLU, SWE-bench) measure task-solving, not review quality. Vendor benchmarks are self-reported. Teams pick a model because it is the default in their tool.

Meanwhile ATCR is already generating the exact data that answers this and throwing it away after each run. Epic 3.3 fixes that by persisting per-reviewer scorecard records and exposing a local `atcr leaderboard` CLI. This epic builds on that foundation: it defines a versioned public submission format, aggregates opt-in submissions, and publishes a static leaderboard site.

The leaderboard's credibility compounds automatically with the roadmap:
- **Stage 1 (today):** corroboration rate — caught-by-2+ vs. solo.
- **Stage 3.0 (adversarial verification):** survived-skeptic rate — a far stronger signal. "This model's findings survive hostile refutation 80% of the time."
- **Stage 11.0 (executing reviewers):** demonstrated rate — findings backed by a failing repro command. Near ground truth.

## Problem Statement

1. The local `atcr leaderboard` (Epic 3.3) answers "which model works best for my codebase" but is siloed per user.
2. There is no shared, reproducible benchmark for code-review model quality. Teams make model selection decisions on vibes.
3. ATCR's reconciler is deterministic and OSS — it is the only tool capable of producing a reproducible, methodology-published leaderboard for this task. Nobody else has built this.

## Proposed Solution

### Phase 1 — Public submission format (1 week)

Extend Epic 3.3's `atcr leaderboard --export` to produce a versioned, anonymized submission record:

```json
{
  "submission_schema": 1,
  "atcr_version": "0.4.1",
  "submitted_at": "2026-06-14T12:00:00Z",
  "reviewers": [
    {
      "model": "claude-sonnet-4-6",
      "persona": "bruce",
      "runs": 12,
      "findings_raised_avg": 10.2,
      "corroboration_rate": 0.61,
      "survived_skeptic_rate": 0.74,
      "cost_per_corroborated_finding_usd": 0.006,
      "latency_p50_ms": 8900
    }
  ]
}
```

Anonymization pass strips: repo identifiers, finding text, provider keys, absolute paths. Only aggregate statistics and model/persona identifiers are included. (Leverages the Strict Redaction Engine from Epic 4.9).

### Phase 2 — Standard fixed-diff suite (1 week)

A curated set of diffs with planted defects (known bugs) that any user can run to produce comparable scores:

```bash
atcr benchmark run --suite standard-v1
atcr benchmark export   # produces a submission record for the public board
```

The suite lives in `github.com/atcr/benchmark-suite` (versioned). Only benchmark-suite runs appear on the public leaderboard — preventing cherry-picked production runs from gaming the board.

### Phase 3 — Public leaderboard site (3 weeks)

- Static site (`github.com/atcr/leaderboard`) generated from aggregated opt-in submissions.
- Methodology published alongside data: the reconciler is OSS and deterministic; anyone can reproduce any row.
- Columns: model, persona, corroboration rate, survived-skeptic rate (when available), cost-per-finding, latency, run count, last updated.
- Rows auto-expire after 90 days without a fresh submission (model churn makes stale rows misleading).
- Sponsorship path: a vendor can pay to have a new model run through the standard suite by the ATCR maintainers and added to the board.

## Acceptance Criteria

- [x] `atcr leaderboard --export` produces a schema-v1 anonymized submission JSON (no PII, no repo content, no provider keys).
- [ ] `atcr benchmark run --suite standard-v1` runs the fixed-diff suite and produces a submission record; the suite diffs are versioned and reproducible.
- [x] `atcr benchmark export` produces a submission record referencing the suite version; record is distinct from the production `--export` format (suite runs only on the public board).
- [ ] Survived-skeptic rate and demonstrated rate appear in export when verification.json / execution evidence are present (populated by 3.0 and 11.0 respectively).
- [ ] Public leaderboard site renders the aggregate table; rows expire after 90 days without a fresh submission.
- [ ] Methodology page published alongside the leaderboard; references the OSS reconciler and standard suite.
- [x] Docs: submission guide, benchmark suite README, privacy model (what is and isn't included in a submission).

## Task Breakdown

1. **Submission format v1**: extend 3.3's `--export` with suite-version tagging and the anonymization pass.
2. **`atcr benchmark` command**: `run` (execute the fixed-diff suite), `export` (produce submission record), `verify` (confirm suite run is reproducible).
3. **Standard benchmark suite v1**: curated diffs with planted defects, expected finding categories, scoring rubric.
4. **Public leaderboard site**: static site generator, submission ingestion pipeline, 90-day expiry, methodology page.
5. **Survived-skeptic / demonstrated columns**: wire in from 3.0 / 11.0 data when available.
6. **Docs + sponsorship path**: submission guide, benchmark README, vendor sponsorship process.

## Out of Scope

- Real-time leaderboard (static, regenerated on submission).
- Team-private leaderboards (Epic 13.0 Team Edition).
- Auto-running the suite on every commit (CI integration — separate).
- Per-language or per-domain sub-leaderboards (v2).

## Dependencies

- Epic 3.3 (per-run scorecard and local `atcr leaderboard`) — the local half is a prerequisite; the public half extends it.
- Epics 3.0 and 11.0 (optional enrichment): the leaderboard works without them and gains richer columns as they land.

## Revenue Model

- **OSS / free**: local scorecard, `atcr leaderboard`, `atcr benchmark`, submission format. These are distribution, not revenue.
- **Attention funnel**: the public leaderboard is top-of-funnel (the lmarena-of-code-review). Drives ATCR adoption.
- **Sponsored evals**: a model vendor pays to have a new model run through the standard suite by the maintainers.
- **Private team leaderboards** (Epic 13.0): paid feature for teams that want regression tracking across model upgrades.

## Moat / Differentiation

- **The reconciler is the moat.** Anyone can call N models; almost nobody has a deterministic, reproducible way to score whose findings were right.
- **Compounds with the roadmap automatically** — survived-skeptic (3.0) and demonstrated (11.0) raise leaderboard credibility without additional leaderboard engineering.
- **Standards gravity** — if "the ATCR benchmark" becomes the cited number for code-review model quality, ATCR is the reference implementation.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Corroboration is a biased proxy at Stage 1 | High | Medium | Label it explicitly as a proxy; lead with survived-skeptic as the headline metric once 3.0 is live |
| Public leaderboard gets gamed with cherry-picked runs | Medium | High | Suite-only submissions on the public board; production `--export` stays local-only |
| Vendor model churn makes rows stale fast | High | Low | 90-day auto-expiry; date-stamp every row; automated suite re-runs |
| Correlated reviewers inflate agreement scores | Medium | Medium | Track persona×model independence; down-weight near-duplicate reviewers in the metric |
