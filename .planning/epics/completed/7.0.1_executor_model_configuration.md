# Epic 7.0.1: Executor Model Configuration

- **Estimated time**: 3 days
- **Tasks**: 4
- **Components**: 1

## Context

Sprint 7.0 introduced the **Executor Model** for generating inline code fixes. While the baseline functionality is powerful, it currently relies on a static, hardcoded prompt and default LLM API parameters (like temperature). 

## Problem Statement

A major pain point in automated remediation is style mismatch. If a project uses tabs but the executor model defaults to spaces, the generated fix will fail linting. If the team enforces specific architectural rules (e.g., "never use `lodash`"), the model might introduce a violation. Furthermore, using default temperatures (e.g., 0.7) can lead to unpredictable or non-deterministic code fixes.

## Proposed Solution

Extend the `executor` configuration schema (introduced in 7.0) to allow developers to tightly control the fix generation behavior:

1. **Temperature Control:** Allow configuring the API temperature (defaulting to `0.0` or a very low value to ensure deterministic, highly predictable code fixes).
2. **System Prompt Override:** Allow users to define a custom system prompt for the fixer persona.
3. **Coding Rules / Style Guidelines:** Allow passing an array of strings (e.g., `rules: ["Use tabs for indentation", "Avoid panic() in Go"]`) that are concatenated into the executor's context.

**Key Decisions:**
- Temperature handling mirrors the existing `AgentConfig.Temperature` pattern: `*float64`, validated to the range `0.0..2.0`, passed through unchanged. No per-provider clamping is added â€” the codebase routes every provider through a single OpenAI-compatible `/chat/completions` endpoint and has no provider-type discriminator; the Risk-table normalization is handled at the LiteLLM/gateway layer before the request leaves the system.
- `system_prompt`, when set, **fully replaces** the default framing line (`"You are {persona}, a code-fix executor..."`) in `buildFixPrompt`. `persona` is superseded for that call. Finding metadata, source snippet, and `rules` are still appended after the custom framing.
- The executor's **effective** temperature defaults to `0.0` (deterministic) when omitted. This is an intentional behavior change: today `callExecutor` sends no temperature at all, letting the provider default apply.

**Scope Boundaries:**
- IN scope: `ExecutorConfig` schema fields (`temperature`, `system_prompt`, `rules`) + validation + defaults; temperature plumbing through `llmclient.Invocation`; prompt assembly (`system_prompt` override + `rules` block); unit tests; example YAML + doc note.
- OUT of scope: reading rules from external files (`.eslintrc`, etc.); per-provider temperature normalization in ATCR code.

**Technical Approach:**
- New fields follow the `AgentConfig` precedent (pointer for temperature so unset is distinguishable from 0.0; control-char guards + length caps for `system_prompt` and each `rules` entry, mirroring the existing `persona` guards).
- `temperature` plumbs through the existing `Invocation.Temperature` field (already marshaled `omitempty`); only `callExecutor` needs to pass the executor's effective temperature.
- An `EffectiveExecutorTemperature` helper resolves the `0.0` default when the field is nil, mirroring the existing `EffectiveExecutorTimeoutSecs` / `EffectiveFixMinSeverity` helpers.
