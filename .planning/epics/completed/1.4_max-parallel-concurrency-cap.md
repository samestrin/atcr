# Epic Plan 1.4: Parallel-Lane Concurrency Cap (max_parallel)

**Estimated Durations**: 1 day

## Objective

Bound the fan-out engine's parallel lane with a buffered semaphore sized from a new `max_parallel` setting resolved through the existing `CLI > project > registry > embedded` precedence chain — so a large roster cannot spawn an unbounded goroutine-per-slot burst against providers.

## Context

Epic 1.0 shipped the fan-out engine with one goroutine per non-serial slot (`internal/fanout/engine.go` `Run`), relying on real rosters being ≤~10 agents — comfortably within AC 01-04's "10 concurrent agent calls" target. The sprint's adversarial review evaluated a concurrency cap and recorded it as accepted risk TD-017: "a concurrency cap adds config surface not requested in v1; fix in v2 with a buffered semaphore sized from a new max_parallel setting." The binding Phase 3 clarification rejected `golang.org/x/time/rate` on the same keep-config-surface-small grounds. This epic is that recorded v2 work.

## Problem Statement

Every non-serial slot spawns its own goroutine with no semaphore or worker-pool bound. A future large roster (or a misconfigured one) fires all provider calls at once: provider-side rate limiting, local resource pressure, and burst cost are all uncontrolled, and there is no knob to tune it per project or per machine.

## Proposed Solution

1. **New setting `max_parallel`**: integer, resolved through the existing settings precedence chain (`CLI flag > .atcr/config.yaml > ~/.config/atcr/registry.yaml > embedded default`). Embedded default chosen to preserve current behavior for v1 rosters (≥10, matching the AC's concurrent-calls target; `0` = unbounded is a candidate escape hatch — decide at /init-plan).
2. **Buffered semaphore in the engine**: `Engine.Run` acquires a semaphore slot before each parallel-lane invocation; the serial lane is unaffected (already sequential). The WaitGroup drain guarantee (no leaked goroutines on cancellation) must be preserved.
3. **Validation**: negative values are a usage error (exit 2), consistent with the existing settings validation tier.
4. **Docs**: registry.md settings table and `atcr init` template comment gain the new key.

## Clarifications

### Recorded 2026-06-12 (rubber duck — /execute-epic)

**Key Decisions:**
- Embedded default `max_parallel` = **10** — preserves current effective behavior for v1
  rosters (≤~10 agents) and matches AC 01-04's "10 concurrent agent calls" target. A
  larger/misconfigured roster is now bounded instead of bursting one goroutine per slot.
- `max_parallel: 0` = **unbounded** (escape hatch), mirroring `payload_byte_budget`'s
  `0 = unlimited` contract. Negative values are a usage error (exit 2). Project/registry/CLI
  tiers carry `*int` pointers so an explicit `0` survives default application (a real
  override, not "unset").

**Scope Boundaries:**
- IN scope: the single global `max_parallel` knob (CLI `--max-parallel` > `.atcr/config.yaml`
  > `registry.yaml` > embedded default 10), a buffered semaphore bounding the parallel lane
  in `Engine.Run`, validation, and docs (registry.md + `atcr init` template comment).
- OUT of scope: rate limiting (requests/second) and per-provider/per-agent concurrency
  bounds — a single global knob only.

**Technical Approach:**
- The semaphore bounds concurrent **provider calls** (the work inside `Completer.Complete`),
  which is what the fake completer's peak counter measures and what Success Criterion 2
  asserts — not the goroutine-spawn count.
- `Engine` gains an optional functional-option constructor (`NewEngine(c, WithMaxParallel(n))`)
  so all existing `NewEngine(c)` / `e.Run(ctx, slots)` call sites stay unchanged. `Run` builds
  a buffered `chan struct{}` of size n; each parallel goroutine does a ctx-aware acquire
  before `invokeSlot` and releases after. `maxParallel <= 0` => no semaphore (unbounded).
- The WaitGroup drain guarantee is preserved: a goroutine whose ctx-aware acquire loses to
  `ctx.Done()` still calls `invokeSlot` (which returns a `StatusTimeout` result for the un-run
  slot) and `wg.Done()`s — no leaked goroutine, no missing result.
- The resolved value threads through `PreparedReview` into `ExecuteReview` exactly like
  `TimeoutSec`.

**Original Questions and Answers:**
1. Q: What embedded default value should `max_parallel` use?
   A: 10 — matches the AC's 10-concurrent-calls target and preserves current behavior.
2. Q: Should `max_parallel` adopt 0 as an "unbounded" escape hatch (mirroring
   `payload_byte_budget`'s 0 = unlimited), with only negative values rejected?
   A: Yes — 0 = unbounded; negative is a usage error.
