# Epic Plan 19.2: Shared Registry Remote Fetch

- **Estimated time**: 2-3 days
- **Tasks/Components**: 3/2
- **Execution**: execute-epic --strong

## Objective

Let a team share one registry by pointing atcr at a remote `registry.yaml` via `ATCR_REGISTRY_URL`, reusing the existing loader and env-var key contract.

## Context

`internal/registry` already loads `registry.yaml` and defers API-key reads to runtime env vars (`TestRegistryLoad_APIKeyNotReadAtLoad`, `docs/registry.md`, `internal/llmclient/client.go:resolveKey`). Sharing a team registry is a thin extension — fetch a remote file — not a new subsystem.

## Task Breakdown

1. Extend `parseRegistryFile` in `internal/registry/overlay.go` to fetch and parse a remote `registry.yaml` when `ATCR_REGISTRY_URL` is set, falling back to local file reads when unset.
2. Validate/merge the remote registry against the existing schema; guarantee API keys are never read from the remote file — env-var-only, unchanged.
3. Document the config-repo pattern (a `registry.yaml` in a shared git repo referenced by URL) in `docs/registry.md`.

## Acceptance Criteria

- [ ] With `ATCR_REGISTRY_URL` pointing at a reachable `registry.yaml`, providers/personas load from the remote source.
- [ ] API keys still resolve only from env vars; a key present in the remote file is ignored (with a warning).
- [ ] An unreachable/invalid URL exits with a clear error and the documented local-fallback behavior.
- [ ] `go test ./...` passes; covered by `internal/registry/*_test.go`.

## Components Touched

- `internal/registry` (loader extension)
- `docs/registry.md` (onboarding note)

## Out of Scope

Secrets-manager integration; auth on the registry URL beyond what `http.Get` supports; registry write/publish tooling.

## Dependencies

- `internal/registry` loader (exists).

## Clarifications

### Recorded 2026-07-05 (rubber duck — /execute-epic)

**Key Decisions:**
- Remote fetch hooks into `parseRegistryFile` (`internal/registry/overlay.go`): when `ATCR_REGISTRY_URL` is set, fetch+parse the remote user registry over HTTP; when unset, `os.ReadFile(path)` as today. This makes both `LoadRegistry` and `LoadMergedRegistry` remote-aware with one change.
- Only the **user** registry is fetched remotely. The project overlay (`.atcr/registry.yaml`) and `trusted_providers.yaml` stay local.
- Remote bytes flow through the unchanged `decodeStrictYAML` → `validate` → `stampSource(SourceUser)` path; key resolution is untouched.

**Scope Boundaries:**
- IN scope: remote fetch of the user registry, validation via the existing strict path, docs for the config-repo pattern.
- OUT of scope: secrets-manager integration, auth beyond `http.Get`, registry write/publish tooling.

**Technical Approach:**
- **AC2 (literal secret in remote file):** Keep the existing strict-decode fail-closed behavior. `Provider` has only `api_key_env`/`base_url` (no `api_key`) and `decodeStrictYAML` runs `KnownFields(true)`, so `api_key: sk-...` in the remote file is already a hard unknown-field decode error. No pre-decode warning scan is added (that path is unreachable — the strict decoder errors first). The structural guarantee (only `api_key_env` names travel; the value is read from the LOCAL env at invoke time via `resolveKey`) is what satisfies "keys never read from remote."
- **AC3 (set-but-unreachable/invalid URL):** Hard error, no silent fallback to the local file. Local reads happen ONLY when `ATCR_REGISTRY_URL` is unset. A present-but-broken remote source (network error, non-200, malformed YAML) is an unconditional failure — matching `parseRegistryFile`'s existing convention (missing/read/parse errors are all hard errors) and the epic's "falling back... when unset" wording.
- **Fetch security posture:** Allow both `http://` and `https://` (mirrors existing `base_url` scheme validation) but emit a one-time stderr warning on a non-https registry URL. Use a dedicated `http.Client` with a bounded ~10s timeout (a distinct, shorter constant than llmclient's 120s completion budget — this is a one-shot YAML fetch).

**Original Questions and Answers:**
1. Q: AC2 — how to handle a literal secret / `api_key` field in the remote file: strict-decode fail-closed (A) or pre-decode warning scan (B)?
   A: Option A. The codebase already fails closed — `Provider` has only `APIKeyEnv`/`BaseURL`, `decodeStrictYAML` uses `KnownFields(true)`, so `api_key: sk-...` is already a hard decode error. Option B's warning path is unreachable.
2. Q: AC3 — behavior when `ATCR_REGISTRY_URL` is set but unreachable/invalid: hard error (A) or warn + fall back to local (B)?
   A: Option A — hard error, no silent fallback. The only existing fallback precedent (`LoadProjectRegistry`) triggers on absence, not failure; a present-but-broken source is an unconditional failure.
3. Q: Fetch security posture — allow both schemes or https-only (a); ~10s timeout OK (b)?
   A: Confirmed both. (a) Allow both schemes with a one-time stderr warning on non-https. (b) Dedicated `http.Client` with a ~10s timeout constant, distinct from llmclient's 120s.
