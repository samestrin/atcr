# 11.1 Dispatcher Structural Gating

- **Estimated time**: 2 days
- **Tasks/Components**: 2/2

## Context

Epic 11.0 added opt-in code execution (`--exec`): execution-enabled skeptics are offered the `run_tests`/`run_script` tools, which run inside the container-isolated sandbox backend. Tool gating today lives at the *offering* layer — `fanout.wireToolDefs` decides which tool definitions an agent is told about, and a non-exec `Dispatcher` never registers the exec handlers (`dispatch.go:78-91`). That offering-layer gate closes the gap for every shipping caller: the sole exec caller, `verify`, enables exec uniformly for all skeptics sharing one dispatcher, so no non-exec agent can reach an exec handler.

## Problem Statement

`Dispatcher.Execute` looks up `d.handlers[name]` with no check that the named tool was actually offered to the *calling* agent. The read-only guarantee for non-exec agents is therefore advisory at the dispatch layer rather than structural: if a future architecture wired `EnableExecution` non-uniformly across agents that share a single dispatcher, a non-exec agent could invoke `run_script` simply by naming it. There is no live exploit today — this is latent hardening, deferred from a `/resolve-td` review of Epic 11.0.

## Proposed Solution

Make the read-only boundary structural at the dispatch layer so it does not depend on every caller wiring exec uniformly. Thread the calling agent's exec eligibility (its allowed tool set, or an `Exec` capability flag) into `Dispatcher.Execute`, and refuse any call whose tool was not offered to that agent — or, equivalently, gate the exec handlers behind a per-call capability rather than a single globally-registered handler shared by the whole pool.

## Requirements

### Functional
- `Dispatcher.Execute` rejects a `run_tests`/`run_script` call from an agent that was not granted exec eligibility, returning a tool error (not a panic).
- Existing exec callers (`verify --exec`, uniform across skeptics) keep working unchanged.

### Non-Functional
- No behavior change for non-exec runs (the default read-only tool set is untouched).
- The change stays within the `tools` package surface plus its immediate callers.

## Technical Approach

- Carry exec eligibility per call: either extend the `Execute` signature / dispatch context with the caller's allowed tool set, or register exec handlers behind a per-call capability gate instead of on the single shared dispatcher.
- Update the exec caller(s) to pass eligibility through.

## Out of Scope

- Any change to the sandbox isolation itself (every run is already isolated identically).
- Re-architecting the shared-dispatcher model; this epic only makes the existing boundary structural.

## Acceptance Criteria

- [ ] AC1: A non-exec agent that emits a `run_script`/`run_tests` `tool_call` is refused at `Dispatcher.Execute`, asserted by a test.
- [ ] AC2: `verify --exec` continues to execute exec tools for all skeptics with no behavior change (existing tests stay green).

## Success Criteria

- [ ] The read-only guarantee for non-exec agents is enforced structurally at dispatch, independent of how callers wire `EnableExecution`.

## Rollback Plan

- Revert to the offering-layer-only gate; the sandbox isolation and the non-uniform-wiring assumption are unchanged, so no runtime risk.

## Security Considerations

- This epic *is* the security hardening: it removes the standing assumption that all callers wire exec uniformly, eliminating a latent privilege-escalation path for any future shared-dispatcher caller.
