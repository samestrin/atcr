# Epic Plan 11.2: Dispatcher Registration Gating

**Estimated Durations**: ~1 hour
**Tasks/Components:** 2/1
**Execution:** execute-epic
**Status:** Ready

> **Origin.** Deferred from sprint `11.1_dispatcher-structural-gating` during `/resolve-td`
> (clarification Q1, 2026-06-26, 92% confidence). Epic 11.1 made the read-only boundary
> structural at *dispatch* (`Dispatcher.Execute` refuses an exec-gated tool unless the caller
> carried exec eligibility). It did **not** harden the *registration* side: the public
> `RegisterTool` API can still register a code-executing handler that is absent from the
> `execTools` gate map. Both of 11.1's acceptance criteria are satisfied — this is new scope
> that extends the public registration contract, so it was deferred here rather than folded in.

## Objective

Close the registration-side gap so the structural exec boundary cannot be bypassed by a future
caller, without weakening 11.1's dispatch-time gate.

Today `Dispatcher.RegisterTool(name, handler)` runs only `guardToolName`, which blocks
*write*-verb names (write/create/delete/...) but not *execution* verbs (run/exec/eval/shell).
The exec-eligibility gate in `Execute` keys off the separate `execTools` map, which is populated
**only** by `EnableExecution` via `mustRegisterExec`. A future caller doing
`d.RegisterTool("run_eval", evalHandler)` registers a code-executing handler that is absent from
`execTools`, so `execGated` is false and the per-call eligibility gate at `dispatch.go:175` is
never consulted — a fully ungated exec tool. The structural boundary thus currently depends on
the unenforced convention that nobody registers an executing handler via the public API.

## Scope

- **In scope:** the `internal/tools` dispatcher registration path (`RegisterTool`,
  `mustRegister`, `mustRegisterExec`, `EnableExecution`) and its tests.
- **Out of scope:** the dispatch-time eligibility gate (shipped in 11.1, unchanged), the
  sandbox backend, and any caller outside `internal/tools`.

## Tasks

1. **Couple exec-gating to registration.** Introduce a single registration path
   (e.g. `registerExec(name, handler)`) that atomically sets both `handlers[name]` and
   `execTools[name]` under `d.mu`, and route `EnableExecution` through it (it already does this
   via `mustRegisterExec` — consolidate so there is exactly one way to register an exec handler).
   Decide and document the policy for the public `RegisterTool` when handed an execution-verb
   name: either reject it (mirroring `guardToolName`'s write-verb rejection) or require it to go
   through `registerExec`. Rejection is the simpler fail-closed default.

2. **Invariant test.** Add a test asserting that every registered handler whose body reaches
   `runInSandbox`/`execBackend` is present in `execTools` (so no exec handler can be registered
   ungated), plus a behavioral test: register an executing handler via the public API and confirm
   it is either rejected or gated (refused for a non-eligible caller), never silently ungated.

## Acceptance Criteria

- An exec-capable handler cannot be registered through the public API without being placed in the
  `execTools` gate map (enforced by test, not convention).
- `EnableExecution` continues to register `run_tests`/`run_script` gated, and the 11.1 dispatch
  gate behavior is unchanged (all existing `internal/tools` tests stay green).
- `go test -race ./internal/tools/...` passes.
