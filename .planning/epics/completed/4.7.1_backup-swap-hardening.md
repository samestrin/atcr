# Epic 4.7.1: Crash-Safe Backup Swap Hardening

- **Estimated time**: ~120 min (design-bearing follow-on to Epic 4.7 `.bak` mechanism)
- **Status**: active
- **Source**: deferred from `/resolve-td` (group 3, 2026-06-19) ÔÇö TD item `internal/fanout/reviewdir.go:284`

## Current State

The `--force` re-run backup mechanism (Epic 4.7) preserves exactly one prior
generation by writing the new backup over the old one. Two sites implement this
with a destroy-before-write ordering that loses the prior backup if the write
fails partway:

- `internal/fanout/reviewdir.go:284` `backupExisting()` ÔÇö `os.RemoveAll(backup)`
  then `os.Rename(path, backup)` (move-based).
- `internal/atomicfs/atomic.go:72` `BackupToDotBak()` ÔÇö `os.RemoveAll(bak)` then
  `copyTree` (copy-based).

In both, the single recoverable prior `.bak` is already destroyed when the
subsequent rename/copy fails (cross-filesystem `EXDEV` when `--output-dir` and
its `.bak` sibling are on different mounts, disk-full mid-copy, or `SIGKILL`).
The live/original tree survives (good), but the prior backup is lost for no
benefit ÔÇö counter to the safe-to-retry goal of Epic 4.7.

## Pain Points

### Backup loss on failed swap
The destroy-before-write order means any failure between `RemoveAll` and the
completion of the rename/copy leaves the user with neither a complete new backup
nor the prior one.

### Inconsistent hardening
Fixing only one site leaves the codebase inconsistent ÔÇö the identical weakness
lives at both call sites and must be hardened together.

## Proposed Solution

Stage the new backup first, swap atomically, and only then remove the old
generation. The two sites need different staging strategies because one moves
and one copies:

- **`backupExisting()` (move-based):** rename the old `.bak` aside to `.bak.old`
  first, then `os.Rename(path, .bak)`, then `RemoveAll(.bak.old)` on success /
  restore `.bak.old` Ôćĺ `.bak` on failure. Detect `EXDEV` and fall back to
  copy+remove.
- **`BackupToDotBak()` (copy-based):** stage into `.bak.new`, confirm the copy
  is complete, then atomically swap `.bak.new` Ôćĺ `.bak` and remove the old
  generation.

## Scope

### In Scope
- Crash-safe staging at `internal/fanout/reviewdir.go` `backupExisting()`.
- Crash-safe staging at `internal/atomicfs/atomic.go` `BackupToDotBak()`.
- `EXDEV` detection with copy+remove fallback for the move-based path.
- Fault-injection tests asserting the prior `.bak` survives a failed swap at both
  sites.

### Out of Scope
- Multi-generation backup retention / `.bak` GC (Epic 4.7 explicitly keeps one
  generation; this epic preserves that contract, only making the swap crash-safe).

## Acceptance Criteria

- [x] AC1: `backupExisting()` leaves the prior `.bak` intact when the swap fails
      (rename old `.bak` aside, swap, remove old on success / restore on failure).
- [x] AC2: `BackupToDotBak()` stages into `.bak.new` and swaps atomically, leaving
      the prior `.bak` intact on a failed/partial copy.
- [x] AC3: `EXDEV` (cross-filesystem) on the move-based path is detected and falls
      back to copy+remove without losing the prior `.bak`.
- [x] AC4: Fault-injection tests cover a failed swap at both sites and assert the
      prior `.bak` survives.
- [x] AC5: The one-generation `--force` contract from Epic 4.7 is preserved (no
      backup accumulation).

## Risks

| Risk | Mitigation |
|------|------------|
| Partial `.bak.old` / `.bak.new` left on crash | Reconcile on next `--force`: prefer the newest complete generation, clean stragglers |
| EXDEV fallback copy is non-atomic | Stage to `.bak.new` on the destination filesystem, then rename-swap (same-fs) |
| Prior decision treated atomicfs twin as acceptable | This epic explicitly overrides the "acceptable for a recovery aid" call at `internal/atomicfs/atomic.go:72` ÔÇö uniform hardening is the goal |

## Clarifications

### Recorded 2026-06-19 (rubber duck ÔÇö /execute-epic)

**Key Decisions:**
- `BackupToDotBak()` (`internal/atomicfs/atomic.go:72`) was already converted to
  staged-temp-then-swap by commit `9cc9d84` (#53), merged ~1 min before this run. AC2's
  intent is already satisfied on `main` (stages into `.bak.tmp-*`, defers `RemoveAll(bak)`
  until after the copy, renames staged temp over `.bak`). Treat AC2 as **verify, do not
  rewrite**; the `.bak.new` naming in the plan was illustrative, not a contract.
- For `backupExisting()` EXDEV (AC3), the realistic trigger is `path` itself being a
  mountpoint (since `path` and `path+".bak"` are same-directory siblings). The fallback:
  copy `path`'s tree to a same-fs staged `path+".bak.new"`, rename it over `path+".bak"`,
  then `RemoveAll(path)` to replicate the move's "leave path vacant" postcondition.
- Deterministic EXDEV / failed-swap injection uses a **package-level rename-function seam**
  (a `var` overridable in tests) ÔÇö the only CI-safe way to exercise the `syscall.EXDEV Ôćĺ
  copy fallback` branch without a real cross-mount.
- **Straggler cleanup at entry is in scope and required**: `RemoveAll` any leftover
  `.bak.old` / `.bak.new` / `.bak.tmp-*` at the top of each backup entry. These are
  atcr-owned temp state, not a backup generation ÔÇö this is the "Reconcile on next `--force`:
  clean stragglers" mechanism the Risks table names, and is categorically distinct from
  multi-generation GC. Required to honor AC5 across a crash-then-retry sequence.

**Scope Boundaries:**
- IN scope: crash-safe staging at both sites; EXDEV copy+remove fallback for the move path;
  fault-injection tests (failed swap at both sites); entry-time straggler cleanup of
  atcr-owned `.bak.old`/`.bak.new`/`.bak.tmp-*`; preserve the one-generation contract.
- OUT of scope: multi-generation `.bak` retention / GC; rewriting the already-hardened
  `BackupToDotBak()`; extending `guardForeignBackup` to `.bak.old`/`.bak.new`.

**Technical Approach:**
- `backupExisting()` (move-based, AC1): clean stragglers Ôćĺ rename existing `.bak` Ôćĺ `.bak.old`
  Ôćĺ `os.Rename(path, .bak)` Ôćĺ on success `RemoveAll(.bak.old)` / on failure restore
  `.bak.old` Ôćĺ `.bak`. Detect `EXDEV` on the `path Ôćĺ .bak` rename and fall back to
  copy-into-`.bak.new` + rename + `RemoveAll(path)`.
- `BackupToDotBak()` (copy-based, AC2): keep #53's implementation; only add a fault test for
  the **Rename step specifically** (copy succeeds, rename fails, prior `.bak` survives) if
  `atomic_errors_test.go` lacks one.
- `guardForeignBackup` stays scoped to `.bak` only; `.bak.old`/`.bak.new` are atcr-internal
  names no foreign toolchain produces, so the foreign-data heuristic has nothing to test.
