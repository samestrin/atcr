## Current State

The scorecard store and emitter write operational diagnostics directly to the
process-global `os.Stderr` rather than to a writer threaded from the caller:

- `internal/scorecard/store.go` â€” `ReadRecords` ("skipping malformed record", "skipping over-long line"), `FindByRunID` ("run spans adjacent month files")
- `internal/scorecard/scorecard.go` â€” `Emit` / `verdictTallies` ("write failed", "verification read/parse failed", "verification finding â€¦ has no matching raised finding")

Because the sink is the global `os.Stderr`, these warnings cannot be captured,
redirected, or asserted in tests, and they bypass any writer the CLI already
threads for the rest of its output.

## Pain Points

- **Not testable** â€” orphan-verdict and malformed-record warnings can only be
  verified by capturing the global `os.Stderr`, so the diagnostics added in
  Sprint 3.3 are guarded only behaviorally (count assertions), not by their text.
- **Not redirectable** â€” the CLI threads `cmd.ErrOrStderr()` for its own output,
  but scorecard diagnostics escape that, so `--quiet`/redirection cannot reach them.
- **Inconsistent** â€” the rest of the command tree sources its error writer from
  cobra; the scorecard package is the outlier writing to a package-global.

## Proposed Solution

Thread a single injectable `io.Writer` (or a small logger) through the scorecard
read + emit entry points and their callers as one designed package-API change:

- Add the writer to `Append`, `ReadRecords`, `FindByRunID`, `ReadAll`, `Emit`, and `EmitForReconcile` (or a package-level option struct that carries it).
- CLI caller (`cmd/atcr/reconcile.go`) passes `cmd.ErrOrStderr()`.
- Default to `os.Stderr` when no writer is supplied so existing behavior is preserved.

## Key Design Gap (why this is not an inline fix)

The MCP entry point `internal/mcp/handlers.go:218` calls `EmitForReconcile` with
**no cobra `cmd`** to source a writer from, so the CLI's `cmd.ErrOrStderr()`
precedent does not transfer. The package change must define how the MCP path
supplies (or defaults) its writer â€” a deliberate API decision, not a mechanical
substitution. This is precisely why it crosses the group-8 boundary and was
deferred from `/resolve-td` rather than resolved inline.
