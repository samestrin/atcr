# Epic Plan 26.0: .atcrignore Token Waste Protection

- **Estimated time**: TBD
- **Tasks/Components**: 3 tasks / 2 components
- **Execution**: execute-epic

## Objective

Prevent ATCR from wasting LLM tokens (and money) on auto-generated code, vendored dependencies, or massive lockfiles by respecting `.gitignore` and introducing a native `.atcrignore` filtering layer.

## Context

If a developer runs a codegen tool or bumps a massive `vendor/` dependency, the resulting git diff can be hundreds of thousands of lines. While Epic 19.5 handles truncation and dropping files to fit the token budget, it does so based on size/priority. It is far safer to explicitly never send these files to the payload builder in the first place.

## Proposed Solution

1. **File Filtering Layer**: Update the diff ingestion logic in `internal/payload` (`diff.go`'s `changedFiles`/`changedFilesMemo`, ahead of `budget.go`'s `ApplyByteBudget` truncation pass) to completely exclude files that match standard ignore patterns.
2. **.gitignore Support**: By default, parse and respect the repository's `.gitignore`.
3. **.atcrignore Support**: Support a dedicated `.atcrignore` for files that should be committed to git but ignored by the AI (e.g., `go.sum`, `package-lock.json`, `docs/`). Repo-root-only, one gitignore-syntax pattern per line, purely additive to `.gitignore`'s exclusions (no `!` negation support).

## Acceptance Criteria

- [x] Files matching `.gitignore` are excluded from the review payload.
- [x] Files matching `.atcrignore` are excluded from the review payload.
- [x] `atcr review` logs a trace/debug message noting which files were skipped due to ignore rules.

**Key Decisions:**
- Use the third-party library `github.com/sabhiram/go-gitignore` for pattern matching rather than hand-rolling a matcher (no gitignore-matching capability exists in the codebase; hand-rolling risks silent under/over-exclusion).
- `.gitignore` support is repo-root-only, matching the `.atcrignore` model â€” no nested-tree git semantics (the file list comes from a single repo-root-scoped `git diff --name-status`; nested precedence would need new tree-walking infrastructure out of scope for this plan).
- Filtering is always-on with a `--no-ignore` opt-out flag on `atcr review`, mirroring the existing `--no-cache` pattern (`cmd/atcr/review.go` â†’ `ReviewRequest.NoCache` â†’ `internal/fanout/review.go`). This gives the `cmd/atcr` component an actual CLI-surface change rather than only log wiring.

**Scope Boundaries:**
- IN scope: filter the git-range changed-file list (`internal/payload/diff.go` `changedFiles`/`changedFilesMemo`, ahead of `ApplyByteBudget`) against repo-root `.gitignore` + repo-root `.atcrignore`; `--no-ignore` flag; slog debug logging of skipped files.
- OUT of scope: nested `.gitignore` files; `!` negation in `.atcrignore`; the diff-file ingestion path (`BuildEntriesFromDiff`, used by autofix and `--diff`/`--diff-file` review) â€” only the git-range path is filtered; complex globbing beyond standard `.gitignore` syntax.

**Technical Approach:**
- Ignore files are read from the working tree at the repo dir (`g.dir` / `e.root`), not from a git ref.
- Filtering matches on the head-side path for every change kind (added/modified/deleted/renamed via new path).
- The filter lives at a single chokepoint over the changed-file list so `BuildEntries` and `ChangedFileCount` agree on the filtered set.
- Skipped files are logged at slog debug level via the existing `gitRunner` logger.
