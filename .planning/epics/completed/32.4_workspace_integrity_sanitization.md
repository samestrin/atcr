# Epic Plan 32.4: Workspace Integrity & Indirect Sandbox Escape Prevention

- **Estimated time**: 2 days
- **Tasks/Components**: 6 tasks / 10 components
- **Execution**: init-plan

## Objective

Harden ATCR against **Indirect Sandbox Escape** attacks (as disclosed in recent AI coding agent vulnerabilities affecting Cursor, Codex CLI, and Gemini CLI), where a contained sandbox execution is bypassed by modifying host configuration files (`.git/config`, `.githooks/`, `.github/workflows/`, `.vscode/`) that execute on the developer's host machine post-review.

## Context

Recent security disclosures from Pillar Security (and CVE-2026-48124) demonstrated that AI coding agents often fail not because the container/sandbox breaks, but because of **Host Trust Transposition**:
1. The AI agent operates in a confined sandbox during execution.
2. Prompt injection or hallucinated patches cause the agent to write a malicious configuration artifact inside the project workspace (e.g. `.git/config` setting `core.pager`, `.githooks/pre-commit`, or `.vscode/settings.json`).
3. Once the agent finishes, host-side tools (Git CLI, IDEs, build scripts) execute those modified workspace files with full host developer privileges outside the sandbox.

While Epic 32.0 and Epic 32.3 hardened ATCR's Docker container runtime, ATCR's `--auto-fix` pipeline applies patches to the host repository via `atomicfs` and runs host-side `git` commands. We must ensure LLM-generated patches cannot write malicious triggers to host configuration paths and that ATCR's host Git invocations cannot be hijacked.

## Proposed Solution

1. **Strict Path Protection Guard (`internal/security/pathguard.go`):**
   Enforce an un-bypassable path inspection check before `--auto-fix` applies any patch. Block modification or creation of critical host-execution and configuration paths:
   - `.git/*` and `.githooks/*`
   - `.github/workflows/*`, `.gitlab-ci.yml`, and CI definitions
   - `.vscode/*` and `.idea/*`
   - `.env*` files
   - `.planning/`
   - `.atcr`   
   Provide an explicit CLI flag `--allow-config-edits` for operators who legitimately need to refactor build configs.

2. **Host Git Subprocess Hardening (`internal/gitexec`, new package):**
   `internal/git/` does not exist — host `git` subprocess execution is scattered across six production call sites (`cmd/atcr/autofix.go`, `internal/fanout/review.go`, `internal/gitrange/resolver.go`, `internal/payload/diff.go`, `internal/personas/submit.go`, `internal/stream/fileindex.go`), each hijackable by a poisoned `.git/config` left behind by an earlier `--auto-fix` run — not just calls made during the auto-fix pass itself. Build one shared `internal/gitexec` wrapper around `exec.Command`/`exec.CommandContext` that injects `GIT_CONFIG_NOSYSTEM=1`, `GIT_CONFIG_GLOBAL=/dev/null`, and `--no-ext-diff` (where applicable), then migrate all six call sites to use it. Partial coverage leaves the exact gap this epic exists to close, so this is not narrowed to auto-fix-time calls only.

3. **Non-Blocking Risk Flag for Executable/Build-Script Patches (`internal/security/pathguard.go` extension):**
   Extend pathguard with a second, non-blocking check — executable-bit changes (via `go-gitdiff`'s `OldMode`/`NewMode`, already available in `internal/autofix/apply.go`'s `applyOne`) and a soft list of build-script paths (`Makefile`, `*.sh`, `package.json`, `Dockerfile`, CI configs living outside `.github/`) — that does not block the patch but surfaces a warning in the generated PR body. `--auto-fix` already never auto-merges (`cmd/atcr/autofix.go` always opens a PR via `CreatePullRequest` and never merges it), so human review is already the terminal gate; this makes elevated-risk patches visible in that existing review rather than adding a new gate or a new agent role.

## Proposed Tasks

- **T1**: Build `internal/security/pathguard.go` implementing `IsProtectedPath(path string) bool` with a comprehensive blocklist covering `.git/`, `.githooks/`, `.github/`, `.vscode/`, `.idea/`, and `.env*`.
- **T2**: Wire `pathguard.go` into `internal/autofix/apply.go`'s `applyOne`, immediately after `containedPath` succeeds and before the delete/modify/create branches, to intercept and reject patches targeting protected paths unless `AllowConfigEdits` is true.
- **T3**: Build `internal/gitexec` (new package) wrapping `exec.Command`/`exec.CommandContext` for git invocations, injecting `GIT_CONFIG_NOSYSTEM=1`, `GIT_CONFIG_GLOBAL=/dev/null`, and `--no-ext-diff` (where applicable). Migrate all six existing host git subprocess call sites to it: `cmd/atcr/autofix.go`, `internal/fanout/review.go`, `internal/gitrange/resolver.go`, `internal/payload/diff.go`, `internal/personas/submit.go`, `internal/stream/fileindex.go`.
- **T4**: Add `--allow-config-edits` flag to `cmd/atcr/`, document security implications in `docs/security.md`, and add `docs/security.md` to `docs/README.md`'s index (the canonical doc list the website build consumes).
- **T5**: Add unit tests in `internal/security/pathguard_test.go` and `internal/gitexec/gitexec_test.go`; add regression coverage confirming each of the six migrated call sites (T3) invokes git through `internal/gitexec` rather than a bare `exec.Command`.
- **T6**: Extend `internal/security/pathguard.go` with a non-blocking `FlagsForReview(path string, oldMode, newMode int) (bool, reason string)` check (executable-bit change via `OldMode`/`NewMode`, plus the soft build-script path list). Thread flagged entries out of `internal/autofix.ApplyPatch` (alongside the existing `BackupMap`) so `cmd/atcr/autofix.go`'s PR-body builder (near its `CreatePullRequest` call) can append a visible warning section when any flagged path is touched.

## Acceptance Criteria

- [ ] AC1: `--auto-fix` refuses to modify or create files under `.git/`, `.githooks/`, `.github/workflows/`, `.vscode/`, `.idea/`, or `.env*` and returns a security error unless `--allow-config-edits` is set.
- [ ] AC2: All Git subprocesses executed by ATCR carry `GIT_CONFIG_NOSYSTEM=1` and `GIT_CONFIG_GLOBAL=/dev/null` in their environment variables.
- [ ] AC3: Unit tests in `internal/security/pathguard_test.go` verify 100% path matching across canonical, relative, and symlink-traversal path formats.
- [ ] AC4: All six identified host git subprocess call sites (`cmd/atcr/autofix.go`, `internal/fanout/review.go`, `internal/gitrange/resolver.go`, `internal/payload/diff.go`, `internal/personas/submit.go`, `internal/stream/fileindex.go`) are migrated to `internal/gitexec`; no bare `exec.Command("git", ...)` / `exec.CommandContext(ctx, "git", ...)` call sites remain outside it.
- [ ] AC5: A generated `--auto-fix` PR whose patch touches an executable-bit change or a build-script path (per `FlagsForReview`) includes a visible warning section in the PR body naming the flagged path(s) and reason; a patch with no flagged paths has no such section.

### Items needing user confirmation (4)

- ⏸️ **"Skeptic" already means something different in ATCR; AC3 / Proposed Solution #3 is unimplementable as scoped, and no task builds it.** The plan's Proposed Solution #3 and AC3 describe "the Skeptic agent" scoring patch risk and flagging `HIGH_RISK_REQUIRES_HUMAN_REVIEW`, citing `internal/personas/community_schema.go`. That file does not exist — the identical stale citation was already caught and corrected in epic 32.1's `/refine-epic` run on 2026-07-17 (see `.planning/epics/completed/32.1_multi_tier_fix_execution.md`). More importantly, "Skeptic" is already a precisely-defined ATCR role: a `role: skeptic` persona invoked by `atcr verify` (`internal/verify`, documented in `docs/verification.md`) that tries to *refute* an already-reported finding, returning `confirmed`/`refuted`/`unverifiable` — not a risk-scorer for patch content. `HIGH_RISK_REQUIRES_HUMAN_REVIEW` does not exist anywhere in the codebase; the real vocabulary is CRITICAL/HIGH/MEDIUM/LOW severities plus a fixed lowercase CATEGORY set (security, correctness, performance, testing, style, docs), defined in `personas/_base.md`. None of T1–T5 actually implements this. Suggested action: either (a) drop AC3 and Proposed Solution #3 from this epic and track "flag config/script risk" as a separate, properly-scoped follow-up that doesn't overload the existing "skeptic" vocabulary, or (b) rewrite AC3 concretely — name which persona(s) gain the new instruction, add a Task for it, and define where the new risk signal is surfaced (a new CATEGORY value? a new field in the findings schema? a gate in `internal/verify`?) instead of an invented verdict string.

- ⏸️ **T2's integration point is wrong: the real patch-write path is `internal/autofix/apply.go`, not `internal/verify/executor.go` + `internal/atomicfs`.** `internal/atomicfs` is a generic, low-level atomic-write/backup utility shared by ~8 unrelated packages (`internal/reconcile`, `internal/cache`, `internal/verify` emit files, `internal/payload`, `internal/fanout`, `internal/debate`, plus `internal/autofix` itself); none of its functions (`WriteFileAtomic(path, data)`, `BackupToDotBak(path)`) carry any concept of "is this an --auto-fix write" or an `AllowConfigEdits` flag, so gating there would need a signature change threaded through every unrelated caller, or would incorrectly gate writes that have nothing to do with `--auto-fix`. `internal/verify/executor.go`'s `generateFixes` only generates fix *content* via the LLM — it never touches the filesystem. The actual host-repo write path for `--auto-fix` is `internal/autofix/apply.go`'s `applyOne`, which already has a defense-in-depth path-containment check (`containedPath`) with a single, well-scoped choke point right after it (before the delete/modify/create branches) where a `pathguard.IsProtectedPath` check belongs. Suggested action: retarget T2 (and its `COMPONENTS_TOUCHED` entry) to `internal/autofix/apply.go` instead of `internal/verify/executor.go` + `internal/atomicfs`.

- ⏸️ **T3/T5's "internal/git/" package doesn't exist; git subprocess calls are scattered across at least 6 production packages, so AC2's "ALL Git subprocesses" is far bigger than one task.** `internal/git/` does not exist anywhere in the repo. `exec.Command("git"...)` / `exec.CommandContext(ctx, "git"...)` production call sites live in `cmd/atcr/autofix.go`, `internal/fanout/review.go`, `internal/gitrange/resolver.go`, `internal/payload/diff.go`, `internal/personas/submit.go`, and `internal/stream/fileindex.go` — six distinct packages, each constructing its own call with its own env handling (or none). There is no shared git-exec helper to update in one place. As scoped, T3 ("Update host Git execution helpers (in internal/git/ or cmd/atcr/)") implies a single-location fix; AC2 ("All Git subprocesses executed by ATCR carry GIT_CONFIG_NOSYSTEM=1...") implies touching all six. Suggested action: either (a) scope T3 down to introduce ONE new shared helper (e.g. `internal/gitexec`) that every call site migrates to — itself a bigger, cross-cutting refactor than "update host Git execution helpers" suggests — or (b) narrow AC2 to name the specific call sites actually in scope for this epic and defer the rest as follow-up technical debt.

- ⏸️ **COMPONENTS_TOUCHED is understated even by the plan's own count, and `/execute-epic`'s scope guard is already violated.** The plan's header claims "4 components" and lists 4 bullets under Components Touched, but two of those bullets each span two distinct directory trees (`internal/verify/` & `internal/atomicfs/`; `internal/git/` / `cmd/atcr/`), which — per this skill's component-counting rule (co-located files in the same tree count once; distinct trees count separately) — is already 6 distinct components before factoring in the corrections above (adds `internal/autofix`, up to 5 more git-exec call sites, and whichever persona/verify file ends up hosting the risk-scoring logic). Suggested action: re-derive `COMPONENTS_TOUCHED` after resolving the three items above, then decide between narrowing scope to fit `/execute-epic`'s ≤2-component limit or routing this epic through `/init-plan` for the full sprint pipeline (see Advisory observations below).
