# Tech Debt: Cluster-ID on Finding (6.2)

- **Estimated time**: ~30-60 min (LOW severity, self-healing, no urgency)
- **Tasks**: 1-2
- **Components**: 1 (internal/reconcile + internal/debate)

## Current State

`filterMergedClusters` (internal/debate/cluster.go) keys gray-zone merge idempotency
solely on `locationKey(File, Line)`. Once one gray-zone cluster at a canonical
File+Line is merged (its survivor record carries `ClusterMerged: true`), a SECOND
distinct gray-zone cluster sharing that same File+Line is also filtered out
pre-debate — even though it was never merged.

The behavior is **self-healing and non-corrupting**: the over-suppressed second
cluster re-surfaces on the next fresh reconcile. It is consciously accepted today
and pinned by `TestFilterMergedClusters_CoLocatedDistinctClustersOverSuppressed`
(internal/debate/cluster_test.go) and documented at the surrounding code comments.

This plan is the "tracked separately" vehicle for the structural fix that the TD
item and code comment defer.

## Pain Points

### Over-suppression of co-located distinct clusters
- Two distinct gray-zone clusters at the same canonical File+Line: merging the
  first suppresses the second's radar item until the next fresh reconcile.
- Impact: LOW — delays (does not lose) the second cluster's debate; no data
  corruption.

### Why it is not a quick-win TD item
- The fix changes the findings.json schema (an external contract), so it is a
  structural extension rather than an inline TD fix.
- It touches four sites across two packages:
  - `internal/reconcile/emit.go` — add a `ClusterID` field to `JSONFinding`
    (+ `JSONFindings()` converter).
  - `internal/reconcile/merge.go` — carry/set the id through `MergeJSONFindings`
    if the merge path is the producer.
  - `internal/debate/cluster.go` — set the id in `applyOneClusterMerge`, read it
    in `filterMergedClusters`.

## Proposed Solution

Add a `ClusterID` field to `JSONFinding` so `filterMergedClusters` keys
idempotency on cluster identity rather than `File+Line` alone, eliminating
co-located-distinct-cluster over-suppression.

### Strategy
1. Add `ClusterID string` (`json:"cluster_id,omitempty"`) to `JSONFinding` so a
   non-merged record stays byte-identical to pre-6.2 findings.json.
2. Stamp the survivor's `ClusterID` from `AmbiguousCluster.ID` in
   `applyOneClusterMerge` (the cluster already carries a stable `ID`).
3. Change `filterMergedClusters` to suppress a gray-zone radar item only when a
   `ClusterMerged` survivor with the SAME `ClusterID` exists, not merely the same
   location.
4. Co-schedule consideration: the 13.2 DBSCAN roadmap item, where stable cluster
   IDs would emerge organically — revisit whether to fold this in there.

## Scope

### In Scope

- Update `TestFilterMergedClusters_CoLocatedDistinctClustersOverSuppressed` to

### Out of Scope
- DBSCAN clustering rework (13.2) — only coordinate, do not implement here.
