# Feature Request: Quick Start CLI Workflow

- **Estimated time**: 4 days
- **Tasks/Components**: 4/1

## Problem

Users currently have to manually configure API keys, generate `config.yaml`, and write GitHub Action workflows from scratch. This high friction during onboarding leads to churn before they experience the value of the tool.

## Solution

Implement an `atcr init` or `atcr quickstart` interactive wizard. This will drive users to a synthetic sign-up (utilizing referral codes), fetch their API tokens securely, and automatically scaffold default profiles tied to synthetic models.

## Acceptance Criteria

- [ ] AC1: `atcr init` command provides an interactive terminal wizard.
- [ ] AC2: Wizard guides the user through synthetic sign-up (e.g., opening a browser link with referral tracking).
- [ ] AC3: Tool securely receives or accepts the generated API token.
- [ ] AC4: Tool auto-generates `.atcr/config.yaml` with default settings and synthetic model references.
- [ ] AC5: Tool auto-generates boilerplate `.github/workflows/atcr.yml` without overwriting existing critical configs.

## Success Criteria

- New users can run their first review within 30 seconds of running `atcr init`.
- 80%+ completion rate for users who start the initialization wizard.

## Out of Scope

- Full GUI dashboard for configuration (CLI only for now).
- Migration of existing complex configs.

## Clarifications

### Recorded 2026-07-02 (rubber duck — /execute-epic)

**Key Decisions:**
- New command `atcr quickstart` (NOT a change to `atcr init`). `quickstart` reuses `init`'s
  writers for the roster/personas, then layers interactive provider + key-env setup on top.
  `atcr init` stays non-interactive and idempotent.
- "Synthetic sign-up (utilizing referral codes)" = a convenience referral link only, no
  automation. Print a referral-coded signup URL as an OSC-8 terminal hyperlink; optional
  browser open behind a `--open` opt-in. No account creation, no token harvesting.
- API keys follow atcr's existing posture: **never persisted by atcr**. The wizard helps the
  user set an env var (prints `export LLM_SYNTHETIC_API_KEY=…`, optionally appends to a
  user-chosen shell profile). The key value never enters `.atcr/` or `registry.yaml`.
- "Synthetic models" = the synthetic.new OpenAI-compatible provider
  (`base_url: https://api.synthetic.new/openai/v1`, `api_key_env: LLM_SYNTHETIC_API_KEY`)
  plus agents bound to real synthetic.new model ids. Ship a static bundled manifest mirroring
  `quickstart-synthetic.json`, PLUS a scheduled auto-refresh GitHub Action (in scope).
- Config split matches current architecture: provider + agent definitions → user
  `~/.config/atcr/registry.yaml`; roster (agent names) → project `.atcr/config.yaml`.
- Scaffolded `.github/workflows/atcr.yml`: option (i) — install the atcr binary and run
  `atcr review` directly, wired to `LLM_SYNTHETIC_API_KEY`. Minimal review-only template
  (does NOT use `samestrin/atcr@v1`, drops the reconcile/github PR-gate; noted in a comment).
  Per-file non-overwrite guard: refuse to overwrite an existing workflow file unless `--force`,
  and skip just that file rather than aborting the whole quickstart.

**Scope Boundaries:**
- IN scope: `atcr quickstart` interactive command; synthetic provider/agents defaults; user
  registry + project roster writes; env-var key guidance (no persistence); referral link;
  `.github/workflows/atcr.yml` scaffold (per-file guard); scheduled auto-refresh Action for the
  bundled synthetic manifest.
- OUT of scope: automated account creation / token harvesting; persisting API keys to disk;
  changing `atcr init`'s contract; modifying the published `action.yml` public contract; GUI
  dashboard; migrating existing complex configs.

**Technical Approach:**
- Tasks: T1 `atcr quickstart` skeleton (reuse init writers); T2 interactive provider + key-env
  flow; T3 synthetic provider/agents + bundled manifest + registry/roster writes; T4
  `.github/workflows/atcr.yml` scaffold with per-file guard; T5 scheduled auto-refresh Action
  regenerating the bundled manifest from synthetic.new `/models`.
- `LLM_SYNTHETIC_API_KEY` is a new convention this epic introduces (zero hits in repo today).
