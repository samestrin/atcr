# Epic 5.3: Anthropic-Native Conformance

**Status:** ❌ CLOSED — WON'T FIX (2026-06-20) — founding premise factually incorrect (see Resolution)
**Created:** 2026-06-19
**Closed:** 2026-06-20
**Priority:** Medium (regression safety for a primary provider)

## Why this exists

Epic 5.1 (OpenAI-Compatible Conformance Suite) deliberately scopes itself to the
`/v1/chat/completions` surface (OpenAI + Ollama) and **explicitly excludes the Anthropic
native API** ("uses a different wire format; separate concern"). Anthropic is a primary
provider for ATCR's reviewer panel, yet no sibling epic picks that concern up — so it was
silently dropped. This stub makes the gap visible and tracked rather than lost.

## Problem

`internal/llmclient` talks to Anthropic's native Messages API, whose wire format differs
from the OpenAI-compatible surface (system prompt handling, `content` blocks, `tool_use` /
`tool_result` blocks, `stop_reason` vs `finish_reason`). 5.1's fixtures will protect the
OpenAI/Ollama path but give **zero coverage** of the Anthropic decode path. A regression in
how ATCR parses an Anthropic `tool_use` response or `stop_reason` would not be caught.

## Goal

Record real wire fixtures from the Anthropic native Messages API and assert the llmclient
parser produces the correct engine-facing `ChatResponse` — the same protection 5.1 gives
the OpenAI-compatible surface, extended to Anthropic.

## Resolution: Closed — Won't Fix (2026-06-20)

Closed during `/execute-epic` after a rubber-duck clarification gate established that the
epic's founding premise is factually incorrect.

**Finding — there is no native Anthropic decode path to protect.**
`internal/llmclient` is purely OpenAI-compatible wire format. ATCR reaches Anthropic
exclusively through an OpenAI-compatible gateway (`choices` / `finish_reason` /
`tool_calls`), never the native Messages API.

Evidence (`internal/llmclient/chat.go`):
- `ToolDef.MarshalJSON` (chat.go:22) emits the OpenAI tool envelope — described at
  chat.go:13 as "the lowest-common-denominator wire format across OpenAI-compatible
  providers."
- `chatToolResponse` (chat.go:126) decodes `choices[].finish_reason` + `choices[].message`
  — the OpenAI Chat Completions shape. There is no `content`-block parsing, no
  `tool_use` / `tool_result` block decoding, and no `stop_reason` field anywhere.
- The code comment at chat.go:97–101 explicitly treats Anthropic as accessed via an
  OpenAI-compatible gateway ("some Anthropic-via-gateway shims"), not the native API.

Because no native Anthropic wire path exists, the epic's Problem statement (line 17,
"`internal/llmclient` talks to Anthropic's native Messages API") describes code that is
not present. AC4's "`stop_reason` handling" has no implementation to assert against.

**Options weighed at the clarification gate:**
- (A) Reframe to the OpenAI-compatible path (capture Anthropic-via-gateway responses,
  already OpenAI-shaped). Rejected — this collapses entirely into deferred Epic 5.1's
  scope, making 5.3 a pure duplicate.
- (B) Build a native Anthropic Messages API adapter first, then conformance tests.
  Rejected — that is a speculative net-new provider adapter no current code calls; a
  feature, not a regression guard, and out of scope for a conformance epic.
- (C) Close as won't-fix / obsolete. **Selected** (user decision, 2026-06-20, 97%
  confidence). The premise is invalid; Anthropic regression safety is already covered
  by the OpenAI-compatible surface that Epic 5.1 protects.

**If native Anthropic support is ever added** (a real native-Messages-API decode path in
`internal/llmclient`), re-open this concern as a fresh epic scoped to that adapter — not
this stub, whose premise assumed the path already existed.
