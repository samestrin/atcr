## Objective

Implement `atcr benchmark run` — load + validate a suite, execute each case's diff through
the review pipeline (via Epic 10.1's ingestion entry), score the findings against each
case's planted-defect categories, and emit the suite-tagged `benchmark.RunResult` JSON that
`atcr benchmark export` already consumes.

## Context

Epic 10.0 shipped everything around `benchmark run` but not `run` itself: the suite manifest
loads and validates (`internal/benchmark/benchmark.go` `Load`/`Validate`), the reproducibility
hash is computed (`ReproHash`), and `export` already reads a `RunResult` via `--in` it cannot
yet produce (`cmd/atcr/benchmark.go:76` — "produced by `atcr benchmark run` (Epic 10.1)").
The `RunResult{Suite, SuiteVersion, GeneratedAt, Reviewers []scorecard.PublicRecord}` struct
and `BuildSubmission` already exist.

With Epic 10.1 supplying the diff-file ingestion entry, the only remaining pieces are the
command shell in `cmd/atcr` and the scorer in `internal/benchmark`. `internal/scorecard` is
reference-only here — the scorer emits `scorecard.PublicRecord` and re-scrubs via
`scorecard.ScrubPublicRecord` (`internal/scorecard/export.go:154`) but does not modify that
package.

## Problem Statement

1. `atcr benchmark export` is wired to a run-result file that nothing in the tree can
   generate — the public-submission path dead-ends without `run`.
2. There is no scorer that turns a review's findings into a comparable per-reviewer score
   (precision/recall against a case's expected categories) for the leaderboard.
3. The suite-execution loop (load → per-case ingest+execute → aggregate) does not exist.

## Proposed Solution

### Phase 1 — Scorer against expected categories (`internal/benchmark`) (1.5 days)

- Given a case's findings and `Case.ExpectedCategories` (the validated planted-defect set),
  compute per-reviewer precision/recall/corroboration folded into the
  `scorecard.PublicRecord` fields the public envelope already carries.
- Trust the manifest contract: `Validate` already rejects empty/duplicate categories, so the
  scorer matches on the normalized set.
- Re-scrub on the way out: apply `scorecard.ScrubPublicRecord` to each record (defense in
  depth, same pass `BuildSubmission` applies) before it lands in a `RunResult`.

### Phase 2 — `atcr benchmark run` subcommand (`cmd/atcr`) (1.5 days)

- `atcr benchmark run --suite-path <dir> [--out <run-result.json>]`, mirroring the
  `verify`/`export` flag style in `cmd/atcr/benchmark.go`.
- Flow: `Load` + `Validate` the manifest → for each `Case`, ingest its diff via Epic 10.1's
  exported `internal/fanout` entry, `ExecuteReview` against the roster, collect findings →
  score (Phase 1) → aggregate into `benchmark.RunResult` → write JSON.
- `GeneratedAt` is injected (not `time.Now`) so runs are reproducible, matching
  `BuildSubmission`'s `submittedAt` convention.

### Phase 3 — Round-trip test + docs (1 day)

- End-to-end `run` test driven by the existing `internal/benchmark/testdata/suite-valid/`
  fixture with a stub `Completer` (no network); assert the emitted `RunResult` is consumed
  unchanged by `benchmark export --in` (reuse `internal/benchmark/testdata/run-result.json`
  as the expected shape).
- Document the `run → export` flow in the benchmark README; note the public board accepts
  only `source == "benchmark-suite"` (`SourceBenchmarkSuite`) and that curated `standard-v1`
  content is external.

### Recorded 2026-06-25 (rubber duck — /execute-epic)

**Key Decisions:**
- The benchmark scorer populates the existing `scorecard.PublicRecord.CorroborationRate` field
  with per-reviewer **category recall**: the fraction of a case's `expected_categories` for
  which the reviewer raised ≥1 finding whose normalized category matches, averaged across
  cases. This is the "proxy" the risk table sanctions ("consistent with 10.0's corroboration
  caveat"); the fixture `run-result.json` already populates this field. The repurpose is
  documented in the scorer doc comment and the benchmark README so the column is not silently
  overloaded (production corroboration vs. benchmark recall; disambiguated by `source ==
  "benchmark-suite"`).
- `findings_raised_avg` = mean findings per case; `runs` = number of cases scored per reviewer;
  `cost_per_corroborated_finding_usd` / `latency_p50_ms` derive from pool usage (0 under the
  stub Completer). `PublicRecord.Persona` = reviewer/agent name, `Model` from the pool summary;
  both scrubbed via `ScrubPublicRecord`.
- **Precision is intentionally NOT computed or persisted.** Precision against
  `expected_categories` is misleading by design: `expected_categories` is the planted-defect
  *subset*, not exhaustive ground truth, so precision-vs-planted penalizes thoroughness (a
  reviewer finding 20 real issues incl. the 3 planted scores worse than one finding only the
  3). Recall measures "did you catch the planted defects"; `findings_raised_avg` already
  exposes volume without the misleading penalty.

**Scope Boundaries:**
- IN scope: scorer in `internal/benchmark`, `benchmark run` in `cmd/atcr`, round-trip test +
  README `run → export` section.
- OUT of scope (confirmed untouched): `internal/scorecard` package — `PublicRecord` stays the
  single versioned public reviewer schema shared with production `leaderboard --export` and the
  external leaderboard site (Epic 15.3); adding precision/recall fields would bump a public
  schema version and pollute production submissions with meaningless fields. `benchmark.RunResult`
  stays byte-compatible with the existing fixture (no new fields). `verify`/`export`/`Submission`
  envelope frozen.

**Technical Approach:**
- `run` flow: `Load`+`Validate` → per case: read diff bytes → `fanout.PrepareReviewFromDiff` →
  `fanout.ExecuteReview` → read findings from the pool artifacts + `ReadPoolSummary` for per-agent
  model/tokens/latency → score → aggregate per reviewer into `scorecard.PublicRecord` →
  `ScrubPublicRecord` → `benchmark.RunResult` → write JSON.
- The `Completer` is injectable: the cobra `RunE` calls an internal run function with
  `llmclient.New()` (real, network); the round-trip test calls the same function with a stub
  `Completer` (no network). `GeneratedAt` is an injected parameter (not `time.Now`), so two runs
  over the same suite + transcript are byte-identical.
- Category match is case-insensitive + whitespace-trimmed on both sides, matching
  `reconcile.ModalCategory` normalization (`strings.ToLower(strings.TrimSpace(...))`).

**Original Questions and Answers:**
1. Q: Should `corroboration_rate` carry per-reviewer category recall (the proxy)?
   A: Yes — recall → `corroboration_rate`.
2. Q: Should `scorecard.PublicRecord` be made in scope to add precision/recall fields?
   A: No — keep `scorecard` and `RunResult` untouched; drop precision (misleading vs. a planted
   subset); recall lands in `corroboration_rate`, volume in `findings_raised_avg`.
