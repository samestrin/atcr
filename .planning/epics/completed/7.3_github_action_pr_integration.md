# Epic 7.3: GitHub Action / PR Integration

**Status:** Draft (stub — created 2026-06-19 to track a gap surfaced in epic reconciliation)
**Created:** 2026-06-19
**Priority:** High (the primary delivery surface for findings; differentiator)
**Estimated time:** 1 week
**Tasks/Components:** 3/1

## Why this exists

Epic 7.0 (Executor Model for Fix Generation) Phase 4 was written as "**update** the GitHub
Action to post inline comments with fixes" — but **ATCR has no GitHub Action today**. The
repo has only `.github/workflows/ci.yml` (its own CI); there is no `action.yml`, no
published Action, and no PR-comment surface. 7.0 assumed an artifact that does not exist, so
the PR-delivery work was unowned. This epic owns it.

## Problem

ATCR produces excellent findings artifacts (`reconciled/findings.txt`, 9-column format) but
there is no first-class way to surface them on a GitHub pull request. Teams that want
"review my PR" must wire up their own glue. Without a maintained Action, the findings — and
the `FIX` column that Epic 7.0 populates — never reach the developer where they work.

## Goal

Ship a maintained, composable GitHub Action that runs `atcr review` + `atcr reconcile` on a
pull request and surfaces the result as a PR check and (optionally) inline comments. When
Epic 7.0's executor has populated the `FIX` column, the Action renders the fix inline —
the "like lgtmaybe" experience, on top of ATCR's multi-model panel.

## Scope (draft)

### In Scope

- An `action.yml` (composite or Docker action) that installs/locates the `atcr` binary, runs
  `atcr review --output-dir … <range>` + `atcr reconcile`, and parses `reconciled/findings.txt`.
- A summary PR check (pass/fail honoring `--fail-on`) with a findings table.
- Optional **inline comments** keyed off `FILE:LINE`, rendering `PROBLEM` and, when present,
  the `FIX` column (Epic 7.0) and executor attribution.
- A toggle to post artifacts only (no inline comments).
- An example workflow in `docs/` and a real-PR integration test.

### Out of Scope

- Fix **generation** (Epic 7.0 owns the `FIX` column; this epic only renders it).
- Hosting/SaaS; non-GitHub forges (GitLab/Bitbucket) — separate epics if demanded.
- Auto-applying fixes / opening fix PRs.

## Acceptance Criteria (draft — refine before scheduling)

- [x] AC1: `action.yml` exists; a workflow can call it on `pull_request` and it runs `atcr review` + `atcr reconcile`.
- [x] AC2: The Action posts a PR check summarizing findings and honoring `--fail-on` for the merge gate.
- [x] AC3: Inline comments render `PROBLEM` at `FILE:LINE`; when the `FIX` column is populated (Epic 7.0), the fix and executor attribution are shown — format "ATCR found: [problem]. Fix: [fix]. Suggested by: [executor]".
- [x] AC4: A toggle disables inline comments (artifacts/check only).
- [x] AC5: Real-PR integration test demonstrates the end-to-end flow.

## Dependencies

- **Epic 7.0 (Executor Model)** — supplies the `FIX` column this Action renders (soft: the Action also works on findings without fixes, posting `PROBLEM` only).
- A published/locatable `atcr` binary (release artifact or build step in the Action).

## Clarifications

### Recorded 2026-06-22 (rubber duck — /execute-epic)

**Key Decisions:**
- PR-rendering logic lives in a new Go subcommand `atcr github` (wrapped by a thin composite `action.yml`), NOT bash glue — so all non-trivial logic stays inside `go test ./...` coverage.
- The subcommand reuses the existing structured reader `reconcile.ReadReconciledFindings` (`internal/reconcile/emit.go:235`) over a review dir rather than reimplementing the pipe parser; this yields `JSONFinding` records (incl. `Evidence`).
- The composite action builds the binary via `actions/setup-go@v5` (go-version `1.25`, cache) + `go build ./cmd/atcr` — no goreleaser/release artifacts (out of scope).
- The action runs the full pipeline (`atcr review` + `atcr reconcile`) with the OpenRouter API key as a required input.
- Executor attribution for AC3 is parsed from the `; fix by <name>` token embedded in the `EVIDENCE` field (written by `appendFixAttribution`, `internal/verify/executor.go`); the "Suggested by" clause is omitted when the token is absent. There is no 10th column.
- Inline comments default OFF (opt-in via an `inline-comments` input, default `false`); the PR check + artifact are always produced.
- `action.yml` lives at the repo root so consumers reference `uses: samestrin/atcr@<ref>`.

**Scope Boundaries:**
- IN scope: `action.yml` (composite), the `atcr github` Go subcommand, a PR check honoring `--fail-on`, opt-in inline comments rendering PROBLEM + FIX + (when present) executor attribution, an `inline-comments` toggle, an example workflow under `docs/`, and Go tests against an `httptest`-backed fake GitHub API.
- OUT of scope: fix generation (Epic 7.0), auto-applying fixes / fix PRs, non-GitHub forges, release artifacts/goreleaser, and any live external-PR posting.

**Technical Approach:**
- AC5 ("real-PR integration test") is satisfied per the repo's established pattern — `net/http/httptest` fake GitHub server in Go tests + a committed `on: pull_request` example workflow + a documented manual smoke-test procedure. No live network calls (consistent with all existing integration/e2e tests; CI has only `contents: read`).
- GitHub interactions (check-run create/update, PR review comments) are made from Go via `net/http` against the GitHub REST API using `GITHUB_TOKEN`; no new heavy SDK dependency unless required.

   A: (a) New Go subcommand `atcr github`/`atcr pr-comment` wrapped by a thin composite action; reuse `internal/reconcile` parser; keeps logic in `go test` coverage.
2. Q: How does the action obtain the `atcr` binary?
   A: (a) `actions/setup-go@v5` + `go build ./cmd/atcr`, replicating the existing CI step; no releases/goreleaser (out of scope).
3. Q: How is AC5's "real-PR integration test" satisfied?
   A: (a) Example `on: pull_request` workflow + Go tests against an `httptest` fake GitHub API + documented manual smoke test; no live PR (CI has only `contents: read`).
4. Q: Where does executor attribution (AC3 "Suggested by") come from?
   A: (a, refined) Parse the `; fix by <name>` token from the `EVIDENCE` field (Epic 7.0 already landed it there); omit when absent. No new column.
5. Q: Inline-comment default — OFF or ON?
   A: Default OFF (opt-in) via an `inline-comments` input (default `false`).
6. Q: Does the Action run `atcr review` itself or consume pre-produced artifacts?
   A: Runs the full pipeline (`atcr review` + `atcr reconcile`) with the API key as a required input.
7. Q: Publish path — root `action.yml` or subdirectory?
   A: Top-level `action.yml` at repo root (`uses: samestrin/atcr@<ref>`).
