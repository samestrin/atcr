# Feature Request: Expand Brace-Language AST Parsers (Java, Kotlin, C/C++, C#)

- **Estimated time**: 4 days
- **Tasks/Components**: 6/1

## Context

Epic 13.1 introduced the AST-isomorphism plugin architecture (`wazero` runtime host, line-to-node mapping, Merkle hashing) to group duplicate code review findings across line-number drift. Epic 13.4 expanded this via a generic, parameterized `braceparser` to cover TypeScript, PHP, Rust, and Bash. This epic (13.6) leverages the existing `braceparser` infrastructure to add support for the next four major brace-based languages (Java, Kotlin, C/C++, C#).

## Problem Statement

- **Coverage Gap**: Findings in `.java`, `.kt`/`.kts`, `.c`/`.cpp`/`.cc`/`.cxx`/`.h`/`.hpp`, and `.cs` currently fall back to the $\pm3$ line-proximity gate.
- **Brittle Grouping**: The proximity gate under-groups (misses duplicate findings that drift by >3 lines) and over-groups (merges unrelated adjacent findings), resulting in noisy reports for projects using these languages.

## Proposed Solution

1. **Per-Language Configurations**: Define syntax rules (comments, string delimiters, raw/triple-quoted string syntax, char literals, and keywords) for Java, Kotlin, C/C++, and C# in [configs.go](file:///Users/samestrin/Documents/GitHub/atcr/internal/astgroup/parsers/src/braceparser/configs.go), with matching per-language unit tests in `configs_test.go`.
2. **Shared Scanner Extensions** (two small, language-agnostic additions to [parse_core.go](file:///Users/samestrin/Documents/GitHub/atcr/internal/astgroup/parsers/src/braceparser/parse_core.go), gated/shared — no per-language control flow):
   - **Trailing-identifier `funcParen` naming**: rework `funcParenName` to take the trailing identifier run before `(` instead of requiring the whole pre-paren header to be a single token, so methods carrying modifiers/return types (`public void execute() {`, `void Foo::bar() {`) are named `execute` / `bar` rather than falling through to an anonymous `block`. Keyword precedence (`classifyHeader` checks keywords first) means existing TS/PHP/Rust/Bash behavior is unchanged; the only new behavior is naming previously-anonymous keyword-less method headers. Add `parse_core_test.go` cases, including a guard that a keyword-less call-shaped header (`foo.bar()`) still does the right thing. **Known limitation (acceptable — block-level grouping still works):** `funcParenName` only fires when the header ends in `)`, so signatures with trailing tokens after the parameter list — Java `throws IOException`, C++ `const`/`noexcept`/`override`/trailing-return `-> int`, C# `where T : ...` — remain anonymous `block` (unnamed, but still a covering block). Pick `corpus.json` cases accordingly; do not assert a recovered method name on those shapes.
   - **`"""` triple-quoted-string state**: add a `tripleQuote bool` config flag and a `stTripleString` scanner state so Kotlin multiline strings, Java text blocks, and C# raw string literals are opaque (their braces/quotes never skew block depth). Data-gated: only languages that set the flag pay for it.
3. **Build Targets**: Add compile-time build tag files (`active_java.go`, `active_kotlin.go`, `active_cpp.go`, `active_csharp.go`) in `braceparser`.
4. **Compilation**: Update [build.sh](file:///Users/samestrin/Documents/GitHub/atcr/internal/astgroup/parsers/build.sh) to compile the four new `.wasm` targets (`java.wasm`, `kotlin.wasm`, `cpp.wasm`, `csharp.wasm`) and extend the `SHA256SUMS` filename list to all ten `.wasm`.
5. **Host Integration**: Embed the new `.wasm` files via `go:embed`, add them to `builtinParsers`, and map their extensions in `LanguageForExt` in [embed.go](file:///Users/samestrin/Documents/GitHub/atcr/internal/astgroup/embed.go); extend `host_test.go` (`TestLanguageForExt` per-extension asserts and the `TestHost_BraceParsersLoadAndParse` `cases` slice) for the four new languages, and add the four languages to the README's AST-grouping coverage list.
6. **Accuracy Verification**: Update `testdata/corpus.json` with labeled drift test cases for these languages to assert grouping accuracy. Populate only on the supported surface (see Out of Scope) — e.g. method-body drift, not findings inside C++ raw / C# verbatim strings.

## User Journeys

### Reconciling Java Code Review
1. A developer runs `atcr review && atcr reconcile` on a Java codebase.
2. Reviewer A flags line 50 (`public void execute()`); Reviewer B flags line 53 (inside the same method) due to line drift or minor diff context.
3. The reconciler maps both to the `execute` block node and correctly groups them into a single, high-confidence finding rather than two separate issues.

### Handling C/C++ Header/Implementation Drift
1. A developer updates a C++ class definition.
2. Model findings drift due to header include blocks or macro expansion.
3. The C/C++ AST parser maps findings to their enclosing struct/class or function block, ensuring accurate clustering.

## Non-Functional Requirements

### Performance
- Module instantiation must be cached per language so repeated parsing of the same language is fast (<10ms overhead).

### Security
- No cross-compilation or native binary dependencies (zero-CGO constraint).

## Out of Scope

- Keyword-based languages like Ruby or Python (Python uses `pyparser`; Ruby is out of scope).
- Handling complex C-preprocessor directives (`#ifdef` blocks that structurally change brace alignment). Standard brace matching is sufficient for heuristic grouping.
- Backslash-continued / multi-line C preprocessor macros (`#define FOO ... \` spanning lines). `#` is treated as a line comment (per the Risks table), which covers single-line directives only.
- C++ raw string literals (`R"delim(...)delim"`) and C# verbatim strings (`@"..."` with `""` escaping). These have no data-only representation and differ from the existing Rust `rawStrings` machinery; findings inside them degrade to $\pm3$ line proximity (a misparse never breaks reconcile). Note: triple-quoted strings (`"""`) ARE supported via the `tripleQuote` scanner state (AC1).
- Semantic type resolution or type-aware analysis.

## Known String-Form Limitations (all brace parsers)

The `braceparser` is a heuristic structural parser for GROUPING ONLY — a misparse degrades a finding to $\pm3$ line proximity and can never break a reconcile. Some exotic string forms are intentionally not modeled and fall back to proximity. This applies family-wide, not just to the four new languages:

- **Existing (TS/JS, PHP, Rust, Bash):** deeply nested template-literal interpolation (`` `…${ `…` }…` ``) can mis-pair backticks; these were accepted limitations in 13.4.
- **New (this epic):** C++ raw strings (`R"(...)"`) and C# verbatim strings (`@"..."`) are out of scope (see Out of Scope). Triple-quoted `"""` IS handled.

This note exists so the data-only / minimal-scanner posture is honest: coverage is "the common brace/comment/keyword/string surface," not "every string-literal grammar."

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| C/C++ preprocessor directives (`#define`, `#ifdef`) cause mismatched/skewed brace counts | Medium | Low | Skip `#` lines during tokenization or treat them as line comments; on parse failure, degrade gracefully to $\pm3$ line proximity. |
| Kotlin string interpolation/raw strings miscount braces | Low | Medium | `$`/`${...}` interpolation inside `"..."` is already opaque via the scanner's string state (braces ignored). Raw/multiline `"""..."""` is made opaque by the new `tripleQuote` scanner state (AC1, Proposed Solution item 2), so its braces never open/close a block. |
| Trailing-identifier `funcParen` change regresses an existing brace language (TS/PHP/Rust/Bash) | Low | Medium | Keyword precedence in `classifyHeader` means the change only affects keyword-less method headers; bare `name()` headers (TS/Bash) resolve identically. Guard with `parse_core_test.go` cases for the existing languages plus a call-shaped header (`foo.bar()`). |

### Items needing user confirmation (3)

⏸️ **1. The "data-only / no new parser engine code" claim does not hold for several of the new languages' string forms.** AC1 ("no new parser engine code"), the Technical Constraint ("Guest code changes in `braceparser` should remain data-only in `configs.go`"), and the Kotlin risk mitigation all assume the shared scanner can be parameterized purely by data for all four languages. Verified against `internal/astgroup/parsers/src/braceparser/parse_core.go`: `langConfig.strChars` is a set of **single-byte** delimiters, and `rawStrings` is hardcoded to Rust `r"..."` / `r#"..."#` syntax (`rawStringStart` requires a leading `r`). There is **no** representation for: triple-quoted strings `"""..."""` (Kotlin multiline strings, Java text blocks, C# raw string literals), C++ raw strings `R"delim(...)delim"`, or C# verbatim strings `@"..."` (with `""` escaping). Supporting any of these requires new control flow in `parse_core.go`, which contradicts the data-only claim and AC1. **Suggested resolution (recommended):** declare these string forms OUT OF SCOPE — a misparse only degrades a finding to ±3 line proximity and can never break reconcile, and this mirrors the 13.4 Bash "function-level granularity only" precedent. Then (a) correct AC1 to "no new parser engine code for the supported brace/comment/keyword surface," (b) add the unsupported string forms to Out of Scope, and (c) populate `corpus.json` only with cases on the supported surface. **Alternative:** explicitly add a `parse_core.go` scanner task (multi-char string delimiters) and drop the unqualified "no new parser engine code" wording — this widens guest-code scope beyond data-only.

⏸️ **2. The Kotlin risk-table mitigation is not achievable as written.** Risk row reads: "Kotlin string interpolation/raw strings miscount braces → Mitigation: Define correct string delimiters (`\"`, `\"\"\"`) ... in the config table." `"""` cannot be defined in the current `strChars` (single-byte set) — the mitigation presumes a scanner capability that does not exist (see item 1). Also note the interpolation half is already safe: `$var` / `${...}` inside a normal `"..."` string is consumed by string state, so its braces are already opaque. **Suggested rewrite:** "`$`/`${…}` interpolation inside `\"…\"` is already handled by string state (braces opaque); `\"\"\"` raw/multiline strings are NOT handled and degrade to ±3 proximity (acceptable — a misparse never breaks reconcile)."

⏸️ **3. C++ out-of-line method definitions will group at block granularity, not as named functions.** User Journey 2 ("C/C++ … maps findings to their enclosing struct/class or function block") implies named-function recovery. But `funcParenName` in `parse_core.go` requires the text before `(` to be a single all-identifier token; `void Foo::bar() {` contains `::`, so it falls through to an anonymous `block` rather than a named `func`. Findings still GROUP (block-level), so this is a precision/naming limitation, not a correctness break. **Suggested action:** note in the plan (or a risk row) that out-of-line C++ member definitions group at block granularity unless `funcParen` handling is extended, and populate `corpus.json` accordingly — OR accept as a documented limitation.
