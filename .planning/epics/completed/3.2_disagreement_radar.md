# Epic Plan 3.2: Disagreement Radar

**Estimated Durations**: 1.5 weeks

## Objective

Promote reviewer disagreement from residue to a first-class output: a "radar" report surface that points human reviewers at the highest-tension spots in a change — severity splits, solo findings from strong models, and gray-zone ambiguous clusters. Almost entirely a presentation/projection over data the reconciler already produces.

## Context

The reconciler writes `ambiguous.json` for gray-zone clusters and keeps severity conflicts inline rather than averaging them away. But that preserved signal is currently treated as overflow — the stuff the deterministic merge couldn't resolve, punted to a human or (later) to cross-examination. This undersells it. Where two strong, independent models disagree on a diff is exactly where the ambiguity, the subtle risk, and genuinely interesting design tension lives. Epic 6.0 (Cross-Examination) will resolve these disputes, but without this epic, 6.0 has no visible input queue and no "before" story.

This epic also receives richer data from Epic 3.0: skeptic-vs-finder disagreements that collapsed to `unverifiable` in verification.json can feed the radar as an additional tension class.

## Problem Statement

1. `ambiguous.json` contains signal that is never surfaced to the user — it exists as an implementation artifact, not a readable output.
2. Severity conflicts are preserved inline in `reconciled/findings.txt` as `disagreement:` annotations but no summary view calls attention to them.
3. 6.0 (Cross-Examination) resolves disagreements but has no stable input queue or visible pre-resolution view.

## Proposed Solution

### What the radar surfaces

- **Severity splits** — reviewer A says CRITICAL, reviewer B says LOW on the same location. The spread is the signal.
- **Solo findings** — a finding only one strong reviewer raised (not noise to discard, a spot the others may have missed).
- **Gray-zone clusters** — existing `ambiguous.json` contents, presented as insight rather than overflow.
- **Verification disagreements** (post-3.0) — findings where skeptic verdict was `unverifiable` due to conflicting skeptic votes.
- **Persona tension** — correctness reviewer and design reviewer pulling in opposite directions on the same hunk.

### Surfaces

- `atcr report --disagreements` — focused view: ranked list of the N highest-tension spots in the diff, each with model positions side by side.
- **Radar section in standard `report.md`** — top disagreements surfaced above the consensus findings. Section is empty/collapsed when there are no meaningful disagreements.
- **Cross-exam handoff schema** — stable format so Epic 6.0 consumes the radar as its item queue directly, without re-parsing ambiguous.json.

### Scoring

Rank by: severity spread (tier distance between highest and lowest severity on the same location) × reviewer independence (down-weight near-duplicate models). Solo findings from high-confidence reviewers rank above disagreements among weak ones.

## Acceptance Criteria

- [x] `atcr report --disagreements` produces a focused ranked list of disagreement items (severity splits, solo findings, gray-zone clusters).
- [x] The standard `report.md` includes a "Disagreements" section above consensus findings; section is omitted/collapsed when the list is empty.
- [x] Disagreement items are ranked by severity spread × reviewer independence; ranking is deterministic (same input → same order).
- [x] The cross-exam handoff schema is defined and stable — Epic 6.0 can consume it without further changes to this epic's output.
- [x] Verification disagreements (`unverifiable` by conflicting skeptic votes from 3.0) appear in the radar when verification.json is present.
- [x] Existing `report.md` output and `reconciled/` artifacts are unchanged for reviews with no disagreements.
- [x] Docs: report.md section documented; `--disagreements` flag documented in `atcr report --help`.

## Task Breakdown

1. **Disagreement scoring**: rank existing `ambiguous.json` clusters and inline severity conflicts by spread × independence.
2. **`--disagreements` report mode**: focused side-by-side rendering of top-tension items.
3. **Radar section in `report.md`**: inject above consensus findings; collapse/omit when empty.
4. **Verification disagreement intake** (post-3.0 data): when `reconciled/verification.json` present, fold `unverifiable`-by-disagreement entries into the radar.
5. **Cross-exam handoff schema**: define stable JSON schema for Epic 6.0 consumption; write to `reconciled/disagreements.json`.
6. **Docs and tests**: report --disagreements flag, section format, handoff schema contract.

## Out of Scope

- Resolving disagreements (Epic 6.0).
- Changing the reconciler's clustering or dedup logic.
- Auto-tuning the ranking weights.
- UI beyond the existing report.md and CLI flag.

## Dependencies

- Epic 1.0 reconciler (`ambiguous.json`, inline severity conflict annotations) — complete.
- Epic 3.0 verification.json — optional; radar works without it and gains the verification-disagreement tier when 3.0 data is present.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Disagreement is mostly noise, not signal | Medium | Medium | Rank by reviewer strength and independence, not raw count; only surface splits among credited reviewers |
| Radar section adds noise for simple diffs | Low | Low | Section collapsed when empty; threshold for inclusion configurable |
| Cross-exam handoff schema becomes a compatibility surface | Low | Medium | Version the schema from day one; document as the Epic 6.0 contract |

## Clarifications

### Recorded 2026-06-14 (rubber duck — /execute-epic)

**Key Decisions:**
- Independence factor = count of distinct reviewers on a finding (the deduplicated `Reviewers []string`). No model-strength/near-duplicate-model dampening — that data does not exist. Documented as a v1 proxy in the handoff schema.
- Solo findings are scored by their own severity rank (CRITICAL=4 … LOW=1) on the same numeric axis as severity spread (spread range 1–3), so a CRITICAL solo can outrank a small LOW-vs-MEDIUM split.
- Inclusion threshold is a fixed hardcoded default (severity spread ≥ 1 tier, all gray-zone clusters, all verification ties). No config.yaml plumbing this epic — follows the project precedent (Epics 4.1/4.5).

**Scope Boundaries:**
- IN scope: disagreement scoring, `atcr report --disagreements` mode, radar section in report.md (above consensus findings, omitted when empty), verification-disagreement intake, `reconciled/disagreements.json` versioned schema, docs + tests.
- OUT of scope: persona tension (no persona/role metadata exists; absent from ACs and Task Breakdown), resolving disagreements (Epic 6.0), changing clustering/dedup, auto-tuning weights, config-driven thresholds, any UI beyond report.md + CLI flag.

**Technical Approach:**
- Severity splits sourced from `Merged.Disagreement` ("lo vs hi"); spread = severityRank[hi] − severityRank[lo].
- Solo findings = merged findings with exactly one distinct reviewer (Confidence MEDIUM).
- Gray-zone clusters = `ambiguous.json` contents.
- Verification disagreements = `verification.json` entries with Verdict "unverifiable" from a skeptic-vote tie; scored by the finding's own severity rank.
- `disagreements.json` carries a `schemaVersion` field from day one as the Epic 6.0 contract.
