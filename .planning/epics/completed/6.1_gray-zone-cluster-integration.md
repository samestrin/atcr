# Tech Debt: Inline application of judge gray-zone cluster rulings (Epic 6.1)

- **Estimated time**: TBD (multi-package integration)
- **Tasks**: TBD
- **Components**: internal/debate, internal/reconcile

## Current State

In the cross-examination stage (Epic 6.0), when the judge rules on a gray-zone
cluster (merge / separate), the decision is recorded into `reconciled/debate.json`
but is **not** physically applied to `findings.json`. The guard at
`internal/debate/debate.go` deliberately excludes gray-zone items from the
per-finding `rulings` map:

```go
// Apply verdicts only to single-finding items. A gray-zone ruling is a
// cluster-level merge/separate decision recorded for the existing
// adjudication path, not a per-finding verdict (see Clarifications).
if ir.Outcome != OutcomeUnresolved && it.Kind != reconcile.KindGrayZone {
    rulings[FindingKey{...}] = ruleApply{...}
}
```

Gray-zone clusters therefore continue to resolve through the **existing
adjudication path** (`internal/reconcile/ambiguous.go`, `gate.go`, `dedupe.go` —
the `adjudication.json` mechanism), not inline from the judge's ruling.

## Pain Points

- Unattended/CI runs that produce a judge cluster decision still require the
  separate adjudication path to physically merge/separate the cluster in
  `findings.json`; the judge's `ClusterDecision` is audit-only.
- The two mechanisms (judge ClusterDecision vs. `adjudication.json`) encode the
  same merge/separate intent in two places.

## Why This Is a New Epic (not a TD quick-fix)

This was deferred from the `/resolve-td` group-1 run on 2026-06-21 (TD item
`internal/debate/debate.go:165`). The clarification answer (82% confidence) was:

> Keep recording gray-zone cluster rulings in debate.json for the existing
> adjudication path. The `it.Kind != reconcile.KindGrayZone` guard is a BINDING
> design decision enshrined in both the code comment ("see Clarifications") and
> the epic plan — the AC "Unattended CI run resolves ambiguous clusters without
> Skill involvement" is already VERIFIED under this behavior.

Wiring `ClusterDecision` inline would reverse that recorded clarification and is
a **multi-package change** (debate → reconcile adjudication path), which exceeds
the debate-package TD boundary and warrants its own scoped plan.

## Proposed Solution (to be designed)

Wire the judge's `ClusterDecision` (recorded per gray-zone item in
`reconciled/debate.json`) into the reconcile adjudication application so an
unattended run can auto-merge/separate gray-zone clusters inline, without a
separate `adjudication.json` round-trip.

### Likely touch points

- `internal/debate/debate.go` — emit cluster decisions in a form the reconcile
  adjudication path can consume (or call into it directly after the debate loop).
- `internal/reconcile/ambiguous.go` — `Adjudication` / `Decision` model and
  `LoadAdjudication` validation.
- `internal/reconcile/gate.go` — the adjudication re-invocation / apply path.
- `internal/reconcile/dedupe.go` — `dedupeCluster` merge application.

## Scope

### In Scope
- Translate judge `ClusterDecision` (merge / distinct / skipped) into applied
  cluster state in `findings.json`.
- Preserve the existing `adjudication.json` path and its audit guarantees
  (baseline hash, original sidecar preservation).

### Out of Scope
- Changing the judge protocol or ruling envelope.
- Single-finding verdict application (already handled in Epic 6.0).

## Open Design Questions

- Does inline application replace the `adjudication.json` path for gray-zone
  clusters, or run alongside it (and how are conflicts resolved)?
- How is the baseline-hash / re-run idempotency guarantee preserved when the
  decision originates from debate rather than from an authored `adjudication.json`?
- Does reversing the 6.0 clarification require updating the Epic 6.0 AC
  "Unattended CI run resolves ambiguous clusters without Skill involvement"?

## Acceptance Criteria

- [x] AC1: A judge gray-zone `merge` ruling physically unions the cluster's
  findings in `findings.json` on an unattended run (no `adjudication.json`).
- [x] AC2: A judge `distinct`/`separate` ruling leaves the cluster unmerged.
- [x] AC3: The existing `adjudication.json` path continues to work unchanged for
  inputs that do not carry a judge cluster decision.
- [x] AC4: Re-run idempotency holds — a second debate run does not re-merge or
  corrupt an already-applied cluster.
- [x] AC5: The Epic 6.0 clarification / AC is reconciled with the new behavior
  (updated or explicitly superseded).

## Clarifications

### Recorded 2026-06-21 (rubber duck — /execute-epic)

**Key Decisions:**
- **Option A locked.** Debate applies a gray-zone `merge` decision **in-memory to
  `findings.json`** — union the cluster's member findings by location and re-emit inside
  debate's existing atomic group (debate.json + findings.json + manifest.json). It does
  **not** call `RunReconcile`.
- **Option B rejected.** Synthesizing `adjudication.json` and re-invoking `RunReconcile`
  would rebuild `findings.json` from `sources/`, **erasing** the verify verdicts (Epic 3.0)
  and single-finding debate verdicts (Epic 6.0) already written into `findings.json`.
- **Union ALL cluster members, not a hardcoded 2-finding pair.** Forward-compatible with
  Epic 13.2 (DBSCAN produces variable-size clusters). Today `AmbiguousCluster` is a pair;
  the apply must not assume that.
- **Idempotency (AC4):** a dedicated finding-level flag (e.g. `cluster_merged: true`) on
  the surviving merged `JSONFinding`, mirroring `Verification.ChallengeSurvived`, filtered
  from the radar/selection on re-run — a cluster analog of `filterAlreadyDebated`.
  `debate.json` continues to record the `ClusterDecision` for the audit trail.
- **Third-finding wrinkle handled inline:** when a gray member was already merged with a
  third finding (≥0.7), union all records the cluster spans at its location(s). Not punted
  to the authored-adjudication path.
- **No `AmbiguousID` reconstruction or `baseline_hash` synthesis needed** — Option A applies
  by location on the live record, bypassing the ID-keyed adjudication MergeSet entirely.

**Scope Boundaries:**
- IN scope: applying judge gray-zone `merge`/`separate` to `findings.json` inline in the
  debate stage; the finding-level idempotency flag; the Epic 6.0 AC reconciliation note.
- OUT of scope: changing the judge protocol / ruling envelope; single-finding verdict
  application (Epic 6.0); rewriting the similarity scoring or clustering algorithm (that is
  Epic 13.0 / 13.2); changing the authored `adjudication.json` mechanism (stays intact for
  AC3).

**Technical Approach:**
- Adjust the `it.Kind != reconcile.KindGrayZone` guard at `debate.go:184` so a gray-zone
  `merge` decision drives the new in-memory union path (NOT the per-finding `rulings` verdict
  map, which stays single-finding-only).
- The apply unions `findings.json` records at the cluster's location(s): merged PROBLEM =
  longest, severity = max, reviewer set = union, verification block preserved/merged.
- `merge` → union; `separate` → leave unmerged. Re-runs skip already-flagged clusters.
- The authored `adjudication.json` path (`ambiguous.go` / `gate.go` / `dedupeCluster`) is
  untouched — AC3 holds for free.

**13.* Roadmap Compatibility:**
- Option A attaches to the clustering **output** (the gray-zone radar item + `findings.json`),
  not the scoring/clustering **internals** that Epic 13.0 (NCD scoring) and Epic 13.2
  (bipartite + DBSCAN) rewrite. Those epics can swap the clustering engine without touching
  6.1's apply logic. Nothing in the 13.* roadmap is blocked or lost. (Option B would have
  coupled 6.1 to `dedupeCluster` / `opts.Merges` / `baseline_hash` — exactly what 13.* rips
  out — and is rejected partly for that reason.)

**Vehicle:** plain `/execute-epic` (no `--strong`). ~4 contained tasks across
`internal/debate` + `internal/reconcile`; the three unconditional adversarial layers cover
the review need for a change this size.
