# Epic Plan 19.1: Audit Trail

- **Estimated time**: 2-3 days
- **Tasks/Components**: 3/2
- **Execution**: execute-epic --strong

## Objective

Emit an append-only audit log per run and generate a one-page compliance report for a given PR.

## Context

Compliance-minded teams need a tamper-evident record of what atcr reviewed and when. The review pipeline has the base/head SHAs and PR number in hand but doesn't persist them as an audit record.

## Task Breakdown

1. Write an append-only `audit.log.jsonl` in the review output directory with run timestamp, base/head SHA, PR number, and findings summary (counts by severity).
2. Add `internal/audit` record type + reader.
3. Add `atcr audit-report --pr <n>` cobra command in `cmd/atcr/audit_report.go` that renders a one-page markdown compliance report for that PR's runs.

## Acceptance Criteria

- [x] Each `review` run appends exactly one audit record.
- [x] `atcr audit-report --pr 1234` renders a report with SHAs, timestamp, and findings summary.
- [x] An unknown `--pr` value exits non-zero with a clear message.
- [x] `go test ./...` passes; covered by `cmd/atcr/audit_report_test.go` + `internal/audit/*_test.go`.

## Components Touched

- `cmd/atcr` (new `audit_report.go`, hook in review output)
- `internal/audit` (new package)

## Out of Scope

Cryptographic signing/notarization; external SIEM export; retention policy.

## Dependencies

- Review output directory + PR-number plumbing (exists).

## Clarifications

### Recorded 2026-07-05 (rubber duck — /execute-epic)

**Key Decisions:**
- The `atcr review` pipeline has NO PR number today (PR numbers live only in `internal/ghaction` via a live GitHub API call, not env). The epic's "PR number in hand" premise is corrected: PR number must be threaded in explicitly.
- Add a `--pr <n>` int flag to `atcr review`, threaded onto `ReviewRequest.PRNumber`; when unset, fall back to parsing `GITHUB_REF` (format `refs/pull/<n>/merge`). No new GitHub API client. Matches atcr's flag-wins/env-fallback convention (`--repo`/`GITHUB_REPOSITORY`, `--token`/`GITHUB_TOKEN`).
- Every review run appends exactly one audit record even when the PR is unresolved; the PR field is stored as `json:"pr,omitempty"` (empty/zero), never skipping the write.
- The audit ledger is a single repo-level accumulator at `.atcr/audit.log.jsonl` via `filepath.Join(req.Root, ".atcr", "audit.log.jsonl")`, written regardless of `--output-dir`, mirroring Epic 19.0's `findings-history.jsonl` pattern (hook near `cmd/atcr/review.go:335-339`).

**Scope Boundaries:**
- IN scope: `--pr` flag + `GITHUB_REF` fallback on `atcr review`; `ReviewRequest.PRNumber`; repo-level `.atcr/audit.log.jsonl` ledger; `internal/audit` package (record type + writer + reader); `atcr audit-report --pr <n>` command rendering one-page markdown.
- OUT of scope: cryptographic signing/notarization; external SIEM export; retention policy; any new GitHub API client for PR resolution.

**Technical Approach:**
- Mirror `internal/history/` layout: record.go (types), writer.go (Append via O_APPEND), reader.go (Load), capture.go (hook), render.go (markdown). Respect `internal/boundaries_test.go` — add the minimal allowed dependency entry for `internal/audit`.
- Severity summary counted over {CRITICAL, HIGH, MEDIUM, LOW}, sourced the same way `internal/history` sources findings from the review dir.
- `audit-report` command modeled on `cmd/atcr/history.go`: `repoRoot()`, `usageArgs`, `cmd.OutOrStdout()`, `usageError` for exit-2; registered in `cmd/atcr/main.go`. Unknown `--pr` (no matching records) exits non-zero with a clear message.

**Original Questions and Answers:**
1. Q: How should a review run obtain the PR number to record?
   A: (c) — add `--pr <n>` flag threaded onto `ReviewRequest.PRNumber`, falling back to parsing `GITHUB_REF` (`refs/pull/<n>/merge`) when unset. No new API client. (Note: epic's claim that this mirrors ghaction env auto-detection is inaccurate — ghaction uses a live API call.)
2. Q: When a run has no PR available, still write the record or skip?
   A: Always write the record with an empty/omitted PR field (`json:"pr,omitempty"`), never skip — matches the unconditional AC wording and the codebase's omitempty-for-optional pattern.
3. Q: Ledger location — single repo-level accumulator or per-review-dir?
   A: (a) — single repo-level `.atcr/audit.log.jsonl`, written regardless of `--output-dir`, mirroring `findings-history.jsonl`.
