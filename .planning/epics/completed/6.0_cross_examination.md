# Epic Plan 6.0: Cross-Examination (Debate Stage)

**Estimated Durations**: 2 weeks  
**Tasks/Components:** 5/2
**Status:** Draft (queued; deps 2.0 + 3.0 complete — ready)

## Objective

Resolve reviewer disagreements through structured multi-turn exchange: where reviewers conflict (severity disputes) or the reconciler's similarity gray zone leaves clusters ambiguous, run a proposer/challenger/judge debate and let the outcome — not a heuristic — settle the finding. Findings that survive hostile challenge form the top confidence tier.

## Context

After Epic 3.0, two classes of items still resolve crudely: severity disagreements (currently severity-max with an inline `disagreement:` annotation) and `ambiguous.json` clusters (currently punted to optional Skill adjudication). Both are exactly the cases where models disagree — which is signal worth extracting, not noise to flatten. Epics 2.0/3.0 provide all the machinery: the tool loop, role plumbing, transcripts, verdict envelopes.

## Problem Statement

1. Severity-max is a blunt rule: one alarmist reviewer inflates the merged severity and the gate.
2. `ambiguous.json` requires a human or the host Skill; unattended runs (CI) leave it unresolved.
3. Skeptic-vs-finder disagreement in 3.0 collapses to `unverifiable` — a debate would often settle it.

## Proposed Solution

- **Triggers:** (a) severity disagreement spanning ≥2 tiers, (b) ambiguous clusters from reconcile, (c) `unverifiable`-by-disagreement verdicts from verify. Config: `debate.triggers`, `debate.max_items` (cost cap, priority by severity).
- **Protocol (bounded, 3 turns):** proposer (a crediting reviewer's model, tool loop available) defends the finding; challenger (different model, `role: skeptic`) attacks; judge (`role: judge`, third distinct model, strongest available) rules with a strict envelope: outcome (uphold / overturn / split), settled severity, merged-or-separate ruling for ambiguous clusters, reasoning.
  - *Single-Key Fallback:* If 3 distinct models are not available or if the user is operating with a single API key, ATCR will fall back to using different system prompts/personas (e.g., `proposer`, `skeptic`, `judge`) on the same model to simulate a debate.
- **Confidence:** upheld-under-challenge becomes the top tier (DEBATED ≥ VERIFIED ordering decided during design — likely merged into VERIFIED with a `challenge_survived` marker to avoid tier proliferation).
- **Artifacts:** `debate/` directory with per-item transcript.jsonl; `reconciled/debate.json`; manifest stage `"debate"`; report.md gains a "Contested findings" section showing rulings with one-line rationales.
- **Command:** `atcr debate [<id-or-path>]`, chained via `atcr review --verify --debate`; MCP tool `atcr_debate`.

## Success Criteria

- [x] Severity disputes ≥2 tiers and ambiguous clusters route through debate; judge rulings replace severity-max and resolve merge/split.
- [x] Three-distinct-models rule enforced across proposer/challenger/judge.
- [x] Bounded protocol: hard 3-turn cap, per-item budget, `debate.max_items` priority ordering with overflow recorded (never silent).
- [x] Unattended CI run resolves ambiguous clusters without Skill involvement. *(Superseded by [Epic 6.1](6.1_gray-zone-cluster-integration.md): as of 6.1 a judge gray-zone `merge` ruling is applied **inline** to findings.json in the debate stage — Option A — instead of only being recorded for the Skill/adjudication path. The "without Skill involvement" guarantee is now stronger; this checkbox stays VERIFIED for the 6.0 behavior as shipped.)*
- [x] Transcripts replayable; rulings auditable in report.md.

## Task Breakdown

2. Debate protocol engine (turn management, role casting, distinct-model enforcement) over the 2.0 loop.
3. Judge envelope + ruling integration into reconciled artifacts and confidence.

5. Report + docs.

## Out of Scope

- Free-form multi-agent discussion (unbounded turns); execution-based evidence (11.0); debates on undisputed findings.

## Dependencies

Epics 2.0 and 3.0 complete (loop, roles, verdict plumbing). Field data from 3.0 informs trigger defaults. New `debate.*` config keys (`debate.triggers`, `debate.max_items`) extend the **Epic 4.2** config-validation schema — update it in lockstep.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Judges favor verbose/confident arguments over correct ones | Medium | High | Judge prompt demands evidence citations from transcripts; strongest model cast as judge; rulings always auditable |
| Cost per contested item is high | High | Medium | Debate only triggered items, max_items cap, 3-turn hard limit |
| Tier proliferation confuses users | Medium | Low | Fold survived-challenge into VERIFIED with a marker rather than a new tier (settle in design) |

## Clarifications

- **Q: Does this replace the Skill adjudication path from 1.0?** A: Supersedes it for unattended runs; the Skill path remains as a manual override. (2026-06-10)

### Recorded 2026-06-21 (rubber duck — /execute-epic)

**Key Decisions:**
- **Confidence (Q1 → A):** Keep `VERIFIED` as the top confidence tier. Survived-challenge is recorded as a boolean `challenge_survived` marker (omitempty) on the existing `reconcile.Verification` struct — NOT a new `DEBATED` tier. Rationale: the gate (`IsFailing`, `internal/reconcile/gate.go:96-113`) keys on `Verification.Verdict`, not the `Confidence` string, so a new tier would be invisible to the gate without patching it; `confidenceV2()` is the single write path to `Confidence` and the marker follows the additive omitempty pattern of `PathWarning`/`PathSuggestion`.
- **Single-model fallback (Q2 → A):** When fewer than 3 distinct models are available in the proposer/challenger/judge roles, debate **skips the affected items and records them as unresolved/overflow (never silent)** by default. Same-model persona fallback is opt-in only via `debate.allow_single_model: true` (config) or `--single-model` (flag). The current registry has zero agents in skeptic/judge roles (`AgentsByRole` normalizes all to reviewer; roles parsed but inert pre-6.0), so single-model is today's baseline — physical model count (11 IDs) is sufficient; the gap is role assignment.
- **Triggers + cost cap (Q3):** All three triggers default **on** — `severity_split` (≥2 tiers), `gray_zone` (ambiguous clusters), `verification_disagreement` (post-verify, so no cost when verify is skipped). `debate.max_items = 5` default cost cap with `0 = unlimited` semantics (mirrors `verify.votes`). Overflow beyond the cap is always recorded, never silent.
- **Gate semantics (Q4 → confirmed):** Judge **overturn** → writes `verdict: "refuted"` (retained, never blocks `--fail-on`). **Uphold** → writes `verdict: "confirmed"` (counts toward gate, satisfies `--require-verified`) and sets `challenge_survived: true`. **Split** → overwrites the finding's `Severity` with the judge's settled value before `Emit`, then the finding gates normally. No new gate logic required.

**Scope Boundaries:**
- IN scope: T1 trigger detection + item selection (priority, caps); T2 debate protocol engine (proposer/challenger/judge, bounded 3-turn, distinct-model enforcement over the 2.0 tool loop) with a `harnessFunc` seam for scripted-completer tests; T3 judge envelope + ruling integration into `Verification`/`reconciled/debate.json` + `challenge_survived`; T4 `atcr debate [<id-or-path>]` command, `atcr review --verify --debate` chaining, `atcr_debate` MCP tool; T5 `report.md` "Contested findings" section + docs. New `debate.*` config block (`triggers`, `max_items`, `allow_single_model`) on `registry.Registry`, validated in `validate()` in lockstep (Epic 4.2 accumulate-faults pattern).
- OUT of scope: free-form/unbounded debate; execution-based evidence (Epic 11.0); debating undisputed findings. The Skill adjudication path remains as a manual override.

**Technical Approach:**
- Mirror the verify stage: new `internal/debate` package with `Debate(ctx, repoRoot, reviewDir, reg, opts)` orchestrator + `harnessFunc` seam (unit-testable with scripted `fanout.ChatCompleter`, no live provider in tests).
- Trigger input is the pre-built `reconciled/disagreements.json` (`DisagreementsSchemaVersion "1.0"`); the three trigger classes map to its existing `kind`s. No new queue is invented.
- Distinct-model enforcement generalizes `SelectEligibleSkeptics`' "exclude matching model" rule across the 3 roles. Proposer uses a crediting reviewer's model with the read-only snapshot tool loop (`buildDispatcher` pattern).
- Artifacts: `debate/<item-id>/transcript.jsonl` via the existing `tools.Transcript` writer; `reconciled/debate.json`; manifest `Stages` gains `"debate"`.
