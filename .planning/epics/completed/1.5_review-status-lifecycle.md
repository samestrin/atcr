# Epic Plan 1.5: Review Status Lifecycle (artifact contract, stale detection)

**Estimated Durations**: 1-2 weeks

## Objective

Make the review status lifecycle coherent end-to-end: one designed write-ordering contract for the status-bearing artifacts (manifest.json, summary.json), atomic-feeling status reads, and a distinct terminal state for dead reviews — without adding a competing state-bearing artifact or breaking the pinned `ReviewStatus` consumer shape.

## Context

Epic 1.0 shipped status as a derivation over two artifacts: manifest.json (roster, provenance) and sources/pool/summary.json (completion signal — absence means in_progress). AC 04-04 pins the `ReviewStatus` shape `{review_id, status, agent_count, agents_done, agents_pending}`; the MCP `StatusResult` is a type alias of it and the CLI `atcr status` consumes the same struct. In-sprint triage deferred the two-step torn-read concern as TD-023. The clarification adjudication (2026-06-11) settled the design direction: summary.json already IS the completion sentinel; a third snapshot artifact would recreate the dual-source-of-truth problem; the two-step read, dead-review terminal state, and partial-flag ordering are one coupled design on one writer/reader pair and must land together. (The partial-flag write reorder — finalized manifest before summary.json — was adjudicated as an immediate re-attempt fix; this epic owns the remaining lifecycle work and must not contradict it.)

## Problem Statement

1. **Torn reads (TD-023)**: `ReadReviewStatus` reads manifest.json then summary.json in two separate steps while the background fan-out is writing both; each file is atomic (temp+rename) but the pair is not.
2. **No terminal state for dead reviews**: absent summary.json is reported as in_progress unconditionally. A fan-out that dies after scaffolding (post-fan-out persistence failure, killed MCP server) leaves `atcr status` reporting in_progress forever, the Skill polling loop cannot distinguish running from dead, and `.atcr/latest` keeps pointing at the orphan.

## Proposed Solution

1. **Keep summary.json as the sole completion sentinel** — no new status.json snapshot, no new sentinel file. Design the read path so the manifest+summary pair is interpreted consistently with the (re-attempted) manifest-before-summary write ordering.
2. **Persist the effective timeout in the manifest**: add an optional `timeout_secs` (omitempty) field to `payload.Manifest` — additive to the plan-owned schema, degrades gracefully for old manifests (zero value = unknown).
3. **Stale detection in `ReadReviewStatus`**: when summary.json is absent AND `StartedAt + timeout_secs` (plus a grace margin) has elapsed, report status `"stale"` — a new additive value beside in_progress/completed/failed (AC 04-04 enum extension recorded as a contract amendment). The state is honestly labeled as inferred, not observed.
4. **Post-fan-out persistence failure maps to `failed`**, not a second new state: on a WritePool/WriteManifest failure, ExecuteReview writes a best-effort failure record the reader understands through the existing summary-derived path where possible.
5. **Preserve the `ReviewStatus` shape**: `{review_id, status, agent_count, agents_done, agents_pending, partial}` stays intact for the CLI and MCP alias; only the `status` value set grows.
6. **Docs/AC**: record the enum extension and the stale semantics in the AC 04-04 contract and skill polling guidance (poll loop treats `stale` as terminal).

## Clarifications

### Recorded 2026-06-12 (rubber duck — /execute-epic)

**Key Decisions:**
- The write-ordering dependency is already satisfied by the current code — no change to `ExecuteReview` is in scope. Task 4 documents and tests the ordering the code already produces (manifest always present from `PrepareReview` scaffold; `WritePool`/summary.json and the finalized `WriteManifest` run at completion). The concurrency test covers the "summary present, manifest not-yet-finalized" interleaving and asserts no torn-pair misreport; a manifest-absent read returns `os.ErrNotExist` (mapped to "review not found"), never a false `in_progress`.
- Stale grace margin is a flat `staleGraceSecs = 60` constant added atop the persisted effective timeout (not a percentage). The post-deadline write path is local-filesystem, synchronous, no network I/O — 60s is conservative and absorbs minor clock skew.
- Post-fan-out persistence failure (`WritePool` error in `ExecuteReview`) writes a best-effort minimal `summary.json` (`total=roster, succeeded=0, failed=roster, partial=false`) so the existing reader path reports `failed` (`succeeded==0 → RunFailed`). No new sentinel file. If even that best-effort write fails, summary.json stays absent and stale inference promotes the review to `stale` after the timeout.

**Scope Boundaries:**
- IN scope: backward-compat tests for the existing `timeout_secs` manifest field; stale inference + `RunStale` constant + injectable clock for tests; failure-marker path in `ExecuteReview`; read-side torn-pair invariant + concurrency test; CLI/MCP pass-through tests for the new value; AC 04-04 enum amendment + skill polling guidance.
- OUT of scope: a consolidated `status.json` snapshot or any new sentinel file (rejected by adjudication); per-agent incremental progress mid-run; changing the `ReviewStatus` field set or the MCP tool schema shape; any reordering of `ExecuteReview`'s artifact writes.

**Technical Approach:**
- Old manifests without `timeout_secs` (zero value) disable stale inference and keep reporting `in_progress` — backward compatible.
- `ReadReviewStatus` gains an injectable "now" (package-level clock var, default `time.Now`) so a stale manifest can be tested deterministically.
- New status value `stale` flows through `StatusResult` (type alias of `ReviewStatus`), `atcr status` CLI, and `atcr_status` MCP handler unchanged — all are pure pass-throughs; no consumer switches on the three literals.

**Original Questions and Answers:**
1. Q: Does this epic include the `ExecuteReview` manifest/summary write reorder, or does the read path stay order-agnostic while the reorder remains a separate fix?
   A: The reorder is already satisfied by the current code; no write-order change is in scope. Task 4 documents and tests the existing ordering. The interleaving to cover is "summary present, manifest not yet written"; a manifest-absent read returns `ErrNotExist`, never a false `in_progress`.
2. Q: Is a flat `staleGraceSecs = 60` appropriate, or should a percentage-of-timeout be used?
   A: Flat 60s is appropriate and sufficient; the post-deadline write path completes in milliseconds locally. A percentage adds complexity with no benefit and could collapse to near-zero for short timeouts. Keep the flat constant.
3. Q: Is a best-effort minimal `summary.json` (succeeded=0, failed=roster) the intended failure marker?
   A: Yes. `ReadReviewStatus` derives `RunFailed` when `succeeded==0`, so the minimal record maps directly to `failed` via the existing path. The fallback (absent summary → stale) is already handled. No new sentinel file.
