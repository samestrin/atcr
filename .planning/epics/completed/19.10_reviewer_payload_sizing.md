# Per-Model Payload Sizing & Graceful Degradation for the Multi-Agent Reviewer

- **Estimated time**: 4–6 days
- **Tasks/Components**: ~12/3 (sprint-sized — see Scope & Sizing Note)

## Context

The atcr multi-agent reviewer ships a single global **byte** budget to a heterogeneous roster whose models span 32k→144k-token windows. Confirmed from the `19.6_community_registry_hub` multi-agent review run (2026-07-08): a **101 files / 6,429 insertion** diff returned **1 finding from 11 reviewers (5 ok, 3 timeout, 3 failed)**. A code-review product could not review its own large sprint.

This plan replaces the original single-file `19.10` epic (the earlier byte-budget-resize scope) in place. During refinement (2026-07-08) the scope resolved from "resize the byte budget per model" into a broader, higher-fidelity design — **chunk-to-fit rather than shed-to-fit**, plus a configurable degradation policy — which exceeds the `/execute-epic` scope guard (≤6 tasks / ≤2 components). It is therefore authored as sprint requirements. See **Decisions from Refinement** for the full rationale.

## Problem Statement

The payload sizer has **no per-model token awareness**. Four independent failure axes, all confirmed in the 19.6 run:

1. **Global byte budget, applied identically to every agent.** `.atcr/config.yaml:21` `payload_byte_budget: 524288` (512 KiB). `ApplyByteBudget` (`internal/payload/budget.go:46`) *sheds files by bytes* against that one value, applied identically at `internal/fanout/review.go:464` and `:726`. There is **no per-model context-window table** and **no bytes↔tokens conversion** anywhere in the codebase (verified). Every agent got the same payload and dropped the same 102 files; `blocks.txt` came out at 506 KB.
2. **Output budget never reserved.** `internal/fanout/review.go:948-958` defines `const defaultMaxTokens = 8192`, applied via `maxTokensPtr()` to every call, model-agnostic. `dax`'s litellm error reported the boundary as `24577` input tokens = **32768 − 8192 + 1** — input+output blew the window by exactly one token past the never-reserved output budget.
3. **No graceful degradation.** litellm `context_window_fallbacks` is unset ("No fallback model group found"), so overflow **kills** the agent instead of falling back or truncating. `otto`'s error literally names the escape hatch: "enable `--allow-auto-truncate`". 506 KB ≈ ~147k tokens shipped unchanged to `otto` (144,941-token model, over by ~2.4k → failed) and `dax` (32,768-token model, over by ~115k → failed).
4. **Timeout too tight for slow backends, made worse by oversized payloads.** `greta`/`vera`/`brad` hit `context deadline exceeded` against the local litellm endpoint; `brad` at exactly 599,998 ms — the `timeout_secs: 600` wall (`.atcr/config.yaml:20`, `DefaultTimeoutSecs = 600` at `internal/registry/config.go:25`).

There is also a **latent instance of the same bug in the chunked path**: the Epic 14.3 chunker (`internal/fanout/chunker.go`, `review_strategy: "chunked"`) bin-packs by diff **line** count (`MaxContextLines`, default 1500 — `internal/registry/config.go:36`), *not* by token window. It splits instead of sheds (no content loss), but 1500 lines of dense code can still overflow a 32k model and wastes a 144k model. The current plan touches only the shed path; the chunk path stays uncalibrated.

## Proposed Solution

Size each reviewer's payload to **its own model's token window** (reserving the output-token budget), and when a payload still doesn't fit, **chunk it to fit** using the existing 14.3 chunker made window-aware — degrading gracefully via a configurable `on_overflow` policy rather than silently gutting the review.

**Core design decisions (locked in refinement):**

- **Chunk-to-fit, not shed-to-fit (Fork B).** A code reviewer dropping files to fit is worse than reviewing the diff in pieces. Extend the existing Epic 14.3 chunker so its per-chunk budget derives from each model's token window instead of the global `MaxContextLines`. Small-window models (e.g. `dax`@32k) get more chunks; large-window models (e.g. `otto`@144k) get one. Zero content is dropped.
- **In-memory per-agent sizing at dispatch.** Build the full `FileEntry` set once, then per agent compute the effective budget and chunk plan at dispatch time. Keep a **single** on-disk audit artifact (the union payload); record each agent's effective budget, resolved window, reserved output tokens, chunk count, and any degradation action in the manifest/summary. Resume works for free because sizing is deterministic from `(entries, model window)`.
- **`on_overflow` is a configurable policy.** A degradation ladder that maps onto existing code paths:

  | Policy | Mechanism | Content | Model identity |
  |--------|-----------|---------|----------------|
  | `chunk` (default) | 14.3 chunker, window-aware | whole | preserved |
  | `truncate` | in-repo auto-truncate signal (drop lowest-priority tail, flag it) | lossy | preserved |
  | `fallback` | litellm any→any fallback (e.g. universal high-context net model) | whole | **swapped — must record provenance** |
  | `fail` | hard fail, loud | — | — |

  For this sprint, implement `chunk` (default) and `truncate`; recognize `fallback` and `fail` as valid config values with `fallback` requiring the provenance-recording work below. Ship `on_overflow` as a real config key regardless.

## Requirements

### Functional

- **F1 — Per-model context-window resolver.** A static table keyed by model id, with a conservative default for unknown models, exposing each roster model's token window. Deterministic; no hot-path network call. Name it distinctly (e.g. `ContextWindowTokens`) so it is never confused with the unrelated per-chunk diff-line budget `MaxContextLines`.
- **F2 — Reserve output, size per model.** Effective per-agent input budget derived so estimated input tokens ≤ `contextWindow − defaultMaxTokens − promptOverhead`. The `24577 + 8192 > 32768` overflow (the `dax` arithmetic) can no longer occur.
- **F3 — Window-aware chunking.** Convert each model's effective token budget into a per-model chunk budget (`maxLines`) feeding the existing 14.3 chunker. On overflow with `on_overflow: chunk`, the diff is delivered whole across N appropriately-sized chunks per model; findings are merged. Respect the existing 64-chunk/agent ceiling.
- **F4 — `on_overflow` policy.** Config-driven degradation: `chunk` (default) and `truncate` implemented; `fallback` and `fail` recognized. Unimplemented arms error clearly rather than silently no-op.
- **F5 — Fallback provenance.** When a slot is served by a fallback model (different from the persona's configured model), record the substitution in `summary.json` so `reconcile` does not count the fallback as the original distinct reviewer (protecting the distinct-reviewer CONFIDENCE calculus).
- **F6 — Timeout scales with load.** The request timeout scales with payload size / chunk count (or a raised ceiling for slow local backends), so a large-but-valid payload — especially a small-window model fanned out into many chunks on a slow local backend — no longer hits the 600 s wall.
- **F7 — Cache-key correctness.** The fan-out cache key folds in the per-agent effective budget / chunk plan, so a per-agent-sized payload is never served a stale full-payload cache hit.
- **F8 — Diagnosability.** `summary.json` records, per agent: effective input budget, resolved model window, reserved output tokens, chunk count, degradation action taken, and fallback substitution (if any).
- **F9 — Configurable Sprint Plan Limit.** Replace the hardcoded `MaxSprintPlanBytes` constant (16KB) in `sprintplan.go` with a configurable value parsed from `.atcr/config.yaml` (`max_sprint_plan_bytes`, defaulting to 65536 / 64KB). This allows operators to leverage larger context windows by providing richer sprint/epic context.


### Non-Functional

- **Determinism:** window resolution and chunk planning are deterministic from `(entries, model, config)`. No live/network resolution on the hot path.
- **Conservatism:** the byte→token ratio must **over-reserve**. Use the worst-case observed ratio (~3.5 B/token from the 19.6 run, where 506 KB ≈ 147k tokens) plus a safety margin — **not** the codebase comment's optimistic ~4.1 B/token (`internal/registry/project.go:89`). An under-filled payload is strictly better than a failed agent; the `on_overflow` net catches the residual tail.
- **No content loss on the default path:** `on_overflow: chunk` never drops a file.
- **Provenance integrity:** heterogeneity is the panel's value. Prefer same-model degradation (chunk, truncate) over model-swapping (fallback); when fallback fires, the swap is always recorded, never silent.

## Technical Approach

### Chunk vs shed — why chunk

The codebase already contains both degradation primitives:

- `internal/payload/budget.go` `ApplyByteBudget` — **sheds** (drops files) to meet a byte budget. Current default path.
- `internal/fanout/chunker.go` `chunkDiff` — **splits** a diff into ≤`maxLines` chunks on file boundaries, capped at `maxChunksPerAgent = 64`, delivering the whole diff (Epic 14.3, `review_strategy: "chunked"`). Currently line-gated, not window-gated.

Chunking preserves both full content and model identity; shedding loses content. This plan makes the chunker window-aware and routes overflow through it by default. The shed path remains for `on_overflow: truncate` (last-resort, flagged).

### Fallback semantics (recorded, opt-in)

litellm fallback is arbitrary model→model — a single universal high-context net model (e.g. a 1M-context model) can back every persona. Because each persona is **prompt + model**, a fallback preserves prompt-driven differentiation but loses the model-diversity axis, and multiple personas collapsing to one net model would inflate `reconcile`'s distinct-reviewer CONFIDENCE if unrecorded. Overflow-after-sizing is a bounded edge case (only the small-window subset on very large diffs), so fallback is an opt-in net, not the default — and every substitution is recorded (F5).

### Timeout coupling

F6 is load-bearing under Fork B, not cosmetic: a 32k model fanned into 6+ chunks on a slow local backend multiplies wall-clock. Timeout scaling and chunking must be designed together. Prefer applying the scaled timeout in `internal/fanout` where the deadline is set (`review.go:516`, `engine.go:610`) by reading the already-resolved `EffectiveTimeoutSecs` value — avoid modifying `internal/registry`'s resolvers unless a config-schema field proves necessary.

## Implementation Components

1. **Context-window resolver** (`internal/payload`) — static table + conservative default (F1).
2. **Per-agent effective-budget + chunk-plan derivation** (`internal/payload` / `internal/fanout`) — in-memory at dispatch, output reserved, conservative byte→token ratio (F2).
3. **Window-aware chunker** (`internal/fanout/chunker.go`) — per-model `maxLines` from token budget; wire overflow → chunk path (F3).
4. **`on_overflow` policy** (config + `internal/fanout`) — `chunk`/`truncate` implemented, `fallback`/`fail` recognized (F4).
5. **Fallback provenance** (`internal/fanout` summary + `internal/reconcile` awareness) — record and de-weight swapped-model slots (F5).
6. **Timeout scaling** (`internal/fanout`) — scale with chunk count / payload; read resolved values, don't touch registry resolvers (F6).
7. **Cache-key correctness** (`internal/fanout/cache.go`) — fold effective budget / chunk plan into the key (F7).
8. **Diagnosability fields** (`internal/fanout` `PoolSummary` in `artifacts.go`) — per-agent budget/window/output/chunks/degradation (F8).
9. **Live audit harness** — scripted, standalone-runnable, also invoked in the execution loop (see Acceptance AC-Live).
10. **Configurable sprint-plan limits** (`internal/registry` / `internal/payload`) — add `max_sprint_plan_bytes` config parsing and pass it to `ScopeConstraint` (F9).

## Constraints

- Reuse the Epic 14.3 chunker and its 64-chunk ceiling; do not reimplement chunking.
- Keep byte→token conversion conservative (over-reserve).
- Do not modify `internal/registry`'s timeout/budget resolvers, except to add the new `max_sprint_plan_bytes` configuration schema and resolver to support F9. Keep other fanout logic local.
- The live audit (AC-Live) is env-coupled (needs `orchestrator.lan` litellm proxy + the real roster) and therefore is a scripted integration verification, **not** a `go test`.

## Out of Scope

- **Live/dynamic context-window resolution** from provider `model_info` / catalog endpoints — belongs to Epic 19.7 (Live Model Resolution). This sprint ships a static, deterministic table; 19.7 can later feed it live windows. Soft, non-blocking relationship.
- **Full prompt-cache-for-determinism layer** — orthogonal to sizing. The existing `cache.go` is touched only for key correctness (F7); a broader pseudo-deterministic prompt cache is its own effort.
- **Config-schema per-model window field** on `Provider`/agent config — deferred; the static table with a conservative default stands alone. Promote only if planning resolves it necessary.
- **Reconciler / severity-gate behavior and wording** beyond the fallback-provenance de-weighting (F5) — the "severity gate tripped upstream (informational only)" path is already correct.

## Decisions from Refinement (2026-07-08)

Captured verbatim so nothing is lost from the refinement discussion:

- **Fork B chosen (chunk-to-fit) over Fork A (shed-to-fit).** A reviewer should review the whole diff in pieces, not review a truncated diff. The 14.3 chunker already exists; make it window-aware.
- **`on_overflow` is a policy, not a hardcode.** Ladder: chunk (default) → truncate → fallback → fail. Chunk and truncate ship in this sprint; fallback needs provenance; fail is trivial.
- **Fallback is any→any and prompt-diversity survives a model swap** — so collapse-to-one-net-model is tolerable but must be recorded; the surviving concerns are the model-diversity axis and reconcile's distinct-reviewer CONFIDENCE.
- **Overflow-after-sizing is a bounded edge case** — correlated to the small-window subset on large diffs, not a random all-11 collapse.
- **In-memory per-agent sizing, single audit artifact, provenance in manifest/summary.** Avoids a per-agent artifact/resume rewrite.
- **Conservative byte→token ratio (~3.5, not ~4.1).** Over-reserve on purpose.
- **Timeout scaling is load-bearing under chunking**, must be co-designed (F6).
- **Cache-key correctness is a real, silent-failure risk** (F7).
- **Live audit is (A): in-loop + standalone-runnable, hard-gated on the deterministic assertions, soft on finding quality.** `orchestrator.lan` is a 24/7 battery-backed homelab server; its unavailability is a bigger-problem signal.
- **Scope:** promoted from a single-file epic to sprint requirements, kept in the `19.10` slot (replacing the original byte-budget-resize plan).
