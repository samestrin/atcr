## Current State

The `VerificationResult` record written to `verification.json` has three structured audit fields that are never populated correctly:

- `TrippedBudgets []string` ŌĆö always empty; `invokeSkeptic` folds tripped budgets into free-text `Notes`, never into the slice
- `Model string` ŌĆö hard-coded to `skeptics[0].Config.Model` regardless of which skeptic produced the winning verdict in a multi-vote run
- `DurationMs int` / `TrippedBudgets` ŌĆö lost entirely on the skip-already-verified rebuild path (pipeline.go:199), which re-synthesizes a compact record from the on-disk Verification block

Additionally, `Model` is empty on the `no_eligible_skeptic` and `tool_harness_unavailable` early-return paths in `verifyFinding`.

## Pain Points

### Degraded audit trail
A human reviewing `verification.json` cannot determine:
- Which skeptic's model was actually responsible for the winning verdict (multi-vote runs)
- Whether a budget was tripped during verification (tripped budgets buried in prose Notes)
- What the original DurationMs/Model was for a finding that was skipped on a re-run

### Misattribution risk
In a 3-skeptic Thorough run where skeptic[0] is refuted and skeptics[1]+[2] confirm, the recorded `Model` is skeptic[0]'s ŌĆö the loser's, not the majority winners'.

## Proposed Solution

Thread the tripped-budget slice and the winning skeptic's model up from `invokeSkeptic` into the `VerificationResult` struct, and preserve `Model`/`DurationMs`/`TrippedBudgets` on the skip-already-verified rebuild path.

### Key changes

1. **`internal/verify/invoke.go`** ŌĆö change `invokeSkeptic` return to surface `trippedBudgets []string` alongside the verdict (currently only folded into Notes)
2. **`internal/verify/pipeline.go` / `verifyFinding`** ŌĆö after aggregating votes via `aggregateVerdicts`, identify the winning skeptic by matching the aggregated verdict against per-skeptic results (mirroring `joinSkeptics`); set `base.Model` to the winner's model, join models when multiple voters agree

## Clarifications

### Recorded 2026-06-14 (rubber duck ŌĆö /execute-epic)

**Key Decisions:**
- T4 (AC3): BOTH early-return paths in `verifyFinding` ŌĆö `no_eligible_skeptic` and `tool_harness_unavailable` ŌĆö set `base.Model = ""`. The audit principle "attribute only to what actually ran" governs: a skeptic that never reached the execution loop contributes no model. Candidate models present in the `skeptics` slice but never invoked are NOT recorded, avoiding a misleading attribution and the new sentinel convention that recording them would require. AC3 explicitly permits `""` on both paths.
- T2 (AC2): "Winning" skeptics = those whose per-skeptic verdict equals the aggregated (majority) verdict; `base.Model` is the deduplicated join of just those winners' models. Tripped budgets attach only to `unverifiable` verdicts (a budget trip collapses a skeptic to unverifiable), so a confirmed/refuted winner correctly carries an empty `TrippedBudgets`.
- T3 (AC4): The rich fields (Model/DurationMs/TrippedBudgets) live only in on-disk `verification.json` ŌĆö the findings.json compact block carries only Verdict/Skeptic/Notes. Preserving them on the skip-already-verified path requires reading the prior `verification.json` and matching by FindingKey, with graceful fallback to the current compact synthesis when no prior record exists.

**Scope Boundaries:**
- IN scope: `internal/verify/{invoke.go, pipeline.go, emit_verification.go}` + their tests.
- OUT of scope: downstream consumers (3.3 scorecard, 10.x leaderboard); the verification.json JSON schema (already carries the fields); the gate/confidence logic.

**Technical Approach:**
- `invokeSkeptic` signature becomes `(*reconcile.Verification, []string, error)` ŌĆö the tripped-budget slice surfaces separately from the Notes string.
- `verifyFinding` builds a nameŌåÆmodel map from the `skeptics` slice, identifies winning per-skeptic verdicts by matching against the aggregated verdict, and sets `base.Model` / `base.TrippedBudgets` from those winners.
- A new reader parses on-disk `verification.json` (VerificationFile ŌåÆ []VerificationResult) keyed by File/Line/Problem for the skip-path carry-forward.

**Original Questions and Answers:**
1. Q: On the `tool_harness_unavailable` early-return path, what should `VerificationResult.Model` record ŌĆö the joined candidate skeptic models (a), or empty string (b)?
   A: (b) empty string. Rationale: `no_eligible_skeptic` and `tool_harness_unavailable` are structurally parallel (neither reaches the skeptic execution loop), the codebase already attributes only to what actually ran (Skeptic is always the executing name), Model has no sentinel convention so recording candidates would read as the executing model, and AC3 explicitly permits `""`. Key change #4's "set base.Model to the skeptic's model" is read as applying to paths where a skeptic actually ran (including inside invokeSkeptic on budget-trip/provider-error), not to pre-invocation harness failure.
