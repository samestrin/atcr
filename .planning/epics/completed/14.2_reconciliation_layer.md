# Epic 14.2: Reconciliation Layer

- **Estimated time**: 1 week
- **Tasks/Components**: 2/1
- **Execution**: execute-epic

## Context

The Reconciler (the "Host" persona) is responsible for taking the output of multiple fanned-out reviewer personas, deduplicating them, and producing a final report. Currently, the Reconciler is allowing hallucinated TD items through to the final output.

## Problem Statement

Because individual personas can hallucinate, passing all items to the final report creates noise. The Reconciler needs to act as a strict gatekeeper against false positives.

## Proposed Solution

Leverage the multi-persona architecture to implement consensus rules and strict anti-hallucination prompting for the Reconciler itself.

1. **The "Two-Persona" Consensus Rule**: If a TD item is only raised by a single persona out of the panel, it is statistically more likely to be a hallucination or an extreme nitpick. We will implement a rule in the Reconciler (or pre-reconciler grouping step) to drop "singleton" TD items unless they meet specific criteria.
2. **Explicit Anti-Hallucination Prompting**: Modify the Host/Reconciler prompt. Explicitly state: *"Your primary job is to aggressively filter out false positives. If a persona suggests a TD item that is not explicitly supported by the provided code diff, you must reject it."*

## Acceptance Criteria

- [ ] AC1: Host prompt is updated with strict anti-hallucination instructions.
- [ ] AC2: The Reconciler logic implements a "consensus filter" that flags or drops singleton TD items.

## Technical Constraints

- The deduplication logic (~~Semantic NCD deduplication or~~ bipartite DBSCAN — **13.0 NCD not adopted, AC3 falsified 2026-06-27; see `.planning/epics/completed/13.0_semantic_ncd_deduplication.md`**) must be updated to identify singletons vs. consensus items accurately before applying the Two-Persona rule.

### Correction 2026-06-30 (Phase 6 independent review — activation gate)

**What changed:** The Q4-recommended activation gate `len(sources) >= 3` was replaced with `panelReviewers(sources) >= 3` (count of distinct non-empty reviewers across all findings).

**Why (discovered fact that overrides the Q4 recommendation):** The independent-review subagent found, and `internal/reconcile/discover.go` confirms, that ATCR discovery returns one source per top-level directory and **flattens every pool persona into a single `pool` source** (`sources/pool/raw/agent/<name>/findings.txt`), distinguishing personas only by the per-finding `REVIEWER` column. So in the standard workflow `len(sources)` is `host` + `pool` = **2 regardless of persona count**, and the `len(sources) >= 3` gate would have left the consensus filter **permanently inert** — AC2 would never fire for the exact multi-persona pool the epic targets. Q4 itself flagged its answer as "my recommendation, not a grounded answer... a scope decision rather than a discoverable fact"; this is that discoverable fact. Gating on the distinct-reviewer count is also more faithful to the epic's "Two-Persona / single-persona" language, which is about personas, not directories.

**Also corrected:** `consensusSingleton` now keys on "confidence below HIGH" (`!ConfidenceAtOrAbove(_, ConfHigh)`) rather than `== ConfMedium`, so a reserved-but-unused `ConfLow` untrusted-source singleton is also dropped and authority/verify-promoted HIGH/VERIFIED findings are always kept. Two pre-existing sort tests (`TestReconcile_SortedBySeverityThenLocation`, mirrored in both modules) were adjusted to share one reviewer so the now-functional filter does not fire in a test that only exercises sort order.
