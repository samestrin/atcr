# 12.1 Technical Debt Format Migration

- **Estimated time**: 2-3 days
- **Tasks/Components**: 4/1

## Context

Technical debt is tracked in a single Markdown table at `.planning/technical-debt/README.md`. As the project grows this one file has become unwieldy, and two structural problems now dominate: the table format is hostile to long-form text, and a single shared file is a merge-conflict magnet when multiple review/sprint runs write to it concurrently.

A first attempt at this epic migrated the table to a directory of **one file per item**. That was reverted before merge: at code-review volume (a single multi-agent review can emit 50-100 findings in one run), per-item files explode into hundreds-to-thousands of tiny files, churn filenames on every regeneration (IDs are positional), and make the corpus impossible to skim. Per-item optimizes the wrong axis — the real access pattern is *batch-by-review*, and the real pains are long-form text and concurrent-writer conflicts.

## Problem Statement

The current Markdown table format:
1. **Is hostile to long-form text.** It encourages brevity, which frequently results in "won't fix, description too vague" closures.
2. **Causes merge conflicts.** Multiple developers / review runs adding items to the one file collide.

And the rejected per-item approach adds a third problem to avoid:
3. **File-count explosion.** One file per finding does not scale to review-volume output.

## Proposed Solution

Move technical-debt **storage** to a **directory of structured files sharded by source** — one file per `### [date] From <Sprint|Review>: <label>` provenance unit, each holding *all* of that run's items as a list with unconstrained multi-line text. The README Markdown table is regenerated from those shards as a **human-readable table-of-contents / summary** (it stays excellent at the at-a-glance overview the flat table is genuinely good at).

This directly resolves all three problems:
- **Long-form text:** each item gets structured metadata plus multi-line description/problem/fix fields.
- **Merge conflicts:** each review/sprint run writes its **own new file**; two concurrent runs never touch the same file, so cross-run conflicts effectively disappear (the original motivation).
- **File-count:** file count is **O(number of runs)** — dozens — not O(number of items). A 100-finding review is one file, not 100.

### Coexistence (additive, like the reverted attempt)

This remains **additive**: the README table stays the authoritative machine-read source for the existing tooling (the 6 Go MCP TD binaries — `group_td`, `td_stats`, `td_filter`, `td_dedupe`, `td_clean`, `td_validate` — and the 9 skills that read/write it). The sharded store is generated alongside and is not yet machine-read by any tool. The cutover that makes the shards canonical (and updates the binaries/skills) is deferred to the follow-on epic (18.0, or renumbered 12.3). No tooling is modified in 12.1.

## Requirements

### Functional
- Store technical debt as structured files **sharded by source** under `.planning/technical-debt/items/` (one file per provenance section).
- Migrate all existing table items into the sharded store with zero data loss.
- Provide a tool to **generate the README table (ToC/summary) from the shards** (reverse direction), proving round-trip fidelity.
- Provide a **validation command/gate** that strictly loads and schema-checks every shard.

### Non-Functional
- Zero data loss during the migration.
- The new format must be human-readable and (occasionally) human-editable.
- The new format must allow unconstrained multi-line strings for descriptions.
- A malformed shard must fail **loudly** (CI), never corrupt silently.

## Technical Approach

### Storage shard granularity: by SOURCE (chosen)

One file per `### [date] From <Sprint|Review>: <label>` section (e.g. `items/2026-06-26_epic-12.1.yaml`), containing that run's items as a list.

Rejected alternatives:
- **Per-item file** (the reverted attempt): does not scale to review volume; filename churn on regeneration.
- **Per-reviewer:** post-reconciliation items carry a *set* of reviewers, so an item has no single home file — does not partition cleanly.
- **Per-group:** a "group" is a path-theme cluster `group_td` re-computes every run (unstable numbering), and two concurrent runs both producing items for the same path theme would write the same file — reintroduces the concurrent-writer merge conflict this epic exists to kill. `group` is kept as a per-item **field** and a **presentation facet** (the generated ToC can cluster by path theme on the read side) — not a storage key.
- **Single JSONL file / SQLite:** JSONL re-creates tail-append conflicts on one shared file; SQLite is a binary blob whose merges are unresolvable and which is not human-readable — both defeat the epic's core goals.

### File format: YAML (`gopkg.in/yaml.v3`, already a dependency)

Each shard is a YAML document: a top-level list of item objects. Structured metadata as scalar keys; long-form `problem` / `fix` / `notes` as multi-line block scalars.

YAML-safety plan (the format must not become a silent failure source):
1. **Generate, do not hand-author the risky parts** — `yaml.v3` Marshal emits valid, correctly-quoted YAML by construction.
2. **Strict-load validation gate** — load every shard with `Decoder.KnownFields(true)` + schema checks (required fields present; `severity`/`status` valid enums). Unknown field or malformed file → hard error in CI.
3. **Adversarial fixture corpus** — round-trip tests for known YAML footguns: Norway problem (`no`/`yes`/`on`/`off`), version-like floats (`1.10`), leading zeros, colons in values, leading dashes, multi-line block scalars, unicode, the `est_minutes` "0"-string-vs-int trap, tab rejection.

*Fallback:* if YAML risk is judged unacceptable, switch the inner format to **JSONL-per-source** (one JSON object per line). JSON removes every ambiguity above at the cost of `\n`-escaped multi-line readability. The shard-by-source storage model is unchanged either way.

### Implementation language/location

Testable Go in `internal/tdmigrate/` + a thin one-off `cmd/td-migrate` entry (removable after cutover; named outside Epic 18.0's planned `atcr debt` namespace). Not a shell script — the repo's TDD + 80%-coverage mandate requires testable Go.

## Implementation Components

1. **Schema & item model**
   - Go types for an Item and a per-source Shard (list of items); `yaml.v3` marshal/parse; status↔checkbox mapping; severity/status enums.

2. **Migration + generator tool**
   - `td-migrate migrate`: parse the README table → write one shard file per source. Idempotent (prunes its own prior output).
   - `td-migrate generate`: read shards → regenerate the README ToC table (round-trip / zero-data-loss proof). The ToC may group by path theme for presentation.
   - Run `migrate` to produce the real sharded store.

3. **Validation gate + adversarial YAML fixture corpus**
   - `td-migrate validate`: strict-load + schema-check every shard.
   - Adversarial fixture tests covering the YAML footguns above.

4. **Format / docs update**
   - Update `.planning/technical-debt/README.md` to document the sharded format + additive model; add a schema doc for `items/`.

## Constraints

- Existing tooling and review loops must not break — the README table stays authoritative and unchanged in shape (the regenerated ToC must remain parseable by `td_stats`/`td_filter`/`td_matrix`/`group_td`/`promote-tech-debt`).
- All columns from the current table must map losslessly to the shard schema.

## Out of Scope

- Making the shards canonical / dual-write / updating the 6 TD binaries or 9 skills (deferred to the follow-on epic).
- The `atcr debt` CLI namespace (Epic 18.0 / 12.3).
- Automated resolution of technical-debt items.

## Acceptance Criteria

- [ ] AC1: A tool parses the current Markdown table and writes one structured shard file per source.
- [ ] AC2: All existing items are migrated to the sharded store with zero data loss (verified by full round-trip: table → shards → table).
- [ ] AC3: The new format supports unconstrained multi-line descriptions.
- [ ] AC4: A validation command strict-loads and schema-checks every shard; malformed input fails loudly.

## Success Criteria

- [ ] A review producing 50-100 findings adds **one** shard file, not 50-100.
- [ ] Concurrent review/sprint runs do not merge-conflict on technical-debt storage.
- [ ] Developers can log technical debt with full multi-line context.
- [ ] A bad shard file is caught by the validation gate, never silently ingested.

## Rollback Plan

- Additive and not-yet-canonical: reverting the migration commit removes the shards and tool; the README table (authoritative) is untouched.

## Security Considerations

- None. Documentation/planning format change. Item filenames are sanitized; the new package is not imported by any production code path.

## Decisions / Rationale (carried from the design discussion)

- **Storage shard = by source.** Natural writer boundary → conflict-free + bounded file count. `group` kept as a field + read-side presentation facet, not a storage key (per-group storage reintroduces concurrent-writer conflicts).
- **Format = YAML (`yaml.v3`) with a strict-load validation gate + adversarial fixtures.** Preserves human-readability; converts "silent bad YAML" into "loud CI failure." JSONL-per-source is the safer fallback if YAML risk is unacceptable.
- **Per-item files rejected** (the reverted first attempt) — does not scale to review volume.
