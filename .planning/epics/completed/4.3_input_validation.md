# Epic 4.3: Input Validation Layer

**Status:** Draft  
**Created:** 2026-06-14  
**Priority:** Medium (security, error prevention)

## Problem

ATCR accepts user inputs (git refs, file paths, review IDs, severity thresholds) but validates them ad-hoc across different commands. This creates inconsistencies:

1. **Git ref validation.** `atcr review --base invalid-ref` fails with a git error, not a clear validation error.

2. **Path validation.** `atcr review --output-dir /etc/passwd` is rejected by the filesystem, not by input validation.

3. **Severity validation.** `atcr review --fail-on INVALID` fails late during reconciliation, not at flag-parse time.

4. **Review ID validation.** `atcr review --id "../../../etc/passwd"` is not sanitized; path traversal is possible.

5. **Inconsistent error messages.** Some commands say "invalid value", others say "bad input", others return raw git errors.

## Root Cause

There is no centralized input validation package. Each command validates inputs independently, using different patterns and error messages. Some validation happens at flag-parse time, some during execution.

## Solution

Create a centralized `internal/validation` package that validates common input types: git refs, file paths, review IDs, severity levels, and enum values. All commands use this package for consistent validation and error messages.

### Core Design

```go
// internal/validation/validation.go
package validation

import (
    "fmt"
    "regexp"
    "strings"
)

// ValidationError is a user-facing input validation error.
type ValidationError struct {
    Field   string
    Value   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("invalid %s %q: %s", e.Field, e.Value, e.Message)
}

// GitRef validates a git ref (branch name, tag, commit SHA).
func GitRef(ref string) error {
    if ref == "" {
        return &ValidationError{"git ref", ref, "must not be empty"}
    }
    // Git ref rules: no .., no ~, no ^, no :, no space, no control chars
    if strings.Contains(ref, "..") || strings.ContainsAny(ref, "~^: \t\n") {
        return &ValidationError{"git ref", ref, "contains invalid characters"}
    }
    if len(ref) > 255 {
        return &ValidationError{"git ref", ref, "must be <= 255 characters"}
    }
    return nil
}

// FilePath validates a file path (no path traversal, no special files).
func FilePath(path string) error {
    if path == "" {
        return &ValidationError{"file path", path, "must not be empty"}
    }
    // No path traversal
    if strings.Contains(path, "..") {
        return &ValidationError{"file path", path, "must not contain .."}
    }
    // No absolute paths starting with /etc, /proc, /sys
    if strings.HasPrefix(path, "/etc") || strings.HasPrefix(path, "/proc") || strings.HasPrefix(path, "/sys") {
        return &ValidationError{"file path", path, "must not reference system directories"}
    }
    return nil
}

// ReviewID validates a review ID (alphanumeric, dash, underscore).
func ReviewID(id string) error {
    if id == "" {
        return &ValidationError{"review ID", id, "must not be empty"}
    }
    matched, _ := regexp.MatchString(`^[a-zA-Z0-9_-]+$`, id)
    if !matched {
        return &ValidationError{"review ID", id, "must contain only alphanumeric characters, dash, and underscore"}
    }
    if len(id) > 100 {
        return &ValidationError{"review ID", id, "must be <= 100 characters"}
    }
    return nil
}

// Severity validates a severity level.
func Severity(s string) error {
    valid := map[string]bool{"LOW": true, "MEDIUM": true, "HIGH": true, "CRITICAL": true}
    if !valid[strings.ToUpper(s)] {
        return &ValidationError{"severity", s, "must be one of: LOW, MEDIUM, HIGH, CRITICAL"}
    }
    return nil
}

// Enum validates a value against a set of allowed values.
func Enum(field, value string, allowed []string) error {
    for _, a := range allowed {
        if value == a {
            return nil
        }
    }
    return &ValidationError{field, value, fmt.Sprintf("must be one of: %s", strings.Join(allowed, ", "))}
}
```

### Integration

```go
// cmd/atcr/review.go
func runReview(cmd *cobra.Command, _ []string) error {
    base, _ := cmd.Flags().GetString("base")
    if err := validation.GitRef(base); err != nil {
        return usageError(err)
    }

    outputDir, _ := cmd.Flags().GetString("output-dir")
    if outputDir != "" {
        if err := validation.FilePath(outputDir); err != nil {
            return usageError(err)
        }
    }

    idOverride, _ := cmd.Flags().GetString("id")
    if idOverride != "" {
        if err := validation.ReviewID(idOverride); err != nil {
            return usageError(err)
        }
    }

    threshold, _ := cmd.Flags().GetString("fail-on")
    if threshold != "" {
        if err := validation.Severity(threshold); err != nil {
            return usageError(err)
        }
    }

    // ... rest of review logic
}
```

## Scope

### In Scope

- `internal/validation` package with validators for git refs, file paths, review IDs, severity, and enums.
- Integration in `cmd/atcr/review.go`, `cmd/atcr/reconcile.go`, `cmd/atcr/verify.go`, `cmd/atcr/report.go`.
- All validation errors are wrapped in `usageError` for exit code 2.
- Unit tests for each validator.
- Integration test: invalid input → clear error message before execution.

### Out of Scope

- Validation of persona prompt content (out of scope; prompts are opaque strings).
- Validation of LLM model names (provider-specific, changes over time).
- Validation of API keys (format varies by provider).
- Validation of configuration files (Epic 4.2 covers config validation).

## Clarifications

### Recorded 2026-06-18 (rubber duck — /execute-epic)

**Key Decisions:**
- `GitRef` ships in `internal/validation` (satisfies AC1/AC2/AC7) but is NOT wired onto `--base`/`--head`: those flags accept git *revisions* (`HEAD^`, `HEAD~3`, `@{...}`, `:/msg`) that the spec'd validator rejects, and existing tests rely on `--base HEAD^`. Option-injection is already closed by `--end-of-options` in `internal/gitrange`, so `GitRef` adds no security value on these flags. (Q1 → Option A)
- `FilePath` is applied to `review --output-dir` AFTER `filepath.Abs` resolution (inside `outputDirFromFlags`), so legitimate relative paths (`../reviews`) resolve to a clean absolute path and pass, while the `/etc`,`/proc`,`/sys` system-dir guard (AC3) still fires. Wire it post-Abs as the sole validator. (Q2 → Option A)
- Existing severity (`reconcile.ParseSeverity`) and format (`report.ValidFormat`) validation paths are left as-is. `validation.Severity`/`Enum` ship in the package for AC5/AC7 + future use but do NOT replace those paths. (Q3 → Option A)
- `--id` is already guarded by `fanout.ValidateReviewID` (path-traversal allowlist `^[A-Za-z0-9][A-Za-z0-9._-]*$`) — it is NOT an unvalidated gap. `validation.ReviewID` is NOT wired onto `--id`, to avoid duplicating/weakening that security guard ("apply new validation only where genuinely missing"). The fanout traversal check is preserved untouched.
- New integration is therefore limited to the genuine path-validation gaps: `validation.FilePath` on `review --output-dir` (post-Abs) and on `report --output` (post-Abs, only when non-empty). `reconcile`/`verify` inputs (id-or-path arg via `anchorDir`/`resolveReviewDir`, `--fail-on`/`--min-severity` via `ParseSeverity`) are already validated, so no duplicate validation is added there.

**Scope Boundaries:**
- IN scope: `internal/validation` package (`GitRef`, `FilePath`, `ReviewID`, `Severity`, `Enum`, `ValidationError`) with 100% coverage; `FilePath` wiring on `review --output-dir` and `report --output`; all such errors wrapped in `usageError` (exit 2).
- OUT of scope: wiring `GitRef` onto `--base`/`--head`; replacing existing severity/format validation; wiring `ReviewID` onto `--id` (already guarded); config/persona/model/API-key validation.

**Technical Approach:**
- `FilePath` wirings validate the absolute (cleaned) form so relative paths are preserved; each guarded by a non-empty check (empty `--output` = stdout, empty `--output-dir` = default `.atcr/reviews/<id>/` layout).
- AC8 (`atcr review --base "invalid..ref"` fails fast): satisfied by the pre-existing fast-fail — `gitrange.Resolve` rejects `invalid..ref` via git before any agent/API call and returns `usageError` (exit 2). Accepted trade-off of Q1→A: the message stays git-flavored rather than validation-flavored, because wiring `GitRef` onto `--base` was deliberately rejected to preserve revision syntax.

**Original Questions and Answers:**
1. Q: GitRef on `--base`/`--head` — which wiring (ship-only / loosen / wire-as-spec'd)?
   A: Option A — ship `GitRef` in the package, do NOT wire onto `--base`/`--head` (they take revisions, not ref names; option-injection already closed by `--end-of-options`; `--base HEAD^` is the documented CI-gate pattern with many existing tests).
2. Q: FilePath on `--output-dir` — validate raw value or post-Abs?
   A: Option A — apply `FilePath` after `filepath.Abs` (relative paths resolve clean and pass; system-dir guard still fires); wire post-Abs as the sole validator on that call site.
3. Q: Severity & format already validated — adopt new validators or replace existing paths?
   A: Option A — leave existing paths as-is; ship `Severity`/`Enum` in the package for AC5/AC7 + future use. Corrections: `ParseSeverity` is NOT shared with Epic 4.2 config validation (separate paths); `--id` is already guarded by `fanout.ValidateReviewID` and must NOT be removed or weakened.
