# Epic Plan 11.0: Executing Reviewers

**Estimated Durations**: 3 weeks
**Tasks/Components:** 5/3
**Execution:** execute-epic --strong
**Status:** Ready — elaborated 2026-06-25 (Task Breakdown T1–T5 + concrete Success/Acceptance Criteria added; depends on Epics 2.0–6.0, all complete)

## Objective

Let agents *demonstrate* findings instead of asserting them: an opt-in execution mode where designated agents can run the project's test suite and minimal repro scripts inside a sandbox, attaching command + output as evidence. The EVIDENCE column becomes executable proof.

## Context

Stages 2–4 ground findings in read code and adversarial challenge, but a reproduced failure is a categorically stronger deliverable: it cannot be a hallucination, and it hands the resolver a failing command to start from. Execution requires real sandboxing — untrusted-model-authored code running on the user's machine — which is why this is the last rung and strictly opt-in.

## Problem Statement

1. Even verified findings are static analysis; some bug classes (race conditions, environment-dependent behavior, integration failures) only show under execution.
2. Resolvers spend time reconstructing repro steps that the reviewing agent already reasoned out.
3. Arbitrary execution of model-authored code without containment is an unacceptable security posture for a shareable tool.

## Proposed Solution

- **Opt-in only:** `atcr review --exec` / `verify --exec`; off by default; refuses to run without an explicit sandbox backend configured.
- **Sandbox backend:** container-based (Docker/Podman) executor: snapshot mounted read-only, writable scratch overlay, no network, CPU/memory/time caps, non-root. Backend pluggable; bare-metal execution intentionally unsupported.
- **New tools (execution-enabled agents only):** `run_tests(target?)` (project test command from config), `run_script(content, timeout)` (repro scripts in scratch). Outputs captured, truncated to budget, attached to the transcript.
- **Findings integration:** `evidence_exec` block in findings JSON — `{command, exit_code, output_excerpt}`; report renders "Reproduced" badges; confidence: reproduced findings are VERIFIED by definition.
- **Primary consumers:** Epic 3.0 skeptics (refute or confirm by running) and a dedicated `repro` role for high-severity findings.

## Task Breakdown

Five tasks, T1–T5, dependency-ordered. Each is implemented via TDD (complex tasks use RED → GREEN → ADVERSARIAL → REFACTOR).

### T1 — Sandbox backend package + Docker executor + preflight check
- **New package `internal/sandbox/`:**
  - `sandbox.go` — `Backend` interface: `Preflight(ctx) error`, `Run(ctx, RunSpec) (RunResult, error)`. `RunSpec{ Command []string; Script string; Timeout time.Duration; SnapshotDir string }`. `RunResult{ Command string; ExitCode int; Output string; TimedOut bool }`.
  - `docker.go` — Docker backend. Container flags: snapshot mounted **read-only** (`-v <snap>:/work:ro`), writable scratch overlay (tmpfs `/scratch`), `--network none`, `--memory`/`--cpus`/`--pids-limit` caps, non-root (`--user`), `--security-opt no-new-privileges`, `--read-only` rootfs.
  - `preflight.go` — verify `docker` CLI present + daemon reachable (`docker info`), base image available, trivial container runs and exits 0.
- **Backend pluggable; Docker only initially** (host has Docker v29 at `/opt/homebrew/bin/docker`; no Podman).
- TDD: **complex**. Preflight failure paths, timeout, non-zero exit, output capture.

### T2 — run_tests / run_script tool-loop extensions
- Register two tools, available **only to execution-enabled agents**:
  - `run_tests(target?)` — runs the component `testing.cmd` from config inside the sandbox; `target` scopes it.
  - `run_script(content, timeout)` — writes `content` to sandbox scratch and executes it.
- Output captured, **truncated to budget**, attached to the transcript.
- TDD: **complex**. Tool-gating (non-exec agents must NOT see these tools), truncation, timeout propagation.

### T3 — evidence_exec block in findings JSON + repro write-back + determinism rule
- **Schema (additive, `omitempty`):** add `*EvidenceExec` to `JSONFinding` in `internal/reconcile/emit.go` with `json:"evidence_exec,omitempty"`. Struct `{ Command string; ExitCode int; OutputExcerpt string }`. **Internal only** — do NOT touch the public library (`reconcile/finding.go`, `reconcile/verification.go`).
- **Repro write-back:** on a reproduced finding, stamp `Verification.Verdict = confirmed`, `skeptic = "repro"`, AND the `EvidenceExec` block on the same `JSONFinding`.
- **Determinism (2-run rule):** execute the repro command **twice**. Both non-zero with the **same exit code** → `confirmed` (→ VERIFIED via existing `ConfidenceForVerdict`). Exit codes disagree → `Verification.Verdict = unverifiable` (existing constant). Boundary tests: one run (no verdict), two matching (valid), two mismatching (unverifiable).
- **Gate/confidence: no new code** — `ConfidenceForVerdict` (`reconcile/confidence.go:18`) already promotes confirmed→VERIFIED; `IsFailing` (`internal/reconcile/gate.go:96`) already treats VERIFIED correctly.
- **Planted-bug fixture (SC-3 e2e):** integration test — repro runs against a fixture with a known failure, attaches failing output, finding marked Reproduced.
- TDD: **complex**.

### T4 — --exec opt-in flag + config validation + refuse-without-backend gate
- Add `--exec` to `atcr review` and `atcr verify`; **off by default**.
- Config: sandbox backend block in `.atcr/config.yaml` (backend type, image, caps).
- **Refuse-to-run gate:** `--exec` without a configured backend that passes preflight → hard error, no execution.
- TDD: **complex**. Default-off, refuse-without-backend, refuse-on-preflight-failure paths.

### T5 — Report "Reproduced" badge + docs
- `internal/report/render.go` `writeSkepticBlock` (~line 365): new branch keyed on `f.EvidenceExec != nil` → render a **"Reproduced" label** (NOT a new verdict enum — keeps the library `Verification` type stable) plus command / exit code / output excerpt.
- Docs: `--exec` mode, container-runtime requirement, sandbox security posture, opt-in surface.
- TDD: **complex** (rendering branch + golden-file assertions).

## Success Criteria

- [x] **SC-1 (T4):** Execution is unavailable unless `--exec` is explicitly passed AND a configured sandbox backend passes a preflight check; otherwise the command hard-errors without executing anything.
- [x] **SC-2 (T1):** Sandboxed runs are network-isolated (`--network none`), resource-capped (memory/cpu/pids), non-root, and leave the read-only snapshot unmodified.
- [ ] **SC-3 (T2+T3):** A planted-bug fixture is reproduced end-to-end — an execution-enabled skeptic writes a repro, runs it in the sandbox, attaches failing output, and the finding is marked Reproduced (`evidence_exec` populated, verdict `confirmed`).
- [ ] **SC-4 (T3+T5):** Execution evidence flows through `findings.json` (`evidence_exec` block) → `report.md` ("Reproduced" badge) → gate semantics (reproduced ⇒ VERIFIED ⇒ counted by `IsFailing`).
- [x] **SC-5 (T1–T5):** All quality gates green — `go test ./...`, `go vet ./...`, `golangci-lint run`, `go build`, coverage ≥ 80%.

## Acceptance Criteria

- [x] **AC-SECURITY (HARD STOP — overrides Phase 8 auto-merge):** `/execute-epic` MUST NOT auto-merge this epic. After CI passes, HARD STOP and emit a security-review checklist covering: container escape vectors, network-isolation enforcement, resource-cap configuration, read-only snapshot integrity, and the opt-in flag surface. The human owner reviews and merges manually via `gh pr merge --squash --delete-branch`.

## Out of Scope

- Auto-fix / patch generation; performance benchmarking; bare-metal execution; long-running service testing.

## Dependencies

Epics 2.0–6.0; container runtime present on the host; project test command configured in `.atcr/config.yaml`.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Sandbox escape / resource abuse by model-authored code | Low | Critical | Container isolation, no network, caps, read-only snapshot, opt-in; security review before release |
| Flaky tests poison evidence | High | Medium | Repro must be deterministic (re-run twice rule); flaky results downgraded to `unverifiable` |
| Setup friction (container runtime) limits adoption | High | Low | Strictly optional; all prior stages work without it |
