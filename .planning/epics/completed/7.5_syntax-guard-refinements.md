## Current State

The Epic 7.1 local syntax guard (`internal/verify/syntaxguard.go`) parses a generated fix with `go/parser` and flags it only when the content is plausibly Go code yet fails to parse. The guard is deliberately calibrated for **conservative recall**: it prefers a false negative (a malformed fix passes unflagged) over a false positive (a legitimate fix is flagged), because a false "invalid" flag degrades trust ‚Äî the exact failure mode the guard exists to prevent.

`looksLikeGoCode` keys on line-structural signals (declaration keyword, trailing open brace, leading close brace). Fenced non-Go content is already suppressed via the `nonGoFenceLangs` map. The residual gap is **unfenced** non-Go content with block braces.

## Pain Points

### Unfenced JSON/config false positive (residual)
An unfenced, multi-line JSON or config snippet with block braces (`{` at line end, `}` at line start) satisfies `looksLikeGoCode`, fails to parse as Go, and is flagged `invalid_syntax` ‚Äî a spurious flag on non-Go content. This is the residual false-positive surface after the 7.1 heuristic hardening. The common LLM output shape (a fenced ```` ```json ```` block) is already handled by `nonGoFenceLangs`; only the unfenced case remains, which is narrow.

## Proposed Solution

Detect obviously non-Go brace content before treating block braces as a Go code signal ‚Äî e.g. recognize JSON-object / `key: value` line shapes (quoted keys, trailing commas, no Go declaration keywords) and exclude them from `looksLikeGoCode`. The detection must stay conservative: it should only suppress flagging (reduce false positives), never add new flagging, so it cannot regress the 7.1 trust guarantee.

## Clarifications

### Recorded 2026-06-22 (rubber duck ‚Äî /execute-epic)

**Key Decisions:**
- `looksLikeGoCode` is consulted ONLY after `parseGoFix` has already failed
  (`validateGoFixSyntax`), so valid Go never reaches it. The new non-Go detection only
  affects already-failed-to-parse content, making "looks like JSON ‚Üí suppress" correct
  and AC2 ("no input previously unflagged becomes flagged") structurally guaranteed ‚Äî
  the change only adds a `return false` path; nothing newly becomes flagged.
- Non-Go detection anchors on QUOTED keys: a line beginning with `"key":` plus the
  absence of any Go declaration keyword. Unquoted `key:` is intentionally NOT a non-Go
  signal ‚Äî Go uses bare `ident:` in struct literals, labels, cases, and map entries, so
  keying on it would risk misclassification. Follows the plan's "quoted keys" wording.
- A broken Go map/struct literal with string keys that fails to parse would be
  suppressed (a false negative). Explicitly acceptable under the 7.1 conservative-recall
  policy (false negative preferred over false positive) ‚Äî AC4.

**Scope Boundaries:**
- IN scope: `looksLikeNonGoBraces` (JSON/config) detection in `internal/verify/syntaxguard.go`
  ahead of the block-brace signal in `looksLikeGoCode`; unfenced JSON/config
  characterization tests in `syntaxguard_test.go`; all existing Epic 7.1 tests still pass.
- OUT of scope: multi-fence validation and `parseGoFix` strategy tightening (both recorded
  WONTFIX); any change that increases flagging / false positives.
