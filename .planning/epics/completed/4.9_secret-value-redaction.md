# Epic 4.9: Exact-Value Secret Redaction in Logs

- **Estimated time**: 0.5–1 day
- **Status**: active
- **Source**: deferred from `/resolve-td` (sprint 4.0_structured_logging code review)

## Current State

The log Redactor (`internal/log/redact.go`) supports two redaction layers:
1. **Exact-value secret scrubbing** — `NewRedactor(root, secrets...)` replaces each
   configured secret value (and its URL-encoded form) with `[redacted]`.
2. **Token-shape scrubbing** — regexes for `sk-…` and `Bearer …`, always applied.

In production the exact-value layer is **dead**: every construction site passes
only a path root and **no secret values**:
- `cmd/atcr/run_helpers.go:24` — `NewRedactor(resolveRedactRoot(...))` (per-review redactor, shared by `review` and `resume`; was `review.go:172`)
- `internal/mcp/handlers.go:97` — `NewRedactor(root)` (per-review)
- `cmd/atcr/main.go` (setupLogger) — `NewRedactor("")` (root logger, AC5 token shapes only — added in 4.0 TD resolution)

So redaction of real provider keys relies entirely on the `sk-`/`Bearer` regexes,
which miss raw keys that lack those prefixes: Google `AIzaSy…`, Azure `api-key`,
JWTs `eyJ…`. The AC5 "no API key in log output" guarantee holds only for
sk-/Bearer-shaped keys; the passing integration test likely uses an sk- key,
masking the gap.

## Pain Points

- A non-sk-/non-Bearer provider key logged in an error or diagnostic line leaks verbatim.
- The dead exact-value loop gives false confidence that AC5 fully holds.

## Proposed Solution

Thread the resolved registry API key values into the Redactor at every
construction site so the exact-value scrub is live in production.

1. Enumerate the configured key values — discoverable from `prep.Slots` /
   `cfg.Registry` (the resolved API keys per slot/provider).
2. Pass them as the variadic `secrets...` to `NewRedactor(root, keys...)` at:
   - `cmd/atcr/run_helpers.go:24` (per-review redactor shared by review/resume)
   - `internal/mcp/handlers.go:97`
3. Decide whether the root logger (`setupLogger`) can also receive key values, or
   whether that stays token-shape-only (keys may not be resolved at root-logger
   construction time — likely yes, stays shape-only there).

## Scope

### In Scope
- Key enumeration helper over `prep.Slots` / `cfg.Registry`.
- Wiring resolved key values into `NewRedactor` at the two per-review sites.
- An integration test using a non-sk-shaped key (e.g. `AIzaSy…`) asserting it is
  redacted in log output.

### Out of Scope
- The root-logger construction path (already covered for token shapes by 4.0).
- The scorecard Diag→logger adapter (separate deferral; the store-path leak was
  base-name mitigated in 4.0).

## Clarifications

### Recorded 2026-06-20 (rubber duck — /execute-epic)

**Key Decisions:**
- The enumeration helper applies a minimum key-length guard: any resolved env value with
  `len(s) < 8` is silently skipped before being passed to `NewRedactor`, symmetric with the
  existing empty-string guards at `internal/log/redact.go:49,59`. Threshold = 8 because no
  real provider key is shorter (OpenAI ~51, Google `AIzaSy…` 39, Azure 32-hex, JWTs longer),
  and an exact 8+ char verbatim collision in a log line is negligible.

**Scope Boundaries:**
- IN scope: key-enumeration helper over `prep.Slots` (resolving `Agent.Invocation.APIKeyEnv`
  via `os.Getenv` at redactor-construction time, deduped, length-guarded); wiring the resolved
  values into `NewRedactor` at the two per-review sites (`cmd/atcr/run_helpers.go:24` and
  `internal/mcp/handlers.go:97`); an integration test using a non-sk-shaped key (`AIzaSy…`).
- OUT of scope: the root-logger construction path (`cmd/atcr/main.go`); the scorecard
  Diag→logger adapter.

**Technical Approach:**
- Actual secret strings are never persisted; the only live source is `os.Getenv(APIKeyEnv)`
  resolved from `prep.Slots` (Primary + Fallbacks). `cfg.Registry` holds env-var names only.
- The helper mirrors `preflightAPIKeys`' slot→agent→`APIKeyEnv` walk, dedupes by resolved
  value, and is reachable by both `cmd/atcr` and `internal/mcp` (lives on `fanout`).
- `correlateAndRedact` and `reviewContext` signatures are extended to receive the prepared
  review (or its resolved secrets); both call sites already hold `prep`.

**Original Questions and Answers:**
1. Q: Minimum key-length guard for the enumeration helper — which threshold (8, 16–20, or other)?
   A: Go with 8 (option a). `len(s) < 8` → skip, symmetric with the existing empty-string
      guard at `internal/log/redact.go:49,59`. (Confidence: 92%)
