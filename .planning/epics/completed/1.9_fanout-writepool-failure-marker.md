# Tech Debt: fanout-writepool-failure-marker

**Epic:** 1.9
**Estimated Duration:** TBD (small: ~2–4h)

## Current State

When `WritePool` fails mid-loop after one or more agents have already had their
per-agent artifact files written to disk, `ExecuteReview` calls
`writeFailureSummary` to leave a terminal signal for the status reader. The
failure summary correctly uses `summarize(results)` (so `Succeeded`/`Failed`
counts reflect in-memory outcomes), but `Partial` may be `false` even when some
agents succeeded — because the failure happened during the write phase, not the
run phase. A reconcile invoked against such a review can walk the surviving
per-agent artifact directories (via `Discover`), find real findings, and emit a
non-partial verdict that silently drops the one agent whose artifacts were not
yet flushed.

The reconcile package itself does **not** read `pool/summary.json` — it receives
`opts.Partial` threaded from the caller (`ExecuteReview` → reporting layer).
Adding a struct field alone (without updating the caller) is a no-op. Both
changes must land together.

## Pain Points

- A mid-loop `WritePool` write failure after partial success produces a
  `PoolSummary` with `Partial: false` even though one or more agents succeeded.
- The caller (`ExecuteReview`) does not re-read the failure marker before
  threading `opts.Partial` to any downstream reconcile.
- Post-hoc analysis tools that do read `pool/summary.json` will see a summary
  that contradicts the agent artifact files on disk.

## Proposed Solution

Two changes that must land in the same commit:

1. **Add `FailureMarker bool` to `PoolSummary`** (`internal/fanout/artifacts.go`)
   — `writeFailureSummary` sets it `true`; `WritePool` never sets it. This makes
   the summary unambiguously machine-readable as a best-effort marker rather than
   a real run record.

2. **Caller-side `opts.Partial` correction** (`internal/fanout/review.go` or
   whichever layer reads `pool/summary.json` before calling `RunReconcile`) — if
   `PoolSummary.FailureMarker` is `true` AND `ps.Succeeded > 0`, force
   `opts.Partial = true` so reconcile does not emit a non-partial verdict from a
   subset of agents.

The reconcile package (`internal/reconcile/`) requires **no changes** — it is
the caller's responsibility to thread the correct `opts.Partial`.

## Scope

### In Scope

- `internal/fanout/artifacts.go` — add `FailureMarker bool json:"failure_marker,omitempty"` to `PoolSummary`; set it in `writeFailureSummary`.
- `internal/fanout/review.go` (or the reporting layer that threads `opts.Partial`) — read `FailureMarker` from the failure summary and force `opts.Partial = true` when needed.
- Tests: a forced-`WritePool`-failure test asserting the `FailureMarker` is set and that a subsequent reconcile call receives `opts.Partial = true`.

### Out of Scope

- Changes to `internal/reconcile/` — the reconcile package's contract is that `opts.Partial` is threaded by the caller.
- Changes to the `atcr status` / `ReadReviewStatus` path — stale inference already covers the no-summary case.
