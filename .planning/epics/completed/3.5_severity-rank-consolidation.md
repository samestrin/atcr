# Tech Debt: Consolidate the severity-rank rubric into one canonical owner

- **Estimated time**: ~120 min
- **Plan #**: 3.5
- **Slug**: severity-rank-consolidation
- **Source**: Deferred from `/resolve-td` (Sprint 3.3_per_run_scorecard code review, Q1)

## Current State

The severity ordinal rubric `{CRITICAL:4, HIGH:3, MEDIUM:2, LOW:1}` plus its
`normalizeSeverity` helper are redefined in more than one place rather than
sourced from a single owner:

- `internal/reconcile` — `SeverityRank` is the **de facto canonical owner**; both
  `internal/report` and `internal/verify` already import it.
- `internal/fanout/postprocess.go:14-17` — an **independent** rank map, added by
  Epic 2.2. It looks up `severityRank[strings.ToUpper(...)]`.
- `internal/registry/config.go` — `reviewSeverities`, a **presence-only validation
  set** (not a rank map), so it would not consolidate cleanly into a rank map.

The lookup asymmetry is the real hazard: `postprocess` upper-cases the key before
lookup while `reconcile/merge.go:104` looks up the raw value. A future severity
rename or a non-canonical casing would silently desync fan-out truncation from
reconcile merging.

## Pain Points

- **Silent desync risk** — two independent rank maps with different casing
  contracts; a change to one does not propagate, and the divergence is invisible
  until rankings disagree in production.
- **Casing asymmetry** — `severityRank[strings.ToUpper(...)]` in postprocess vs a
  raw-value lookup at `internal/reconcile/merge.go:104`.
- **No single source of truth** — adding/renaming a severity requires editing
  multiple files with no compiler or test enforcing they stay in lockstep.

## Proposed Solution

Extract the canonical ordered rank map plus `NormalizeSeverity` into
**`internal/stream`** and migrate consumers to it.

`internal/stream` is the correct owner: it has **zero internal dependencies** and
is **already imported by both `internal/fanout` and `internal/reconcile`**, so it
introduces no import cycle. It is deliberately NOT a new `internal/severity`
package and NOT `internal/reconcile` — placing the canonical map in `reconcile`
would force a `fanout → reconcile` dependency that `postprocess.go:14-16`
intentionally avoids.

- Add the exported rank map + `NormalizeSeverity` to `internal/stream`.
- Migrate `internal/reconcile` (`SeverityRank` and the raw lookup at
  `merge.go:104`), `internal/fanout/postprocess.go`, `internal/verify`, and
  `internal/report` to the shared definition.
- Fix the raw-vs-upper-cased lookup asymmetry as part of the migration so all
  consumers normalize identically.
- Leave the registry `reviewSeverities` presence-set as-is (it is validation, not
  ranking) unless it can derive its key set from the canonical map cleanly.

## Scope

### In Scope
- New canonical rank map + `NormalizeSeverity` in `internal/stream`
- Migration of `reconcile`, `fanout/postprocess`, `verify`, `report`
- Fixing the casing-asymmetry lookup at `internal/reconcile/merge.go:104`
- Tests proving a single source of truth (delete local maps; suite still green)

### Out of Scope
- The registry `reviewSeverities` presence-only validation set (different shape)
- Any change to severity values or ordering (pure consolidation, no behavior change)

changes) and was therefore deferred to its own plan rather than resolved inline.

All other reconcile lookup sites operate on provably-canonical data, so normalizing them would be a no-op and is out of scope.
- Cross-package consumers (`fanout/postprocess`, `verify`, `report`) all route their lookups through `stream.SeverityRank` + `stream.NormalizeSeverity`; their local maps/normalizers are deleted.

**Scope Boundaries:**
- IN scope: stream rank map + `NormalizeSeverity`; migrate reconcile (alias + merge.go:104 fix + disagree.go:338/360 fix), fanout/postprocess, verify, report.
- OUT of scope: `registry.reviewSeverities` presence-set and `registry.normalizeSeverity`; the `reconcile.Sev*` constants (stay in reconcile); any severity value or ordering change.

**Technical Approach:**
- `stream.SeverityRank` keyed by string literals; `stream.NormalizeSeverity(s) = strings.ToUpper(strings.TrimSpace(s))` — identical to the three existing normalizers being consolidated.
- Severities reaching real lookups are already canonical upstream, so adding `NormalizeSeverity` at the two boundary sites is behavior-preserving for valid data and only fixes mixed-case/whitespace handling (the intended fix). Pure consolidation — no severity value/order change (AC6).
