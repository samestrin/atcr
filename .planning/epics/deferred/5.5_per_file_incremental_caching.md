# Epic 5.5: Per-File Incremental Diff Caching

**Status:** Deferred (far-future; depends on Epic 5.2 — complete)
**Source:** Deferred Option B from the Epic 5.2 rubber-duck clarification (Q1, 2026-06-20)
**Scope:** Re-architect the review fan-out from payload-level caching to per-file (per-chunk) incremental caching. Significant architectural change — explicitly out of scope for 5.2.

## Problem

Epic 5.2 caches at **payload granularity**. `buildPayloads` (internal/fanout/review.go) concatenates every changed file into one text blob per payload mode before any agent is called, so the cache unit is per-agent-per-payload. A re-run over a byte-identical diff is a full hit and skips the LLM call entirely — but changing even one file in a 50-file diff alters the blob's hash and is a **full miss for every agent**. The cost model therefore scales with `#agents × payload`, not with `#changed-files`: a 1-line change on a large branch still re-reviews (and re-pays for) all 50 files.

This is acceptable for the dominant 5.2 use case (re-running an unchanged review iteratively), but leaves the original epic's stated motivation only partially served:

> "Running a review on a branch with 50 changed files costs the same whether 1 file or 50 files changed since the last run."

5.2 does not fix that case; this epic does.

## Goal

Serve unchanged file chunks from cache even when *other* files in the same diff changed, so a re-review only spends tokens on the files that actually changed since the last run — true incremental caching.

## Why This Needs Architectural Change

The 5.2 design cannot deliver per-file hits without re-architecting two chokepoints:

1. **`buildPayloads`** flattens all files into one blob per mode (review.go ~491). Per-file granularity requires keeping the per-file boundary intact through payload assembly.
2. **The engine** sends exactly one prompt per agent (`runEngine` → `NewEngine().Run`). Per-file caching requires either many calls per agent or a payload-splicing scheme.

Two candidate architectures, both substantial:

- **(A) Per-file API calls.** Each agent reviews each changed file in a separate call; unchanged files are served from a per-file cache. Pro: clean cache semantics, exact incremental cost. Con: loses whole-diff context (a reviewer seeing files in isolation can miss cross-file issues — broken call sites, contract drift), multiplies prompt/system-overhead tokens per file (can be *more* expensive for small diffs), and changes latency/concurrency characteristics.
- **(B) Payload splicing.** Split the payload into cached (unchanged-file) and fresh (changed-file) segments; send only the fresh segment to the LLM and merge the cached findings for the unchanged segment into the pool. Pro: keeps one call per agent and most cross-file context. Con: cached findings were generated against a *different* surrounding payload, so their validity under the new context is not guaranteed; merging stale and fresh findings needs a defensible correctness story.

Choosing between (A) and (B) — or rejecting both — is the core design question and the reason this is far-future rather than a 5.2 follow-up.

## Acceptance Criteria

- [ ] A re-review where 1 of N changed files differs re-reviews only the changed file(s); unchanged files are served from cache.
- [ ] Cache keys are per-file (per-chunk) and still incorporate the model + persona + tuning (the 5.2 invalidation contract, applied at file granularity).
- [ ] Cross-file review quality is measured against the whole-payload baseline and the regression (if any) is quantified and accepted, OR a context-preserving design (B) is proven correct.
- [ ] Cost is genuinely incremental for partial-change diffs and not worse than 5.2 for small/whole-diff changes (benchmark both).
- [ ] Tool-enabled agents and the verification stage remain excluded from caching (unchanged from 5.2).
- [ ] `--no-cache` and `cache_max_bytes` semantics carry over unchanged.

## Out of Scope

- Changing what is cached for tool/verify stages (still excluded).
- Cross-run caching keyed on commit SHAs rather than content (a separate idea).

## Dependencies

- Epic 5.2 (Diff Caching & Incremental Reviews) — complete. This epic replaces 5.2's payload-level cache unit with a finer-grained one; the cache package (`internal/cache`), config plumbing (`cache_max_bytes`), and `--no-cache` flag are reused.

## Risks / Open Questions

- Cross-file context loss (design A) could degrade finding quality more than the cost saving justifies — needs an empirical quality gate before committing.
- Findings cached under a different payload context (design B) may be stale/invalid — needs a correctness argument, not just a hash match.
- For small diffs, per-file overhead may make caching a net cost increase — the design must detect and fall back to whole-payload review below a file-count threshold.
- Est: 1–2 weeks of design + spike before any implementation estimate is meaningful.
