### 1. [P1 / api] ListReviewComments is not paginated — dedup silently breaks past 30 existing comments
- **Cluster:** providers-io
- **Location:** `internal/ghaction/comments.go:87`
- **Confidence:** HIGH
- **Problem:** ListReviewComments GETs /repos/{owner}/{repo}/pulls/{pr}/comments with no per_page parameter and no Link-header pagination loop (the shared get() helper never follows pagination). GitHub returns at most 30 results per page by default, so on a PR that already carries more than 30 inline review comments only the first page is returned. deduplicateComments (cmd/atcr/github.go:299) builds its 'already present' set solely from this truncated list.
- **Why it matters:** On any actively-reviewed PR, repeated CI runs accumulate ATCR inline comments. Once the PR has >30 review comments, every re-run re-posts the comments that live on page 2+ because they are absent from the dedup set, spamming the PR with duplicates — the exact failure the dedup pass exists to prevent. It is invisible in tests because no test exercises more than one comment.
- **Suggested fix:** Append ?per_page=100 and follow the RFC 5988 Link 'rel=next' header in get()/ListReviewComments, accumulating all pages before returning; or at minimum request per_page=100 and document the residual cap. Add a test with a paged httptest server (page 1 sets a Link: <...page=2>; rel="next" header) asserting all comments are collected.
- **Verification:** Confirmed at internal/ghaction/comments.go:87-94. ListReviewComments GETs /repos/{owner}/{repo}/pulls/{pr}/comments with no per_page param and no Link-header pagination. The shared get() helper (internal/ghaction/client.go:200-247) issues exactly one request and json.Unmarshals a single body — it never inspects the Link header or loops, so it cannot follow pagination (the 8MB io.LimitReader does not help; GitHub paginates server-side at a 30/page default for list endpoints). The truncated list flows directly into deduplicateComments (cmd/atcr/github.go:208-212), which builds its 'already present' set solely from `existing` (github.go:303-308). Re-run idempotency is the explicit purpose of this dedup pass (see github.go:259 'Re-runs are idempotent because...'). On a PR carrying >30 inline review comments, ATCR findings anchored on page 2+ are absent from the dedup set and get re-posted on every CI re-run via CreatePRReview/CreateReviewComment — the exact duplicate-spam the dedup exists to prevent. No guard, retry, or caller mitigates it. ListReviewComments is the only list-returning consumer of get(). P1 is defensible for a CI tool whose core idempotency guarantee silently fails under realistic heavy-review conditions; severity is mildly conditional (requires >30 comments) but not inflated.

### 2. [P2 / resilience] git ls-files in BuildFileIndex is not cancellable (no context/timeout)
- **Cluster:** core-pipeline
- **Location:** `internal/stream/fileindex.go:46`
- **Confidence:** HIGH
- **Problem:** BuildFileIndex runs `exec.Command("git", "-C", root, "ls-files", "-z").Output()` with a plain exec.Command and no context. It is called from validateFindingPaths -> RunReconcile, which threads a context.Context and checks ctx.Err() between the Discover/Reconcile/Emit stages specifically so a client cancel or MCP server shutdown aborts the pipeline. The git invocation here sits inside the Reconcile->Emit window and cannot be cancelled: on a pathological or hung repository (huge tracked file set, slow/locked .git, a wedged git process), reconcile will block in this call regardless of context cancellation. The sibling git runner in internal/payload/diff.go:96 correctly uses exec.CommandContext for exactly this reason.
- **Why it matters:** An MCP server reconcile (or a CI reconcile under a deadline) can hang here unkillable, holding the request goroutine and ignoring graceful-shutdown signals, instead of returning the documented context error. The cost of a hung index build is borne by the whole reconcile rather than degrading gracefully (BuildFileIndex already has a clean nil-returns-degrade path for git failures).
- **Suggested fix:** Thread the reconcile context to BuildFileIndex and use exec.CommandContext(ctx, "git", ...). RunReconcile already has ctx in scope; pass it through validateFindingPaths into BuildFileIndex so a cancel/shutdown kills the git child and the existing nil-index degrade path (existence-only, no suggestion) takes over, matching how diff.go handles git cancellation.
- **Verification:** Confirmed at internal/stream/fileindex.go:46: BuildFileIndex uses plain exec.Command("git","-C",root,"ls-files","-z").Output() with no context/timeout. The call chain is exactly as claimed: BuildFileIndex (via validate.go:24) <- validateFindingPaths (gate.go:225) <- RunReconcile(ctx, ...) (gate.go:167). RunReconcile is explicitly context-aware: it checks ctx.Err() at gate.go:168, 175, and 227 (the 227 check is immediately AFTER validateFindingPaths), and its doc comment (gate.go:159-160) states ctx is checked between Discover/Reconcile/Emit so a client cancel or server shutdown aborts the pipeline. The git call sits inside that window and is NOT cancellable, so a hung/slow/locked git process blocks the reconcile goroutine and ignores cancellation. The sibling runner at internal/payload/diff.go:96 correctly uses exec.CommandContext(g.ctx, ...) with explicit ctx.Err() handling (diff.go:102-104), proving the project's own established pattern. BuildFileIndex already has a clean nil-returns-degrade path (fileindex.go:48-57) for git failures, so the hung-process case is the gap. Severity P2/resilience is justified, not inflated: it requires a pathological repo/wedged git (not a routine path), and the MCP-shutdown / CI-deadline impact is real but conditional.

### 5. [P2 / resilience] Server-advertised Retry-After is honored verbatim with no upper bound
- **Cluster:** providers-io
- **Location:** `internal/llmclient/client.go:667`
- **Confidence:** HIGH
- **Problem:** parseRetryAfter accepts an arbitrarily large delta-seconds value and dispatch sets honorExact=true so the resulting delay bypasses clampBackoff (maxBackoff=30s). time.Duration(secs)*time.Second also overflows int64 for secs greater than ~9.2e9, producing a garbage (possibly negative) duration. A provider can therefore dictate a multi-hour-to-multi-century retry sleep, or an overflowed one.
- **Why it matters:** A hostile or misbehaving endpoint controls how long an agent blocks between retries. The caller's per-agent/global context deadline (via sleepCtx) is the only backstop; a call with a generous or absent deadline can be pinned for an unbounded time, tying up a fan-out worker slot. The deliberate no-clamp on Retry-After trades robustness for politeness without a ceiling.
- **Suggested fix:** Cap the honored Retry-After at a sane ceiling (e.g. a few minutes, or maxBackoff) and reject/clamp values that would overflow time.Duration before multiplying by time.Second.
- **Verification:** Confirmed at the cited location. parseRetryAfter (internal/llmclient/client.go:667-684) returns time.Duration(secs)*time.Second for any positive integer with no ceiling. In dispatch (lines 497-499) a positive retryAfter sets delay=retryAfter and honorExact=true; honorExact then makes the sleep loop skip jitter (line 458) AND skip clampBackoff (the clamp at line 465 runs only AFTER the sleep, on the NEXT delay), so the raw server value is slept verbatim via sleepCtx (line 461). maxBackoff (30s) is deliberately bypassed — confirmed by the comment at lines 686-688 and the absence of any large-value/clamp test in TestParseRetryAfter (largest tested value is "5"). Overflow is real: I compiled and ran the math — secs around 9.22e9 overflows int64, e.g. "9223372037" yields a negative ~-2562047h. However the negative-duration sub-claim is defanged by the `if retryAfter > 0` guard at line 497, which routes negative results back to normal backoff. The exploitable cases remain: (a) a legitimately huge positive value like 9223372036 -> ~292 years, and (b) an overflow that wraps back POSITIVE, e.g. "99999999999" -> ~246 years, both of which pass the >0 guard and are slept verbatim. The only backstop is the ctx deadline in sleepCtx; the per-agent timeout in fanout/engine.go:507 is applied only when a.TimeoutSecs > 0, so a TimeoutSecs==0 agent under a deadline-less parent ctx can be pinned, tying up a fan-out worker slot — impact claim holds. P2 is justified (requires a hostile/misbehaving provider plus a deadline-less caller; not a default crash, but a genuine unbounded-block resilience hole). Minor correction to the finding's wording: the negative/overflowed-negative duration is NOT actually slept (guarded), but the positive multi-century / positive-wrapped-overflow sleep is.

### 8. [P2 / resilience] ghaction retry backoff uses non-interruptible time.Sleep
- **Cluster:** providers-io
- **Location:** `internal/ghaction/client.go:171`
- **Confidence:** HIGH
- **Problem:** Both postDo and get sleep between retries with time.Sleep(backoff). ctx cancellation is only checked once before the sleep, not during it, so a cancellation that arrives mid-sleep is not observed until the sleep completes (up to 250ms*2^n). llmclient correctly uses a context-aware sleepCtx (select on ctx.Done()) for the same purpose.
- **Why it matters:** In a GitHub Action a cancelled job (timeout, user cancel) should abort promptly; a blocked time.Sleep delays teardown and can outlive the surrounding step's grace window. Minor in absolute terms (sub-second to a few seconds) but an inconsistency with the project's own context-cancellation discipline.
- **Suggested fix:** Replace time.Sleep(backoff) with a select on ctx.Done()/time.After(backoff) (or reuse a shared sleepCtx helper) so a cancellation interrupts the backoff.
- **Verification:** Verified at the cited location. internal/ghaction/client.go:171 is time.Sleep(backoff) inside postDo's retry loop, preceded by a single ctx.Err() check at line 168 (before the sleep, not during). The same pattern repeats in get() at line 231 (ctx check at 228). Backoff is 250ms doubling (lines 159/172, 219/232), confirming the up-to ~seconds blocking claim. The project's own llmclient uses a context-aware sleepCtx (internal/llmclient/client.go:714) that selects on ctx.Done() vs timer.C and is called at client.go:461 — exactly the pattern the finding says ghaction should mirror. So the problem exists, is not handled elsewhere (the only ctx check is the pre-sleep one), and the documented inconsistency with llmclient is accurate. P2/resilience is justified, not inflated: bounded sub-few-second delay, teardown-promptness concern rather than a correctness bug.

### 9. [P2 / security] Skeptic prompt-injection sentinel uses math/rand, not crypto/rand like its two sibling paths

### 11. [P2 / resilience] Successful backup reported as a failure when superseded .bak.old cleanup fails

- **Location:** `internal/atomicfs/atomic.go:180`

- **Problem:** In swapStagedBackup, after the staged copy is successfully renamed into place over <src>.bak (the new generation is now durable and correct), a failure of os.RemoveAll(bakOld) returns a non-nil error to the caller. - **Suggested fix:** After a successful renameFn(staged, bak), treat RemoveAll(bakOld) as best-effort: log/swallow the error (or return bak with a nil error) rather than failing the whole backup, since the one-generation contract is already satisfied and the stale .bak.old is cleaned at the next BackupToDotBak entry.
- **Verification:** Confirmed at internal/atomicfs/atomic.go:179-183. In swapStagedBackup, after renameFn(staged, bak) succeeds at line 168 (the new .bak generation is now durable and correct), a failure of os.RemoveAll(bakOld) at line 180 returns a non-nil error (line 181) to the caller, conflating "backup succeeded but superseded prior generation could not be deleted" with "backup did not happen".
