## TD-001 — @stable segment matching must strip :variant suffix + vendor prefix before tokenizing (MEDIUM)
**Origin:** Phase 1, task 1.4 gate review, 2026-07-08
**File:** internal/personas/catalog.go (Phase 3, not yet created) — spec in plan/documentation/openrouter-catalog-api.md
**Issue:** The `@stable` preview-token segment-matching rule is defined only over hyphen delimiters (segment-equals / suffix `-token` / interior `-token-`). It does not specify stripping the `:variant` suffix (e.g. `:free`, `:thinking`) or the vendor prefix before tokenizing, so a preview model carrying a variant suffix (e.g. `...-preview:free`) matches neither suffix `-preview` (string ends in `:free`) nor interior `-preview-` (followed by `:`), and escapes the `@stable` exclusion — a latent false-negative.
**Why accepted:** No live model today carries a preview token combined with a `:variant` suffix (forward-looking rule only). Deferred so the normalization lands in Phase 3's `catalog.go` with its own unit test rather than as Phase-1 prose.
**Fix in:** Phase 3 (task 3.10/3.11, `@stable` channel) — normalize the slug (strip `:suffix` and vendor prefix) before hyphen-segment tokenization; add a `...-preview:free` test case.
**Resolved:** 2026-07-08 — Phase 3 task 3.11 (`0c7e9bd2`/`18f55171`): `slugHasPreviewSegment` strips the `:variant` suffix and vendor prefix, then hyphen-segment-matches `previewTokenSet`; `TestResolveModel_Stable_TD001_StripsVariantSuffix` covers `...-preview:free`.

## TD-003 — resolved-lock advance re-fetches/deletes co-located .md via writePersonaUnit (MEDIUM)
**Origin:** Phase 4, task 4.2.A adversarial review + task 4.11 phase gate (two independent reviewers), 2026-07-09
**File:** internal/personas/upgrade.go (upgradeResolvedLock → writePersonaUnit)
**Issue:** The binding/resolution path advances only the local `model:` lock but persists through `writePersonaUnit`, which (a) re-fetches the persona's co-located `<name>.md` from the community base URL (`ATCR_PERSONAS_URL` — a different service than the catalog that produced the slug), (b) on a 404 for that `.md`, DELETES the local `.md`, and (c) aborts the whole advance on any transient non-404 `.md` error. Data-loss path (gate framing): a binding persona with a locally-authored/customized custom prompt but no upstream `.md` loses its prompt on a mere model-slug bump. Also couples a catalog-resolved advance to the personas `.md` endpoint being reachable, and leaves a dry-run/real-run parity gap (dry-run never exercises the `.md` fetch/delete step, so a dry-run can report an advance a real run then fails on).
**Why accepted (deferred, not fixed inline):** Not exploitable today — ZERO personas ship a `binding:` field, so the resolution path (and thus the `.md` re-sync) is exercised only by test fixtures; the data-loss scenario requires a future binding-carrying persona with a locally-customized `.md`. Reusing `writePersonaUnit` is the story's explicit mandate ("writes continue to flow through writePersonaUnit so install and upgrade stay consistent"), and the correct decoupling interacts with how future personas ship bindings + prompts — a design decision, not a hot fix. The misleading `setModelField` doc-comment (over-claimed "only change written to disk") was corrected inline at the 4.11 gate.
**Fix in:** A future robustness pass (or whenever the first binding-carrying persona ships) — give the resolved path a write-only-yaml lock advance (`refuseSymlinkedIntermediate` + `writeFileAtomic(dest, newYAML)`, no `.md` fetch/delete), so a lock advance never touches the prompt or depends on a second network fetch; this also closes the dry-run `.md`-step parity gap. Add tests: a local custom `.md` survives a resolve-path advance; dry-run vs real-run parity across the full write.
**Resolved:** 2026-07-09 — Phase 4 (post-gate, at maintainer request): the resolved path now writes only the model-bumped YAML via `writeLockYAML` (`refuseSymlinkedIntermediate` + `writeFileAtomic`), never fetching or deleting the co-located `.md`. Data-loss risk removed, the advance no longer depends on the personas `.md` endpoint, and the dry-run/real-run `.md`-step parity gap is closed (neither touches the `.md`). `TestUpgrade_BindingAdvancePreservesLocalMarkdown` locks a local custom `.md` surviving a real advance; the `setModelField` doc-comment now accurately states the resolved slug is the only change written. `upgradeResolvedLock` no longer takes/needs `baseURL`.

## TD-006 — bindingless newer-member fallback can't group a `<family>-<version>-<tier>` slug with its tier siblings (LOW)
**Origin:** Phase 5, task 5.11.A adversarial review, 2026-07-09
**File:** internal/personas/drift.go (deriveFamilyPrefix)
**Issue:** `deriveFamilyPrefix` strips only a single TRAILING version-like hyphen segment, so a slug whose version is not last — e.g. `openai/gpt-5.4-mini` (version `5.4`, then tier token `mini`) — derives to itself (`openai/gpt-5.4-mini`) rather than a family key. Teaching the fallback to strip an interior version segment when a known tier token trails risks reintroducing cross-tier bleed and adds speculative complexity against no shipping persona (all 10 are bindingless pinned slugs whose versions ARE trailing).
**Fix in:** A later epic, only if a real bindingless persona adopts a `<family>-<version>-<tier>` slug shape — strip an interior version segment guarded by a known tier-token suffix allow-list, with a cross-tier-bleed regression test.

## TD-010 — major-jump classifier and advance predicate disagree on version-shaped non-semver tokens (MEDIUM)
**Origin:** Phase 6, task 6.2.A adversarial review, 2026-07-09
**File:** internal/personas/upgrade.go (isMajorJump vs isNewer default branch; versionSegRe at :204)
**Issue:** `versionFromSlug`'s `versionSegRe` (`^v?\d+(\.\d+)*$`) accepts version tokens that `semver.IsValid` rejects — 4+ dotted components (`4.1.2.3`) or leading-zero/date segments (`0125`). For such a token `isNewer` still reports an advance via its `default: local != remote` string-inequality branch, but `isMajorJump` returns false because it requires valid semver on both sides. So a persona locked at `deepseek/deepseek-v4.1.2.3` advancing to `deepseek/deepseek-v5.1.2.3` would advance the lock across the v4→v5 boundary with `MajorJump=false` — no verify flag and no fixture re-check, the exact silent cross-major advance the gate exists to prevent.
**Why accepted:** Requires a version-shaped-but-invalid-semver token on BOTH the locked slug AND the resolved slug that also differ and cross what would be a major boundary. Every real catalog slug characterized by the Phase 1 spike is ≤3-component semver-shaped (`claude-opus-4.8`, `glm-5.2`, `deepseek-v4-pro`, `qwen3-coder-plus`), so no shipping persona can hit this. The gate's stated invariant still holds for every real transition; the divergence is a theoretical hole against a slug shape that does not exist in the live catalog. Fixing now would harden `versionSegRe` (risking Phase 4 `versionFromSlug` regressions) or add a fail-safe branch against inputs no persona produces — speculative complexity this phase.
**Fix in:** A later phase/epic, if a real slug adopts a 4+-component or leading-zero version — make the classifier fail-safe (when `isNewer` advances but both normalized tokens are non-semver, treat as a major jump so the gate + flag still fire) rather than silently advancing, OR tighten `versionSegRe` to only match `semver.IsValid`-acceptable tokens; add a regression test for the divergent-token case.
**Resolved:** 2026-07-09 — fixed inline in Phase 6 (6.5.A) after a SECOND independent adversarial reviewer converged on the same finding and it maps to Story 06's HIGH-impact "silent cross-major advance" risk. `isMajorJump` now falls back to a leading-numeric-component comparison (`leadingMajor`) for version-shaped tokens `semver.IsValid` rejects; establishing-from-empty and alias (no-version) slugs stay ungated. Covered by `TestUpgrade_NonSemverMajorJumpStillGates` + extended `TestIsMajorJump` cases.

## TD-011 — index-derived roster gives a malformed published index entry no per-name tolerance (MEDIUM)
**Origin:** Phase 7, task 7.2.A adversarial review, 2026-07-09
**File:** cmd/atcr/init.go (installCommunityPersonas — nil-roster derivation + install loop)
**Issue:** The Option B nil-roster path derives the install set from the fetched index's own `e.Name` values, which `FetchIndex` does not validate. An index entry whose name is empty or contains invalid characters is a member of `indexed`, so it is NOT skipped, then fails inside `InstallUnit`→`validatePersonaName`, tripping the all-or-nothing `rollback()` that discards every already-installed valid persona and aborts online `init`/`quickstart` for every user. The explicit-roster path tolerates index imperfections via skip-with-warning; the derived path has no equivalent per-name tolerance, so a single malformed PUBLISHED entry becomes a hard first-run dependency.
**Why accepted:** The all-or-nothing rollback on a per-entry INSTALL failure (404 unit, transient fetch error) is AC-mandated (AC 07-03 Edge Case 1) and regression-locked by `TestInstallCommunityPersonas_MidRosterFailure_RollsBack` — it must NOT be softened, so the reviewer's "skip-and-continue on InstallUnit failure" fix is rejected as AC-violating. The genuinely-new surface is narrower: an empty/invalid NAME in the community index (curated + validated by the atcr project itself) turning a would-be-skip into a hard abort. That is low-probability on a curated index, and failing closed with an `--offline` hint is a defensible, non-silent posture (a malformed published entry SHOULD surface loudly). Path traversal via such a name is NOT exploitable — the only writer, `InstallUnit`→`personaPath`, validates and fails closed before any disk write. No path-traversal write, no security exposure.
**Fix in:** A future robustness pass — when building the derived roster from `entries`, pre-filter names failing `validatePersonaName` (skip-with-warning), distinct from the AC-mandated all-or-nothing rollback on genuine per-unit INSTALL failures; add a malformed-index-entry test asserting the valid entries still install.
