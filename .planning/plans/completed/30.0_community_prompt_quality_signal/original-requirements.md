# Epic Plan 30.0: Community Prompt Quality Signal (opt-in, content-free)

- **Estimated time**: TBD
- **Tasks/Components**: TBD
- **Execution**: init-plan

## Objective

Close the persona living-library flywheel by teaching the maintainer *which* reviewer prompts need tuning — from **opt-in, aggregate, content-free** signal. Because atcr reviews **private code**, this signal must be collected so that **never a line of code or a finding body leaves the machine**: per-persona+model counters and identifiers only, opt-in, with a visible local preview of exactly what would be sent. This is the **signal half** split out of Epic 19.9 (which owns intake + curation); it depends on 19.9's community personas plus the dismissal signal (24.0) and the transport (28.0), so it is numbered to run after all three.

## Context

Model-tuned reviewer prompts are refined *in production*: they hallucinate, over-report, flag non-bugs, or miss real ones, and get tuned over time until they earn trust. The **quality signal** ("this prompt over-reports") is *data*, distinct from the prompt itself (which Epic 19.9 handles as a PR). That data must ride content-free telemetry: Epic 24.0 (false-positive / wontfix dismissal) *produces* the raw `dismissed`/`confirmed` events; Epic 28.0 (telemetry / cloud sync) is the *transport*. This epic is the consumer that aggregates them per persona+model, gates them behind an explicit opt-in with a local preview, and surfaces the result to the maintainer.

This closes the flywheel: **drift detected (19.7) → hermes drafts (19.8) → maintainer + users refine in production → dismissal signal (24.0/28.0) flags weak prompts → refined prompt submitted back (19.9) → graduates into the vetted library.** This epic is the "flags weak prompts" arc that points 19.8's drafting and 19.9's submissions at the right targets next.

## Proposed Solution

1. **Quality signal (opt-in, content-free).** Aggregate counters per persona+model (e.g. `dismissed`/`confirmed` derived from the 24.0 dismissal signal) transported via 28.0. Hard privacy line: **zero code and zero finding content ever leaves the machine** — counters and identifiers only, opt-in, with a visible local preview (`atcr … --preview` or equivalent) of exactly what would be sent. Nothing is sent by default.
2. **Close the loop.** Surface the aggregated signal to the maintainer (which prompts over/under-report) so it drives which personas the hermes drafting agent (19.8) and community submissions (19.9) target next.

## Acceptance Criteria

- [ ] AC1: Quality signal is opt-in, aggregate, and content-free — only per-persona+model counters and identifiers are transmitted; a local preview shows exactly what would be sent; nothing is sent by default.
- [ ] AC2: The signal is surfaced to the maintainer in a form that identifies over/under-reporting prompts, closing the loop to 19.8's drafting and 19.9's submissions.
- [ ] AC3: `go test ./...` passes; docs document the exact content-free telemetry contract (fields transmitted, opt-in mechanism, preview behavior, and the absolute no-code/no-finding-content line).

## Out of Scope

- **The submit flow and two-tier curation** — owned by Epic 19.9; this epic consumes its community personas as the keying dimension.
- **The dismissal signal source (24.0) and telemetry transport (28.0) themselves** — this epic consumes them; it does not build them.
- **Transmitting any code or finding content** — the privacy line is absolute; only content-free aggregate counters + identifiers, opt-in.
- **The hermes drafting agent (19.8)** — separate; this epic feeds it targets via the signal loop.

## Dependencies

- **Epic 19.9 (Community Prompt Submissions)** — the community personas + `submitted`/graduated status this signal is keyed by, and one of the loops it closes.
- **Epic 24.0 (false-positive / wontfix dismissal)** — produces the raw `dismissed`/`confirmed` quality signal.
- **Epic 28.0 (telemetry / cloud sync)** — the opt-in, content-free transport for the signal.
- **Epics 19.7 / 19.8** — the drafting/drift arcs this epic's signal steers.
