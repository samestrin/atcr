# Epic Plan 32.3: Sandbox Writable Overlay (Polyglot Auto-Fix Support)

- **Estimated time**: 2-3 days
- **Tasks/Components**: 6 tasks / 3 components
- **Execution**: init-plan

## Objective

Fix the `EROFS` (Read-Only File System) limitation in the ATCR Docker sandbox by implementing an opt-in ephemeral copy strategy, unlocking `--auto-fix` validation for non-Go ecosystems (Node, Rust, Python) that require write access to the project directory during testing.

## Context

Epic 32.0 containerized the `--auto-fix` validation step. To protect the host machine, the user's project directory is mounted read-only (`-v $PWD:/work:ro`). This works perfectly for Go (which writes its caches to a redirected `/scratch` mount), but fails closed for many other ecosystems.
When `npm run build` writes to `dist/`, or `cargo build` writes to `target/`, or Python generates `__pycache__` next to source files, the read-only mount blocks the write. The validator crashes, the fix is assumed invalid, and the PR is silently discarded.

`dockerRunArgs`/`Run` (`internal/sandbox/docker.go`) are shared infrastructure also used by `--exec` (Epic 11.0, via `internal/tools/exec_tools.go`), which documents and pins a hard read-only-`/work` guarantee (`docs/execution.md`, `internal/sandbox/sandbox.go`'s package doc). The fix below is therefore **opt-in per `RunSpec`**, not a global behavior change: `--exec` keeps its exact current read-only-`/work` mount and semantics untouched; only `--auto-fix`'s validation call site opts into the writable ephemeral copy.

## Proposed Solution

1. **New opt-in `RunSpec.Writable` field** (`internal/sandbox/sandbox.go`): defaults to `false`, preserving every existing caller's current behavior (`--exec`'s two call sites in `internal/tools/exec_tools.go`, and the existing `internal/sandbox` test suite) with zero change.
2. **New `DockerConfig.WorkSize` field** (`internal/sandbox/docker.go`), mirroring the existing `ScratchSize` pattern, sized for a full source-tree copy rather than `ScratchSize`'s build-cache-sized default.
3. **Conditional mount in `dockerRunArgs`:** when `spec.Writable` is true, mount `SnapshotDir` read-only at `/src` instead of `/work`, and add a new `--tmpfs /work:rw,exec,size=<cfg.WorkSize>` mount so `/work` actually has writable backing under the container's global `--read-only` rootfs flag. When `spec.Writable` is false (the default), the mount is unchanged: `SnapshotDir` binds directly to `/work:ro`, exactly as today.
4. **Setup-step injection at the actual execution layer, not `renderCommand`:** when `spec.Writable` is true, wrap the real payload — for both `RunSpec` modes, since `--auto-fix` exclusively uses `Command` mode (direct argv exec, no shell) while `--exec` can use either — with a `cp -a /src/. /work/ && cd /work` setup step executed via an explicit shell, before the user's command or script runs. This happens where `dockerRunArgs`/`Run` actually build the argv/stdin the container receives, not in `renderCommand` (which only renders a display string for evidence/logging and has no effect on execution).
5. **`--auto-fix` opts in:** `RunSandboxedValidation` (`internal/verify/sandboxvalidate.go`) sets `Writable: true` on the `RunSpec` it constructs, so non-Go `validate_command`s can write into their working directory. `--exec`'s call sites (`internal/tools/exec_tools.go`) are left unchanged (`Writable` stays false), preserving their documented read-only guarantee with no code or doc changes required there.

## Tasks

- **T1**: Add `Writable bool` to `sandbox.RunSpec` (`internal/sandbox/sandbox.go`), defaulting to `false`; add `WorkSize string` to `DockerConfig` (`internal/sandbox/docker.go`) with a sane default in `DefaultDockerConfig()`, mirroring `ScratchSize`.
- **T2**: Modify `dockerRunArgs` (`internal/sandbox/docker.go`) so that when `spec.Writable` is true it mounts `SnapshotDir` at `/src:ro` and adds a `--tmpfs /work:rw,exec,size=<cfg.WorkSize>` mount; when false, the existing `-v SnapshotDir:/work:ro` mount is unchanged.
- **T3**: Modify the argv/stdin construction in `dockerRunArgs`/`Run` (`internal/sandbox/docker.go`) so that when `spec.Writable` is true, both `Command`-mode (wrap via an explicit shell: `/bin/sh -c 'cp -a /src/. /work/ && cd /work && exec "$@"' -- <command...>`) and `Script`-mode (prepend the copy step to the script body before it is fed to `sh -s` over stdin) execute the ephemeral-copy setup before the real payload.
- **T4**: Update `RunSandboxedValidation` (`internal/verify/sandboxvalidate.go`) to set `Writable: true` on the `RunSpec` it builds for `--auto-fix` validation.
- **T5**: Add unit tests in `internal/sandbox/docker_test.go`/`sandbox_test.go`: (a) argv/stdin assertions proving `Writable:true` produces the `/src:ro` + `/work` tmpfs mount and the wrapped setup step for both `Command` and `Script` modes; (b) a fakeDocker-based test proving a mock validation script can write a file under `/work` when `Writable:true`; (c) a regression assertion proving `Writable:false` (the default) still produces byte-identical output to today's mount/argv — keeping `TestDockerRunArgs_HardeningFlagsPresent` green unmodified.
- **T6**: Update `docs/auto-fix.md`'s "What runs in the sandbox" section (lines 38-53, including the unconditional "mount mode is still read-only" claim at line 47) and its EROFS blockquote (lines 62-70) — plus the duplicate limitation note in `internal/verify/autofix_exec.go`'s `ResolveAutoFixSandbox` doc comment (lines 47-55) — to reflect that non-Go `validate_command`s are now supported via the opt-in writable overlay.

## Acceptance Criteria

- [ ] `RunSpec.Writable` defaults to `false`; every existing caller (`--exec`'s two call sites, the full existing `internal/sandbox` test suite including `TestDockerRunArgs_HardeningFlagsPresent`'s `/work:ro` assertion) is provably unaffected — same mount, same argv, same behavior as before this epic.
- [ ] When `Writable` is true, a validation command or script that writes into its working directory (e.g. `npm run build` → `dist/`, `cargo build` → `target/`, `python` → `__pycache__`) succeeds instead of failing with `EROFS`, proven for both `Command`-mode and `Script`-mode `RunSpec`s.
- [ ] `--auto-fix` validation (`RunSandboxedValidation`) sets `Writable: true`, so non-Go `validate_command`s no longer produce a false-negative validation failure and silently-discarded PR.
- [ ] The container's snapshot (`/src` when `Writable` is true) remains read-only for its entire lifetime; only the ephemeral `/work` tmpfs copy is writable, and it — along with any writes into it — dies with the container. No host file is ever mutated.
- [ ] `docs/auto-fix.md`'s EROFS limitation note and its adjacent "mount mode is still read-only" claim, and `internal/verify/autofix_exec.go`'s duplicate doc-comment note, are all updated to match the new behavior.

### Items needing user confirmation (5)

- ⏸️ **Task 2 targets the wrong function — it cannot inject anything into the actual execution.** `renderCommand` (`internal/sandbox/docker.go:151-156`) only builds a human-readable log string used for evidence/logging (`res.Command`); it has zero effect on what the container actually executes. The real execution content is computed independently: `dockerRunArgs` appends `spec.Command` directly to argv, or `Run` sets `cmd.Stdin = strings.NewReader(spec.Script)` — neither reads `renderCommand`'s return value. A prior Epic 11.0 code review already scrutinized this exact function's fidelity to the real command (`.planning/epics/code-reviews/11.0_executing_reviewers/multi-agent/raw/bruce/review.md:19-29`, "Issue 1: `renderCommand` doesn't reflect the actual command") and concluded it is display-only — independent confirmation this is not a new observation. Suggested action: relocate the setup-injection logic into `dockerRunArgs`/`Run`, where the real argv/stdin is actually constructed.
- ⏸️ **Even relocated, Task 2's mechanism doesn't reach `--auto-fix`'s actual code path.** `RunSandboxedValidation` (`internal/verify/sandboxvalidate.go:43`) — the function that serves `--auto-fix` validation, this epic's whole objective — exclusively constructs `RunSpec{Command: argv}`, never `Script`, and this is a pinned invariant (`internal/verify/sandboxvalidate_test.go:71`: *"Script must never be populated on the argv path (RunSpec exactly-one invariant)"*). In `Command` mode, `dockerRunArgs` execs the image's argv directly (`args = append(args, cfg.Image); args = append(args, spec.Command...)`) with **no shell at all** — there is nothing to interpret a prepended `cp -a /src/. /work/ && cd /work` shell snippet even if it reached the container. As literally described, Task 2 has zero effect on `--auto-fix`'s validation runs — the epic's own AC2 would remain false for the code path it exists to fix. Suggested action: wrap the `Command`-mode exec in a shell invocation (e.g. `/bin/sh -c 'cp -a /src/. /work/ && cd /work && exec "$@"' -- <command...>`), mirroring whatever setup gets added for `Script` mode.
- ⏸️ **`/work` has no writable backing under the container's `--read-only` flag.** `dockerRunArgs` sets a global `"--read-only"` flag on the entire container rootfs; the only currently-writable location is the explicit `"--tmpfs", "/scratch:rw,exec,size=" + cfg.ScratchSize` mount. If Task 1 is implemented literally (rename the `-v` target from `/work:ro` to `/src:ro` and nothing else), `/work` becomes an ordinary path on the read-only rootfs with no volume/tmpfs backing it — `cp -a /src/. /work/` would still fail with `EROFS` (or the container may fail to start if the runtime can't even create the `--workdir` path on a read-only root). Suggested action: Task 1 needs an additional writable mount for `/work` (e.g. a new `--tmpfs /work:rw,exec,size=<budget>`, mirroring the existing `/scratch` pattern) sized for a full source-tree copy rather than `/scratch`'s 64MB build-cache-sized default — likely a new `DockerConfig` field, not a one-line mount-target rename.
- ⏸️ **Blast radius is wider than stated — this changes shared sandbox infrastructure, not just `--auto-fix`.** `dockerRunArgs`, `Run`, and `renderCommand` are shared by `--exec` (Epic 11.0, via `internal/tools/exec_tools.go:178,215`) as well as `--auto-fix` (Epic 32.0). The epic's Objective/Context frame this purely as an `--auto-fix` fix and Components Touched lists only `internal/sandbox/docker.go`/`docker_test.go`, but the change alters execution semantics for `--exec`'s Script-mode invocations too, and touches a documented security guarantee: `docs/execution.md:56-57,86,89` states *"/work (the snapshot) is read-only; the run cannot mutate your working tree"* as one of the sandbox's stated guarantees for `--exec`, and the package doc (`internal/sandbox/sandbox.go:9`) states *"a read-only view of the snapshot (the run cannot mutate the work tree)"* as a hard MUST for every `Backend.Run`. Suggested action: either explicitly scope this as a shared-infrastructure change (add `docs/execution.md` + `internal/tools/exec_tools_test.go` coverage to scope), or make the ephemeral-copy behavior conditional/opt-in so `--exec`'s existing guarantee is preserved for callers that don't need it.
- ⏸️ **An existing, currently-passing test pins the mount contract Task 1 would change.** `TestDockerRunArgs_HardeningFlagsPresent` (`internal/sandbox/sandbox_test.go:55`) asserts `assert.Contains(t, joined, "/tmp/snap:/work:ro", "snapshot must be a read-only mount")`. Task 1 as described would break this test; it isn't mentioned anywhere in Task 3's scope (T3 only describes *adding* new tests, not updating this existing pinned assertion). Suggested action: add updating this test to Task 1 or Task 3's scope explicitly.
