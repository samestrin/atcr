# User Story 5: Regression Proof and Documentation Parity

**Plan:** [32.3: Sandbox Writable Overlay for Polyglot Auto-Fix](../plan.md)

## User Story

**As a** Go engineer maintaining the atcr sandbox internals (`internal/sandbox/docker.go`, `internal/verify/autofix_exec.go`)
**I want** unit tests that pin `Writable:false` as byte-identical to today's mount/argv, a fakeDocker-based test proving `Writable:true` actually lets a validation script write under `/work`, and `docs/auto-fix.md` plus `autofix_exec.go`'s doc comment rewritten to drop the stale "effectively Go-only" claim
**So that** the plan's central safety property â€” zero regression for `--exec`'s read-only guarantee â€” is provable by CI rather than by inspection, the new writable-overlay mechanism is verified end-to-end without a real daemon, and operators reading the docs are told the truth about what `--auto-fix` can now do

## Story Context

- **Background:** Stories 1-4 added `RunSpec.Writable`/`DockerConfig.WorkSize` (Story 1), the conditional `/src:ro` + `/work` tmpfs mount branch and `cp -a` setup-step injection in `dockerRunArgs`/`Run` (Story 2 and 3), and `RunSandboxedValidation` opting in with `Writable: true` (Story 4). This closing story adds no new production behavior â€” it is the test and documentation pass that proves the mechanism is correct and safe, and that brings the project's own docs back into agreement with the shipped behavior. `TestDockerRunArgs_HardeningFlagsPresent` (`internal/sandbox/sandbox_test.go:35`, assertion at `:55`) is the regression anchor asserting the joined argv contains the literal `/tmp/snap:/work:ro`; it must stay green and unmodified as proof `Writable:false` is untouched. `TestDockerRunArgs_ScriptUsesStdinShell` (`sandbox_test.go:83`) pins the Script-mode argv shape and is the pattern anchor for the new `Writable:true` Script-mode assertions. `writeFakeDocker` (`sandbox_test.go:24`) is the package's POSIX shell shim impersonating the `docker` CLI without a real daemon (skips on Windows); Task 5(b) reuses it, keyed on the `Writable:true` argv, to emulate a container that actually receives and executes the `cp` wrap. `fakeDockerRecording` (`internal/verify/autofix_exec_test.go:23`) plus `runArgsLine` (`:43`) is the resolver-level recording variant used by `TestResolveAutoFixSandbox_BuildsAndPreflights` (`:56`); Preflight always runs with `Writable:false` and forms a second control group proving the writable-mount branch never leaks into the preflight path. On the docs side, `docs/auto-fix.md:38-53` ("What runs in the sandbox") states unconditionally that "the mount mode is still read-only," `docs/auto-fix.md:55-60` is a standalone "Limitation (read-only `/work`)" paragraph sitting in the gap between the two originally-cited doc ranges, and `docs/auto-fix.md:62-71` is the EROFS blockquote claiming sandboxed `--auto-fix` is "effectively Go-only today" â€” all three are now false and must be rewritten together. `internal/verify/autofix_exec.go:47-55`'s `ResolveAutoFixSandbox` doc comment verbatim-duplicates the same stale limitation and must be rewritten in the same pass so the two descriptions never disagree again. `docs/execution.md` (lines 51-62, 86-90) and the `internal/sandbox/sandbox.go` package doc's hard MUST ("a read-only view of the snapshot ... the run cannot mutate the work tree") describe `--exec`'s guarantee and must remain textually accurate for `Writable:false` â€” this story's regression tests are what prove that claim still holds.
- **Assumptions:** Stories 1-4 are merged, so `RunSpec.Writable`, `DockerConfig.WorkSize`, the conditional mount branch, the `cp -a`/shell-wrap setup step, and `RunSandboxedValidation`'s `Writable: true` opt-in all exist and are exercised by this story's tests rather than built by it. The `writeFakeDocker` shim and `fakeDockerRecording` helper already exist in the package and need no new scaffolding beyond new script bodies/assertions. The new implicit constraint â€” Command-mode `Writable:true` requires the validation image to ship `/bin/sh` and `cp -a` (true for `alpine`/`golang`-family images, false for `distroless`/`scratch`) â€” is a documentation fact to record, not a runtime check to add.
- **Constraints:** `TestDockerRunArgs_HardeningFlagsPresent` must pass with zero edits to its existing assertions â€” any diff to that test's `/tmp/snap:/work:ro` check would itself indicate the regression this story exists to prevent. `TestResolveAutoFixSandbox_BuildsAndPreflights` and Preflight's control-group argv must likewise show no `Writable:true` leakage. This story touches only test files (`internal/sandbox/docker_test.go`, `internal/sandbox/sandbox_test.go`) and documentation (`docs/auto-fix.md`, the doc comment in `internal/verify/autofix_exec.go`) â€” no production code in `docker.go`, `sandbox.go`, or `autofix_exec.go`'s logic changes.

## Technical Considerations

- **Implementation Notes:** For argv/stdin assertions, extend the table-driven structure already used by `TestDockerRunArgs_HardeningFlagsPresent` and `TestDockerRunArgs_ScriptUsesStdinShell` with new `Writable:true` cases rather than introducing a parallel test structure â€” this keeps the regression anchor's own table untouched while adding sibling cases that exercise the new branch. For the fakeDocker write-proof test, write a shim script body (via `writeFakeDocker`) keyed on the presence of the `--tmpfs /work` flag and the `cp -a`/`exec "$@"` wrap in the recorded argv, then have the shim itself perform a file write to prove the mechanism would work against a real daemon; assert the written file's existence/content from the test. For the regression assertion, either add a dedicated `Writable:false` case to the same table with a full-string argv comparison against the pre-story literal, or add a standalone test that snapshots the `false`-path output and compares it byte-for-byte â€” either approach must not require editing `TestDockerRunArgs_HardeningFlagsPresent`'s existing assertions. For docs, rewrite `docs/auto-fix.md:38-53` to state the mount mode is read-only by default and writable only when `Writable:true` is set by the caller (not operator-configurable â€” it is `RunSandboxedValidation`'s internal choice), replace the `:55-60` limitation paragraph and the `:62-71` EROFS blockquote with a description of the ephemeral `/src:ro` + `/work` tmpfs copy and the fact that non-Go `validate_command`s are now supported, and add the `/bin/sh` + `cp -a` image-requirement note to the sandbox-requirements section per plan.md's Refinement Decisions (2026-07-21). Rewrite `internal/verify/autofix_exec.go:47-55`'s comment to match the new docs wording exactly (or close enough that the two cannot drift apart in meaning again).
