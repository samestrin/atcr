# User Story 3: Ephemeral-Copy Setup Injection

**Plan:** [32.3: Sandbox Writable Overlay for Polyglot Auto-Fix](../plan.md)

## User Story

**As a** Go engineer maintaining the atcr sandbox internals (`internal/sandbox/docker.go`)
**I want** `dockerRunArgs`/`Run` to prepend a `cp -a /src/. /work/ && cd /work` setup step to the real payload â€” for both `RunSpec.Command` and `RunSpec.Script` modes â€” whenever `spec.Writable` is true, executed via an explicit shell wrap for Command mode and a prepended script line for Script mode
**So that** a validation command or script that needs to write into its working directory (Node/Rust/Python toolchains) actually finds a populated, writable `/work` tmpfs copy to write into, instead of failing with `EROFS` against the read-only snapshot mount â€” without ever string-interpolating the real argv or script body into shell text

## Story Context

- **Background:** This plan's earlier stories add the `RunSpec.Writable` / `DockerConfig.WorkSize` fields (Story 1) and the conditional mount branching that gives a `Writable:true` run a writable `/work` tmpfs alongside the read-only snapshot mount (Story 2, which remounts the snapshot at `/src:ro` and adds `--tmpfs /work:rw,exec,size=<WorkSize>` when `spec.Writable` is true). Story 2's branch makes the writable overlay *exist* (its AC 02-03 pins the combined mount-plus-copy behavior at integration level), but an empty tmpfs is useless without the source tree copied into it first. This story owns the payload-wrapping specification in per-mode detail: it makes `dockerRunArgs`/`Run` actually populate `/work` from `/src` before the user's real command or script executes, so the "opt-in ephemeral copy" mechanism described in the epic's Proposed Solution item 4 is realized. Two independent `/refine-epic --deep` passes already corrected the design point that matters most here: `renderCommand` (`internal/sandbox/docker.go:153`) only builds a human-readable evidence/log string â€” `res.Command` is never read by execution â€” so setup injection must happen where `dockerRunArgs`/`Run` build the real argv/stdin the container receives, never in `renderCommand`. Corroborated independently by a prior Epic 11.0 code review (`.planning/epics/code-reviews/11.0_executing_reviewers/multi-agent/raw/bruce/review.md:19`, "Issue 1: renderCommand doesn't reflect the actual command").
- **Assumptions:** Command mode today has no shell at all â€” `dockerRunArgs` (`docker.go:145-147`) appends `cfg.Image` then `spec.Command...` directly as argv, so there is nothing to interpret a prepended `cp` shell snippet unless it is explicitly wrapped in a shell invocation. `RunSandboxedValidation` (`internal/verify/sandboxvalidate.go:43`, `spec := sandbox.RunSpec{Command: argv, ...}`) exclusively constructs Command-mode `RunSpec`s â€” Script mode is never used on the `--auto-fix` path, a pinned invariant in `internal/verify/sandboxvalidate_test.go:71` ("Script must never be populated on the argv path"). This means the Command-mode shell wrap is the code path that actually matters for unlocking `--auto-fix`, even though Script mode must also be handled for completeness and to keep `--exec`'s script path correct under `Writable:true`. `TestDockerRunArgs_ScriptUsesStdinShell` (`internal/sandbox/sandbox_test.go:83`) already pins Script mode's stdin-delivery shape (`-i <image> /bin/sh -s`, script body absent from argv) â€” the copy-step prepend must preserve that invariant, travelling over stdin, never appearing in argv.
- **Constraints:** The existing no-interpolation invariant (`RunSpec.Command` doc comment, `sandbox.go:29-30`: "No shell interpolation is performed") must hold for the wrapped payload too â€” the real argv must be passed to the shell wrap via positional expansion (`-- "$@"`), never built into the `-c` script string via string formatting/concatenation of user-controlled values. This is a hard security constraint, not a style preference: `spec.Command` can contain arbitrary operator- or model-influenced strings, and interpolating them into a `sh -c` string reopens the exact shell-injection class the current argv-only Command mode was designed to close. Setup injection is conditional on `spec.Writable` only â€” when `Writable` is false, both Command mode's argv-only shape and Script mode's stdin delivery must remain byte-for-byte unchanged (this is `--exec`'s default path and must not regress). Injection must live inside `dockerRunArgs` (argv construction) and/or `Run` (stdin construction) â€” never in `renderCommand`, which stays a display-only function per the epic's corrected design.

## Success Criteria (SMART Format)

- **Specific:** When `spec.Writable` is true, `dockerRunArgs` wraps Command-mode argv as `/bin/sh -c 'cp -a /src/. /work/ && cd /work && exec "$@"' -- <original command...>` (image, then the wrapped invocation), and `Run` prepends `cp -a /src/. /work/ && cd /work\n` to the Script-mode script body before it is written to `cmd.Stdin`; when `spec.Writable` is false, both modes are byte-for-byte identical to today's output.
- **Measurable:** New unit tests in `internal/sandbox/docker_test.go` assert: (1) Command mode with `Writable:true` produces argv containing `/bin/sh`, `-c`, the copy-step string, and `--` followed by the original command tokens, with the original command never appearing string-interpolated inside the `-c` script text; (2) Script mode with `Writable:true` has the copy step present in `cmd.Stdin`'s content (not argv) preceding the original script body; (3) `Writable:false` (or zero-value) argv/stdin for both modes remain unchanged from the pre-existing behavior, verified against the existing `TestDockerRunArgs_ScriptUsesStdinShell` and Command-mode argv assertions with zero required edits to those pre-existing tests. `go test ./internal/sandbox/...` passes with all existing and new tests green.
- **Achievable:** The wrap is a self-contained addition to `dockerRunArgs`'s Command-mode branch (`docker.go:145-147`) and `Run`'s Script-mode stdin construction (`docker.go:205-210`), gated by a single `spec.Writable` check in each; no changes to `RunSpec.validate()`, mount-arg construction, or callers are required by this story.
- **Relevant:** This is the story that makes the writable mount (Story 2) actually useful â€” without the copy step, `/work` is an empty tmpfs and every polyglot toolchain command still fails, just with a different error (missing files) instead of `EROFS`. This directly satisfies the epic's first acceptance criterion: a validation command/script that writes into its working directory succeeds under `Writable:true`, for both dispatch modes.
- **Time-bound:** Completed as the third story of this plan, after Stories 1-2 land the data model and mount branching, and before Story 4 (`--auto-fix` Opts Into the Writable Overlay) sets `Writable: true` on the `--auto-fix` path.

## Original Criteria Overview

1. Command mode: when `spec.Writable` is true, `dockerRunArgs` produces argv of the shape `... <image> /bin/sh -c 'cp -a /src/. /work/ && cd /work && exec "$@"' -- <spec.Command...>`, with `spec.Command`'s tokens passed as separate argv elements after `--` (positional `$@` expansion), never concatenated into the `-c` script string; when `Writable` is false, argv is unchanged from current behavior (`<image> <spec.Command...>`, no shell).
2. Script mode: when `spec.Writable` is true, `Run` (or `dockerRunArgs`'s caller) prepends the literal setup line `cp -a /src/. /work/ && cd /work` to `spec.Script`'s content before it is written to `cmd.Stdin`, with the setup step never appearing in argv (argv stays `-i <image> /bin/sh -s` exactly as today, per `TestDockerRunArgs_ScriptUsesStdinShell`); when `Writable` is false, stdin content is unchanged (`spec.Script` verbatim).
3. Both modes' wrapped payload runs `cd /work` after the copy so the process's working directory is the writable copy, not the read-only `/src` snapshot, and the copy step (`cp -a`) is asserted present in unit tests as a literal substring of the constructed argv (Command mode) or stdin reader content (Script mode) â€” not merely inferred from a passing end-to-end container run, since `internal/sandbox` tests are argv/stdin-level and daemon-free per the package's existing test pattern (`dockerRunArgs` is documented pure/I-O-free specifically so hardening flags can be asserted without a daemon).


## Technical Considerations

- **Implementation Notes:** In `dockerRunArgs` (`docker.go`), replace the current unconditional `else { args = append(args, cfg.Image); args = append(args, spec.Command...) }` Command-mode branch with a conditional: if `spec.Writable`, append `cfg.Image, "/bin/sh", "-c", "cp -a /src/. /work/ && cd /work && exec \"$@\"", "--"` followed by `spec.Command...`; otherwise keep today's unwrapped form. In `Run` (or a small helper `dockerRunArgs`/`Run` shares), when `spec.Script != "" && spec.Writable`, build the stdin reader from `"cp -a /src/. /work/ && cd /work\n" + spec.Script` instead of `spec.Script` alone; when `Writable` is false, `cmd.Stdin` stays `strings.NewReader(spec.Script)` exactly as today. The mount paths `/src` and `/work` are assumed to be exactly what Story 2 wires up (`/src:ro` snapshot mount, `/work:rw` tmpfs) â€” this story does not itself change mount arguments, only the payload that runs against them.
- **Integration Points:** `internal/sandbox/docker.go` (`dockerRunArgs` Command-mode branch, `Run`'s Script-mode `cmd.Stdin` construction). No changes to `internal/sandbox/sandbox.go` (`RunSpec`, `validate()`), `internal/verify/sandboxvalidate.go`, or any `--exec`/`--auto-fix` call site â€” those consume `RunSpec.Writable` as an opaque bool and are unaffected by how the wrap is implemented internally.
- **Data Requirements:** None. This story is pure control-flow/string-construction inside the existing `dockerRunArgs`/`Run` functions; no new struct fields, config, or persisted data.

## Potential Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Interpolating `spec.Command` into the `sh -c` script string (instead of passing it via `-- "$@"` positional expansion) reopens shell injection â€” a command token containing shell metacharacters (`;`, `$()`, backticks) would be interpreted rather than passed literally, defeating the no-interpolation invariant `RunSpec.Command`'s doc comment guarantees | High | Use `/bin/sh -c '<fixed script text>' -- <command tokens as separate argv elements>` exclusively; the fixed script text is a Go string literal containing no user data, and `spec.Command`'s elements are appended as distinct `[]string` entries after `--`, exactly mirroring how `exec.Command` itself avoids shell interpolation. A unit test asserts a command token containing `;` or `$(...)` survives as a literal argv element and is never treated as script text. |
| The operator's validation image lacks `/bin/sh` or a `cp` supporting `-a` (distroless/scratch images) â€” the wrap silently assumes both exist, so a shell-less image fails at container start/exec with an opaque backend fault rather than a clear error | Medium | Documented as a known implicit constraint carried over from the epic (integration_gap): the wrap only works for shell-and-coreutils-bearing images (alpine/golang-family). Out of scope to fix in this story (no image-capability detection exists), but the doc comment on the `Writable` field (Story 1) and/or this story's implementation notes should flag it so a future distroless-image failure is diagnosable rather than mysterious. |
| Prepending the copy step to Script mode changes the script's effective working directory (`cd /work` runs before the user's script body), which could surprise a script that assumed it started in `/work` already under the old read-only mount semantics | Low | `cd /work` after the copy restores the same working directory the container already starts in (`--workdir /work` is set in `dockerRunArgs` regardless of mode), so the net effect for well-behaved scripts is unchanged cwd, just now writable; documented in the setup-step's inline comment so the intent (populate, then land back in the expected cwd) is clear to future maintainers. |
| Forgetting to gate the Script-mode prepend behind `spec.Writable`, causing every `--exec` script run (which never sets `Writable`) to pick up an unwanted `cp`/`cd` prefix and silently change behavior for the read-only default path | Medium | Explicit `if spec.Writable` branch (never a default-on prepend); regression coverage via the existing `TestDockerRunArgs_ScriptUsesStdinShell` test, which must continue passing unmodified for the `Writable:false` (zero-value) case, proving no behavior change for current callers. |
