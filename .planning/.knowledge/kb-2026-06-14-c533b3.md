# How should verdict parsing handle real LLM output variations

## Decision

LLM outputs frequently vary in casing and whitespace (e.g., {"verdict":"Confirmed"} or {"verdict":"refuted "}). The parseVerdict function must normalize input before switching on the enum: verdict := strings.ToLower(strings.TrimSpace(parsed.Verdict)). This prevents real verdicts from falling to the default branch and being coerced to unverifiable, which would drop confirmed findings from --require-verified counts (a false pass). Context: Epic 3.0 adversarial verification. The sibling function hasTrustedVerdict already normalizes with ToLower+TrimSpace, proving the pattern. Always normalize LLM envelope fields before enum matching.
