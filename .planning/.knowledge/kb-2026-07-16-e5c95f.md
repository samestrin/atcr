# Where must a deferred --sync-cloud push be sequenced in runR

## Decision

runReview registers the deferred --sync-cloud push via defer inside `if result != nil`, so it fires on every later return path — including exit-2 usageError returns from in-process reconcile/verify/debate infra failures. reconcile.go is NOT symmetric: its I/O-failure path returns before the push block. Two related bugs surfaced a phase apart: (1) a post-run 401/403 auth rejection unconditionally overrode the run's real exit code, masking exit-1 and exit-2 failures alike — fixed with resolveSyncCloudOutcome(runErr, syncErr), which lets an auth rejection override only success or a plain uncoded exit-1 failure, never an already-coded exit-2 error; (2) a later TD pass found the guard still let exit-2 usageError paths trigger the push itself, reporting outcome="failure" for a run whose tooling broke, not the review — fixed by only executing the deferred push when the run reached a finalized review outcome, mirroring reconcile.go's return-before-push ordering. General pattern: any deferred side-effect (cloud push, telemetry, audit record) registered via `if result != nil { defer ... }` must be guarded against firing on infra/tooling failure paths, not just success/findings-gate paths.
