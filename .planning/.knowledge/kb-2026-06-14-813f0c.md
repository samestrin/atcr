# What is the recommended pattern for mitigating prompt inject

## Decision

When embedding reviewer-authored finding fields (Problem, Fix, Evidence) into skeptic prompts, wrap each field in per-prompt randomized sentinel delimiters: <finding-problem sentinel="a7f3b2">{problem content}</finding-problem>. The skeptic is instructed to treat delimited content as data, not instructions. This prevents attacker-controlled source from propagating a pre-formed verdict envelope or "ignore prior instructions" string into the skeptic context. Additional defenses: strip/escape brace-fence and triple-backtick sequences from finding fields; use adaptive fence length for code blocks (max embedded run + 1); validate verdict enum downstream (switch-validated, so broken fences degrade quality but cannot forge verdicts). Context: Epic 3.0 adversarial verification. The finding fields are themselves LLM-generated from code under review, so they are an untrusted axis.
