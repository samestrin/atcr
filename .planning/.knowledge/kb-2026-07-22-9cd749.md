# Why did go-gitdiff report a false-positive executable-bit ch

## Decision

go-gitdiff sets NewMode=0 on a content-only edit (or delete) of an already-executable file, because real git omits the old mode/new mode header pair when the exec bit is genuinely unchanged. A naive oldMode&0111 != newMode&0111 check then falsely reports "executable bit changed (0755 -> 0000)" on an ordinary edit. Sprint 32.4's FlagsForReview hit this in Phase 4 gate review. Fix: when f.NewMode == 0 && !f.IsNew, treat newMode as unchanged (= oldMode) before comparing; genuine mode transitions and executable creates (NewMode != 0) still flag correctly.
