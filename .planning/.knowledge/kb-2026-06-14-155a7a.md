# How should gate thresholds be resolved consistently across C

## Decision

Both atcr review and atcr reconcile must resolve gate thresholds using the same resolver: flag > project-config fail_on > registry fail_on. The resolver returns "" when nothing is configured, so the threshold!="" guard still holds. Problem: The one-shot review gate resolved its threshold via failOnThreshold(cmd) (flag-only), whereas standalone atcr reconcile used resolveGateThreshold(cmd) (flag > config > registry). A repo with fail_on:HIGH configured was gated by atcr reconcile but NOT by atcr review (no --fail-on flag), so review silently exited 0 on findings the configured gate should fail. Fix: have runReview call resolveGateThreshold(cmd) instead of failOnThreshold(cmd). Why: Two entry points disagreeing on "the gate" breaks the CI contract. --require-verified's precondition keys off the same source, so inconsistency propagates. Additional normalization: IsFailing normalizes severity with strings.ToUpper but must also normalize threshold; category canonicalization (lower+trim) must be consistent across gate, summary count, and report rendering. Context: Epic 3.0 adversarial verification. The gate is the CI trust boundary; inconsistency here makes the gate untrustworthy.
