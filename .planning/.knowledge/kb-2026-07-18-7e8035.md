# Why does atcr's --axi mode keep the existing 0/1/2/3 exit-co

## Decision

Epic 31.0's original proposal was a 3-code scheme (0=clean, 1=findings, 2=internal/syntax error). During sprint execution this was evaluated and explicitly rejected: atcr already has a precedent-setting 0/1/2/3 exit-code contract (established by atcr verify, documented in plan/documentation/exit-code-cli-mcp-precedent.md), and --axi reuses it unchanged rather than introducing a fifth, AXI-specific meaning for 2. New AXI-introduced error sources were classified into the existing buckets: an unsupported --axi+--auto-fix (or --debate/--single-model/--exec) flag combination is a usageError (exit 2), and an internal AXI rendering fault is left unwrapped so it defaults to exitFailure (exit 1) - mirroring how non-AXI internal errors are classified. exitCode() in cmd/atcr/main.go has no --axi-specific branch at all; the dispatch is generic errors.As regardless of output format. The rejected epic proposal and the reconciliation rationale are recorded in cmd/atcr/main.go's exit-code comment block and docs/ci-integration.md, cross-referenced from AC 02-03.
